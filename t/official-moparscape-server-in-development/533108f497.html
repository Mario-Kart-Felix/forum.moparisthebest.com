Miss Silabsoft:

Can someone care to explain why the settings should be bulked into the player? the column count would easily go over 200

200 sounds like an exaggeration. There’d be maybe 15? Public/private chat, trade/duel, and a bunch of buttons in the settings section are what come to mind. What are you thinking of?

Anyway, it should be noted that most relational databases support column counts in the thousands (see http://en.wikipedia.org/wiki/Comparison_of_relational_database_management_systems#Limits).
Also, if you’re specifically naming each setting in the relationship’s tuples (i.e., public_chat_state, private_chat_state, trade_state, etc) then the relation is still going to be in 3NF.

@David:
Ironically, this approach also leads to a space saving over the current one. You aren’t maintaining an extra primary key index, nor would you be maintaining a unique index over the [player_id, setting] combination, which you would need to do if you wanted to enforce the constraint that any setting appeared 0 or 1 times. You could just have a primary key over player_id/setting but your example suggests that isn’t the case (i.e., you appear to be using an auto incrementing id as the primary key).
Further, you don’t need logic to check for missing settings when performing queries in the application.

So anyway, that’s the rationale for not separating those settings into a second table.
