[quote=“Ollie123, post:22, topic:446163”]Happy to see that this is back in development, good luck :)[/quote]Thanks

I finally got my new computer set up and have resumed work on it. So far, I’ve copied over some more stuff from Rarity and fixed a bug that only occurs on multi-core processors, which causes the client to crash about 75% of the time a player logs in. I’m working on implementing a new way to decrease the player updating time to take less than 600ms always, and 2000 players will definitely be supported. It’s a simple method of only adding/removing players when other players move around. Right now, for every online player, 4096 possible tiles will be iterated and players will be checked. With this new method, only 64 or 128 (depending on if the player is running) tiles will be iterated and there will be no process of checking if the player already exists in the update list (and mind you this only occurs when there’s movement).

EDIT: I actually forgot about the actual process where it updates/removes existing players. Therefore, only 64 tiles max will have to be iterated and I might make an adjustment on that updating procedure to make it not iterate through the current players. It would be simple - instead of going through each player currently in the update list and checking if they logged out or moved out of scope, I’d make a list for each player with players to remove and update and only then will the server cycle through the current players. I’m not sure. I’ll have to see how adding players works out.
