A Short Introduction
gpg is software which handles public-key cryptography. This works by having keys, each person has a private key which they keep secret, and a public key which they spread around.
Suppose Alice wants to send an encrypted message to Bob, she encrypts her message with Bob’s public key. Only Bob can decrypt it because only Bob knows his private key.

There is also a feature called digital signatures. This is a way of mathematically proving that a message came from that sender and that it hasnt been modified in transit.
Suppose Alice wants to send a message to Bob and be sure that no-one else has modified it along the way. She signs her message with her private key and Bob can verify it with her public key. Only Alice can sign because only she has her private key, but anyone can verify because anyone has her public key.

For this to work, you need to have the public key of everyone you’re communicating with, hence this thread.

Almost all linux distros come with gpg already installed, if you’re lucky, you may have a graphical front end such as kgpg or seahorse.

If not, you can use these steps on the terminal to generate you own key pair.

$ gpg --gen-key
$ gpg -a --export your-name-here

digitally sign file “cleartext”

gpg --clearsign cleartext
OR
gpg --clearsign < cleartext

verify digital signature file “cleartext.asc”

gpg --verify cleartext.asc
OR
gpg --verify < cleartext.asc

encrypt file “cleartext” with the public key of “yakman”

gpg -a -r yakman -e cleartext

decrypt file “cleartext.asc”

gpg -d cleartext.asc

Whenever you send an encrypted message to someone, it should be signed too. If you dont, it opens up a variaty of cryptoanalysis attacks.
encrypt file “cleartext” with the public key “yakman”, sign it and store in file “ciphertext”

gpg -a -r yakman -e < cleartext | gpg --clearsign > ciphertext

if you receive a message like this, a combined verification of the signature and decryption

gpg -d ciphertext | gpg
