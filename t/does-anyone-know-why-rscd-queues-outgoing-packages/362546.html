Thats the thing, the packets aren’t processed or anything. They are simply queued and then sent off.

/**
	 * Sends queued packets to each player
	 */
	public void sendQueuedPackets() {
		try {
			for (Player p : players) {
				List<RSCPacket> packets = p.getActionSender().getPackets();
				for (RSCPacket packet : packets) {
					p.getSession().write(packet);
				}
				p.getActionSender().clearPackets();
				if (p.destroyed()) {
					p.getSession().close();
					p.remove();
				}
			}
		} catch (Exception e) {
			e.printStackTrace();
		}
	}

CodeForFame August 1, 2016, 12:32pm #9

I can’t think of a valid reason for it… Reading and writing are handled asynchronously by Mina, so calling the write method wouldn’t block…

Literally nothing is done between the packets being queued and sent, besides the processEvents method being called. All that they would be doing is possibly adding packets to the queue, but could just be sent straight off as well. Unless there was an event that removes redundant packets from the queue, or canceling a packet from being sent. The ability to do that is kind of nice, but I don’t see a reason to do so, unless there is just too much network traffic being sent. As it stands, the only event that is being processed saves players, which makes the queue completely useless.

Removing the queue wouldn’t change the synchronization of the clients at all… Pretty much the only difference would be that not all packets are processed before it starts asynchronously sending packets, and clients can update in smaller portions. I’m fairly certain Mina uses Nagle’s algorithm (TCP_NODELAY) by default, so packets would still be grouped, for each client, to an extent. If anything, it would make the game look smoother…

Anyway, the most likely reason for it would have to be that he wasn’t using Mina initially, and it’s just left over from the previous networking code.
