sinisoul August 4, 2016, 2:17pm #2

Think it through. Obviously you want to control the rate at which each item is dropped to manage the available supply. If you take the average time to kill a creature, then factor that in with the drop rate you can come up with a figure of new stock/time. The factor of this is obviously a percent which represents rarity. There’s more you should take into account. It’s a good system but for terms of readability a number between 1-100 would suite best. I liked how jagex did it in their scripts with a subroutine for each case instead of a table so that they could integrate multiple drops.

Cres August 4, 2016, 2:19pm #14

Count the amount of drops listed for a specific NPC.
While parsing, remember what the highest “estimated_kills” value is and store that value along with the drops.
Sort the droplist from rarest to most common.

Upon a kill, roll a random number ranging from 0 to 1. (only this part may need to be adjusted, because the value will actually range from 1/highest_estimated_kills to 1 in steps of 1/highest_estimated_kills.)
Loop through the list, comparing your random number to each “rarity” factor of each item.
If your rarity matches (is higher than the item’s rarity), get a list with all other items with the same rarity.
Return a random item from that list.

^ +/- how I’d do it
