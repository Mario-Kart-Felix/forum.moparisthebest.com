Brief Setup Instructions:

First you need to add the cache for your client to the appropriate folder. For 317, add the cache to apollo/data/fs/317. For 377, add the cache to apollo/data/fs/377.

You can build Apollo with Apache Ant and Apache Ivy. First download and setup Ant and Ivy, info on that can be found here: http://ant.apache.org/ and here: http://ant.apache.org/ivy/.

In command prompt, in the apollo directory, run these commands:

ant resolve

‚Äúant resolve‚Äù only needs to be called once when you first download Apollo and again when you update the versions of the dependencies. It downloads third-party libraries such as JRuby and Netty.

ant build

‚Äúant build‚Äù compiles all the source files.

ant run

‚Äúant run‚Äù starts the server running, by default it tries to load the 377 cache and packet codecs. You can change this by editing the version attribute in the build.xml file, or by running it manually without ant.

Other ant targets you may find useful:

doc - generates javadocs, if you have graphviz installed it also includes nice graphs of the class hierarchy
test - runs (the very limited set of) unit tests

For those who are lazy to lookup the dev thread:

[quote=" Graham"][size=18pt]Introduction[/size]

Apollo is a RuneScape emulator which aims to encourage a fundamentally-different alternative to the way in which private server development is done today. It consists of a high-performance, modular server written in Java as well as a collection of utilities for managing the data files and plugins.

[size=18pt]Plugins[/size]

Currently people download a server, read through tutorials and apply their modifications or write their own code on top of it. They‚Äôll then host it or release it. The result is we currently have a complete mess of servers (this includes every current base - Hyperion, rs2hd, winterlove) all created from cobbled-together code.

Apollo is going to change that through its plugin system. Instead of throwing everything into one big application, Apollo consists of:

[ul][li]A small ‚Äòcore‚Äô application which provides features necessary for the server to operate.[/li]
[li]A set of plugins, and the tools to manage them (install, uninstall, publish, download, etc).[/li]
[li]Some additional tools and utilities for managing the data files.[/li][/ul]

This will make it much easier for everyone to develop a private server, no longer restricting it to people who can (or can‚Äôt as the case is here) program. Instead of messing around copying and pasting lines of code from a woodcutting tutorial, people will simply have to download and install a woodcutting plugin.

There is also a plan to have a plugin repository (or maybe multiple repositories) and a set of tools to make the experience very much like a package manager on Linux.

It also means updates can be provided for the core server (e.g. security, stability, optimizations) very easily. Users will just have to download the new jar, overwrite the current one with the new one, and reboot their server.

And best of all, inexperienced users are protected from making fatal mistakes in the core and are kept away from stuff that they shouldn‚Äôt be editing at their experience level.

Plugins are currently written in Ruby, however, in the future other languages could be added. Here is what a plugin looks like:
hello.rb

[size=18pt]Core Features[/size]
Some of the significant (technically) current core features include:

[ul][li]Packet encoding/decoding has been split from the representations of the packets themselves. This allows the potential for encoding/decoding to go on in parallel and also allows multiple revisions to be supported. Currently 317 and 377 are both completely supported.[/li]
[li]Update server support (JAGGRAB, ondemand and HTTP)[/li]
[li]Packet handler chaining: this allows multiple plugins to be able to intercept a single packet and deal with it appropriately. For example, a quest plugin could intercept searching a bookshelf for instance, if the behaviour needed to change in certain cases.[/li]
[li]Parallel execution of player updating for multi-core machines - this has a significant benefit on my dual core machine used to test the server.[/li][/ul]

[size=18pt]Source control/building[/size]

Git is used for source control and Maven for building. It is likely that I will change to Ant in the future, perhaps with Ivy.

Currently the repository is private (it isn‚Äôt ready yet, and will not be released until it is ready!).

[size=18pt]Server architecture[/size]

[size=14pt]Third-party libraries[/size]

The server uses the Netty API for the networking code. Netty supports a variety of transports such as NIO and blocking IO, so you can choose what is right for your environment/use case.

The Commons Compress library is used for decompressing bzip2-compressed files in the cache.

The JUnit library is used for unit testing.

The JRuby library is used to execute the Ruby plugins.

APIviz is used to draw pretty diagrams in the Javadocs.[/quote]
