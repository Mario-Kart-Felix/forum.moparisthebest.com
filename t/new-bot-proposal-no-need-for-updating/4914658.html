Basically this is how I planned (I also had a basic implementation going, but because serp sucks it didn't go too well) on doing this,

Load the loader.jar from rs.com with a modified url class loader so you can jerry rig/scan the classes for the methods/fields you need as they're loaded into a frame of an applet so you can post awt events to it. Then, with the vars we know we need, dynamically hack up a stub sort of accessor methods class to get the data through reflection.

It's pretty trivial to load up the applet in a frame and make it think it's coming from rs.com without needing to touch any code inside the client itself, but because matching the vars you want from straight bytecode is a lot harder and more annoying to track down than by source I thought that wouldn't be the best way to go, although it is quite possible and with a framework in place you can delegate sifting through the bytecode to the plebs that hang around.
