I would suggest reading up on Event Programming Fundamentals. I haven’t seen anyone in this thread distinguish the difference between a task and an event. Your event should ‘trigger’ (tell your) task to initiate. You create an event handler to ‘listen’ for new ‘incoming’ events that were created from some action (in your case item on item?) from there, you have your handler trigger the needed tasks based off either event types (ugly design tactics), or handling the task & event handling in abstraction.

[quote=“zyle1992, post:16, topic:434829”][quote author=Taharok link=topic=537546.msg3930899#msg3930899 date=1330656448]
Either way, you’re going to have to program it. My idea is: why not program less? It’s an easier approach, regardless of where you put the code. By using a domain-specific language (whether customly interpreted or defined in a dynamic language like lisp), you can specialize constructs just for handling the specific cases for each and every even permutation. It would same a lot of programming time, not to mention a large event system has a lot of resource overhead.
[/quote]

So you’re suggesting just programming it in the traditional way?[/quote]

Not at all. I’m suggesting the complete opposite: use a specific language just for this problem. A language that utilizes as little text to do as much as possible, such as:

"(*) bar" on "anvil":
	first.$1 = "bronze":
		startBronzeDialogue()
	first.$1 = "rune":
		startRuneDialogue()

Etc.

Edit: As far as processing is concerned, it doesn’t matter whether it’s event-driven or not. The idea is to minimalize the complexity behind the system, which simplifies development time, collaboration time, and leads to overall less bugs and problems. I also believe Jagex may take an approach similar to this, as they often discuss their serverside “rune script” language.
