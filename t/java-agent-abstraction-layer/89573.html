newbiehacker July 30, 2016, 8:38am #1

Now, this is only a simple idea, and it stems from the inspiration regecks’ (sp?) post gave me.

Basically, I think we should build a small java agent abstraction layer which loads transformers specified by a configuration file (or maybe at run time [a static constructor?]). Basically the idea is that we use an installer similar to regecks’ (sp again =/) (obviously with a bit more feedback to the user) to shove out java agent abstraction framework into the properties files. Once install, either using the run-time method or editing a configuration file we could simply plug in new agents.

Usages for this could be simply reverse engineering the client by following all method calls, logging packets and comparing against illegally modified clients, making the client draw offscreen, etc.

Post your comments, opinions, criticisms etc. below.

pplsuqbawlz July 30, 2016, 8:38am #2

It’s an abstraction layer, and depending on how the client is being executed (and loaded into runtime) severely limits any and all transformers.

However, minimalistic information displays are possible such as versions, and other classes that are relatively easy to detect and swap, such as signlink and it’s uid, or FileWrapper (which implements randomaccessfile for… access?) to redirect where the game loads from specifically.

Not that you’re the first person to have such a thought, but hacking up the Graphics-Configuration to render off screen is relatively simplistic, even, unnecessary. The client renders to Canvas, trick canvas (by injecting invalid getGraphics() method to return our own buffered image for example) into sending our own object and we’re set. Of course the method in Applet where they check the clips can give us issues…
