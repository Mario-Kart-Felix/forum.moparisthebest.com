xeez July 31, 2016, 9:57am #4

When running the JVM through JDK’s “java” application, it assumes that any “dot” (.) refers to specific package names. With this in mind, it interprets the directory: “/Users/tempjoes/Desktop/St/RMScape/Client/EGUI.class” as “/Users/tempjoes/Desktop/St/RMScape/Client/EGUI/class”. As you can see, it interpreted “EGUI” as a directory rather than the actual file name. Instead of passing the class name (EGUI) along with the file format (.class), you simply pass the class’ name. So to run it, you’d do:

java /Users/tempjoes/Desktop/St/RMScape/Client/EGUI

Just a fore-warning: Most current runescape private server’s clients are limited to work only within windows due to the faulty programming.

moparisthebest July 31, 2016, 10:08am #20

I have programmed many programs in java (and other programming languages, but we will ignore those for now) that work in multiple operating systems.

The separator ‘/’ works on all systems, I also believe that ‘\’ also works on all systems.

The things that break cross platform compatibility are stupid things that some people do, like ‘~/somefile’ will only work on unix systems, and ‘C:\somefile’ will only work on windows systems (and further only on windows systems with a C: drive, which not all of them have one)

Another thing that breaks cross platform compatibility is case-sensitivity. Windows, regardless of the file system it runs on, ignores case. To windows, ‘bob’ is the same as ‘BoB’ or ‘Bob’, whereas linux running on a case-sensitive filesystem (most of them) sees those same strings as 3 different files. You can see the havoc that inevitably results. (the same thing applies on other programming languages, I remember in my opengl class the teacher’s example programs never ran until I lower-cased all of the file strings, it was a pain in my ass, don’t do it.)

In conclusion, when programming java use ‘/’ as a file separator and use all lowercase files and strings (or at least be consistent in code and on the file).
