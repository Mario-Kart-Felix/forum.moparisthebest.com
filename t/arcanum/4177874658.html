Great. I’d like to help you if you want. and for the fullscreen interfaces based on the client, is there something that’s server sided to make the interface loads fullscreen or will it load fullscreen it self?

The 317 client uses one packet for the Welcome Screen - Data for it + Opening it
The 377 client uses three packets: One packet to send Welcome Screen Data, one to open a Fullscreen interface, and another to open a normal interface (which is shown below the welcome screen - the MOTW graphic, the text is set seperate.)

So that’s how I made it do it - I made a different packet to open the welcome screen. e.g:

  # Open the welcome screen
  player.open_welcome_screen

  # Send the message of the week
  player.send_message_of_the_week MOTW_INTERFACES[:VIRUS] , "Welcome to the game!" , "@red@DON'T FUCK UP"

where openWelcomeScreen sends filler data and opens the fullscreen interface

hidendra August 1, 2016, 3:42pm #38

[quote=“Wintercanon, post:37, topic:417787”]Nice work with RuneSource Hidendra, how are you finding it’s single threaded performance with the features you’ve implemented? Also I’m guessing the Ruby scripts are interpreted upon load up. I imagine that’s quite lengthy?

Also did you consider converting the protocol to 377 instead of hacking up a 317 client? As I’m sure you know the protocol is very similar.[/quote]
Thanks!

I’m finding performance just fine. JRuby has a slow startup (3-5 seconds for me), but after that the whole Ruby state can be reloaded almost instantaneously (that is, everything scripted on the ruby side can be reloaded instantly.) Speed of going from Java -> Ruby (and even sometimes from Ruby -> Java again) is not very bad at all and has been working wonderfully and works just as if it were pure Java. Any very important logic or framework-y stuff is generally going to stay on the Java side, while Ruby is providing the implementations (although some packets are implemented Ruby-side, and others are on Java, I hope to move them all to the Ruby side at some point.)

As of right now with what I’m implemented there isn’t really anything that uses a lot of CPU - it’s all quite simple. In the future there’ll be a need for work to be done on seperate threads/cpus but of course that isn’t right now (for example, logging queries (which aren’t urgent) will be offloaded to a seperate thread and batched - the server will at some point be logging a lot of data, most likely)

I did run into some minor issues with many clients running concurrently but I may need to write my own (simple, of course) bot that can connect to the server with a large amount of clients (Blitzbot is slow). If I use some other bot utils, I need to disable RSA, welcome screens, etc… Pretty annoying!

I have indeed considered using a different protocol (e.g 377) altogether. I stuck with 317 because so much is already known about it and at the moment it suits my needs (an oldish server like cheezscape + rswiki and you’re good!)
