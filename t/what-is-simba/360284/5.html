Simba is a free, open-source replacement for the venerable SCAR, it has reached a point where it is actually better than SCAR ever was and runs on multiple platforms (currently Linux and Windows, with Mac OSX soon to follow).

It is written and actively developed and maintained by a group of programmers over at SRL headed by Wizzup?, the following is what he wrote in the official release thread about Simba:

[ul][] Simba is free to the bone; you don’t have to pay anything for it and you can discover exactly how it works. It is open source.
[] Cross platform. Simba runs on Linux and Windows, and Mac support is hopefully following.
[] Actively developed; it is still under heavily development; so don’t be amazed when you see regular updates that touch core features of Simba (the GUI).
[] Open minded. We welcome all sorts of criticism and ideas.
[] Documented. There is documentation on how to use Simba as well on how Simba actually works.
[] Extensible; you can write your own extensions (or plug-ins, depending on how you want to call them) for Simba. You don’t need anything but Simba to do this; and the extensions can vary from updaters, little user interface enhancements, to powerful tools like DTM editors.
[] Simba is stable and quite fast
[] Upcoming support for other programming languages.[/ul]

Simba features:

[ul][] Run multiple scripts
[] Code hints and completion
[] Colour Picker History
[] Auto updates
[] Handful of useful tools to aid you while scripting.
[] Full access to all settings[/ul]

So I would recommend everyone to switch from SCAR to Simba immediately, there are no downsides and only many benefits!

Download it here: https://www.moparisthebest.com/smf/index.php/topic,456711.0.html

Ah, I didn’t express myself that well, I’m afraid.
libmml is NOT Simba functionality rewritten at all; it is simply a C-alike interface (wrapper) for Simba functions. We chose to use a C approach for obvious reasons; it is widely supported. libmml is written in FPC (Pascal); but it doesn’t use any of the Pascal types in the functions that it exports.

For example, findcolors() does not return a pascal array, but rather a pointer to point data. The reason is that FPC (and other Pascal implementations) all have managed arrays, which means we cannot be sure when they are freed, moved, etc. So what libmml does is call the pascal function and then perform a flatcopy of the array data onto data that we allocated ourselves (not using some super-managed system), and pass that to whatever program calls the library function. It is also the duty of the program to free this data. The performance hit caused by the copy is minimal.

Freeing and allocating memory that is shared among the program/libmml probably has to be done with the functions provided in libmml, but that shouldn’t cause a lot of issues.
The types exported aren’t too complex and translated to the proper types on the fly, so it should be easy to use.

But yes, easy ``port’’ to another language will have to implement libmml (write an interface for it).

If you have a better solution, I’d like to hear them.
