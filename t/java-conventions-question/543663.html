I’m confused as to your first question, but when I write my code, Everything that should not be changeable in scope I mark a final, so that I don’t accidentally do so.
In this case

public final class Foo {
    public Foo(String value) {
        value = value;
    }

    private final String value;
}

In the constructor, you see value = value. In this context, all this is doing is setting the parameter passed to itself, not the instance variable. Like in this example:

public final class Foo {
    public Foo(String value) {
        this.value = value;
    }

    private final String value;
}

It’s a simple mistake, but one I’ve seen countless times, especially in winterLove-based servers.

More to the point of the second question, should you make it final. In my experience, if it doesn’t need to be changed, don’t allow it to be in the first place.

public final class Foo {

    public List<String> isValid(List<String> list) {
        list = null;
    }

    public void doSomething(List<String> list) {
        if (isValid(list)) { // This nulls the list
            System.out.println("List is valid");
            System.out.println("List size = " + list.size()); // Throws NPE
        }
    }
}

In this case, when checking if the list is valid, it actually sets the list reference itself to null.

public final class Foo {

    public List<String> isValid(final List<String> list) {
        list = null; // This will throw a compile-time error.
    }

    public void doSomething(List<String> list) {
        if (isValid(list)) { // This nulls the list
            System.out.println("List is valid");
            System.out.println("List size = " + list.size()); // Throws NPE
        }
    }
}

Another thing you should be careful of are classes that are not immutable. Strings and primitives are immutable, Strings because they are immutable, and primitives because they get passed by value. But if for example you pass a non-immutable object, final doesn’t matter much during runtime

public final class Foo {

    public static void main(final String[] args) {
        final Date date = Date.parse(System.currentTimeMillis());
        final Foo foo = new Foo(date);
        date.setDate(4);
        foo.printDate();
    }

    public Foo(final Date date) {
        this.date = date;
    }

    public void printDate() {
        System.out.println(date.toString());
    }

    private final Date date;
}

Now I now the set* methods in Date are deprecated, but this is just for an example.

What we see is we created a Date object (which is mutable), passed it to Foo, change the date of the Date object, and had Foo print the date.

Now, if you don’t know how this works, you might expect that when foo.printDate() is called, it would print out the Date that was used in the constructor, since the parameter was final, and the instance variable was. This is not the case. Since the Date object was passed as a reference, the final field date in Foo was just a reference to the Date that was created in main(). When you change a field inside of Date with the setDate() method, it also will change the value inside of the date in Foo.

If you’re doing something like that then a simple listener pattern would suffice

class EventDispatcher {
    Set<Event> events = new HashSet<>();

    public EventDispatcher() {
        events.add(new Event1());
        events.add(new Event2());
        ...
    }

    public void execute(int value) {
        for (Event event : events) {
             if (event.getValue() == value) {
                  event.execute();
             }
        }
    }
}

    public abstract void execute();
}

class Event1 extends Event {

    }
}

That’s more of an event queue (Itemize events, run on call). Dispatcher is dispatching to something, often times a reusable handler (or sink, or listener). Depends on if you want the logic for the event internally held or externally held which is where the difference is exemplified! They both are similar in the fact that there is a timeline to when the logic for events can be called, which is often held on a trigger method.

Also, I just said this:

I doubt any applications you work with will REALLY require you to be anal about using the final keyword on methods. I use it more or less as a tool to assist my direction in development. It's like people who use final with inline method variables/parameters, why? Say the specifications for the method change, then you have to delete it. Unless you really are required to declare THOSE final such as when you are passing a parameter/inline method variable to an anonymous class, it's in my suggestion of practice to not use it. The only place I see using final outside of the context of a constant is with classes to imply hierarchy. Tagging a class with final is fine because then it reminds that it cannot extend it which is a design piece that you have to explicitly state or construct in your model.

I know most of this extends the scope of your question, but there’s a good explanation about when to use the final keyword.

Edit: A waterfall way to put it is that the keyword final is good for explicit behavior WHICH is good for class structure, bad for anything else unless you absolutely need it.

Your first question depends on what you try to achieve. If you have a fixed set of pre-defined sub-types of a given super-type, you may very well hard-code the values provided to the constructor of said super-type. But you don’t necessarily have to restrict yourself to one or the other. If no sub-type add additional methods or fields, you don’t really have to expose the implementation classes. Instead, use one Static Factory class with Factory Methods creating the instances (all of which use a package private or package protected access modifier - that is, no access modifier at all). This approach will significantly reduce the complexity while learning your API.

Now to your second question. It’s up to you, whether you want to add final to all parameter arguments in all methods that should not be changed. If anything, it will help the compiler help you enforce that no parameter argument is changed when it shouldn’t be. When it comes to performance… Yes, it used to help the compiler (or HotSpot, JIT-compiler) optimize code. But that time is practically over. Nowadays it doesn’t really matter.
