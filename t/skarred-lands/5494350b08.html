I’ve decided my current skill system isn’t up to par (yet again…) because I had to keep writing new cycle events and so on. It was rather nasty.

So here I am, again, writing a new skill base. But this time it will be a base for all skills to work off.
It will do/have the following (But not limited to):
A single method to check if the player is able to do a skill action (Ex: cut a tree down, cook fish, fish for fish) - This will check things such as if the player is already skilling, is trading/dueling, the player level, etc.
A single method to check if the player has all materials for the skill action (This method will be used in the aforementioned method - and other events that may do skill like actions outside of skilling)
A single method to run the skill action. - Instead of writing a new cycle event for each skill, there will be one “mother” cycle event that each skill will use, where they all send whatever data required (Anim, skill item, etc, etc)
A SkillItem object - Instead of declaring an object for each skill, or 1000 enums, I will be using a SkillObject that will be able to contain all the data any single skilling action would need. Some data may get broken up to multiple hunks, to fit better. What I mean is instead of putting ALL data for a single herb into a “HerbloreItem” (ALL as in unclean herb, clean herb, unfinished pot, finished pot, ingredients, w/e) it will be in multiple SkillItems, ex: A dirty > clean. A set for clean > unf. and a set for unf > finished. It probably isn’t the most efficient, but it’s better than my current.

The first skill I’m rewriting for the new system is Herblore (Kind of why it was referenced above). If all goes well the conversion process will be fast.

For instance, if you look at this post I keep doing things that changes the back end, preventing me from adding unique content.

However, that doesn’t mean there isn’t any on there currently. Here is a couple (That means “NOT ALL”) of unique features I have:

Drop All - If you have multiple of an item, you can “Drop all” by right clicking the first item (I’ve been considering changing this to work on any item in the inventory as long as they have multiples)
Bury all bones - (Works by right clicking a bone, this idea was inspired by arham and he deserves credit for it)
My yell title system - Players are able to create custom yell titles (I haven’t fully decided how players will obtain this ability) using a message prompt, and then customizing it. I eventually want to put this on it’s own interface.
Yell chat - The yell chat is it’s own chat type. None of that game message yell nonsense. This makes it easier for users to read the yell messages.
XP locks - Most servers usually have it so you either gain xp or you don’t. My xp lock system allows you to lock an individual skill (using an interface I created) or unlock it.
Dungeoneering - Most servers throw you into a room with npcs and each one gives you dungeoneering XP and tokens. My server has custom floors for dungeons that have npcs, a boss, hidden chests, and more features to come. Your xp gained is based on the floor you’re on, how many kills and deaths you have, chests opened, and time. I’ve been considering making party mode, which would allow people to group up, but I need to get a better understanding of the construct region packet before that.
Dungeoneering Rewards - Instead of just giving people the weapons from dungeoneering (As seen on LOTS of servers), I’ve decided to give users rewards that will have effects as they wear or use them. A quick example: Wearing the Shadow Silk Hood will make NPCs un-aggressive towards you.
