system August 1, 2016, 12:46am #38

Ehhh, from what I can tell, this will not allow you to have 2,000 players, nor will it improve server performance. The lag isn’t caused by not being able to accept new clients fast enough, it’s caused by player updating and lots of client IO threads. It’s a pretty neat design idea (I was using a similar system for servers a few years back), but ultimately it doesn’t do much for your server because this is not where the performance bottleneck lies.

It increases stability because your class has exception handling, so that if an exception occurs while the PlayerHandler process method is running, it will catch the exception and the thread wont terminate. In the old implementation (without the Executor), the thread would just die and the server would crash. This is decent, except that you aren’t actually multi-threading the PlayerHandler processing (because you use a single thread for the logic executor).

tl;dr:
In my opinion, this does effectively stabilize your server, but it does not increase performance very much.
