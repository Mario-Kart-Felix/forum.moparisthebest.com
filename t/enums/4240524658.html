People use maps to bind the enumerated object to its ID. Here‚Äôs an easy example I wrote - I‚Äôm not maintaining that this is good practice though:

import java.util.HashMap;
import java.util.Map;

public enum Block {
    
    DIRT(0),
    GRASS(1),
    WATER(2, false);
    
    private final int id;
    
    private final boolean walkable;
    
    private Block(int id) {
        this(id, true);
    }
    
    private Block(int id, boolean walkable) {
        this.id = id;
        this.walkable = walkable;
    }

    public int getId() {
        return id;
    }

    public boolean isWalkable() {
        return walkable;
    }    
    
    private static Map<Integer, Block> map;
    
    static {
        map = new HashMap<Integer, Block>();
        for (Block block : values()) {
            map.put(block.getId(), block);
        }
    }
    
    public static Block getById(int id) {
        return map.get(id);
    }

}

Edit: also you could just use:

map.put(block.ordinal(), block);

If you didn‚Äôt want to manually assign each item an ID.

[quote=‚Äúzyle1992, post:20, topic:424052‚Äù]Enums should be used to represent fixed data types (ie. Planets, colors, etc). Enums are constant structures and should not be changed after creation. Therefore, all pertinent data should be supplied in the constructor, and all data you want visible should have an accessor method. (Note that it is not necessary, but is accepted as the coding standard)

As Davidi2 will admit, its quite easy to abuse them. Take for example a class that deals with orientation. One could say that there are only 4 orientations, and could use an enum like so

public enum Orientation {
    NORTH,
    SOUTH,
    EAST,
    WEST;
}

At first glance, this may seem acceptable.[/quote]

And it may be, depending on your application. But what if you wanted something like North-West? You might say ‚ÄúSure, I‚Äôll just add some more element and make it look like this‚Äù

public enum Orientation {
    NORTH,
    NORTH_WEST,
    WEST,
    SOUTH_WEST,
    SOUTH,
    // ...
}

And you may think that‚Äôs fine. But what if you wanted to have ‚ÄúNorth North-West‚Äù (like in wind direction), then you would have to add even more element, and the enum becomes inappropriate for that case. Instead, you should use a class, with some predefined values

public class Orientation {
    public static final Orientation NORTH = new Orientation(0, 1);
    public static final Orientation SOUTH = new Orientation(0, -1);
    public static final Orientation EAST = new Orientation(1, 0);
    public static final Orientation WEST = new Orientation(-1, 0);

    private final int deltaX;
    private final int deltaY;

    public Orientation(final int deltaX, final int deltaY) {
        this.deltaX = deltaX;
        this.deltaY = deltaY;
    }

    public int getDeltaX() {
        return deltaX;
    }

    public int getDeltaY() {
        return deltaY;
    }
}

As for the HashMaps, some people use them to make accessing them easier. Although, accessing them shouldn‚Äôt be a problem, because enums should be used to express (in my opinion) small, finite data types. For example, planets are a good use of enums, the Periodic Table of Elements however probably isn‚Äôt.

@Taharok: I think the reason most people get NullPointerExceptions using enums is because they don‚Äôt check for nulls before using them, even though enums are still Objects. For example, if you take my NESW orientation enum, one could have a method like this

public void print(final Orientation orientation) {
    System.out.println(orientation.getDeltaX() + ", " + orientation.getDeltaY());
}

This might seem fine on the surface, but calling the method is perfectly legal, and will compile without error, and throw a NullPointerException if you call it like so

print(null);

So to fix it, you just need to add a validity clause to the beginning of the method

public void print(final Orientation orientation) {
    if (orientation == null) {
        return;
        // throw new NullPointerException("orientation cannot be null");
    }
    System.out.println(orientation.getDeltaX() + ", " + orientation.getDeltaY());
}
