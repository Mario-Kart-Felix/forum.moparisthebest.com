The Tutorial:

This tutorial will focus on the main base for your bot. The loader. The loader is what “loads” the runescape applet into the GUI of your bot. Without using a loader, you would be forced to use the applet right off the runescape site, which would prevent minimization. A good example of this kind of bot is SCAR.

The loader is a very useful tool, but it should be done correctly. A “dirty” loader could easily result in a ban.

There are two types of loaders, a URL loader (a URLClassLoader) and a jar loader (a JarClassLoader). The URL loader loads all the data and the applet right off the runescape site. The jar loader loads runescape from a jar file on the local filesystem (this method is used for all bytecode bots).

We need to create a new frame.

Now an empty loader will appear. Before we can start the loader, we need to set the stub with the parameters we need. There are two ways to go about doing this. We can hardcode each parameter into the loader, or we can parse the parameters off of the runescape website.

Obviously, hard coding them is much easier, but parsing them is much safer.

However, since hardcoding them is still acceptable (seeing as RSBot uses hardcoded parameters), I will start with doing it this way, then show you how to parse them later.

First, we need to create a new HashMap to hold our values.

private static final HashMap<String, String> parameters = new HashMap<String, String>();

For your ease of use, and my time, I have copied a small excerpt from the JavaDocs regarding the HashMap class. I do highly suggest you look into the Javadocs yourself and explore further, I am simply placing this here for simplicity. The java 6 docs can be found at http://java.sun.com/javase/6/docs/api/ .

Hash table based implementation of the Map interface. This implementation provides all of the optional map operations, and permits null values and the null key. (The HashMap class is roughly equivalent to Hashtable, except that it is unsynchronized and permits nulls.) This class makes no guarantees as to the order of the map; in particular, it does not guarantee that the order will remain constant over time.

This implementation provides constant-time performance for the basic operations (get and put), assuming the hash function disperses the elements properly among the buckets. Iteration over collection views requires time proportional to the “capacity” of the HashMap instance (the number of buckets) plus its size (the number of key-value mappings). Thus, it’s very important not to set the initial capacity too high (or the load factor too low) if iteration performance is important.

An instance of HashMap has two parameters that affect its performance: initial capacity and load factor. The capacity is the number of buckets in the hash table, and the initial capacity is simply the capacity at the time the hash table is created. The load factor is a measure of how full the hash table is allowed to get before its capacity is automatically increased. When the number of entries in the hash table exceeds the product of the load factor and the current capacity, the hash table is rehashed (that is, internal data structures are rebuilt) so that the hash table has approximately twice the number of buckets.

So, basically, we are using the hashmap to store our values.

These are really the only parameters we need to start the loader:

parameters.put("height", "762"); parameters.put("width", "503"); parameters.put("worldid", "37"); parameters.put("members", "0"); parameters.put("modewhat", "0"); parameters.put("modewhere", "0"); parameters.put("safemode", "1"); parameters.put("lang", "0");

Next, we create a new instance of the “client” class. For those who don’t know much about client hacking, the “client” class is the main class in the client. It is what loads the game etc.

So, our goal is to make a new instance of the client class. For this, we need to load the client class. For that, we use a client class loader :D.

clientClassLoader = new URLClassLoader(new URL[] { new URL( "http://world37.runescape.com/runescape.jar") });

Once again, take note of “world37”. In your bot, you may want to replace this with a user selected variable.

Basically, this creates a new URLClassLoader which we use to load classes from the runescape JAR, without having it on our current filesystem.

Here is another excerpt from the javadocs regarding the URLClassLoader:

This class loader is used to load classes and resources from a search path of URLs referring to both JAR files and directories. Any URL that ends with a '/' is assumed to refer to a directory. Otherwise, the URL is assumed to refer to a JAR file which will be opened as needed.

The AccessControlContext of the thread that created the instance of URLClassLoader will be used when subsequently loading classes and resources.

The classes that are loaded are by default granted permission only to access the URLs specified when the URLClassLoader was created.

Our next step is to setup the “stub” of the applet. A stub is a routine that doesn’t actually do anything other than declare itself and the parameters it accepts. Stubs are used commonly as placeholders for routines that still need to be developed. The stub contains just enough code to allow it to be compiled and linked with the rest of the program.

Setting the stub for our loader couldn’t be easier. Since we are already implementing AppletStub, we just go:

loader.setStub(this);

[size=20pt]Jar Class Loader: [/size]

The entire idea behind a jar class loader is the ability to load a modified client. Bytecode bots inject their accessor methods and interfaces into the client, and the bot then uses that client which is stored on the filesystem.

If anyone knows, I’m pretty sure it is possible to use a URL loader with a modified client, if you upload it on the web.

There are two steps to creating a loader for a modified client.

  1. Creating the jar loader
  2. “Hacking” the runescape loader to load your classes instead of theirs.

In this tutorial, I will be covering the first part, and briefly giving a taste of how the second part is done.

I would further cover the second part except it isn’t something I understand to a great detail. And since I acknowledge that the amount of things I put in this tutorial does not make my epeen any bigger, I would rather put in information which I am sure is correct and will not confuse you.

NOTE: Some of the code you will see in this section is from RSBot. I strongly believe in code reusability, and I also strongly believe in credits being given where they are needed. Therefore everything which is not mine will include the author tag of the actual creator.

Ok, so moving on.

A common misconception, in a jar loader, there are actually two different jars needed. You need the runescape jar, but you also need the loader jar. The loader is what loads the runescape client. Also make note that it will be treating your filesystem like a website. Since the runescape jar and loader jar are in the same directory on the runescape site, they must be in the same directory on your filesystem.

You can download the runescape jar from: http://world37.runescape.com/runescape.jar
And the loader jar from: http://world37.runescape.com/loader.jar

[size=20pt]Troubleshooting: [/size]

I will address a few of the primary concerns and problems with loaders.

OutOfMemory

This error is thrown when the JVM is out of memory to run the rs client. The client uses a fair amount of memory, so don’t run too many “big” applications in the background. This error occurs at runtime.

Could Not Reserve Heap

This error is thrown when the JVM can’t reserve enough memory to run the application. Close any big programs you are running and check what processes you have going. This occurs before runtime.

Other JFrames freeze

If you are making a bot and when you start the loader all the other frames in your bot freeze, it means you have this issue. As already stated, the rs client uses a lot of memory. The only way to get around this is to run your loader in a separate thread.

Invalid Version

If you get any sort of error regarding a version when you try your jar loader, it is because your jar files are out of date as runescape has had an update. Redownload the jar files and it should run.

If you have any more errors, feel free to post!
