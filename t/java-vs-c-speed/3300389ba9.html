Stork July 31, 2016, 2:58pm #53
zstars:

JIT compilers can indeed be theorethically as fast as native programs.

Well, actually, theoretically they can be much faster than native programs.

Think of it like this - the more high level a program, the more you are telling the compiler about what you want to do (as opposed to how you do it), so the compiler can infer: ‘I can see you’re trying to do this, but I can do it faster like this’.

Obviously this depends entirely on the compiler being smart enough to understand this shit, but in theory the more a compiler knows about what you’re trying to do, the better it can optimise what you wrote.

[quote=“Stork, post:53, topic:330038”][quote author=zstars link=topic=426425.msg3135695#msg3135695 date=1266945841]
JIT compilers can indeed be theorethically as fast as native programs.[/quote]
Well, actually, theoretically they can be much faster than native programs.

Think of it like this - the more high level a program, the more you are telling the compiler about what you want to do (as opposed to how you do it), so the compiler can infer: ‘I can see you’re trying to do this, but I can do it faster like this’.

Obviously this depends entirely on the compiler being smart enough to understand this shit, but in theory the more a compiler knows about what you’re trying to do, the better it can optimise what you wrote.[/quote]

Compilers have great trouble finding out what you are trying to do, and the information they can infer is quite limited. However, there’s nothing preventing such information from being also available to a native compiler. Either through one that builds self-optimizing programs (Teorethical too, most likely) or through one that can instrument and profile the code and automatically use that information to infer what little it can ( Visual Studio Team System’s compiler can already do this ). Though I do agree, a JIT for a non-limiting language which is built for optimization and speed and not so much for safety and reliability and which doesn’t impose a memory model would have the same or greater potential than a native compiler and be easier to auto-optimize.

On a different note, it is indeed true that most programmers can’t optimize better than a compiler. However, while this is fully applicable to micro-optimizations, the compiler is almost useless with algorithmic optimizations or with a few other things. While this is not so significant in languages such as Java, because there are less no-do’s, in C++ the performance difference between similar code can be significant if you make certain mistakes.

The more common ones would be passing by value (syntactically, Java style) what should be passed by ref (which can give you an orders of magnitude slower program, and which the compiler won’t optimize, because it can’t know whether you really want to do that or not), and abusing the default new heap allocation (default new tends to be particularly slow in C++ under most operating systems, unlike Java’s). Stack allocation is generally faster when it’s appropiate (small objects), and the default new should not be used where performance is important (and overload new to use a memory pools based one, which can be provided by boost.pool, or other solutions).
