zstars July 31, 2016, 2:54pm #28

Java is significantly slower in practise than C++ (or most similar languages) on most applications where such a thing matters. But IMHO everyone actually knows that already.

A proper JIT compiler is or would certainly be capable of generating code that is similar to C++s, and it even has an easier time optimizing certain things (such as using processor specific instructions), but Java is burdened by the language itself. The garbage collector, checked arrays / references, heap-based memory model, etc.

And most of all, in real applications performance does not only depend on the actual processing speed, but on memory efficiency as well. And though the Java JIT compiler is currently kinda nice, its memory usage tends to be insane. Even Mono seems to be significantly more memory efficient. Things might actually get better when they add the new Java GC in 1.7, but I doubt it. We shall see.

Might be noteworthy tho, that it takes significant knowledge and experience to be able to optimize a C++ or even a C program properly, wheras Java is simple and protective towards the programmer by design, so it is easier.

BAsh July 31, 2016, 2:55pm #32

In my opinion: Java + JIT is as fast and can be faster than C. Why? Everything C can do the JVM can do with the bonus that the JVM can optimise to suit the CPU architecture and other bottlenecks that can change depend on system loads etc.

As an example take a math intensive algorithm. The JIT should have no problem taking a Java implementation and optimising it to suit the specific CPU instructions available to it. In the BEST case the C program can do: if(cpu1) { program1(); } else if (cpu2) {…} The issue with that is that it takes up more space and so there is a trade off. The JVM does not have that trade off. While the JIT code may be bigger it only needs to support <super_fast_instruction> once rather than any number of times in multiple C libraries spread throughout the machine.

So from a theoretical standpoint, ignoring startup / compiling times, Java will always be faster since C will always have a trade off. The only limitation on that is if the ingenuity of the C programers is better than the Java programers. But of course due to the nature of open source one can simply steal from the other.

So the only slowness can be attributed to the startup time. What I don’t understand is why all Java classes can’t be JITed and cached. It would significantly cut down on startup time since what you are effectively caching is the internal in-memory representations of Java classes. All you have to do is ‘link’ them which you would have to do with C anyway. Why Java doesnt implement cached JITed classes is beyond me.

JIT compilers can indeed be theorethically as fast as native programs. They simply aren’t. It would certainly be nice if they were, but at the moment it’s just not that way, probably because of the issues I already mentioned, and some others. In the case of Java and other modern languages, the memory model they work under is meant to be friendly towards the programmer and safe (Garbage Collector, bounds checking, etc), which has a cost. This is especially true for the Java one, as C# does have some native unsafe ops.

It might be noteworthy that theorethically as well there’s nothing preventing a C++ compiler from using processor-specific instructions or extensions (Visual Studio for instance allows that option) or even optimizing itself on runtime polymorphically.

Also, C++ is just a language and a JIT compiler / VM for C++ would be perfectly possible. In fact, it exists already (LLVM). Performance wise the potential of a C++ JIT compiler would be greater, as C++ is significantly less restrictive (no need to check for array overflows, for valid pointers, etc), unlike Java, which does require such checks. Of course, you might be willing to make that performance sacrifice (LLVM itself probably is) to get some of these benefits.
