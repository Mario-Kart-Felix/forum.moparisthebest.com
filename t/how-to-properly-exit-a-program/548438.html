Okay, I’ve been curious about this for a while and I have googled it and found the same answer most every time:

System.exit(0);

I’m here to ask for a better answer.

I’ve seen other ways to exit a program such as:

Thread exitThread = new Thread() { @Override public void run() { try { Thread.sleep(1000); } catch (InterruptedException e) { e.printStackTrace(); } System.exit(0); } }; exitThread.setDaemon(true); exitThread.start();

Or simply:

Thread.sleep(1000);
System.exit(0);

(I know Thread.sleep needs a try and catch but I left it out)

imthenull August 4, 2016, 2:05pm #2

The JVM will shutdown and return a exit code of 0 when all non-daemon threads have finished running. You can immediately shutdown the JVM by explicitly invoking the System.exit function with the proper exit code. Thus, calling the latter with a status code of 0 is redundant.

Protosstribe August 4, 2016, 2:06pm #20

We use System.exit(int) in the application we’re writing at work. One of our GUI menus have two buttons; one for restarting the computer and one for restarting the application. We also have a kill-command that can force us to shutdown. To restart the program, a specific status code is used. Otherwise, 0 represents a graceful shutdown and anything else represents an erroneous shutdown, as the result of an unrecoverable fatal error. Our application (process) should not crash entirely. And no cryptic error messages should be shown to our customers. Although we try to log everything. If that’s not enough, we log a core dump, should the entire application (process) crash.
