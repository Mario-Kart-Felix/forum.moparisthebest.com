lothy August 4, 2016, 1:04pm #4

To be honest, the ‘engine’ as you’ve described it is ultimately a mapping of actions to outcomes. That is, you might map an action (i.e., prospecting a rock, or more generally object option 2) to a script, and that script runs subject to some predicate being true.

For example: You map the using a bank booth to a handling method that sends a message to the user. The predicate might only return true if the user is within a certain location (i.e., within a bank that, in your server, is abandoned). So whenever a player clicks a bank booth object while standing in an abandoned bank, they get the message (which might be something like “This bank is not operating.”).
To complicate it a little further, you might want to chain handlers so that one action maps to zero to many handlers. You might also want to introduce the notion of priority, and finally the notion of either firing the action down the chain or aborting the action partway through the chain when appropriate.

For example, you might have the following actions for mining iron ore:
Priority 1: Mining iron ore in a particular location during a quest - when this handler’s predicate is true, the handler fires and the quest progress is updated. This is a special case, so it doesn’t fire the event further down the chain.
Priority 2: Regular iron ore mining. This is the default, and should always run if reached. You would likely place the regular action as the lowest priority so that special priority handlers have the chance to run when appropriate.

Importantly, a real game would have hundreds or thousands of handlers. So you need to use efficient data structures (typically structures that run in logarithmic time in the worst case – basically your tree structures). Otherwise the solution doesn’t scale up to those hundreds/thousands of handlers very well.

xEthanx August 4, 2016, 1:04pm #6

It’s not as bad as you might think, but it isn’t quick either. I haven’t done it personally but Kane handles game development for our RSPS and showed me articles that he followed that literally list all the protocol you’ll need for the entire server engine. I’m sure it’d be tedious, but worth it. Being able to fully understand the entirety of your server structure would be really nice, especially considering a lot of these clients and server sources that people just edit pieces of code from we’re written in like 2001, and are just disgusting looking. You’ve probably seen the bukkshit where they have methods called “method557” or classes called “Class5_Sub1_Sub3_Sub1_Sub2” or something like that. So to avoid that whole mess by creating your own system is definitely worth it.

Plus, if Kane can figure it out a lot of people can, lolz.

Unless you’re just up for a challenge though, what’s the point really? The community behind Java is just insanely huge so help is readily available, and everyone’s pretty much guaranteed to have it installed.

Anyways, good luck if you give a try, you should cause there’s no harm in that.
