[size=24pt][color=red]Wild-Card Importation V.S. Direct Importing[/size]

Hello, and thank you very much for reading this Thread.

As a lot of you may know, I left MoparScape for a brief period of time, but now I have returned. As an apology for my absence, I took the liberty of releasing a 110K.B. Think Tank design for R.S.P.S, which you can find here.

I hope that someone finds that useful, and now, without further ado, I will give my take on ‘Wild-Card Importation V.S. Direct Importing’. I would like to say that I’m not ‘dissing’ one or the other, as they’re both functional, and they both perform their jobs well. I’m simply listing Pros and Cons without bias. Let’s take a moment to peak at some lines of code, shall we?

package com.world;

import com.repack.NPC.*;
import com.pack.core.Client;
import com.pack.player.Player;

Now, you may look at this and say, “Jack, I don’t see anything wrong”, or, “These lines of code don’t need to be re-written or improved upon”.

Let’s take a closer look, except we won’t need to look at the package directory, so we’ll leave that out:

import com.repack.NPC.*;
import com.pack.core.Client;
import com.pack.player.Player;

As you can see, one of the Imports is using an asterisk syntax. The use of a Wild-Card Import (the asterisk syntax) denotes that ALL files inside of the corresponding directory shall be Imported. The other two Imports shown, are specific Direct Imports. I’m sure there are users who will argue that there is no difference between the two, but I can assure you, that is just simply not the case. I’m now going to list both the Pros and the Cons of each of the two:

[size=24pt]Pros:[/size]

  1. Organization: When you’re using Direct Imports - you know EXACTLY which file you’re incorporating. This also makes it easier for human eyes to scan, enables you to instantly recognize which file you’re Importing, and you aren’t including 5 unnecessary files to access one file.

  2. Clutter: It’s no secret that R.S.P.S. are notoriously messy and unorganized. This is taking a step in the right direction, however, there is another side to this ‘plus’ as well – which will be covered very shortly.

  3. Familiarity: When using Direct Imports, you will become more familiar with your Source, Classes, Methods, Conditions, and lines of code. The more knowledge you have of what you’re working with, the more capable you can become.

  4. I.D.E. Knows That Feel, Bro: I’ve found that some people dislike this function. When using a modern I.D.E, it can and will convert Wild-Card Import statements into Direct Imports. EDIT : ADDED BY HcoJustin : This is IDE dependent. IntelliJ IDEA will prefer direct imports until you reach a certain threshold (Default: 50), or you are in a specified package (Default: javax.swing). It is certainly possible to turn it off, and I’m sure Eclipse and Netbeans have similar features that are easily configured.

  5. Build-Time: Although this will most likely cause a large argument, converting Wild-Card Imports to Direct Imports CAN reduce your Build (Compile) time.

[size=24pt]Cons:[/size]

  1. Clutter: Previously, when writing up the ‘Pros’ for utilizing Direct Imports, I mentioned that using them is a step in the right direction - towards organization. However, like I said, there is another ugly side to this. When listing multiple imports, you will start to accumulate a long list of Direct Imports. In this case, a Wild-Card Import would be more practical than a long list of Direct Imports.

  2. Absent Class: When using a Direct Import, the specific file you’re attempting to Import MUST exist. If you’re using a Wild-Card Import, then this is simply not the case.

  3. When using Wild-Card Imports, you could run into a Naming Conflict. This usually occurs when two files in the same directory have the same name.

In closing, I would like to reiterate once more… I’m not saying that one of these is better than the other (although I most certainly have my preferences). I would just like to point out that both of these functions have their uses, and both should be utilized according to situations.

  1. Conflict Resolved: When using Wild-Card Imports, you could run into a Naming Conflict. This usually occurs when two files in different packages, or, directories, have the same name. When using Direct Imports, you can avoid this.

This is misleading, you make it sound like you can avoid conflicting class names using direct imports, when really having to use two classes with the same name can’t be fixed at all the way it’s imported. The only difference is, say you had two packages:

  • p1
  • p1.MyClass
  • p1.MyOtherClass
  • p2
  • p2.MyClass
    MyLoaderClass

If in MyLoaderClass you used import p1.* and import p2.MyClass, yes you would run into a naming conflict. However, if you actually needed to use p2.MyClass AND p1.MyClass, you can’t avoid that using direct imports
