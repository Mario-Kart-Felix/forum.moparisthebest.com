IYAOYAS August 4, 2016, 11:44am #218

[center]This is really a design show-off since it doesn’t have many features but it is the most flexible combat design i have seen yet.

[I]Note: Due to deadlines this system wont be fully implemented upon beta release. After beta I’ll get it up to par and implement it wide scale.[/I]

Files:

As you can tell the [I]locks[/I] package closely resembles the [I]java.util.concurrent.locks[/I] package. The main difference between the two is that the combat locks do not produce a concurrent lock on the mobile entity, they produce an in-game “busy” lock. The provided [I]CombatLock[/I] currently supports the following actions:[/center]

lock(Boolean) tryLock() unlock() getHoldCount() isLocked() getOwner() (<- net.ar.world.mob.Mob) getAttackers()

[center]Unimplemented:[/center]

newCombatCondition() hasWaitingAttackers(CombatCondition) getWaitQueueLength(CombatCondition) EntityList<Mob> getWaitingAttackers(CombatCondition)

[center]Each [I]Mob[/I] has an [I]AttackBuilder[/I] assigned to them by default however you can optionally create new attack builder instances if need be. One of the several measures of flexibility this provides is that you can build an attack with an [B]anonymous attacker[/B]! The reason this would be useful is for quests like Dragon Slayer where the deck hand needs to die without actually being attacked by a registered mobile entity. You could also use a system like this to create custom features like dealing one damage when you walk over a fire on the ground, or whatever else when really nothing attacked you, it was a landscape based attack.

Here is an example of configuring an anonymous attack:[/center]

Mob self = ... AttackBuilder bldr = new AttackBuilder(null).setVictim(self); Attack attack = bldr.setAttackStrategy(MeleeAttackStrategy.getInstance()).setPoisonPotential(70).build(); ...

[center]Here is an example of an attack on another player:[/center]

Mob victim = ... Attack attack = attacker.getAttackBuilder().setVictim(victim).setAttackStrategy(RangedAttackStrategy.getInstance()).build(); ...

[center]By design the [I]AttackBuilder[/I] is the only link between a [I]Mob[/I] and the combat system making this flexible, compact, and conventionally correct because we are not using every class in the combat system in a thousand unnecessary places and referencing everything in a static context like most combat systems do. As you can tell I have provided methods for manually configuring pieces of the combat formula because for things like anonymous attacks we cannot grab a poison chance from a configuration file or whatever and use it like we can for mob vs mob combat. Unfortunately as stated above this system wont be able to be implemented until after the beta release since it will take some time to perfect the system.

[HR]
[SIZE=3]Q&A:[/SIZE]

[QUOTE]Arix: How do the locks effect Npc combat with multi, and non-multi combat Npcs?[/QUOTE]
A [I]CombatLock[/I] is produced by the attacker every time the combat sequence is initiated no matter what. These locks can be configured to restrict mobs to 1v1 combat or 1vX combat and are also useful if the victim can be attacked by multiple mobs but the attacker cannot or vice versa. The combat locks can also be used internally to grab damage statistics such as which mob has done the most damage to the npc and have the npc switch its primary target accordingly (would be a nice feature for bossing), as well as an assortment of other things that would normally be dealt with in a bulky unprofessional fashion (methods in Player, Npc, and Mob classes).

This is a well constructed idea although I think there is a better way of doing this that revolves more around making the action of attacking a strategy rather than building an attack each turn. You can simply just inflict damage and collect statistics internally in the mob through designating a source of the damage because the mob object is supposed to hold such data. I think the responsibilities are a tad bit off.

Mob should hold methods to check:

  • Which source has done the most damage.
  • Who is currently targeting the mob.

This solution seems very decoupled and I think will lead into problems with how it is executed.

The attack builders state is internally maintained if the conditions haven’t changed. The target switching would apply to Npcs mainly and that could be configured to take effect for either bosses only, or npcs with HP >= X making it less taxing on performance since we wouldn’t have every npc on the map switching targets. Emperor got me started on the idea of having the CombatLock implement an ActionLock of a sort which would allow us to produce locks for everything from trading/banking to hectic minigames like pest control.

An example of an ActionLock would be a player X requests a trade with Y which calls the tryLock() method and the player awaits the response. If Y player responds before X calls unlock() then the trade is initiated instantly (real-to-runescape), otherwise Y sends a request. This system is more dynamic than taxing or decoupled.

EDIT: I don’t think you understood the purpose of the attack builder. As stated above the attack builder maintains the combat state relative to the Mob it is attached to. The AttackBuilder.build() method is only called if state has changed, otherwise the previously allocated Attack instance will be reused which simply recalculates a new hit, etc…
