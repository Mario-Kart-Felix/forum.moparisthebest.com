ry60003333 January 21, 2019, 2:37am #39
Gnarly...:

Yz, if you call the Garbage Collector after a socket is destroyed it will clean up any RAM it used while being on the server. How can that be bad? I totally understand and agree why it’s bad to call it every 500ms or on a timer, because it’s useless.

Because Sun designed the garbage collector to run in the background and to not interfere with your program, and recommend that you don’t try to manually invoke it. Plus there shouldn’t be a need to call it after a disconnection, or your ‘client’ class is too big. Your ‘client’ class probably won’t be the most lightweight class in the server, but it should contain only client specific things. Then for client ‘actions’, you pass a client object to another class which ‘acts’ on that specific client. You would only need once instance of the action object for the entire server.

Also, I bet that the combat code IS located in the client class, which it shouldn’t be, AND I bet that you have to write different code for PvP, PvN (Player vs Npc), and NvN (Npc vs Npc). Java interfaces are your friends, use them. Players and Npcs share common characteristics, so make an Entity class like the client has. There both Attackable, so make an Attackable interface. These kind of ‘innovations’ are what make a server good. These are normal practices to good Java programmers (not ‘coders’), but to this community they will seem like something brand new. Its AMAZING how EASY servers are to program once you have a rock solid foundation to build on.

Everything in technology is layered. From the protocols that run the Internet, too how the Java Virtual Machine wraps around the native OS on your computer. You should be able to modify the lowest level framework of your server without having to touch the actual ‘working’ code. Just like if Sun updated the java.io.Socket object, you wouldn’t notice. You should do something like this. Right now, no released ‘source’ besides RS2Dv would even qualify as good.

Need we say more?
