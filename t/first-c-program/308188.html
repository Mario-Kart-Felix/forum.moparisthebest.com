As people have said already, Java and C# work in the same way, both compile to bytecode. (The CLI is a bytecode). There is also no “another” requirement, as Mono by itself can execute .NET assemblies. I’m afraid your claims about memory management and protection and garbage collection are also wrong, as, again, C# and Java’s scheme is the same, with the single exception of C# being more powerful and supporting native interaction as a built-in extra rather than as a different API (The JNI). The performance of each system is relative and discussing it is mostly irrelevant as if we wanted particular performance we would probably use neither. The benchmarks I have seen (Mono ones) seem to suggest that Java has a higher CPU throughput, whereas C# Mono provides better memory efficiency.

On a different note, I do agree that currently Java probably has a higher deployment rate than .NET. However, C# is an ECMA standard and the Mono implementation is Open Source. Though it would obviously be nice that Microsoft explicitally supported it, Linux and other projects do not belong to any company and they fare well enough. Novell happens to be Mono’s sponsor anyway, and several others provide commercial support too (Just like there are companies that provide commercial Linux support).

Nonetheless, originally I was simply referring to C# being significantly more powerful as language. Especifically, it provides the same features (including reflection, a large library, attributes (anotations), compiler speed), plus a lot of important ones which Java (and sometimes even C++) doesn’t support: Unsigned types, properties, operator overloading, stack-based structures, delegates, built-in limited native calls and pointers support, primitive generics (which also generate actual unique code to each instance) automatic type-inferring var declaration support, closures (and lambdas), partial classes, partial methods, class extensions, preprocessing, etc etc.

zstars July 31, 2016, 12:10pm #17

Yep, exactly what qauters says

Although as mentioned above C# is just as safe as Java, it still lets you take the decision to leave that safety aside for a while to do things you otherwise couldn’t. Calling a WINAPI, for example, is maybe a bit more verbose than in C, for example, yet as you can see in Qauters example nobody would find it particularly troublesome, and at the same time nobody would use an unsafe operation by mistake.

In Java however, you’d have to use the JNI which adds a whole new layer of complexity, and in fact requires using a different language ( C / C++…) and an API that is not particularly user friendly. That does not only require much more code, but it also requires you to know to some extent a different language and an obscure particular API for it, plus using two languages hinders maintenability significantly. It also increases the chances of bugs, because now you’ve a significant portion of native code which can easily segfault.

Personally, I wouldn’t touch the JNI with a ten-foot pole if I could avoid it, and I happen to know both Java and C++. The average Java-only programmer would hardly even consider it an option. For a language that strives for simplicity, and that sacrifices a lot of things to achieve it, a system such as the JNI is particularly out of place.
