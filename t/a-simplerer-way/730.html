A short lesson in RuneScape protocol.
At first glance, you appear to have a link set up like this

client <----> server

and that leads many people to believe that the server uses information from the client in some way. This has led to any number of farfetched schemes that might give you a good screenshot at best.

In reality, the link is more like this
server —> <— client
The client just takes it. Nothing there but a 3D graphics system and some mouse logging.

Now for the really fun part - the encryption.
RuneScape protocol is partially encrypted. The data is untouched, but the command ID of each packet is scrambled by adding a random number drawn from (iirc) an ISAAC generator. This generator uses as its seed a session key which is established in a two part handshake sort of arrangement. The final initialization numbers are transmitted in the same packet as the login data, under asymmetric RSA encryption which we cannot practically crack. There is no known way to determine what the next number will be without the seed.

ISAAC hasn't been broken since it was published 9 years ago. No bias has been detected either. I am offering these prizes in an attempt to get others to try to break it and assure those techniques are published. Apologies for having such a small prize. If you find successful attacks or biases in smaller versions of ISAAC, I'll include them in isaac.html, even though there are no prizes for them. I'm interested in attacks and biases against RC4 as well. Other fast stream ciphers are SEAL and WAKE.
It only works because both client and server are using RNGs running in parallel - they are initialized at the same time and every packet that is encoded on one side must be decoded on the other, so the arrays stay synchronized. If you were to inject packets into the stream without working through the client interface, thus without access to the RNG, the server-side RNG will be misaligned and the resulting linkdeath will make it quite difficult for your bot to function.
