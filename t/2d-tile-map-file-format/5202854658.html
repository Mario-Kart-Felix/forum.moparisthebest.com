Fat_Cookies August 4, 2016, 8:11am #27

Sorry if you’ve already accounted for it or someone has already suggested it but 2D maps generally have a lot of contiguous regions, so your map format should feature a method of referencing tiles which have been already been defined (if that makes any sense).

On a game I’m currently fiddling with, the map is stored using ASCII letters, and D-I are reserved for referencing nearby tiles so when parsing if the game sees a D, it means the current tile is the same as the one to left, E means the current tile is the same as the one above etc etc. RLE in then used so these regions are nicely compressed

Having said that, you’d probably get better compression using an algorithm which already exists.

mod_taharok August 4, 2016, 8:50am #29

Or, you can just be lazy and write a format that’s quick to parse but mostly unconscious to compression, then use some sort of RLE or dictionary-based compression algorithm that massively sizes it down. That’s mostly what I do these days, though it’s also really fun to develop your own lossy compression algorithm that stacks with a RLE or dictionary-based, meaning you can even size the data down significantly further, at the cost of minimal precision loss.
