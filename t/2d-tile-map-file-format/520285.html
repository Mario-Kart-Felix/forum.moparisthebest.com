File format for a 2D tile based region (think pokemon-esque)

First byte = region id
4 bits - number of different referenced tilesheets
1 byte for each difference referenced tilesheet id, first one would be relative tilesheet id 0, etc.

For each tile:
1 bit - blocked? (for those that don’t know, means you can’t walk through it)
3 bits - layer empty? (111 would mean all 3 layers have a tile id, 010 would mean only the middle layer has a tile id)
2 bytes for each tile layer: (if 3 active layers, would be 6 bytes for the tile, etc)
AAAABBBB BBCCCCCC
A = 4 bits, relative tilesheet ID
B = 6 bits, tile X from tilesheet
C = 6 bits, tile Y from tilesheet

Example:

Header:
01001011 (Region 75)
0011 (3 different tilesheets referenced in this map)
00000011 [Tilesheet 3, relative ID 0]
00011001 [Tilesheet 25, relative ID 1]
00101000 [Tilesheet 40, relative ID 2]

Actual map:
Tile 0, 0
0 (not blocked)
110 (2 active layers)
0001 (relative ID 1)
010111 010110 (23, 22)
0000 (relative ID 0)
000001 000000 (1, 0)
Tile 1, 0
1 (blocked)
100 (1 active layer)
0010 (relative ID 2)
010110 010110 (22, 22)
etc

Map file size assuming a 64x64 map with all 3 layers always used: (note this is a worst-case scenario, most of the time you wont be using all 3 layers)
26 kilobytes

Thoughts, comments, ideas on how I could make it smaller, what you think I should change, etc?

mod_taharok August 4, 2016, 7:36am #11

To slightly counter the thoughts going on here (for giggles), the format you have will have a tremendous amount of redundant data (especially if you make the change I’ll suggest in a bit), which means any sort of compression algorithm that uses RLE (essentially most of them) will massively compress (hehe oxymoron) your file. If you do the changes I suggest, try creating one of the map files and then compressing it with LZMA or BZIP2 and see how significantly the size drops. Hell, even ZIP will have a noticeable size-down.

Now, my main suggestion is the lack of a need to story tilex/y. From what I’ve seen, region files usually contain information for every single tile in a region. Thus, you really only need to specify width and height, and even that is unnecessary if all regions are the exact same size. Thus, just store the per-tile information in an CONSECUTIVELY INTERLACED way (which will speedup cache coherency when reading in the file format in chunks) one tile after another, for every tile in the region. This may result in a bigger file, but it doesn’t have to deal with maintaining a list of placed tiles. You’ll benefit more from helping a RLE-compression algorithm and designing the format for cache-coherency than trying to make it really small by yourself. There are some circumstances for various different format standards and you really don’t have an outstanding need for anything in particular.

Just a bit of feedback on the per-tile information, I very rarely see games need 3 layers. They may provide the illusion of having three layers, since there are tiles you can walk on (layer below), tiles you cannot walk on (same layer), and tiles that are above you when you walk on them (layer above). Nevertheless, even for tiles that are semi-transparent being drawn on a background tile, that’s still only 2 visual layers. I pretty much never see transparent tiles on transparent tiles on an opaque tile (3 layers). Reconsider just implementing 2 layers and implementing some nice editor tricks for making it seem like 3. Unless, of course, I’m misunderstanding your intended goals.

Really, each tile just needs to store a tile id for the upper layer and the lower layer, then the collision type (walk over, don’t collide, walk under). Also, as far as the x/y from the tilesheet, you can just use a packed index instead of separate x/y values (ie, treat the tilesheet like an array of images rather than a tilesheet).

Just my 2c.

I had a feeling the first bit was unnecessary right when I was submitting it, oh well.

I wasn’t aware you wanted object support. You could support “objects” just by making certain tiles “actionable,” wherein you can perform an action on them by some key or walk over or something. That way you’re only introducing a single bit per tile instead of needing another ID. If you really need object support, then it will be similar to three layers to store the object ID, but you could also store a list of objects separate from the tile data, then store a locator for each ID which represents which tile the object exists on (a bit opposite to I was saying above). This is a good solution because there will be few objects, so you’ll end up saving a lot more space this way versus storing a “null object” for most of the tiles, since most will probably not have an object.

All depends on what you’re planning on doing with it.

I just mean that most situations can be done using two layers, instead of needing more. Some people may think there are different levels of background geometry (ie grass, stone, dirt, etc.) and thus each level can be represented by a different layer, where you just control some blend factor between them (more evident in terrains than in tilemaps, but similar idea). Although this works perfectly well and is quite convenient to develop, it’s not common for a single tile to be blending between all four or even three of the types in question. Usually you only see blending between stone and dirt, dirt and grass, etc. Rarely do you need to support blending (or a mixture) of all three simultaneously. Thus, even though multiple layers is convenient, most of the time you are only going to see a single background type tile and a single foreground type tile together.

You mention lighting tiles. That would be an excellent example of when a third layer would be quite convenient to support. You have several options in this case. You could bake multiple layers together, thus only requiring 2 or even 1 single layer, otherwise you can add more layers and increase the amount of blending. Essentially, it’s a CPU versus memory situation, since baking is a one-time cost with a memory loss and a draw time gain. Adding more layers saves memory but requires more draw time. To each their own.

I’m not exactly sure what you mean by the movement system. Regardless of the region size, you’re still going to need to stitch the regions together so when the camera moves past the edge of a region, it’s viewing parts of multiple regions at one time. The camera being centered on the player is a good option, though. It’s really up to you from a design perspective. If you prefer the way pokemon does it, then center the camera on the player, making sure to add constraints to stop the camera when you’re on the edge of the loaded area (unless it’s truly seamless, in which case you have to be careful when you’re on the edge of the entire game world). Otherwise if you really like the area-by-area feel, you could make it so the camera is centered on the region, like you mentioned, and just load a new region everytime the player goes off screen, moving the player to the other side and continuing with the new area. I can’t give you advice one way or the other, because it’s a completely opinionated decision.
