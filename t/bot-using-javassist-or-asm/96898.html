Krichevskoy July 30, 2016, 9:33am #7

The last time I used Javassist, it wasn’t quite as powerful but it was EASY. I mean really easy - you probably already know about it, but the core technology is a Java-like syntax for bytecode edits. Very nice, but I seem to remember that there were a few things Javassist simply couldn’t do. I definitely remember hacking the Loader class for multiple Translator support, though not precisely why that was necessary. It’s been in development since then, so the functionality has probably improved. Also, I now realize that I might not have been using it correctly - it’s the backend of an AOP framework, and since I now know what AOP is I’m sure I could use it more effectively.

The last time I used ASM, I was impressed by the speed, size, and overall cleanliness…I like ASM for the same reason I like assembly. What you write is what you get. The problem was that I couldn’t figure out how to use it for editing existing classes very well. It worked like a charm for generating new classes, though. The downside is that you have to be familiar with bytecode to use it.

BCEL is large, slow, and powerful. It’s a dead project - as far as I know - so problems don’t get fixed (I don’t know of any offhand) and new features don’t get added. It’s more complete than either Javassist or ASM (BCEL has Visitor classes, though no one here uses them). It comes with a decent search class (InstructionFinder), which Javassist didn’t match and ASM doesn’t try to match. It’s a balance between the two in knowledge of bytecode required - you still have to know it, but there’s some abstraction to keep you from the tedium.

Miscellaneous facts, ideas, etc:
Javassist has a built-in mechanism for runtime editing (Translator), so you don’t have to mess with ClassLoader.

ASM is meant for things like generating getter methods (there’s even an example in the source zip), so it might work better for modern passive bots than it did in the age of Aryan.
