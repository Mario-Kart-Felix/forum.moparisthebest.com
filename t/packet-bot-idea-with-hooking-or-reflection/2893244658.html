marluxia July 31, 2016, 9:21am #27
Jython_:

There is no single RSA key… there are two; it’s public key encryption, meaning there are two keys: the public key, and the private key. The issue isn’t in the public key (we can easily get that one, since it’s in the client), it’s in the private key. The issue is not only in the fact that it constantly changes, it’s in the fact that it are extremely hard to bruteforce and that it would take an extremely long time to crack it. The only reason we would want the private key is because in standalone proxy-type bots, such as autorune, we need to use it to decrypt the login block. Why would we want to decrypt it? Because it holds the client and server session keys, which are used to seed the ISAAC ciphers, and we need the seeded ISAAC ciphers so that we can decrypt and encrypt our packet opcodes. Why would we want to decrypt packet opcodes? So we know what the server is telling us. Why would we want to encrypt our own packet opcodes? So that we can send packets to the server… which is the point of a proxy-type packet bot. But there are also other kinds of packet bots that are possible… but they are no autorune Among these include packet bots that edit the client, like yours, which aren’t worth their struggle… if you’re going to have access to the client, why not just simply use it’s internal data, why not just click and do things that are so much easier… And then there are the client simulators, like zstar’s bot. These are simply custom clients if you think about them, and they are also a huge pain.

xeez July 31, 2016, 9:55am #40

A packet bot would be ideal, but it would be a pain to maintain updated :(. I think a “proxy-bot” would be the optimal solution as it requires no actual updating in order for the bot to survive. Of course, when I say a proxy bot, I mean all but a standalone one. Before launching, the bot could load the client’s bytecode and search for the current public key. Then, when starting the client, the bot could use Java’s Instrumentation API to transform the loaded bytecode on the fly. With this, it could replace the ciphers that are created in the login-block dynamically, with static ones that the bot is already aware of.

Of course, after injecting a packet, the server’s RNG would not be synced with the client’s. To fix this, the bot could have two ciphers; a server cipher and a client cipher. When a packet is received from the server, the client removes the stream cipher sent from the server and adds the “client” cipher. When a packet is sent from the client, the client removes the stream cipher and adds the “server” cipher.
