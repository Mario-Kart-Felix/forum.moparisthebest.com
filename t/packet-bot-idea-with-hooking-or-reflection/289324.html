Jython July 31, 2016, 9:18am #5

There is no single RSA key… there are two; it’s public key encryption, meaning there are two keys: the public key, and the private key. The issue isn’t in the public key (we can easily get that one, since it’s in the client), it’s in the private key. The issue is not only in the fact that it constantly changes, it’s in the fact that it are extremely hard to bruteforce and that it would take an extremely long time to crack it. The only reason we would want the private key is because in standalone proxy-type bots, such as autorune, we need to use it to decrypt the login block. Why would we want to decrypt it? Because it holds the client and server session keys, which are used to seed the ISAAC ciphers, and we need the seeded ISAAC ciphers so that we can decrypt and encrypt our packet opcodes. Why would we want to decrypt packet opcodes? So we know what the server is telling us. Why would we want to encrypt our own packet opcodes? So that we can send packets to the server… which is the point of a proxy-type packet bot. But there are also other kinds of packet bots that are possible… but they are no autorune Among these include packet bots that edit the client, like yours, which aren’t worth their struggle… if you’re going to have access to the client, why not just simply use it’s internal data, why not just click and do things that are so much easier… And then there are the client simulators, like zstar’s bot. These are simply custom clients if you think about them, and they are also a huge pain.

There are two keys involved - the private key and the public key. The public key is hardcoded into the client and, as a result, is distributed to all players. The private key, on the other hand, is kept solely by Jagex (this is why it’s called the private key… because it’s private). The game client can only encrypt data; it can’t decrypt it.

In order for a proxy bot to work, it has to analyze traffic running between the server and the client. Unfortunately, when the game client sends an RSA block[sup]1[/sup], the proxy bot doesn’t know what’s going on because it doesn’t have the means (the private key) to decrypt the traffic. If it doesn’t know what’s going on, how will it modify, insert, or delete packets at will?

You might say, “OK, I’ll just make a WPE Pro style thing and record all the packets I send to cut trees and bank.” That would be a good idea, but Jagex put a stop to this with the implementation of ISAAC and the insertion of random numbers throughout RSA encrypted blocks (each block is unique).

At any rate, if you’re going to develop a proxy bot, you’d probably want an updater (and it doesn’t necessarily have to be with reflection) to retrieve the public key.

One final point I’d like to make. Let’s define a contrived encryption scheme:

To encrypt letters we add 5 to their ASCII decimal counterparts. To decrypt letters we subtract 5 from encrypted text.

Many encryptions schemes rely on running through seemingly obscure mathematical operations. The one above would be considered a very simple form of encryption. By doing the mathematical reverse of what we used to encrypt the text, we are able to decrypt it. Why can’t we just do the reverse with public key encryption like RSA? Well, we should be able to, but RSA relies on the little knowledge we have of factoring extremely large digits. At some point in your reverse RSA routine, you’re going to have to factor a number (the modulus). If the number is huge, you’re going to be screwed. This is why we can’t have nice things.

[sup]1[/sup] There are actually 3 RSA blocks.

Yes, that’s one workaround.

At some point, you’re going to have to ask yourself what you’re trying to accomplish. A proxy bot runs the client (so we don’t save on memory consumption), it requires at least a small amount of client analysis, and one that competes with bytecode/reflection bots would require an extaordinarily intimate knowledge of the JS5 protocol.

At this point, a proxy bot is a novelty. The sad thing is, you will have to destroy the “wow” factor by having a modified client (even if the modification is small). The “wow” factor I’m referring to is firing up a program and then running the game client with no other intervention involved.

The work that someone puts into developing a proxy bot can be transferred over to the development of a packet bot. A packet bot retains most of the benefits of a proxy bot, but it doesn’t need a game client. You can also develop at your own pace because you know for a fact that you’re not aiming for undetectability. The downside is that, unlike a proxy bot, much of the work isn’t done for you; you’ll have to re-implement client functionality.

If you’re going to do this, try making a packet bot that logs in (harder than you’d think) first and then work from there.

unlimitedorb July 31, 2016, 9:20am #17

Alright, you edited your post above asking for where the public key is:

Search for when the method “modPow” is called. This brings you to the method that runs RSA.

What we refer to as a RSA public key is actually made up of two numbers - the public key exponent and the modulus. The first argument to modPow is the “public key exponent” and the second is the “modulus”.

Trace the exponent and the modulus back to their origins.

Alternatively, search for BigInteger. There are only two instances initialized outside of the method that runs RSA (the BigInteger initialized inside RSA is the plaintext we’re trying to encrypt.) The largest BigInteger will always be the modulus; the smallest will always be the exponent.
