That’s not what was meant by binary.

The idea of binary sorting is pretty simple. Based on some comparator factor (in this case, just the integer values you’re using in the switch statement), values are sorted from weakest to greatest (or lowest to highest, depending on how you perceive it).

Now, when you want to look through that array to get the index of a value inside it, you can use a simple function like this:

public int getIndex(int needle, int[] haystack) {
	for (int i = 0; i < haystack.length; i++)
		if (haystack[i] == needle)
			return i;
	return -1;
}

However, this method is conventionally slow when searching an array which is binary sorted, because there are faster ways of searching it. This particular method is of O(n) complexity, meaning it’s linear and is processed n number of needed times. A search will always take between 1 and n (array length) operations to find the index.

With binary, the search algorithm should always take at least half, or less, of the time a linear search would require. I’ll briefly explain why. Binary search algorithms would typically start in the middle of the array, instead of at the beginning with a linear array. If the value is equal to that middle index, great! Otherwise, the algorithm can check if the value is greater or lower, and thereby know whether to go forward or back in the array to search for the value. It’s called binary searching because it always segments the array by 2 (starts off with two halves, selects one and divides it into two again, and so on). It continues this process until it finds a match, or finds where a match ought to be, and returns a negative form of it (or -1, for some implementations).

I’d recommend reading up on it, or looking through [tt]java.util.Arrays.sort()[/tt] and [tt]java.util.Arrays.binarySearch[/tt]. There is also bubble sorting or sequential sorting, as mentioned above, among other more complex strategies. Binary is really basic and powerful for what it achieves, though.
