So privately I’ve been working on a game unrelated to Runescape private servers. It has helped me learn a lot, as well as think of new ideas that could be implemented in Runescape private server emulators/frameworks. I’ve decided to compile these ideas into a small and simple API.

It will take me awhile to convert components from my framework as they are dependent upon each other, while some have irrelevent use to Runescape or other games. So don’t criticize it on its lack of current features. It is still under extensive development!

Some of the upcoming features I plan to add are networking and file IO using the NIO library. The networking design will feature a lightweight reactor design coupled with basic codec handling. The file IO will hopefully implement efficient character data storage using custom serialization.

Another feature will be a centralised command system that will work both in-game and through the console/command prompt. Somewhat similar to how Minecraft’s commands work. It will be linked to a custom logger, which allows logging to files for 3 different logging levels (info, warning and error). Custom commands will be easily implemented due to proper abstraction. No more massive if-else or switch statements.

A small update to the event system planned is instead of setting a tickrate upon construction of the instance, it will default to ticking at 100ms. It will take averages of the time taken to service all the workers and adjust the tickrate accordingly. For example if the thread doesn’t sleep for 5 ticks, it will yield for other threads and then use an average overtime to adjust the tickrate. Hopefully this will optimise memory usage.

At the moment I’m releasing this closed source because I haven’t documented the code as of yet, but I’m sure someone will decompile it and confirm it isn’t malicious.

At the moment the API includes components for:

[size=14pt]Task/event processing:[/size]

Package: com.orathanapi.exe

Description: While there are probably plenty of task or event handlers that are written more efficiently than mine, this seems to prove more accurate than the ones currently used in Runescape private servers. This makes use of timing using nanoseconds as opposed to milliseconds. I suppose Runescape private servers use milliseconds because winterLove did when he wrote his emulator before Java 1.5. Another feature useful to accuracy is this takes into account the time the thread takes to sleep and wakeup.

Usable classes:

[hr]

Name: Executor.class

Description: Handles the processing of operations and processors. Serves as a task or event handler. It runs on its own thread.

Usable methods: [constructor]Executor(int tickrate), start(), register(Worker), unregister(Worker), getExecutorTickrate(), getOperations(), getProcessors(), isRunning(), setRunning(boolean running)

Example:

Executor executor = new Executor(100);
executor.register(aWorker);
executor.register(new Operation(5000) {
	@Override
	public void execute() throws Exception {
		System.out.println("This operation was registered 5 seconds ago.");
	}
});
executor.start();

[hr]

Name: Operation.class [abstract]

Description: A task or event that executes once. It can be given a delay to act as a future task.

Usable methods: [constructor]Operation(int delay), getDelay()

Example:

public class ShutdownOperation extends Operation {

	// Delay is in seconds.
	public ShutdownOperation(int delay) {
		super(delay * 1000);
	}

	@Override
	public void execute() throws Exception {
		System.exit(0);
	}

}

[hr]

Name: Processor.class [abstract]

Description: A task or event that executes once every given amount of time. It cannot cycle faster than the tickrate of the executor.

Usable methods: [constructor]Processor(int tickrate), getTickrate()

Example:

public class PlayerProcessor extends Processor {

	public PlayerProcessor() {
		super(600);
	}

	@Override
	public void execute() throws Exception {
		// Code used to execute player updating.
	}

}
