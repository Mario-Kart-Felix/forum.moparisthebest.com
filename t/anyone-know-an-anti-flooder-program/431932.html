So you believe NULL is a ‘flooder’? I don’t know exactly what a flooder is, but based on the context I believe you think it’s someone committing a malicious act.

The truth of the matter is that NULL is actually a memory address. You see Java, like other languages, has the ability to allocate memory on the heap rather than the stack. This is what the new keyword does in Java, C (malloc), and C++ to name a few. NULL is the memory address of 0, it is reserved for pointers (or references in the case of Java) that do not point to any position in memory.

How does this occur? In Java, two ways:

The reference has been manually set to NULL

Object myObject = null;

Or the memory the reference was pointing to has gone out of scope

//...
private void doSomething() {
     Object myObject = new Object();
}
//...

As you can see the scope of myObject is restricted to that method. When the method returns, the object is no longer accessible and garbage collection comes along and cleans up the no longer accessible memory allocation.

If your issue is reoccurring, possibly simultaneous connections from one source I suggest trying to throttle connections. An easy way to do this is to have a map keyed by connecting address (you can use the InetAddress object in Java which you can get from the incoming connection Socket) and have an long as it’s value (the longs value should be the last successful log in timestamp - you can obtain this using System.currentTimeMillis()). When an incoming connection is found, simply making a conditional like the following:
[tt]
If ConnectionMap contains IncomingAddress
If System.currentTimeMillis() - ConnectionMap.get(IncomingAddress) > 3000L //If the last connection is greater than 3 seconds
Allow Connection
Else
Close Connection
Else
Allow Connection[/tt]
