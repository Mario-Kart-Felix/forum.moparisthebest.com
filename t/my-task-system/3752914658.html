meiscooldude August 1, 2016, 1:54am #22

Cool! Interesting little experiment. A few code smells though…

  1. Why does the Engine class use the ‘start’ method instead of a constructor?

  2. Engine is meant for handling simple tasks and delegating complex tasks (ie. Events) to an EventManager. Wouldn’t it be cool if the same Engine could delegate tasks to multiple types of TaskManagers instead of just an EventManager?

  3. I also feel like Engine is doing too much. Handling simple tasks and delegating tasks is a lot for it to manage. Why not make it just delegate and create a concrete ‘SimpleTaskManager’ for handling simple tasks? That way when Engine has a task submitted it simply delegates it to the appropriate Manager.

  4. Engine shouldn’t check for a specific type of a Task upon submission. Try using a Map of sorts to tie Type(s) to Manager(s). Or perhaps even a cleaver use of Generics?

  5. I don’t see why the mark(), taskCompletion(), and getEventManager() methods should be public. If Engine is mean to be extensible then make the protected, if they are just helper methods, private.

  6. Engine is not very Thread friendly. By means of both efficiency and safety.

6(a). There is no need to check the system clock in Engine. Inside the run method there is an if statement providing execution delay. While this works it waists valuable processor time that could be given to other threads. Java’s Thread.sleep method causes threads to pause execution and give other threads it’s valueable processor time.

while(running) { taskCompletion(); eventManager.research(); Thread.sleep(delay); }

6(b). There is no sure fire way to stop Engine from execution (aside from Thread interupts) with the given method ‘setRunning’. Idealy it seems you want a developer to call ‘e.setRunning(false)’ to stop execution of tasks. If code on (a) separate Thread(s) call the ‘setRunning’ method with ‘false’ then ‘true’ before Engine has a chance to check its loop condition ‘while(running)’ it will continue to run. Try implementing a few thread safe methods like start(), pause(), and stop().

6©. Threading in Java is simple. Threading well in Java is a whole other issue. But there are resources! Here’s a freebee: http://download.oracle.com/javase/tutorial/essential/concurrency/index.html

  1. Your code could use a little better documentation. A method name should give you a pretty good description of what that method does. ‘garbageEvent’ could be better substitued with ‘disposeEvent’. Try and use verbs in method names, because a method does something. Properties are a great place for nouns.

Also, ‘Researches all events to check if they should be executed.’, the documentation for the ‘research()’ method on EventManager doesn’t state that it performs any actions. A developer who may be too busy to look at the underlying code will only see the method heading and the documentation. Noting that it returns a boolean (which isn’t used anywhere btw) would imply it only checks to see if there are any Events in the queue that need to be executed not actually execute the Events, and remove those which had expired. That could really irritate a developer who just wants to check something with timing (for instance) and suddenly all this random code is executing for no reason.

  1. When I first saw you using a Queue I was all gitty inside. But apparently your just itterating over it inside the EventManager instead of actually using the Queue interface. Try looking into PriorityQueues. They are quite nice when dealing with scheduling.

  2. What happens if Engine’s delay is set to 100ms and an event’s delay is set to 50ms? Is this behavior desireable?

  3. Instead of having stop() method on Event, try implementing a ‘RepeatingEvent’ that repeates execution a given number of times with 0 being infinity. That would stop forcing people to remember to have to stop() and remove the possibility of having something you only want ran once (ie. Add something to Database) run as many times as it can before you realize and close that application as fast as you can.

-EOF-ish

See where those get you. I’d be happy to look over it again from there. Remember, Code reviews are AWESOME! =]

Okay well I wanted the mark method to be public because what if somebody wants to run the event faster? I mean yeah this is a really unlikely case, but the whole reason for doing these “reinvent the wheel” experiments is to add more features for ease to the user.

It’s very true that there is somewhat no reason to check system clock in Engine, however, what if somebody wants to slow down the rate at which it processes tasks? Very stupid call of me here though, because why on Earth would anybody want to do that? Plus you’re right about it even more, because 100ms would force the event to call at least 100ms, while the user wants it to be delayed at 50ms. So there is no reason not to process tasks as fast as possible (using while(true)) because it won’t truly IDLE the engine if we set a “delay”, as it’ll still be checking if we were delayed enough.

Hey,

Looked over it again, still a few issues I see. Don’t have time to go through and show you, BUT I do have time to tell you about Unit Testing!

http://www.vogella.de/articles/JUnit/article.html

Unit Testing basically uses code to test code. You write a ‘test’ in code that executes the code you want to actually test. The test will (usually) check the program state (if you changed a variable in your test, was it what you think it would be at the end of your test, did any exceptions throw you were not expecting, etc) to see if the test pasted.

Try and get 100% code coverage (every line in your Task System code is executed by a test). Some of the issues should arise if you do this [usually those edge cases].
