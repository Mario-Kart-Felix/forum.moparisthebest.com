[code]package org.xera.me;

import java.util.LinkedList;
import java.util.Queue;

import org.xera.me.event.Event;
import org.xera.me.event.EventManager;
import org.xera.me.task.Task;

/**

  • Constantly processes tasks of our environment.

  • @author Thomas Nappo
    */
    public class Engine implements Runnable {

    /**

    • Starts a new engine.
    • @return the engine we created, for chaining
      */
      public Engine start() {
      return this;
      }

    /**

    • The scheduled event manager of the engine.
      */
      private EventManager eventManager = new EventManager();

    /**

    • @return The scheduled event manager of this engine.
      */
      public EventManager getEventManager() {
      return eventManager;
      }

    /**

    • A queue of pending tasks.
      */
      private Queue tasks = new LinkedList();

    /**

    • Submits a new task to be processed.
    • @param task The task itself.
      */
      public void submit(Task task) {
      if(task instanceof Event) {
      eventManager.submit((Event) task);
      return;
      }
      tasks.add(task);
      }

    /**

    • Completes all of the tasks in the task queue.
      */
      public void taskCompletion() {
      while(!tasks.isEmpty()) {
      Task t = tasks.poll();
      if(t instanceof Event) {
      continue;
      }
      t.handle();
      }
      }

    @Override
    public void run() {
    while(true) {
    taskCompletion();
    eventManager.complete();
    try {
    Thread.sleep(1);
    } catch (InterruptedException e) {
    e.printStackTrace();
    }
    }
    }

}[/code]

This is our task engine, it’s the core worker of our system. You could also call this “TaskManager” or “TaskEngine”, but I’ve called in Engine for general purposes. It is a runnable thread, and every set amount of time it completes all the tasks in the queue, and calls the event manager (which is like an extension of a task manager, or this class, for the most part) to research all events in it’s separate event queue.

/**

  • Manages events.

  • @author Thomas Nappo
    */
    public class EventManager {

    /**

    • All the scheduled events.
      */
      private Queue events = new LinkedList();

    /**

    • Marks all scheduled events (this should rarely be used!)
      */
      public void markEvents() {
      for(Event e : events) {
      if(!e.running)
      continue;
      e.mark();
      }
      }

    /**

    • Attempts to dispose a certain event.
    • @param e The event to garbage.
    • @return true if we disposed the event, false meaning the event was running and we did not dispose of it.
      */
      public boolean disposeEvent(Event e) {
      if(e.running) {
      return false;
      }
      e = null;
      return true;
      }

    /**

    • Executes all events of the queue that were found to be ready and running.
      */
      public void complete() {
      int s = events.size();
      while(s > 0) {
      Event e = events.poll();
      if(!e.ready()) {
      events.add(e);

      continue;
      }
      e.mark();
      e.handle();
      if(e.running) {
      events.add(e);

      }

This is our event manager. Similar to the task manager, it has a queue of events (scheduled tasks). When it researches through the tasks of the queue, if a task iterated isn’t running, it adds it to a “remove” arraylist, which later removes it from the queue. We cannot directly remove it in the same loop because it’s concurrent modification, just imagine having a few apples, while you’re sorting through all of them you find one that you don’t like, well if you remove it right then and there, that slot, or iteration, becomes null, therefore we just tried to check conditions of a null (or nothing). This solution is like having a few apples, while you’re sorting through all of them you find one that you don’t like, keep it in your head, continue to sort through the rest of the apples, and after you’re done sorting, the apples in your head you then remove. After we check if a task isn’t running, we then check if it’s ready to be executed, this is dependent on whether the current machine time subtracted by the marked machine time of the task is greater than the delay of the event. If so, we continue on iterating through the events. If we passed both conditions, we then mark the event first (because what if the execution time of the event is slow?), then execute the event.

[code]package org.xera.me.event;

import org.xera.me.task.Task;

/**

  • A time-schedueled task.

  • @author Thomas Nappo
    */
    public abstract class Event implements Task {

    /**

    • Constructs a new event.
    • @param delay The amount of delay in execution of the event.
      */
      public Event(int delay) {
      mark(); // Mark the current time of creation.
      this.delay = delay; // Set the delay.
      start(); // Finally allow the event to run.
      }

    /**

    • Marked machine time.
      */
      protected long current;

    /**

    • The amount of delay in execution of this event.
      */
      protected int delay;

    /**

    • Flag in which states whether or not this event is running.
      */
      protected boolean running;

    /**

    • If this event is ready for execution.
    • @return true if ready, false if not.
      */
      public boolean ready() {
      return (System.currentTimeMillis() - current) > delay;
      }

    /**

    • Starts this event.
      */
      public void start() {
      this.running = true;
      }

    /**

    • Stops this event.
      */
      public void stop() {
      this.running = false;
      try {
      this.wait();
      } catch (InterruptedException e) {
      e.printStackTrace();
      }
      }

    /**

    • Marks the event with the current machine time.
      */
      public void mark() {
      this.current = System.currentTimeMillis();
      }

}[/code]

Event means scheduled task in this circumstance, so therefore an event implements a task. The difference between interfaces and abstract classes, are that classes that extend abstract classes can superconstruct it’s parent, or “super”, so it can use all the methods inside the super class without having the rewrite the methods/variables, where interfaces require you to rewrite all methods placed inside.

engine.submit(new Event(2000) { @Override public void handle() { System.out.println("Hello, we were delayed by 2 seconds."); this.stop(); } });

This is an example submission of an event. It’s delay is 2 seconds, so after two seconds it prints to the console “Hello, we were delayed by 2 seconds.”, after, it stops the event so it will only run once.

engine.submit(new Event(2000) { @Override public void handle() { System.out.println("Hello, we were delayed by 2 seconds, and are going to repeat 2 seconds later."); } });

Like the previous example submission of an event, this events delay is two, but it does not stop. It will print out “Hello, we were delayed by 2 seconds, and are going to repeat 2 seconds later.” every 2 seconds until the event is stopped.

[code]package org.xera.me.tickable;

import org.xera.me.event.Event;

/**

  • Simple special event that can execute at 600ms intervals.

  • @author Thomas Nappo
    */
    public abstract class Tickable extends Event {

    /**

    • Constructs a new tickable.
    • @param amount The amount of cycles it takes up.
      */
      public Tickable(int amount) {
      super(600 * amount); // Superconstructs an event with 600 milliseconds multiplied by the amount of ticks specified.
      }

}[/code]

This is a tickable event. Tickables (also known as RuneScape cycle) are cycles of 600 milliseconds. They are multiples of 600ms events. They are useful directly for RuneScape, as near all actions in RuneScape are done in tickables.
