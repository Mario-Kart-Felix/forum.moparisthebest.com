In the Java context, a byte is a small unit of data, it can be stored as an electrical impulse, and then changed into something storable on a computers memory. It is assigned a numerical value, which can range between -1 and 255. -1 is a special kind of byte, it somehow signifies the “end” of bytes. Bytes are mainly used in the describing of files, and files can easily be translated into bytes and then assembled into something readable by a Java application. When you are using MoparScape, there are two components:

a Server,
a client.

The server is what you connect to. It is located on the computer you enter the IP number of. The server is responsible for recieving information from the client, and then making a suitable response. In this case, it is recieving that a client has used a cracker with a player. Because of the way servers and clients are set up, before a server can recieve any information from the client, the information has to be sent over the internet.

The method of sending information over the internet is by use of a Socket, a ServerSocket, a InputStream, and an OutputStream. These are basically continous sources of bytes that are being transfered to the sockets, and then through a SocketChannel, then reaching the destination.

When an inputStream reaches the ServerSocket, it is buffered and then changed into something that can responds to the standard context of Java. The server will respond to the information, and then send its response back to the socket, and then to the client socket.

The client. This is the part of MoparScape you see, and that other people see. It is basically a graphic interface - it loads models it needs in byte form from a cache, it displayes the models based on responses from the server, and it records input from mouse clicks and key presses.
All the input the client recieves is then streamed into a socket, and sent to the server.

The glitch can now be explained like this: The server is responding to information that shouldn’t ordinarily be available. When someone uses “::setitem”, the client replaces all the bytes that it was using for a particular object, and then renders the replacement. When the information is about to be sent to the server, it sends a packet with the replaced bytes, and the server responds to it as the way it would if the bytes were decoded.

Its like putting what the player has and does in that players hands, without restrictions. When the client sends packets to the server, the server has nothing else to go on. It will invariably respond to what it has been given. This is because its a bunch of bits and bytes comprising a computer program that cant think for itself. If you want to fix the problem, you have to make the server actually respond to invalid packets, and not take them as if they were ordinary.

The only problem with that is then you would be forced to write a system of acceptable operations of given packets.
