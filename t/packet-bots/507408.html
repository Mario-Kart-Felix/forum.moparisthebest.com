it’s possible in either case, really. the caveat is that it requires you to make the jump from syntactic analysis to semantic analysis.

build a frame updater based on automating identification of code in the client that frames execute regardless of revision.

what does that really mean?

it means instead of figuring out what opcodes do dynamically (clicking a rock object and seeing what opcode is fired at runtime), you need to discover this statically, in the code, and run an updater based on that. For example, there are some things that are fairly consistent in the game: you can click on a rock to mine it. presumably this logic is found in the code base. if you find the logic for this and trace back to the opcode it is associated with, you’ll discover what that particular opcode does.

that’s a problem for people on this forum in general because…

you’re not going to find the answer in “cheating tutorial for the simple moron”. instead, the answer lies in research papers.

developing a reliable packet bot would require solving the halting problem. since that’s not gonna happen, you’ll need to make do with slightly less reliable hacks.

my solution would be to first dynamically discover what each opcode does in one revision. find the code it executes. build an AST for the code. run a hashing algorithm sensitive to structure (not just sum hashing where you count the number of nodes that are the same) then for future revisions identify relevant pieces of code by running an AST similarity algorithm. this should find code that is /semantically/ similar (not just syntactically.) Once you have that, you just trace back to the opcode that is associated with it and you know what kind of frame it is.

how did i figure this out? i googled something along the lines of “semantic program similarity” and whaddya know a research paper popped up. (the structurally sensitive hashing algorithm it used had rabin-karp)

I’ve been fiddling around with using packets for interaction for while, have a look at

The opcodes for each type of interaction do change every revision, so I went ahead and wrote some stuff to find them dynamically. The types of data (e.g both LE and BE short/int) you need to send however do not change from what I’ve seen (IIRC they were the same for object interaction in OS#16 and 317). To see what’s going on when you do a certain game action just inject a callback into every “put” method in the Stream class, it’s quite interesting.

EDIT: And even if the data types change, it’s not unreasonable to hook them dynamically aswell rather than using them as the basis for hooking the opcodes,
as unlimitedorb said there are very fundamental game mechanics for which the chain of events has stayed pretty much identical/consistent throughout 317 to OSRS.
