I wrote the following code Using an online Java IDE, the two classes are an example of a timer using System.currentTimeMillis().

[code=java]public class HelloWorld {
// Ignore below this, it’s just initializing the thread.
public static void main(String[] args) {
Thread t = new Thread(new Example(1000));
t.start();
}//Ignore above this

private static class Example implements Runnable {
	private long lastMessage;// Take note of the 2 variables I use to setup the timer. NOTE: Your "delay" variable would never change, so you could make it a constant.
	private long delay;
	// Ignore below this line
	public Example() {
		this.delay = 500;
		this.lastMessage = System.currentTimeMillis() + this.delay;
	}

	public Example(int delay) {
		this.delay = delay;
		this.lastMessage = System.currentTimeMillis() + this.delay;
	}//Ignore above, except that the lastMessage value is being set to System.currentTimeMillis() + the value of the delay

	@Override
	public void run() {
		while (true) {//Ignore
			if (lastMessage < System.currentTimeMillis()) {// This is what you should be worried about
				lastMessage = System.currentTimeMillis() + delay;
				System.out.println("Hello. This is a timed message that sends every " + this.delay + "ms.");
			}// This is what you should be worried about
		}//Ignore
	}

}

}[/code]
Basically, your timer comes down to these small components:

private long lastMessage; private long delay; // if (lastMessage < System.currentTimeMillis()) { lastMessage = System.currentTimeMillis() + delay; }

Every time the “lastMessage” variable is less than the value of the System’s current time in milliseconds, it executes the code in the if statement. For the timer to work properly, you must reset the “lastMessage” variable (In this example) to the System’s current time PLUS how long you want to wait.

Note: Server’s run on a 600 millisecond ‘tick’, so you will NOT be able to time it down to the last millisecond. You will have to do it based on ticks (1 tick = 600ms, 2 ticks = 1200ms, so on).

The reason the arrows still hit at the same time is because you threw all the code into the case…

If you recall, I stated that the server runs on a 600ms tick.
You need to use that in order for this to work.

This is usually frowned upon, but for this to work you need to use the process() method of your Client.java

The reason for this is because that method is executed every tick, so the timer will be checked each tick. You will need to make sure the player is using the special (You have 2 booleans for that… “usingBow” and “dbowSpec”) for the bow, otherwise every player will be shooting arrows as long as they’re in game.

What you should do is create a method for the timer in your client class, and then use it in the process method.
The first two things you should check are the booleans. Then check that the player is still in combat and has arrows.

As for making sure it doesn’t run more than twice:
if you declare the long “lastMessage” in the Player or Client class, you can do something like this:

if (c.lastMessage < System.currentTimeMillis() && (c.lastMessage > -1)) { c.lastMessage = c.lastMessage == 0 ? (System.currentTimeMillis() + DBOW_DELAY) : -1; }
When you declare a long, it will by default be 0. 0 is obviously greater than -1, so that’s 1 run. Then it checks if the value is 0, and on first run it is. If it is, it will set the value to the current time plus your delay (Clearly above 0). On second run (key work: second), it will set the value to -1, as c.lastMessage is no longer set to 0.

if you declare the long “lastMessage” in the Player or Client class, you can do something like this:

if (c.lastMessage < System.currentTimeMillis() && (c.lastMessage > -1)) { c.lastMessage = c.lastMessage == 0 ? (System.currentTimeMillis() + DBOW_DELAY) : -1; }
When you declare a long, it will by default be 0. 0 is obviously greater than -1, so that’s 1 run. Then it checks if the value is 0, and on first run it is.
