This is the very first tutorial in a series of tutorials which i may or may not finish, but this was requested by someone and im sure it will help out a lot of people, so here we go!

In this tutorial we will:

[ul][li]Learn how to load a sprite into the editor.[/li]
[li]Learn basic positioning techniques.[/li]
[li]Learn a few variables and their functions (for sprites).[/li]
[li]Learn some other skills that come in handy while using this tool.[/li][/ul]

Let’s start by explaining what Lin’s Interface Editor is and what is it good for.

The interface editor is a tool designed by Lin which modifies or edits the interface data file inside of the cache.
It can be used to very accurately reproduce interfaces or modify pre-existing ones to either be completely custom
or mimic the interfaces from later revisions of RuneScape (what I myself have utilized the interface editor for)
and repacks the data back into the format read by the client.

Before we start this tutorial, I suggest we pickup 2 tools, one being the interface editor and the other being Tom’s Cache Suite.

The interface editor can be found here, http://www.moparisthebest.com/smf/index.php/topic,427639.0.htmll

There are many mirrors for Tom’s Cache Suite, but i recommend the modified version, by k4rn4ge, the reason being
most people have very big cache sizes with all of the data packed into them.

http://www.rune server.org/runescape-development/rs2-client/tools/281941-toms-cache-suite-modified.html

Now that thats taken care of, we can get on with the tutorial.

[SIZE=24pt]Step 1: Packing our Sprites[/SIZE]

The Interface Editor grabs all of the sprite data and methods from the RS2 client and cache itself, meaning we first have to import all of
the Sprites that we need before we can start creating or editing an interface.

It is up to you to grab the sprites that you want, i am only explaining what to do with them ;).

First we need to open up our cache with tom’s cache suite. You should know how to do this.
Once your cache is loaded you will need to select the View/Edit Image Archives
and then bring down the dropbown box and select Media.

Like so:

Once you have done this, if your media.jag is not corrupted a new window will show up with a list of archives
and their names. We will be adding a new archive for our interface.

Under the list of archive names you will see a [+] and a [-] button set, these are for adding and removing
archives, we want to click the [+] :P.

Step1_01.png601×551

Once you click on the box, a prompt will open up asking for the name of the new archive. You may name it
whatever you wish but make sure you remember what you input (write the name down somewhere) because any
archive created by the cache suite becomes bugged and the name is randomised into random numbers (Just like
the top archives in the picture above) but the archive can be called by its original name in the interface editor.

Once you have named your archive, select it. Now we move to the right column, at the bottom you will see another
button set with [+] and [-], these do the same thing except instead of a prompt asking you to name an archive,
you will be given a window in which you find the sprites you want to add to the archive.

Note: Sprites must have the desired transparent areas of the sprite to be colored with the
hot pink color, this is the color read by the cache as transparency.

After you have found your sprite you press the save button and the sprite will be added to the archive, it’s name
will be replaced with the number of the position in the archive the sprite is, remember this number as we will
need it later on.

Our sprite is now in the archive and we can get on with the fun stuff, but first remember to hit the
Pack Archive button!

Close the cache suite and open up the Interface Editor

[SIZE=24pt]Step 2: Loading our Sprites[/SIZE]

Once you have the Interface Editor open, we will need to load interface data.

First, click on the file tab in the top left hand corner, and choose the load cache option, find the cache that you
just used to pack the new sprites to and open it.

The tool should pause for a moment and then it should display the modern magics spellbook, what we want to do is create
a new interface for now, select the interface tab and choose the create new option, if you wish to edit an existing
interface, you choose the select option in the interface tab and then input the interface id of choice, but for now
we will create a new interface.

The modern magics spellbook should disappear and the tree plot in the top right hand corner should go blank, this is
normal as our new interface has no child ids, go to the interface tab again and select the show sidebar background
option, this helps us see what we are doing while editing.

Take a look at the single id that shows up in the tree plot to the top right, this is our interface id, remember it!

Next we want to take a look at the content types the editor is able to apply to an interface, you will see a box under
the interface window called “Add Interface Type.”

This box has many options but this tutorial will only cover the use of the “Sprite” button.

Click on the sprite button, a small white box should appear in the top left hand corner of the interface window, this is
our sprite, it seems pretty useless at this point because nothing is defined for the sprite to display.

Before we get to defining our sprite, click on the white box that has shown up in the interface window, notice that the
white box is deselected, but it isnt deselected, you have actually selected the sprite, but we are not in editing mode!

The Interface Editor has 2 modes, emulation mode and editing mode.

Emulation Mode - This mode is used to test functionality of the interface selected, things
such as configuration ids, and hovering.

Editing Mode - Although not properly named as such, this mode is used to quickly select and
change attributes of a certain object, such as position, scale, and definitions.

To do most of our work, we will be using editing mode. Once again, go to the interface tab and deselect Emulate Interface.
We are now in editing mode.

[SIZE=24pt]Step 3: Defining our Sprites[/SIZE]

With our sprite placed and the interface editor in editing mode, we can now get down to business.

Look into the tree plot and you will see that another id has been added to the interface, the sprite should show as indented
from the main interface id, this is because our sprite is a child id of this interface. Select the sprite id in the tree plot
and you will see that our little white box shows up again, you can use this feature to easily point out which ids belong to
which sprite and other content types.

With our sprite selected, there are 2 ways of moving our sprite around, W-A-S-D on our keyboard and click and drag with the
mouse. I recommend dragging a sprite or other content with the mouse to the general area in which you want it and then
precising its location with the WASD keys.

But where do we know where we want a sprite if we dont know what it is?!

Now we will be defining what image our sprite will be utilizing from the cache.
With the sprite still selected, take a look at the miniature spread sheet of data in the bottom right hand corner, this is
where all of our defining will be taken place.

Unfortunately this spread sheet doesnt sitinguish between the different
types of content, so we will have the same number of variables in the table for any content type, the only variables that
need our attention at this point are as follows:

AtActionType
DisabledSpriteName
EnabledSpriteName
Height
ToolTip
Type
Width

These are the basic variables used in defining a sprite, there are other variables in the table used with the functions of
sprites, but this tutorial only covers the basics of sprites, more complex tutorials in the future will cover the usage of
the more complex variables.

AtActionType - This variable determines whether or not a sprite is clickable, setting this variable
to 1 well tell the client this sprite will perform something when clicked (i.e sending an actionbutton id to the server).

DisabledSpriteName - The name of this variable is exactly what its name means, this is where the
directory for a sprite is called whenever a sprite is not enabled (An example of enabled and disabled sprites would be the
sprites lighting up and being darkened out on the spellbook with the given amount of runes etc required. [not all disabled
sprites need to have an enabled sprite defined, infact most sprites have no enabled sprite defined])

EnabledSpriteName - The variable used to define what will be shown when a sprite has been activated
by a configuration id (not explained in this tutorial)

Height - This is how tall or short an area defined by a sprite will be, increasing or decreasing this
variable will not stretch the sprite itself, just the area in which the sprite lies.

ToolTip - Do not let this name decieve you, this is not the yellow tooltip which is displayed when
you hover over the sprite, this is the text that appears in the top left hand corner of the game screen when you hover over
the sprite or other content type.

Type - This doesnt specifically pertain to sprites, but it does however define what content type the
id is, changing the number of the Type will change the type of the content, from sprite to layer and vice versa.

Width - Same usage as Height, except it is how wide or narrow the area will be.

First we will be dealing with the DisabledSpriteName variable, here is where our archive name and sprite position come into
play.

in the box next to the variable name, you will enter the name of the archive and the position of the sprite, separated by a
comma, like so:

Remember that the positions do not start at 1, they start at 0.

Once you have input the archive name and sprite position, it everything is entered correctly, the new sprite should now show
up under the white box.

Note: If for some reason the sprite shows up with a pink border, this means the sprite needs to be imported
into the image archive a 2nd time under the same position, the reason is beyond me, it is just a glitch with the cache suite.

After your sprite is displayed, it is highly recommended to adjust the height and width accordingly to the size of the
sprite, if the sprite is clickable and the area (white box) does not cover the whole sprite, the parts of the sprite that
are now covered by the white box will not be applicable to be clicked by the user (in-game). Also, if the area defined is very
large and overlaps the area of another sprite, this will more than likely cause conflicts with things such as hovering and
button clicking.

Congratulations! You have successfully loaded a sprite from the cache into the interface editor, this tutorial seems very long for 1
simple piece of content, but it is a big part of interface design.

The next tutorial will focus on the functions and uses of the other types of content applicable by the interface editor.
