[quote=“Ashi, post:15, topic:543562”][quote author=Lothy link=topic=662441.msg4427774#msg4427774 date=1401656976]
C# has the same problem. They’ve provided a SecureString class to attempt to rectify it, but various APIs aren’t compatible with it so ultimately you end up converting it back to a string anyway.

Also, for your solution to have any value, you can’t start from a string literal (i.e., “Test”) and then convert it to a character array because by that stage you’ve already stored it as a String in memory.
[/quote]
Okay so how would you do it?[/quote]
You don’t. The string language feature in java is immutable and inextensible. There’s nothing to be done about it as far as being a regular developer goes.

In an ideal world there would be a way built in to the language to mark a string as being secure when you create it. It would still be a string, and therefore still usable with the many APIs that exist. Basically the only difference would be that it receives special treatment - the garbage cleaner would overwrite it immediately after it falls out of scope.

As for the guys joking about ‘security and security and security’, there’s a reason it’s like that. There are no doubt countless stories of real-world lapses in security precisely because someone thought one security ‘thing’ was enough. Of course when that wall falls, whatever happens happens.
By considering security throughout your entire network and system (i.e., from when permitted traffic enters through your firewall to when it receives an application layer response) you drastically reduce the likelihood of breaches.
