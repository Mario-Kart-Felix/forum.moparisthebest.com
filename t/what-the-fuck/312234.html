This is normal. JODE / JAD don’t behave nicely with exceptions even when properly formatted and even when they are not relevant to the bit that its trying to parse. Just strip them out. There is no value in them.

abraham2119:

whats even weirder is that if i open this method in jbe, and click “save method” (without effectuating any changes) it bitches about how BIPUSH instructions at 16, 280, and 295 require a byte to be passed (even though they are bytes). anyways, so what i did was i changed those instructions to SIPUSH instructions, saved the method, and it decompiled perfectly

JBE can’t handle BIPUSH properly. When it parses the class file it sees “BIPUSH 250” so it prints that. When it tries to parse it however it expects the integer to be in the range -128 to 127. Obviously this is in error. It should accept 0 to 256 or output -128 to 127. Using SIPUSH gets around this limitation even though it wastes the a byte.

I don’t know how loading and saving it in JBE fixes it. Probably an issue with jump offsets or something equally mundane that a byte code library like ASM abstract and dispose of. Use ASM or BCEL and run a verifier over the method.
