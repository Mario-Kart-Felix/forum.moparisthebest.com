packets:

login_packet {
         8 id; // 18 or 16 depending if you're reconnecting
         8 size; // 57 + login_data.size
        32 version;
         8 lowmem; // 1 or 0 depending if it's the low memory version or not.
        32 checksum1; // checksum of cache file
        32 checksum2; // checksum of cache file
        32 checksum3; // checksum of cache file
        32 checksum4; // checksum of cache file
        32 checksum5; // checksum of cache file
        32 checksum6; // checksum of cache file
        32 checksum7; // checksum of cache file
        32 checksum8; // checksum of cache file
        32 checksum9; // checksum of cache file
        32 checksum10; // checksum of cache file
        32 checksum11; // checksum of cache file
        32 checksum12; // checksum of cache file
        32 checksum13; // checksum of cache file
login_data data; // rsa encrypted chunk
}

login_data {
     8 id; // always 10
    32 keyA; // client generated
    32 keyB; // client generated
    32 keyC; // server generated
    32 keyD; // server generated
    32 uid;
    64 user; // user name that is encoded into a long
string pass; // password
}

string { 
array8 bytes[]; // array byte values of the string
     8 end; // the end of the string in the packet
}

bit = 1 binary unit
8-bit = 1 byte;
16-bit = 2 byte = 1 short;
32-bit = 4 byte = 2 short = 1 int;
64-bit = 8 byte = 4 short = 2 int = 1 long;
array# = array of bits # large;

Note that the format is “(bits) name // description”.
[hr]
description:
login_packet is the main packet for logging in, it uses the login_data structure after it once it has been RSA encrypted. and login_data uses the structure for streaming a string string. If you look over the login routine, you can easily see how a world switcher that doesn’t LOAD fresh on the new world is detected. The server keys can track you from one server to another, and if Jagex is smart, they should be like a phone number, a range of 4000 keys are assigned to each server. The id’s tell the server how to interpret incoming packets. Key’s A-D are used as the base key-pairs for the frame and bit encryption used in the game from then on out on “Frame” bytes for those who are acquainted with private servers, and “Opcode”, which is the true name for “Frame” bytes… Opcode being a conjunction of Operation + Code, literally meaning: The code which tells a system how to operate. They keys track you through instances, the uid’s track you through computers. On login_packet the length bits are used to extract the RSA encrypted login_packet. The version is used to verify client versions, the byte for lowmem is a boolean telling it whether or not you are a low memory or a high memory client. Lastly, the checksums assure that you haven’t edited the cache such as to make all things invisible except what you want a color clicker to see.
