Massive Process Usage
This is a massive hell no. Why does everyone on this site feel the need for all there shit to be called on ever half a second? Adding all this code to boolean process or whatever you guys use is a very poor way of doing this.
What you are doing:

if (newAnimRequired && newAnimDelay < 1) { outStream.createFrame(1); // Xerozcheez: Resets animation so we can do another one yayyyy! startAnimation(newAnim); newAnimRequired = false; } if(Climbing && ClimbStage == 1 && actionTimer < 1) { // Xerozcheez: Climbing up rocks part 1 teleportToX = absX-1; teleportToY = absY; actionTimer = 5; ClimbStage = 2; }

That is all being called every 500ms (half a second) in boolean process. You are running 2 uneeded checks there (now that i look down further into the code there are many more xero added).
A better way to do this:

public void updateAnim() { if (newAnimDelay < 1) { outStream.createFrame(1); // Xerozcheez: Resets animation so we can do another one yayyyy! startAnimation(newAnim); newAnimRequired = false; } }
There we have taken that check out of process, and added it into a method. Then where ever you would normally put shit about newAnimRequired == true;
or some shit like that you would just put updateAnim();. Basically what I am trying to say is only call on the check when absolutely necessary, not ever half a second.

A fuck load of “global objects”:
If I had a nickle for ever server that has this. When are people gonna learn that adding all these shitty ass objects not only looks like shit, but also lags like hell. What you are doing, is making it so every time a person enters an area with these objects, the client has to load even more objects then before. When you create new object training zones you are adding what, 15-20+ “global objects”? Every time a person goes into that area the client has to show 15-20+ extra objects, AND the server has to send 15-20+ more things to the client (making it even worse).

Training on objects:
Thank god this is starting to disappear. But it is a very bad practice for skills that don’t need it (IE. Combat, fletching, etc.). All this is, is a breeding ground for mass clicking. A great alternative to this is npc training (which I can see a lot of servers are doing). This isn’t just object training though. Anything that involves repetitive clicking is also a breeding ground for mass clicking.

SendFrames:
Fuck we need to get rid of as much of this as possible. Sending frames (as in replacing text as you call it) is a very poor practice. Especially when your changing the frames on something that is updating often and/or being used often (IE. Stats, the famous “Server Consoles”, etc.). This is sending this text over and over again from the server to the client and back. This causes major lag.

Commands:
The way you people do commands is horrible. I am greatly appreciative of Meiscooldude’s tutorial on this (i believe it’s the one on using if/else shit). The way you have it set up is that every time someone tries a command it checks EVERY command.
What you are doing:

if (command.equalsIgnoreCase("fuck") && playerRights <=2) { //do some shit } else if (command.equalsIgnoreCase("motherFucker")) { //do some shit } else if (command.equalsIgnoreCase("bigMotherFucker") && playerRights <= 2) { //do some shit }
What you should be doing:

if (playerRights >= 0) { // Commands normal players can use are here } else if (playerRights >= 1) { // Commands mods can use goes here. } else if (playerRights >= 2) { // Commands admins can use goes here. }
and so on. What that code I just gave you does, is check the player rights first. So if there rights are 0, it only checks the commands in that if/else statement. If there are mods (playerRight 1) it only checks the commands in that if/else statement and so on. The only downside to this, is it takes up a little more space. So what you might want to do is create a few more methods that stores the commands.

Text File Saving/Loading:
This is another very poor practice. Loading from text files creates more lag, and is very insuficient. A good way to load players, npcs, ect, SQL saving. SQL is easy to set up, and allows a lot more things (IE. registering on your site, online highscores, a good message system, ect.). Also, both are hackable. Just because there on your computer DOES NOT mean nobody else can see them. For instance, say you had a friend over to your house. He was looking through your computer and found the char files. He now has all there info. With sql you can take a lot more security measures. Like password protecting the database, encrypting the info (which can be done with both, but I find easier with sql).

Coding Conventions
wL’s coding conventions suck. Which do you think looks better?

public void noConventions() { if (yourMom == 1) { sendMessage("HI"); } else { sendMessage("YOUR MOM!"); } }

Or

public void correctConventions() { if (yourMom == 1) { sendMessage("HI"); } else { sendMessage("YOUR MOM!"); } }

I, along with anyone on this site who knows a thing or two, should agree that the second one is much better to look at. I encourage all of you learn the coding conventions. There is a tool that will work this out for you, but I am not gona mention the name (using a tool isn’t as much fun as learning is it?). This isn’t a major flaw really. But using the coding conventions makes you look like you know a little more about java xD.

Vital Flaws:
There are a good bit of vital flaws in wL that need to be fixed.

[ul][li]1 Thread Per Client[/li]
[li]Massive switch statements[/li]
[li]How packets are handled[/li]
[li]Connections are never actually dropped[/li][/ul]

1 Thread Per Client:
This is probably one of the greatest flaws of wL. It is the cause of your lag, and many crashes. Basically the JVM can only hold so many threads before it fucks itself over. Well, in wL every time a new player is made, a new thread is made. Thats why we can only handle around 50 players at one time, and maybe 200 connections total. What we need to be doing, is like in rs2dv, 1 thread holding all connections. Until you people figure this out, you will keep getting “SYIed” and crashed.

Massive switch statements:
As some of you know, wL handles all players in a big fucking switch statement. I don’t feel like looking all through my server to find this, but that IS how players are handled. It sucks, it’s laggy and there are better ways to do this. I will look for a better way to do this in the near future, but I currently don’t have time xD. This is not the only thing that is a massive switch statement. Look in the next “article” thing. Packets are also handled this way.

How packets are handled:
Packets are handled very poorly in wL. All the major things are in 1 method like this.

public void handlePackets(int packet) { //Shit to recieve the packets switch(packet) { case 45: // Just an example, lets say its action buttons case 1: // button id 1 break; case 2: //button id 2 break; } }
and so on.

What we should be doing:

public void actionButtonExample(int buttonID) {
	switch (buttonID) {
		case 1: //id 1
		break;
		case 2: //id 2
		break;
	}
}
public void handlePackets(int packet) {
	//Shit to recieve the packets
	switch(packet) {
		case 45:
		//Shit to get button id
		actionButtonExample(id);
		break;
	}
}[/code]

See the changes?  Doing that with all your shit will also get rid of the "code to large" error with parseIncoming packets.  As you add more things, you may need to add more methods to hold the action button clicks and the other shit.

[b][u]Connections are never actually dropped:[/u][/b]
Well first of all, thanks Yz for reminding me of this one.  This is a huge flaw that can not only keep your computer vaulnerable, but creates a lot of extra stress on the JVM.  Let's take a look at the logout button case:

[code]case 9154: //Log out
if (LogoutDelay >= 0) {
	sendMessage("You have to wait 30 seconds out of being in combat to logout.");
} else {
	logout();
        savemoreinfo();
        saveStats();
}[/code]
If you notice the only things that are taking place (if the logout delay isnt greater then or equal to 0) is a logout method, saving more info, and saving the stats.  Let's take a look at the logout method.

[code]	public void logout(){
		outStream.createFrame(109);
	}

If you look closely, you will see the only thing that happens, is a packet is sent to get the title screen. But nothing actually gets rid of the player. So what you have to do is close the socket basically, but I’ll let you figure that out for yourselfs ;).
