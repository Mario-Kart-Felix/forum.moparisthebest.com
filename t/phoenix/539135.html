[center][size=10pt]Phoenix is an addition onto my prior framework Aero. Phoenix has had a lot of additional work done to make it even more user friendly.[/size][/center]

[size=14pt]New Content[/size]
[list]
[li]Global Object System. The creations are parsed from an external text file.[/li]
[li]Character Customization. All changes are saved in the JSON file along-side all other data.[/li]
[li]Skilling System. An abstract skilling system has been put in place to keep programming organized.[/li]
[li]Bone Burying. Bone burying is completed and is using the latest skilling system.[/li]
[li]Core Functionality. Many minor changes have been written into the core. Im not going to list them all since most are minor.[/li]
[li]Level Up Interfaces. When a player increases a skill level the correct interface is shown.[/li]
[li]Incoming & Outgoing Packets. Many Misc. packets have been registered.[/li]
[/list]

[size=14pt]Standing Content[/size]
[list][li]Floor Items. Turn global after 90 seconds and disappear after 180.[/li]
[li]Player Rendering, Mob Rendering, Movement, Login, and Packet Reaction. (Obviously).[/li]
[li]Nuked Core. The core is written in NIO and supported 2000 players and mobs within a 500 tile radius.[/li]
[li]Banking. Supports inserting, swapping, withdrawing 1x, 5x, 10x, and a specified amount.[/li]
[li]Equipment And Inventory. Supports switching and handles 2 handed weapons. [/li]
[li]Mob support. Creations are parsed from an external text file with an easy to follow syntax. [/li]
[li]Private Messaging. Supports logging in and out and sending private messages over the network. [/li]
[li]Account Saving. Written with the JSON API for quick effective parsing of character files during run-time. [/li]
[li]Thread Pooling. The core is based around a single thread pool which reads the amount of available processors you have for an adjusted performace.[/li]
[li]Outgoing Packets. Outgoing packets are abstracted out to avoid the clutter of a giant static based file.[/li]
[li]Incoming Packets. Incoming packets use annotation to assign opcodes to specified handlers. [/li]
[li]Running & Walking. Correct accurate formulas the support changing of states while in motion (No walk-back algorithim).[/li]
[li]No global state is used, everything uses polymorphism as a substitute.[/li]
[li]String Layers. String layers are parsed from an external text file to avoid any clutter in an initialization method. [/li]
[li]Built light-weight without all the dead code and unused methods. [/li]
[li]Attribute system to avoid an access of variables. [/li]
[li]All content is current pulse driven and should be written as such. [/li]
[li]Container for Equipment, Inventory, and Banking to avoid any bulk in a single class. [/li]
[li]Teleporting and region changing. Updates for everyone within your bounds. [/li]
[li]Weapon interfaces. For every weapon the correct interface is displayed. [/li][/list]

Clawz_fury August 4, 2016, 11:35am #10
KroniKz_:

Question. What is the Attribute system?

It holds attributes like flags and values. People generally use Attribute systems because they are cleaner and they save memory. I can’t really explain it well but here, instead of:

[code=java]private boolean isFarming;

public void farming() {
isFarming = true;

 if(isFarming) {
     // do something
 }

}[/code]

you can do:

[code=java]
public void farming() {
player.getAttributes().put(“farming”, true);

 if(player.getAttributes().get("farming")) {
      // do something
 }

}[/code]

I personally didn’t really like them all that much, but I’m rethinking it now I might make my own attribute system for Asteria. Keeps things clean and prevents things like this:
