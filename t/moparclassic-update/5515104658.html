I don’t think i ever really contributed to MoparClassic, but i did want to. It highly put me off when people started changing it, adding scala and all of a sudden anyone who doesnt know scala will be put off from it. I didn’t look too much at MoparClassic source, but i did read that people kept breaking shit and changing shit and now its a mess.

I only recently have had the change to go through all this RSC stuff, but i can say in my own opinion the last good place a server was at, was probably Peter’s 2nd Darkquest (the RSCA fork that basically converted a lot of content into java plugins). That was a good point. I know that when i was apart of ProjectRSC one of the devs then added python scripts into the mix and the same thing really, scripting was only efficient to someone who knew python.

In my own opinion, a good server to start collaborating on would be to take something like an early version of ProjectRSC (after the netty implementation, but before the F2P & python changes) and work off that.

Another huge part of active development is accessibility. If people can download a server and it works out of the box that’s great. Being tied to SQL is fine, i think though by default it should be disabled and SQLite should be used instead. Maybe even bundled with a program where you can create accounts.

I think years ago when i forked mopar classic i went to compile/run it and it didn’t have a compile/launch script, or it didn’t have the right library, or the paths were hardcoded to someones computer. fix that and it was a compile error, or an incompatible java error, then you gotta set up sql, import db and overall it was a good hour or 2 of fucking around that people simply don’t want to do.

it could certainly use a bit of easyrsc treatment, without degrading any quality or adding a GUI. if i knew i could click the download button on moparclassic, launch a few files and it all just worked, i’d be keen to develop.
