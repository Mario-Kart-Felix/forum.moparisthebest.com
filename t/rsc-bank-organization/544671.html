int size = itms.size();

for(InvItem takenItem : taken) {
	size = getSize(takenItem.id, takenItem.amount, size);
}

for(int i = 1; i < added.size(); i++) { // i = 1 for the drop 1
	size = getSize(added.get(i).id, -added.get(i).amount, size);
}

It was written quite clearly in Scala… Here is how it could have been written, using some ‘syntactic sugar’:

var size = ((itms.size /: taken)((s, i) => getSize(i.id, i.amount, s)) /: added.tail)((s, i) => getSize(i.id, -i.amount, s))

That is much more ‘WTF’ to people who don’t understand Scala and does exactly the same thing. I’d also rather see either of the Scala versions in the source with a comment above it, explaining it, than see all that Java and have to figure out what it’s doing. The difference is 8 lines (with the comment) vs. 22 (with no comments) to determine if the result of a transaction is limited by the number of slots the container holds.

Also, the codebase is far better than what we started with, and anything that is open sourced. Let’s just look at the sheer number of lines that we had to deal with for the Bank, Inventory, and Shop classes…

Original:
File Lines
Inventory.java 196
Bank.java 146
Shop.java 245
Total: 587

Current:
File Lines
Container.scala 150
Inventory.scala 71
Bank.scala 6
Shop.scala 105
Total: 332

Not to mention the Scala code is written in a very Java developer friendly way. It would be a lot more succinct if it was written in a more Scala-eque way.

Now, let’s look at the differences…

The current version is thread safe; I’d love to see this done with the Java version. Most of the methods in the original version are only a few lines, to make them thread safe, you’d need to add a few more to each, as they’re not using atomic operations.

The current version allows you to easily create new contains that behave very differently. Just look at the bank class… It’s basically a single line, and has no special logic in it, except for the number of slots to allow (which is a simple if statement to see if it’s a member’s world or not). Look at the inventory class… In a total of 71 lines it deals with all the inventory specific behaviors, such as dropping items on the ground if a player can’t hold it, dealing with wielding items, and so on. It also reuses all the logic for adding and removing items from the container class. The shop class only needs to change the behavior of a single method in the container class, and that’s the equals method. Everything else is reused and deals with the logic specific to shops. The part where it’s messy is the init method, caused by part of the codebase that hasn’t been modernized and is extremely clunky. The previous version? Well, nothing is reused between them, at all, so if there needs to be a change to the behavior it needs to be changed in all the classes. In addition, it’s like someone copy and pasted the same skeleton in 3 files and then changed them as needed, so if there is a bug in one, you probably need to go change 3 files instead of 1.

The only real issue is that there aren’t many comments, but the code is pretty clear if you know Scala. The comments in the original implementation tell you absolutely nothing useful, and are written in a way that JavaDoc standards specifically tell you to avoid. Here are all the comments and what they’re commenting in the original:

Bank.java

/**
 * The maximum size of a bank
 */
public static final int MAX_SIZE = 192;

Inventory.java

/**
 * The maximum size of an inventory
 */
public static final int MAX_SIZE = 30;
/**
 * World instance
 */
private static World world = Instance.getWorld();

Shop.java

/**
 * The maximum size of an inventory
 */
public static final int MAX_SIZE = 40;
/**
 * World instance
 */
private static World world = Instance.getWorld();

Now, let’s talk about how easy it would be to extend one of the implementations other than Container, which doesn’t exist in the original. Well, in the original, everything is private or public. About half the public methods are never used outside the class. The current version? Well, things are primarily protected and public, and the only private things are things that should be private, and have appropriate accessors and mutators.

Now that we know you’re not going to be able to extends the old implementation without rewriting all the logic from scratch, let’s look at the container class. The first thing we can do is look at what it can do. Well, via simple parameters in the constructor, we can set the max size, set if everything should be stackable or not, and allow slots to have values of 0 or not. So, what type of container exists in RSC that isn’t covered by those options? None, everything you can conceptually think of in RSC as a container only needs those few options. You can even turn things like the hopper in windmills or even chests into containers and it still would work. How much work would it be to make a special container for those? Almost none. Meanwhile, have fun copy, pasting, and modifying a few hundred lines in any other source to implement them.
