lothy August 4, 2016, 1:30pm #24

^ Yeah.

You’ve got to do it in an intelligent and considered manner, and that really requires a fundamental understanding of concurrency and the happens-before relationship between memory access by different threads.

For what it’s worth, the synchronized keyword (i.e., using an object’s monitor lock) performs well when there’s little or no contention for the resource. In RSC servers, where there’s strict separation of packet handling and issuing world state packets to clients, it will often be the case that contention levels are low for all but the most sought after resources.

When reading other peoples opinions on how to communicate with something (incl. computers) in a turing complete manner, there will not always be an explanation that is acceptable. That is all I was saying–there are MANY a case where I see Java features being used unnecessarily, as do I in pretty much every language but Scheme(almost any dialect/implementation but Racket I’d say), and C(I’m unaware of a standardized version that this occurs in often).
In server daemons designed to synchronize specific things, it is often simpler to do these things in the same thread. Otherwise you’ll need to explicitly state to the computer Be aware of this and await a change,'' or similar pathology. A specific opposite of this comes to mind on a project I am working on currently in C99. I am utilizing SDL for easy cross-platform support of input, graphics, and audio. It provides me with fine-grained control over what gets executed, how it is executed, etc, while being compiled to a small native code binary for the target platform. It provides simplicity in this sense, while also providing all the means to my end. I never would have done something like this in Java, for the simple reason that it is almost always done for you in Java. You then lose the so-stated fine-grained control over execution. As CodeForFame stated, synchronizing everything is not acceptable as it is not needed for everything and is often simpler not to and involves lessboilerplate’’ code to work as expected on target machine.
