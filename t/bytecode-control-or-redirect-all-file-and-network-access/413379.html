General Programming
moparisthebest August 1, 2016, 2:40pm #1

I am trying to brainstorm a good way to do this. The goal is to have an application that forces another application to run in a sandbox or chroot of sorts without the other application knowing about it. Giving me the opportunity to accept, deny, or change all requests to access the network or filesystem on the computer the program is running on. I think what I am going to do is something like this:

Load the jar with a bytecode library, like BCEL or ASM (suggestions/examples welcome), replace all instances of “new File(…)” with “new org.mypackage.MyFile(…)” which is a custom class I would write which would extend Java’s File, therefore providing the same API, but would let me redirect a request for the file ‘C:/Users/firefoxs_password_database’ to perhaps something like ‘C:/mychrootdirectory/Users/firefoxs_password_database’. I could also fix-up windows-specific paths (like C:\ or anything using ) to work on non-windows systems.

The issue is I would need to do this for every single class in the Java Standard Library that can access a file, as well as those that can access the network. The ones that I know of are:

File
RandomAccessFile
FileInputStream

URL
URLConnection
Socket
ServerSocket
DatagramPacket
DatagramSocket
MulticastSocket

Do you know of any others? Or do you have another solution all together? Another option I’m sure some of you will mention is using -Xbootclasspath to over-ride default classes, and that might be nicer, but it can’t be done with an applet can it (without modifying the user’s system beforehand)?

Well that’s the thing. Even if you could change the code, you can’t necessarily redefine a class (noting I am not at all familiar with the instrumentation API, but I’m under the impression that only works via a java-agent, someone please correct me if I’m wrong). Working with the standard API, the only way you could pass along modified or custom classes is by using your own classloader, but the failure there is java.io.File is definitely going to be defined in the boot classloader by the time your applet is even executed. And, of course, you can’t redefine java.io.File even in a custom class loader, because every class loader must be a child of the boot class loader. If there is some workaround to this, I personally don’t know it.
