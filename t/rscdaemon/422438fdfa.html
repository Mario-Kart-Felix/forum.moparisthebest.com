I still don’t see what you’re asking. Do you mean how it is administrated, how the source is written, the content?

After maintaining RSCE for so long, it’s clear that many mistakes were made when choosing the designs of certain parts of the project. First and foremost, I don’t believe that Java is the right language to write a “realtime” (used in the broadest sense) application because of the way that the stock garbage collector operates. Sure, you can tune the GC by making it run concurrently or do generational sweeps, but eventually the application will lose responsiveness when the collector feels that it’s time to compact the heap, even if it is only for a second or two. If you’re interested, watch this presentation that describes it in detail: http://www.infoq.com/presentations/Java-without-the-GC-Pauses For that reason, I’ve been keeping a semi-up-to-date C++11 solution designed the same as the Java one, since I don’t want to purchase Azul System’s VM and C4.

The second “mistake” was the way that content was added. I was hoping for Java 7 to include closures to help reduce the boilerplate of the event system, but that did not happen. This will be the next issue addressed after porting part of the client to native code.

I could go on forever on how I’d do things differently, but instead of that, I am doing them differently.
