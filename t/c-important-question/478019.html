Fat_Cookies August 4, 2016, 2:07am #7

The rule of thumb to stick by is that each function should do one task and one task only.
In the main function you’d generally process all the command line arguments which in turn are used to call other functions that “start” your program.

For trivial short programs there is no point in creating pointless functions.

mod_taharok August 4, 2016, 2:08am #10

This is also a design preference. Personally, I prefer main to do as little as possible. I’d rather shift the main execution thread over to using “my functions” versus main, which is sort of an “out-of-the-program” function. That’s my take on it, anyway.

Getting into larger applications, this becomes easier to do. For example, if I’ writing a game, then the main function is going to handle initializing the core objects and then invoking their initialization methods, then passing on its execution to the main game loop, or block until the main game loop ends (if another thread is created for it instead, etc). Main just does pre-initialization and then starts the application. Most of the remaining work is done elsewhere.
