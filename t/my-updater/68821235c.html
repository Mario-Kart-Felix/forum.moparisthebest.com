Krichevskoy July 30, 2016, 7:11am #122

Here’s how I thought to figure this out.

Exception in thread "main" java.lang.arrayindexoutofboundsexception: -1
First line. This tells you what the problem was. In this case, the code tried to access the -1 index of an array - which cannot exist - so you've probably got a null value or other "can't find it" error happening somewhere.
at org.apache.bcel.generic.constantpoolgen.setconstant(ConstantPoolGen.java:644)

The exception was thrown in BCEL’s ConstantPoolGen class, on line 644 of the source code. Odds are that this is not a BCEL bug (though they do happen occasionally), so it’s best to move on until you find code that you’re working with.

at com.bawt.update.ClientPatcher.hackmouse(ClientPatcher.java:391)

So this is the method that called the BCEL method that threw an exception. Looking at line 391 of ClientPatcher, I saw this:

cpg.setConstant(cpg.lookupUtf8("mouseClicked"), new ConstantUtf8("_mouseClicked"));

From the javadocs, I know that lookupUtf8 returns -1 when it can’t find the string

lookupUtf8

public int lookupUtf8(String n)

Look for ConstantUtf8 in ConstantPool.

Parameters:
    n - Utf8 string to look for 
Returns:
    index on success, -1 otherwise</blockquote>

So the call to setConstant at 391 is trying to place a new ConstantUtf8 entry at constant pool index -1 instead of replacing the constant pool entry for “mouseClicked.” Index -1 doesn’t exist, so it throws an exception.

In short, the problem seems to be that the string “mouseClicked” isn’t where the updater is looking for it.
This might mean that Jagex has moved some methods or caused another class to implement an interface with “Mouse” in the name, which apparently makes the updater register that class as the mouse class with no further checks.
