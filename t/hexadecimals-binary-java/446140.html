It all depends on how you want to represent the data. Hexadecimal is useful with colors (RGB) because it’s split into 3 different channels. It’s a lot easier to tell that 0xffff00 is red and green, with no blue, than 16776960 is. Decimal is useful for most other things (sequences of animations, items, etc). Binary notation is only available in Java 7, so that’s something that you must consider if you add it in your code. It does however make bitmapping easier.

Thanks guys, what else would they be used with if not RGB? Like in the 317 protocol, it’s used for the update flags. 0x1 0x2 0x10 0x20 and so on.

Same reason. It’s to mark the bits. Imagine the update flag like so

00000001 = 0x01
00000010 = 0x02
00000100 = 0x04
00001000 = 0x08
00010000 = 0x10 (16)
00100000 = 0x20 (32)
01000000 = 0x40 (64)
10000000 = 0x80 (128, or -127)

It’s a simple way of sending multiple true/false values to conserve space (instead of 8 bytes, you use 1 byte with 8 values).

As an example, I’ll show you real code I’ve written using the various notations:

private static String decode_utf(int len, ByteBuffer data) throws MalformedClassfileException { char[] code_pts = new char[len]; int ptr = 0; while (ptr < len) { int x = data.get() & 0xff; if ((x & 0b10000000) == 0) { code_pts[ptr++] = (char) x; } else { int y = data.get() & 0xff; if ((x & 0b11100000) == 0b11000000 && (y & 0b11000000) == 0b10000000) { code_pts[ptr++] = (char) (((x & 0b00011111) << 6) | (y & 0b00111111)); } else { ... } } } return new String(code_pts); }
Depending on the context, it can help to express integral values using hex literals or what have you. As seen here, it is much easier to understand how UTF-8 encodes characters using binary literals, while using hex 0xff is more concise and equally as descriptive for treating data.get() as unsigned, just as 0 is fine for the first if conditional.
