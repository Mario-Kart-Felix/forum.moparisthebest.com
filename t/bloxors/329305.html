frank’s idea was using a lazy region quadtree (c|p gave me the adjective ‘lazy’ and i borrowed it cause i’ve been wanting to use it to describe the quadtree but i’m a noob and couldn’t think of it) and then doing a breadth-first search down it until the goal state was hit.

ok… a region quadtree is a prefix tree where each node has four or no children, and it is lazy because we are building to it as we go as opposed to working with a finished product. from any state in the game there are four possible future states (unless that state is ‘dead’); they are ‘up’, ‘down’, ‘left’, and ‘right’, based on the key you hit.

take a look at level 1:

from that root game state, there are four possible future states – up, down, left, and right.

the above image demonstrates the idea of future states; going up will result in being dead, going left will end up in dying, too, while left and right produce ‘promising’ states (i use the term promising loosely). to describe this as a traditional tree:

notice the two dead-end nodes? there is no point in following them, so our next ‘pass’ in the algorithm will work on the two ‘ok !’ nodes which could potentially lead us to the victory. this type of algorithm is called a breadth-first search algorithm. another way to look at it in the traditional region quadtree view is as follows:
