It’s completely fine the way you are doing it. It would just seem like a name you would use if you had multiple methods relating to the same thing, such as:

create3DSpriteIn3DWorld
create3DSpriteIn2DWorld

It would seem appropriate to use the very specific naming then, as it might be a vital component to the rendering process. But then again, there are several factors you have to take into consideration, such as:

[ul][li]Application scope[/li]
[li]Parameters - (are the methods going to do essentially the same thing, but require different formulas); such as render(Shape polygon) or render(Shape ngon); both obviously render shapes, but the design pattern for doing so would be different. This would allow you to pass an argument inside of another method based on the required shape.[/li]
[li]Object-orientation - it would definitely benefit you more in the end to think of all of the components of a given object, rather than thinking of it as one whole thing.[/li][/ul]
