I’ve been working on this for the last few days, as I’m waiting for Silab and Zerofreeze to get back before continuing with the 503.

Ignore itemdefs.txt - that’s just an easier to parse version of the configuration files used by the winterlove servers; I’m yet to parse it into a usable binary file format.

Walking works, public chat should work, BinaryFileManager loads and saves player information to binary files, etc.
Plugin packet handler implementations can be found in plugins/packethandler. Some are done, others have intrinsic variables that are partially self-documenting - if you’d like to write and contribute to any of it, be my guest. I’d like to have most of it done by Monday afternoon.

The only other thing I can think to mention is that ItemList needs a rework to allow for some of the functionality that I’ve forgotten (ie, swapping AND inserting items before other items in the bank).

Download it from here: http://www.MegaShare.com/470077 - The password is L317%40mscp

Oh, and tell me what you think of the definition classes. I’ve given them builder classes so they’re completely immutable after construction, which means that, should anyone decide to make the server multi-threaded in the future, they’ll be completely thread-safe.

TeleNubby July 31, 2016, 12:57am #2

Hmm… Is it just me or does this look like a “expanded version of RS2D”? Either that, or you heavily based your ideas and most namings after it.

I see that in the file logs/out.log you’ve set it up so it dumps the same line that is printed to the console. The only problem with that, though, is the time logging. When reading the logs directly on the console, you can know how long the server has been running. But when reading it from the log file, knowing how long the server has been up isn’t very helpful. My suggestion is to keep it the way it is when printing the log to the console, but when saving it to the file, you should use the time of day instead (or both, if you wish). Just a preference of mine.

Another point of mine, why do you have the JMF library in your lib/ folder. If the server simply uses the console and no graphical user interface, then why include the Java Media Framework in the package? Did you have plans earlier on about using it? If so, let’s hear it. That is, of course, unless I missed something.

Overall, it looks quite good. Never knew you knew anything Java-related until I had that “sneak-peek” at your 503 server when I accidentally had access to it. That, and this, make me want to finish my server (at the moment, 503-based, but doesnt mean I can’t switch). I took a two-thread-total approach, rather than a MINA or any other thread-per-client/packet approach, which I will find out eventually if it was worth the hassle or if it was just another dumb mistake of mine.

lothy July 31, 2016, 12:58am #3

Heh, nobody said it was from scratch.

The goal here is twofold: develop a ready to play out of the box solution, and develop a solution that is as secure as possible, removing the chances for people to coax others into running dangerous code. I presume you noticed the MoparscapeSecurityManager class. I haven’t implemented it yet, but it’s on the TODO list.

Lots of the code (or at least some naming) was taken from RSCD/RS2D/etc; I see no reason to redevelop the wheel. However I did tweak bits and pieces to my liking, as I believe some of the RS2D code is sloppy.
Zerofreeze has already contributed, as the configuration loader was taken from the Hybridscape software that we are writing.

As for the Logger, I’m not overly concerned about it. It’s just there for convenience (mine more than anyone’s) at the moment.
