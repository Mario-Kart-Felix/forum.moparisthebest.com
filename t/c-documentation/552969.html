xx_matty_xx August 4, 2016, 3:55pm #2

That’s funny, I experienced the complete opposite in all of the places I worked after graduating. Maybe it’s supposed to be self-documenting…

One thing I would say is that you should try making your comments more concise and think about what another programmer would want to know at that level. For example, instead of “A bool to check if the transmission was successfull. False when flooding or when the Aucxis service is offline.”, maybe just “True if transmission was successful, otherwise false”. If the programmer needs to know this information it’s too much for a bool imo, and should be a more self-documenting return type.

I think naming should also be a bit more consistent? Doesn’t matter here so much, but I think in general it’s good to be more precise especially when you have more complex stuff it’s annoying for other people to work it all out. For example, the function is called “Send” but the comments refer to it as “passing”.

Other than that it seems pretty good - I can understand what the code is doing, and it goes into just the right about of detail… explaining things that aren’t immediately obvious by reading the code, and without going into detail about irrelevant shit that I could just work out myself if I really wanted to.
