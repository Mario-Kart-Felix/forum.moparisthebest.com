Want to record field modification? Use Instrumentation to redefine all the classes to call your method or however you want to implement it. Costs: the time it takes to loop through all classes. If the client has been running for a while some bits would have been JIT compiled and that would have to be redone. After 5secs or so the client should be running at the speed it was before the hooking. Only difference is the call to your method which should be very quick, too quick to notice.

There is no point using native when you can do something like this.

When you request events with JDI, you can specify whether or not it blocks the entire JVM (SUSPEND_ALL), the relevant thread (SUSPEND_EVENT_THREAD), or nothing at all (SUSPEND_NONE). It queues automatically for you (there’s no other way to receive an event; “There is one instance of EventQueue assigned to a particular VirtualMachine.”). So, I suspect the slowness is simply due to JDI being an abstraction of JVMTI. JNI is not very fast, from what I’ve heard.

You should learn c to better understand the ultimate foundation of java. There’s an old quote that resurfaces every now and again at the client sites I work with. If you know c, java is simple.
