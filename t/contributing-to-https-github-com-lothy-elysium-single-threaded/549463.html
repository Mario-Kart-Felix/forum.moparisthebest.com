I’m told that at least one person has expressed interest in contributing to this project. That’s fine.

lothy August 4, 2016, 2:18pm #5

I’m actually in the process of refactoring some of the components back into the Player class. I’ve also set up continuous integration using travis-ci.org.

It isn’t about importance though. I feel there were a few reasons MoparClassic failed:

  1. People were given direct access to make commits.
  2. These people then started making architectural decisions that they shouldn’t have without consulting others (such as introducing scala everywhere).
  3. No unit testing requirements.
  4. No code review, or at least no genuine code review.

In short, the project succumbed to poor governance and quality control.

imthenull:

Just curious, how stable is this project? Is it functional at all?

Login, walking, picking up and dropping items, trading, dueling is partially done (except for the bit where they actually enter combat), public chat, command handling.

Npcs are loaded from XML, as are objects. Keep in mind a lot of this stuff has just been ported from RSCD with minor refactoring. I’ve decoupled packet decoding from packet handling. I also plan to decouple scriptable content (such as using items on each other to produce different outcomes, eating food, etc) from the packet handling where relevant. There’s still a fair bit to do in this regard.

-Nothing:

This wouldn’t really provide much performance over RSCD with few upgrades.

False. The changes to the underlying network code alone should wipe the floor with RSCD (for example, RSCD makes many syscalls to write and flush packets, and it also doesn’t make use of buffer pooling and so on).
The event handling stuff should also perform better too for very large numbers of events due to the use of priority queues and event sorting.

lothy August 4, 2016, 2:18pm #17

Check out the new script manager:
https://github.com/Lothy/elysium-single-threaded/tree/master/src/main/java/org/moparscape/elysium/script

The idiom for writing a script is quite simple:

        ScriptManager.registerItemOnItemScript(1, 1, ScriptManager.PRIORITY_DEFAULT,
                (player, itemA, itemB) -> {
                    // Perform your task here.

                    // Return true to indicate that other scripts for this event, if they exist, should run.
                    // Return false if execution of scripts for this event should terminate.
                    return true;
                });
