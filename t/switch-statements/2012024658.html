Well, one way to ‘use Strings’ in switch statements is to decode them into longs. There are big limitations here, as a long can only hold so much. But if you’re working with few-character Strings you should be good.

String switchString = getMessage();
switch(java.lang.Long.decode(switchString).longValue())
{
       case Long.decode("test").longValue():
              System.out.println("getMessage() returns 'test'");
              break;
}

But I dont recommend the above solution. Its inefficient and most likely slow. Try using enums. (http://www.java2s.com/Tutorial/Java/0040__Data-Type/Switchstatementwithenum.htm)

I was wrong about hashcode() being runtime dependent. Looking at String it appears it caches a hash of the characters in the string. As for the Long.decode() it wants a String representing a number “-234” is an example so passing “test” will stuff up. If you do “test”.hashcode() it should work.

The hashcode method will also be faster than either intern() or the .equals() methods. Equals() will iterate through the characters looking for equality and this will have to happen for every case you put in the ‘if else’ blocks. Hashcode will only need to iterate once to compute a sum and then it can do a TABLELOOKUP which is much fast than ‘if else’. However long intern() takes it will most likely be a while. Its implemented in native code which will incur a penalty as you switch from one to the other. The code behind it will have to then search through all the strings used in the program and determine which one is equal to the one you give it. Ironically it will probably be optimized with a hashcode type arrangement. Finally it will return the one you want and then we get to go through the ‘if else’ blocks. This is slower than a TABLELOOKUP instruction or otherwise the TABLELOOKUP instruction wouldn’t exist!

There is a possibility of a hashcode collision but I wouldn’t worry about it. Because String.hashcode() doesn’t change between platforms/charsets/runtimes any collision will not be a transient bug and so easy to pick up also the chances of this happening is like the chance of winning the lottery 3 times in 3 weeks.

*Hashtable[br][br][size=1]Posted on: April 24, 2008, 09:05:57 AM[/size][hr][quote=“supamagier, post:27, topic:201202”]off topic question: What is hashcode()? :S[/quote]

[i] The general contract of hashCode is:

* Whenever it is invoked on the same object more than once during an execution of a Java application, the hashCode method must consistently return the same integer, provided no information used in equals comparisons on the object is modified. This integer need not remain consistent from one execution of an application to another execution of the same application.
* If two objects are equal according to the equals(Object) method, then calling the hashCode method on each of the two objects must produce the same integer result.
* It is not required that if two objects are unequal according to the equals(java.lang.Object) method, then calling the hashCode method on each of the two objects must produce distinct integer results. However, the programmer should be aware that producing distinct integer results for unequal objects may improve the performance of hashtables.

As much as is reasonably practical, the hashCode method defined by class Object does return distinct integers for distinct objects. (This is typically implemented by converting the internal address of the object into an integer, but this implementation technique is not required by the JavaTM programming language.)[/i]
