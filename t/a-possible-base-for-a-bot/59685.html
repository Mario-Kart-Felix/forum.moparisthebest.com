Alright, krypton and i have long since abandoned the idea of making a bot because we are both lazy and cant see the point of it much anymore, many of my ideas have changed since we stopped

the system would work in three parts, it can be considered similar to arga; the updater, the slave and the master. though early plans were to update on runtime, but was abandoned due to it not having enough benifits

the updater would download the jar, insert any accessors + interfaces and override/patch and “undesirable” methods. after this, it would then be dumped as a custom formatted file containing information on the classes and their raw bytes

the slave itself would be based around a java agent. it loads the binary file, parses it, then as classes are loaded the byte arrays are swapped into place of the current class being loaded

the master would be the front end for all slaves, possibly even on other machines. it would be a console-styled interface, accepting commands to control the slaves and dispatching the commands using RMI

nothing done so far in the area of input, though its rather obvious what you have to do

ill attach what little we have for the sake of learning from other peoples mistakes you may have to fix some of the imports, as i slapped this together from two different projects

byte July 30, 2016, 5:13am #5

I love the solution. I hate the way you implemented it. -javaagent gives you the ability to load and instrument the RS2 class files through the Jagex website eliminating the ‘download the loader jar store it here and do that with it’ rountines.

I honestly believe that the most robust bot solution would contain 3 modules:

[ul][li]An instrumentation agent able to locate bytecode patterns (BCEL).[/li]
[li]A debugging agent such as JVMTI (maybe even JDI in order to avoid the dreadfully slow JNI calls) which uses the data found by the instrumentation agent to retrieve client data on the fly and pass it on to the next module.[/li]
[li]The front end module which caters to the user and runs plugins all while communicating with the debugger.[/li][/ul]
