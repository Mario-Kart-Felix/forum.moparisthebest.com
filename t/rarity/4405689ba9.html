Lin August 3, 2016, 2:37am #58

Well, I didn’t get to do much work today due to real life stuff as usual, so the project pretty much remains unchanged. I fixed the tiniest of issues with ground items, where the X check would be <= 60 whereas the Y check would be < 64, and I only know that because I was looking at player updating. I also removed connection queuing with the server because it really serves no purpose other than to waste a thread.

Now, while I didn’t actually work, I did some benchmarking and researching some other networking libraries I could use. Sadly, the only one I could really find was Lidgren, which used the UDP protocol so it was out. While benchmarking, I found player updating performance really took a big hit. 251 players in the same region takes a whopping 300ms, and while there isn’t any player processing lag, for some reason updating is taking way longer than it should. Here’s my entire updating engine in a nutshell.

Firstly, there’s a world system that contains a 3D array made up of 32x32 regions. Each region contains a tile dictionary, which is composed of tiles indexed with RegionPositions. RegionPosition is just a structure containing a 3D point. Each tile contains a hashset for entities, a hashset for ground items, an object, and location properties. The reason I use dictionaries/hashsets is for fast indexing and memory saving.

Now, for player updating, to add players, it cycles through the last calculated in-regions for the player, which are calculated when the player moves. All 4 regions are looped through, and then so are all of their tiles, and then so are all of their entities. There won’t be any empty tiles that are looped through unless they contain either a ground item or object. On top of that, I’m checking if the near players list already contains the player ID.

I think the problem here is I’m not adding players only when they walk or even caching anything. When I do the math, that’s potentially over 60000 checks toppled by 251 O(n) operations. I think I need to think of a better system here, because I’m not getting the results I expected. More on this tomorrow.
