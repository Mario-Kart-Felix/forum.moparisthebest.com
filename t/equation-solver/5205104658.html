It’s a good start, but you can definitely build on it. As was mentioned, it would be cool to allow users to input the equations directly, automatically figuring out what variables to extract. This isn’t as complicated as it seems. Just strip the line of all whitespace, then extract the coefficients assuming only addition or subtraction separates the terms, plus there must be an equals to a constant at the end.

While you’re there, why stop at 2 equations? If you are familiar with how reduced row echelon form works, it makes it fundamentally easy to solve systems of n equations using a n x n+1 matrix. The algorithm for solving reduced row echelon form can be broken up into 2 processes, a top-down process for putting a matrix into echelon form, then a bottom-up method for putting it into reduced form. You use 3 elementary row operations (swap, multiply by a non-zero constant, and add a constant-multiplied row into another row), and deciding which to use can be as simple as checking the numbers under (first part) or over (second part) the pivots of each row. The pivots are always along the diagonal, so finding their positions is pretty easy, too. Then you know all the values left of a pivot in each row must be zero and all the values above a pivot must be zero. Then you can just extract the last column’s values, which reads out the solutions of each of the variables (unless there is no solution, in which case the left side of the matrix has a row of all zeros or an impossible solution (wherein it ends up being 0x+0y+…+0z=1 or something)). This is the process of gauss-jordan elimination.

If you do end up taking that route, as far as parsing is concerned, make sure that you ORDER the variables in each equation to match the other equations (ie, all x’s could be first, then y’s, then z’s, etc.) Otherwise gauss-jordan elimination does not work.

Try to use local variables more often, putting everything in the global scope is rather ugly and can be detrimental to memory usage.
Also, use better variables names, not just for others, but also for yourself, the less confusing variable names you use, the easier it is to program.

Also, the YI variable you declared will always evaluate to the same value in your printer() method, the way you’ve done it now ensures the expression gets evaluated once, when the class gets initialized.

Just a little bit of nitpicking, but whenever you use javadoc, every single @-tag should be on it’s own line, this piece of jdoc

[code]/**

  • Solves equation with the gauss method
  • @author Strucker, @author alejandrou7
    */[/code]

should really be

[code]/**

  • Solves equation with the gauss method
  • @author Strucker
  • @author alejandrou7
    */[/code]

Well, I would be careful in regards to completely zero rows, for starters. If I’m not mistaken, you will have a divide by zero problem in your code. You also don’t need to use doubles for something like this, unless you really feel the need to. floats are more than likely to produce the precision you need and, if you’re running in x86 mode (on a 32bit architecture or at least with 32-bit Java VM), they will be faster than doubles in a lot of ways.

Otherwise just quickly glancing shows me something sppt pointed out. You really want to use local variables as much as you can, especially for time-crunching code. You could start looking up other solutions to gauss-jordan. Some are really neat approaches to solving it. People have figured out ways that you’d never think of, so it’s a good learning experience.
