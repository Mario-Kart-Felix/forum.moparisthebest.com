The goal is a content system, not one server for all to use
If we have learned anything from developing and releasing countless well-written, stable, efficient, and relatively content heavy server bases (i.e. Osiris) it is that RSPS hosts are unwilling to adapt to change. It is understood that it is human nature to resist change, so this is not unexpected. The goal is then to develop a solution that will work for those unwilling to submit to changes as large as an entire server base overhaul.

An abstract content machine
Java proudly boasts the “write/compile once, run anywhere” philosophy. I believe that we can apply the same concept and create a RSPS content machine just like Java has the JVM. This machine would be complete with a complete and consistent specification including an API for programming RuneScape content. The machine would then need to be written for (or ported to) all of the popular server bases - rs2d, Hyperion, Apollo, RuneSource, Project Insanity, Deltascape, Osiris, etc. and content can be written once and run anywhere.

Ideally it should be a drop-in JAR file with as minimal code modification as possible. We may need to use java.lang.reflect to hook into the core of a server if it does not hurt performance too much. Remember the goal is to make this as easy as possible.

The API
The API should be simple yet very powerful. It should be highly event based, but also with player action queuing (to allow for pseudo-completions) and other features that will make developing content more like creative design and less like arm wrestling. As Taharok stated, it should fully support and even recommend the JSR 223 specification.

Package management
Hell yeah we should write a package manager for the content system. It should be plugin based as well. It should have version checking and automatic updating, with both a CLI interface and a GUI (ugh) as well. I’m not sure how submissions should be moderated if at all, but those are all minor details.
