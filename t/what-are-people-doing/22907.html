moparisthebest July 29, 2016, 11:51pm #2

Well, my current plan is to write a bot when this semester lets out, I have exactly how it should work in my head, but as you all know everything is easier said than done.

My plan is to not modify the client at all, but using either reflection or bcel (or both?) to output a class of Accessor Methods every update with all of the various functions needed for a bot (getNearestNPC(int id), getHP()…) This part, at least as far as I am concerned, is the hardest part, since you have to keep up with all of the updating.

The difference bewtween your AccessorMethods class and that from previous bots would be this:

[ul][li]You would not call any methods in the client, particularly because of the variable incrementing on every method called. If you find that you must do this, you would either decrement the value of the integer incremented after the call, or copy the code contained within the needed method into AccessorMethods.[/li]
[li]Similar to SCAR, the methods would return the on screen coordinates of the item/object/npc, and not any ‘Tile’ info that a human would not know. This is what is needed to click on things in a human fashion[/li][/ul]

After you have that the rest is rather simple.

  1. Emulate a true browser as closely as humanly possible.

[ul][li]Implement a custom SecurityManager and try to have it exactly like the java plugin’s default for unsigned jars (or signed jars if you want to deal with that headache) a decent reference for what the applet can or can’t do is here. [/li]
[li]Grab the url for the loader jar from the runescape website so that the session id stays intact. And then, using a URLClassLoader, load loader-sessionid.jar from the world you want to play on and start loader.class the same way a browser does.[/li][/ul]

  1. Emulate user inputs correctly

[ul][li]Send all mouse movements, presses, and releases. Use a good algorithm to generate your movement path, please don’t ever move in straight lines :P[/li]
[li]Send key events in the proper way too.[/li]
[li]Make sure you hold each key/click down for a random amount of time as well, make everything as random/human as possible.[/li][/ul]

  1. Use a good scripting system.

[ul][li]This, unlike some past bots, will run in its own thread, much like SCAR does. So you will have to ask it about events and for info instead of being told.[/li]
[li]Personally I see nothing wrong with Aryan’s scripting system, I would probably just use it. (implement text scripting in addition if you wish ;))[/li][/ul]

  1. This is by far the most important step:

[ul][li]Release it as OSS (Open Source Software). This speeds up development and gives a practically endless script base.[/li]
[li]Personally, I may not use the gnu/gpl license, or I would put some further restrictions on it, one being that it cannot be sold.[/li][/ul]

That is the basic plan, and I’m sure it will need changing before the finished product comes out. But that is how I envision it anyway.
