If you’re continuously having to tell your players to “go download the latest jar” or “i updated the client go get it” this is probably for you

To save bandwidth (although small) and to increase loading times I have a suggestion.

Instead of it from loading the entire JAR at startup from the URL and starting it up. Try this small startup method. (I used this for a easy and simple desktop loader on an old RSPS and even caried it over to other java products I worked on.

Step 1: (Getting and checking current installed and newest versions)
1 - Check if version.txt exists within the cache directory. If not, assume this is first startup and move to “Step 2”.
2 - Load version.txt within the cache directory (setable via a string) and then compare that to a version.txt retreived from a URL.
3 - Compare the two, if they match, procede to “Step 3”. If not, continue to Step 2:

Step 2: (Getting the new JAR)
1 - Check if old JAR exists, if so, delete it.
2 - Start downloading new JAR to the cache directory.
3 - Once new JAR is done downloading, (optional) check its MD5 with one stored online.
4 - If everything checks out, delete old version.txt and make new one with the version now installed. Then move to Step 3.

Step 3: (Starting the new JAR)
1 - Check to make sure the JAR exists, if so, check its MD5 with one stored online of newest version.
2 - If it doesn’t match, backup to Step 2 (Maybe have the MD5 check also with the version.txt checking, but this is how I did it)
3 - If it matches, start the JAR like normal and continue all normal operations.

Shouldn’t take much more work except a “JAR” directory or somewhere on your site to store the JAR, a version.txt, and a JAR.md5 that all can be stored on a website, NOT dropbox. To make updating very easy I created a simple PHP page that I made a updating program in JAVA that would upload the new JAR to the proper place (deleting old one first) then open the PHP page to set the MD5 and new version.txt strings.
