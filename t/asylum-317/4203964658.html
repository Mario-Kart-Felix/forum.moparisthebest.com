Update: 11/11/11 will be the hypothetical day for alpha testing. I have spent the last bit working on client some more and more specifically the update server (client and server sided). I have pretty much fully refactored the 317 system, and I have working server sided for an un modded 317 client. However, mine doesnt fit that category, so I have had to make the following changes, and I will probably just rewrite the client and server sided so it better suits my purposes.

Changes made
Previously the system had requests made and sent with.
Cache index - 1; so index 1(models) was sent as 0. Since I use the update server for sending the initial game files, I modified it so index 0 (configs in 317) is 0, and index 1 is 1, etc. Additionally, I added a byte to the server sided to determine file compression since the files in index0 use bzip2 compression, whereas the other files use gzip. I plan on also modding the protocol so that files are sent in once at the whole time rather than in chunks, this is debatable though, I havn’t decided completely yet. Also, the priorities for the cache requests have been changed.
Before, there were 3 priorities which were based around whether the client was logged in, and other stuff. This has been changed so that certain indexes are prioritized client sided, with whether the client is logged in also playing an role in the value sent.

Server sided, the update server is simplistic, and uses the same opcodes as before. Currently, requests are added to a PriorityBlockingQueue, where the comparator compares the values by their priority. I use a threadpool to service the requests in this queue.
