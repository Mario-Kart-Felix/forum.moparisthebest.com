	Asylum is as much a client project as it is a server one. The client is based off of Galkons 474 look alike. All data with the exception of sprites is cache packed.

There are currently two builds of the asylum client.

The first is:
474 gameframe, hiticons, autocasting like rs, 602 animations, items, npcs, gfx, and models. The objects are from the 508 revision. The only non 317 maps added are godwars and Warriors guild.

The second is the unstable build:
This client is based on the last version made of the first client and contains the 474 gameframe, hiticons, autocasting like rs, 602 animations, 377 maps, 650 models, items, maps, npcs, animations, gfx, and objects cache packed.
The client also uses a modified version of the 317 on demand protocol, for file updating instead of the versionlist like 317, it uses a system more like the new engine cache format. The client does not use Jaggrab or the Http servers at all.
Note: Even though the data is cache packed, I have not yet done the headers to load the data for anything besides items.

The server is based off of riotscape. The server uses netty for networking, and features A* pathfinding.
Many of the original flaws and problems with winterlove are completely redone and fixed including but not limited to:

Thread-per client.
Updating.
Walking.
Excessive use of process().
The way packetbuilding/decoding was done.
Ton of un-needed variables related to players and Npc’s.
Additionally, A lot of the excessive instancing of pointless classes has been removed.

Changes made to riotscape before starting:
Removal of Grahams old event manager and the python scripting system. Removal of the object managing system. Implementation of javascript scripting. Implementation of the Tick system that I have been using for months based on the same theory as used by Tickables in google and by graham in his TaskManager he released.

The Server is comparable to the bastard son of hyperion with a winterlove *****. I’ll be honest when I say that at least half of the original changes were made based on how hyperion has influenced my overall concept of server design.

Code taken from hyperion, either by inspiration or just ripped:
Location.java (ripped, and more methods later added on).
Some methods were used for Entity.java, others are self made.
The region system is based around the one from Hyperion, however, Regionmanager class uses a single array of Regions which is organized so that it is numbered from right to left (and then up a row and then right to left and then… etc).
PacketBuilder/ Packet java (before the netty switch).
Event Manager.
Original Update server.
Pathfinding (later modified by myself).
