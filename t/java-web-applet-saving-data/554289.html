With that description, it actually sounds like PHP can do exactly what you need.
Add in MySQL for saving the data, and you’re good to go.

To create a “manager” account in the MySQL, you can just have a flag for their rights. Ex: if set to 0 (FALSE) they are not a manager and cannot edit data. If set to 1 (TRUE), then they can. You can create a standalone page just for the managers, that checks the PHP session at the start, and the user data (Like if they’re even logged in/are a manager). Don’t display the contents of the managers page if they don’t have the rights for it, in any way.

Being a little more complex, you could make a single page that holds all of it (Login/registration/managing), however, you’d have to make sure the person can only see what they’re supposed to (Not hard, but not easy for a beginner).

Mopman August 4, 2016, 5:13pm #19
Bowser jr:

The simplest way, if PHP can’t do it for you, is to trim away anything from the input that isn’t an alphanumerical character

This is bad advice, imo.

Firstly, what is an alphanumeric character? Are we limiting to just basic ASCII alphanumerics, or are we considering Unicode?
Secondly, how do we ‘trim away’ everything else - can you be sure there aren’t bugs in your method for doing that (ie. null bytes and eregi in old PHP code).
Thirdly, it doesn’t stop all SQL injection anyway (think ?id=1 or 2 etc.)
Fourthly, you can’t always only allow alphanumeric obviously…

The solution is to use parametrized queries, and nothing else is acceptable.

Though even that is hard with PHP, because PHP likes to fall back onto emulating prepared statements and opening up potential holes there too. But it’s better than trying to ‘filter’ data which is doomed to failure.
