I have an Entity class and I have the Player and the Enemy as subclasses of the Entity class.

The Entity class has the variables x, y, and z. Should these variables be private or protected? Is it more OOP-like to access these variables from its subclasses using getters or to access them directly?

Ex:

[code=java5]
public class Entity {
private double x, y, z;
public double getX() {
return x;
}

 public double getY() {
      return y;
 }

 public double getZ() {
      return z;
 }

}[/code]

public class Player extends Entity { private void doStuff() { player.move(getX() + 50, getY(), getZ()); } }

OR

[code=java5]
public class Entity {
protected double x, y, z;
public double getX() {
return x;
}

 public double getY() {
      return y;
 }

 public double getZ() {
      return z;
 }

}[/code]

public class Player extends Entity { private void doStuff() { player.move(x + 50, y, z); } }

zyle1992 August 4, 2016, 1:57pm #8

To conform more to OOP, I would encapsulate that as it’s own class. Location, Position, Tile, Point, etc.

public final class Location {

    public Location(final int x, final int y, final int z) {
        // NOTE: if there are any restrictions on the values (ie. must be non-negative)
        // do that here.
        this.x = x;
        this.y = y;
        this.z = z;
    }

    public int getX() {
        return x;
    }

    public int getY() {
        return y;
    }

    public int getZ() {
        return z;
    }

    private final int x;
    private final int y;
    private final int z;
}

Then in your Entity class:

public class Entity {

    public Location getLocation() {
        return location;
    }

    public Location setLocation(final Location location) {
        // NOTE: Do any input validation
        this.location = location;
    }

    private Location location;
}

That would be more OOP. However, I don’t really see a problem in making the Location field in Entity protected. The only thing that might come up is if someone else is looking at the code from a black-box perspective, they might not know where the location field is defined:

public class Player extends Entity {

    public void doSomething() {
        location = null;
    }

    // These are the only fields declared
    private String name;
    private int health;
}

In this context, it might not click immediately that location is actually defined in Entity, rather and Player. Now, in modern programming, most people are using IDEs, and if they select location, there’s usually a “Go to definition” action, and it will bring them to it, so it’s still not that big of a deal. Try to program for the most common denominator.

@Zymus: That idea sounds great, until you get to the point where such a class is passed around an incredible amount of times in the code.
From what I heard and saw in some of Minecraft’s source code, that actually seems to be part of the reason why the performance in some places is less than desirable.
The moment you encapsulate some simple values in a class, you demote those parameters from being on the stack (= fast access) to being on the heap (= slow access, needs to search the object).
This is an inherent issue with Java due to the lack of structs, or something similar. Structs are used in many languages to create objects that exist on the stack.

If you’re gonna end up encapsulating and de-encapsulating values millions of times per second you might hit a bottleneck.

Protosstribe August 4, 2016, 1:57pm #18

You really won’t notice any difference, like others have said. I’ve tested this on my software-based 3D scanline rasterizer project. One instance of that project used immutable types, whereas the other one used mutable types. I noticed no “measurable” difference in the FPS.

I prefer to use immutability, as long as you have no reason to use mutability.

My Java AST (Abstract Syntax Tree) project used to be immutable. But I realized it’s quite cumbersome to “modify” a tree structure that is immutable. What concerns my 3D scanline renderer, I currently use a mutable version of it as well. In a larger scale it may prove beneficial. At least in contrast to an immutable version of it.
