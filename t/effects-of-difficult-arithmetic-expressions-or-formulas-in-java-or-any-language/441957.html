Fat_Cookies August 3, 2016, 2:39am #13

I’m not going to compile the code and look at the bytecode, but I’m pretty sure the java compiler would somehow optimize it so it’s not doing 10 million consecutive additions, etc etc.

The actual number of arithmetic instructions isn’t normally that important, it’s how the number of them increases as you increase the size of the input, hence why people measure an algorithms efficiency using the Big O notation. Anything in linear time O(n), logarithmic time O(log n) is a win, quadratic algorithms O(n^2) can get slow, any any exponential ( O(2^n) etc etc) is just painful.

Using any time methods in java to work out how long something takes is not reliable, shouldn’t be used to benchmark anything.

Would gladly appreciate it.[/quote]in most usages you see big O notation used to describe time complexity, which is roughly how many steps an algorithm takes to return a result given an input

so, an algorithm with O(1) time complexity (constant) supposedly only takes 1 step to return a value regardless of input.
an example is a simple add function:

int add(int a, int b) { return a + b; }

meanwhile, an algorithm with O(n) time complexity (linear) will take n steps dependent upon something about the input.
a good example is array traversal where n describes the size of the array.

int[] doubleElements(int[] intArray) { for (int i = 0; i < intArray.length; i++) intArray[i] *= 2; return intArray; }

there are lots of different types of complexities ie. logarithmic, exponential etc. big O can also describe another type of complexity, space complexity, which shows how much space an algorithm needs in memory.
