Protosstribe August 4, 2016, 2:08pm #12

The reciprocal is the multiplicative inverse of a number.

Because multiplication is faster than division, and you have the opportunity to pre-calculate the reciprocal of the number that you want to divide with, you can save some computation time by multiplying with that pre-calculated reciprocal instead. The result is the same. I use it all the time in my 3D software renderers.

Although, there are places where you shouldn’t use it. Simply because you wouldn’t save anything doing so. If you divide a number once, and you don’t have an opportunity to pre-calculate the reciprocal as a constant field, you would have to perform more computations than if you just divided it in the first place.

On the other hand, multiplying with the reciprocal makes the code harder to understand for people that are not used to it. Take yourself as an example. And, in a lot of cases there would really be no difference. So having it just because, well, that’s up to you to decide, but it’s not necessary.

As others have said, premature optimization isn’t the way to go. First, make it work. Then, if you’re not satisfied with the FPS as in your case, profile your application and try to find bottlenecks in your code. If you change your code based on what you think is an optimization, and it isn’t, change it back. Even though it makes no difference performance-wize, unless of course the “optimization” makes the code more clear. Having clear code weights more than having code with “optimizations” that really makes no difference.
