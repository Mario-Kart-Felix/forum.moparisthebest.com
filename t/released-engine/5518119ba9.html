Ok, Ill try to do an example as close to your code as possible.

Let’s take a different type of concrete object, like a polygon.

public class Polygon {
  private Point[] verts;
}

Now, if we are going to create methods for this object, we want to make sure they fit the class’ definition and don’t extend outside the scope of what this class is for. What is the class definition? To represent a polygon and provide methods to manipulate it. Note that when I say scope it’s not a language definition that can actually result in errors or anything, I mean scope as in the scope of the class theoretical perceived definition.

Here’s an example of a manipulation method:

public class Polygon {
  private Point[] verts;

  public void translate(int tX, int tY) {
    for(Point p : verts) {
      p.translate(tX, tY);
    }
  }
}

Here’s an example of an out of scope manipulation method (what you have):

public class Polygon {
  private Point[] verts;

  public void translateBasedOnSomethingUnrelatedToThePolygon() {
    for(Point p : verts) {
      p.translate(MyRandomStaticClass.translateX, MyRandomStaticClass.translateY);
    }
  }
}

To achieve the same result, you’d have the correct translate method and you’d call [tt]translate(MyRandomStaticClass.translateX, MyRandomStaticClass.translateY)[/tt] instead.

Alternatively in some cases if you are using a more ‘custom’ object (a polygon is a concrete mathematical object), you can pass around your other custom objects (cough camera instance), like [tt]translate(myCustomClassInstance)[/tt]

and have methods like

public class MyCustomObjectThatHasPoints {
  private Point[] verts;

  public void translate(MyCustomClass c) {
    for(Point p : verts) {
      p.translate(c.translateX, c.translateY);
    }
  }
}

I can’t think of a situation where strictfp is useful for games. Portability doesn’t matter and it comes at a cost. Spending extra cycles in the FPU is a waste for games, especially since you can’t see a visual difference when it comes down to actually interacting with the environment you’re rendering. This is why games choose to use floats instead of doubles. Wouldn’t double the precision be better? It turns out to pretty much never matter. If you are keeping your floating point values within the highest area of precision the float can offer (between -1 and 1), then you don’t really need anymore precision than that. (To be fair, Intel processors are a bit weird with this and often extend 32-bit floats to something like 80-bit floats, so precision literally doesn’t matter for performance in this case, but that’s not consistent across architectures).
