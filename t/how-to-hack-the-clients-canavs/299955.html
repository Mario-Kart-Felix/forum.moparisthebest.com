Since this question has come up a few times I thought that I would write a summary listing the pros and cons of each method. The idea is to be able to paint on top of the canvas.

Double Buffering
Unless you want the text to flicker you need to use double buffering. Basically you give the client something to draw on, then you draw on it, then you paint that to the screen. If you don’t use double buffering, you and the client will interfere with each other when you draw on the screen at the same time. Sometimes the client will paint first and then you so your text will be visible. Other times you paint first and then the client paints over the top so your text isn’t visible. Since this is all random you’ll get an annoying flickering as each thread fights to paint first.

Method 1: Simply paint on the canvas
The simplest but inefficient and visually annoying since it doesn’t use double buffering.

Graphics g = client.getComponentAt(1,1).getGraphics();

You paint on that Graphics object.
On the positive side you don’t modify the client, you don’t modify the API objects that the client uses and its simple. Negatively it’ll suck up CPU more than any other method and the flickering will be annoying.

Method 2: Subvert the call to Canvas.getGraphics()
The most efficient way requiring the fewest resources but it requires hacking of the API classes and the client. Contrary to FUD created by certain people who wish to peddle their own colour-based bots: byte code modification is not as detectable as reflection. Why? you can do everything you can do with reflection but with the bonus of being able to do more things. Think of it like a tool. You could wave it about like an idiot and draw attention. Or you could use it properly and more efficiently.

Somehow you make the client call your version of Canvas.getGraphics(). Rather than returning the actual graphics of the canvas, you draw on a buffer, paint the buffer to the Graphics returned by the actual Canvas.getGraphics() method, then return the Graphics of the buffer. What happens is: the client paints on your buffer, you then paint on the buffer, you then paint the buffer onto the screen, then you give the buffer to the client to continue the cycle. The result is that the screen is one frame behind but since there are 25ish frames per second it really doesn’t matter.

Since the Graphics object is quite opaque it is hard to tell where it came from. You could be drawing on an in-memory image or on the screen or anywhere and the client wouldn’t be able to know unless they dug through several layers of API specific (non-standard) variables with reflection. That means you can return any Graphics object.

Method 2a: Make the client’s canvas extend a different class
Rather than the client’s canvas (We shall call it Canvas_Sub1) extending Canvas. We can make it extend our own class that then extends canvas. So: Canvas_Sub1 -> Canvas becomes: Canvas_Sub1 -> HackCanvas -> Canvas.
In HackCanvas you override the getGraphics() method.
The danger is that if the class hierarchy of the Canvas_Sub1 is checked then the hack will be obvious.
RSBot uses this method.

Method 2b: Make the Canvas_Sub1 call a different getGraphics method.
Whenever the client calls a method “getGraphics” where the signature has no parameters and returns a Graphics object, you make it call your own static method instead. So from canvas.getGraphics() it becomes Bot.getGraphics(canvas). No complex byte code modification is involved since only one instruction is changed.
The issue with this method is that the painting code could bypass it with reflection but this is highly unlikely and if you pay attention to the code in the client you’ll be able to see it. If they do bypass it then the only effect is that your text won’t be displayed.

Alternatively if Canvas_Sub1 overrides getGraphics then you just need to intercept calls to super.getGraphics(). Reflection could not bypass it.

Method 2c: Make the Canvas call a different getGraphics method.
A bit more complicated. Basically change the getGraphics method in the Canvas class. You could rename the method and add your own method with just enough logic to call your own method in another class. For example:

public Graphics getGraphics() { return Bot.getGraphics(this, getGraphicsOld()); }
Of course this code would have to be implemented in byte code so you use another class to do the actual logic and just write a shim layer.
The drawback is a deep search with reflection of the Canvas class would turn up this new method.

Alternatively put your shim in when the old method would’ve exited. That way you don’t make another method.

An even simpler method is to completely replace the Canvas class with your own version. Add the option: Xbootclasspath/p:“HackedCanvas.jar” to the VM arguments where HackedCanvas.jar contains your canvas with hacked getGraphics() method. You can take Canvas.java from the jdk’s source. [credits to D e n v e r].

Method 3: Make the canvas a lightweight component.
This is Sun specific. Using reflection change the Canvas’s peer to an instance of a lightweight peer. As a result the getGraphics() method will call the parent component’s getGraphics and so on. To intercept the call you need to make your own component with an overridden getGraphics method. Then make the client a child of that component. The heavyweight peer is still there you need to destroy it so call its dispose() method using reflection.
Since the Canvas is now lightweight you don’t need to use heavyweight menus. Additionally it may make running multiple client’s in tabs easier. An issue is that the client could override the getGraphics() method in which case this can be detected. An advantage is that it doesn’t require byte code editing so it is simpler.

Method 4: Change what the Canvas paints on
This is Sun specific. Change the SurfaceData of the Graphics object to that of a buffered image of the same size. You then use another thread to paint that buffered image onto the original surface data. A simplification of this method allows you to see whats on the screen for a colour bot.
I don’t have eclipse on this computer so I can’t give details but I’ll update this post later.

Method 5: Hack the client itself
Inject code into the client’s paint method. Probably the best way is to intercept the call to Graphics.drawImage().

Hopefully that covers all possible ways to hack with the painting of the client. If you know of any others please comment.
