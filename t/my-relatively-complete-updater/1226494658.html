I took a look at some of your classes. The way you program is generally a bit yucky, at least judging by the few classes I looked over.

Firstly, the if statements without braces. There’s a good chance for logic errors in a bigger program. You’re not doing yourself any favors by not enclosing the body of your if statements. If anything, it’s increasing the chances that a third party like myself might accidentally add a statement that we think will only be executed if the if statement condition evaluates to true. If you absolutely feel compelled to not use braces, then why not put the if statement’s body on the same line?

Secondly, you don’t space your methods out at all. It’s all just one big clump of text. Sure, it’s ok when you’re working alone; but I wouldn’t fancy working in a team with you if you wrote your classes like that.

Another little thing for me was overall readability when it comes to inheritance. You’re using inheritance, which is great. But you could enhance readability by using the ‘final’ keyword in your method declarations. I’m absolutely positive that your subclasses don’t override every single method in your superclass. Having said that, why not make your methods final if you aren’t going to override them? It allows someone like me to be sure one way or another, without digging through other classes to check.

lothy July 30, 2016, 12:40pm #23

I prefer textpad. I hardly bother with IDEA.

As for if statements without braces, yes it’s definitely a preference thing.

But something like this:

if (condition) doThis();

is less likely to be misread then this:

if (condition) doThis();

If you’re working in a team with several people making changes, then you can probably see the likely hood of someone doing something like this when making a quick change:

if (condition) doThis(); andThis();

The logic error there is that the person who added it thought the method would only be called when condition evaluates to true, and if you’re working on a reasonably large program then that bug is going to be a bitch to hunt down.

As for using the final keyword, it’s just a nice thing to do if you’re releasing code for others to continue working on. As I said above, you can work out the class hierarchy faster.

Zappacky:

If everybody followed these (including myself, I dont claim to be anywhere near perfect), the world would be a wonderful place.

  1. It is NOT acceptable. There are no exceptions.

  2. If there is a problem, it will become a big problem unless it is promptly fixed.

  3. Unless it is a typo or a memory lapse, it is a design flaw and the system design should be refined.
    3a. A simple patch is NOT an acceptable substitute.

  4. Reliability and speed should never be a compromise. If it becomes a compromise, see rule number 3.

  5. Smaller is not always better. It usually is, however, because large binaries tend to be slow and full of bugs. If a binary becomes too large, see rule number 3.

  6. If it crashes randomly once, but never does it again, see rule number 2.

  7. No problem is ever too small. If the problem is too small, reread this rule, then see rules 6 and 2.

  8. Errors happen 1000% more frequently and faster on client computers. If an error occurs every 10 runs on your computer, it will be completely unusable on a client computer. If it occurs less frequently than that, see rule number 7.

  9. Code is always better the second time it is written than the first, and better the third than the second, and so on. When you think it is perfect, see rule number 1.

  10. It has not been tested enough. Period.

These can be simplified into two golden rules.

  1. It is never ‘good enough’. Period.

  2. Reliability should never impact speed. A well designed system is both reliable and fast.

Which can be further simplified into the golden rule:

  1. If it’s slow or it crashes, IT’S WRONG.
