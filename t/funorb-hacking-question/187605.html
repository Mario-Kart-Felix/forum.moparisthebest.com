I originally used that design with javascript for the actual editing. Ie there was the definition and then the actual action written in javascript. It was okay in terms of speed providing you compile the scripts before hand so they don’t have to be compiled every time you do the action. Problem with that approach is that the xml is too restrictive so you have to create either a bunch of special cases or glue it together with javascript. Once you get into javascript though things start acting differently. For example you now have everything in doubles so when you xor an int (32bits) you get something completely different to what you were expecting (a large negative number). This caused me to use nested classes to achieve it. For example: new ComplexCondition(insnOpcode, new ValueCondition(…), new AnyCondition()).
