That’s why you implement your own look-and-feel (LAF) model to make it look that way.

As far as using Slick, there should be options for UI implementations that use slick. Maybe there’s even an interface that allows Swing to be rendered using Slick, since Slick is designed to replace the Java 2D pipeline and that’s all Swing/AWT use anyway.

Finally, I do not recommend implementing your own UI. You will spend a considerable amount of time implementing something that will not be as good as Swing functionally and possibly even efficiently. Swing is incredibly bloaty, but it’s efficient enough for most applications. Do not implement an UI system unless you can clearly justify that you have no other option. In this case, you do have many other options.

mod_taharok August 4, 2016, 10:18am #23

A StringBuilder is a good way to do it, especially if you don’t need to pull the text back out (such as for network transfer, editing previous lines, copying lines, etc.) Arrays simplify some other features that you obviously don’t need, but they come at a cost. StringBuilder will be more efficient since you didn’t roll out your own display component, so it’s the better choice here.
