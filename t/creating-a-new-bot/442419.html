Aftermath_961 August 3, 2016, 2:43am #3

Programming languages are just tools. That being said, different tools and toolkits are better suited to different approaches. For reflection, Java is really your only option; it’s also your best bet if you want to do bytecode injection. If you want to make an OpenGL bot you’ll probably want a toolchain that can compile to native, so C/C++ (or almost any of the Visual Studio languages if you’re on Windows) would work. For a screen scraper / color bot you can essentially use anything that has the appropriate I/O libraries.

Of course, you can build all the other parts that don’t interface with the game itself (e.g. the scripting system, user interfaces) in whatever language you see fit.

As for which approach is ‘better’, that’s your call really. All of them aim at solving the same problem. At the moment I’d probably say a OpenGL / DX bot is your best bet since it’s been three months since we’ve made any changes and the bot Silab + I worked on still works just fine. Reflection / injection bots require quite a bit of maintenance and updating due to obfuscation.

We already have a working GL bot that works by replacing the opengl32.dll file. It interfaces with Simba: https://github.com/silabsoft/SimbaGL

As for the logging, you can use GLIntercept. It works the same way (by replacing the opengl32.dll file) and you can configure it to log to a file and have pictures of each modification to a frame. I don’t know if it supports automatically associating a command with a frame, you might have to do a bit of manual work on that.
