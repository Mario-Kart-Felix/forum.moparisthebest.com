 Styx is a NIO founded framework which I have high hopes for. I finally decided to work out a repository to hopefully get some criticism on the current code. I'm not going to go into listing all the features since you'll have a pretty good idea of my progress based on the repository. I'll provide media as soon as I have something worth depicting since the only feature as of now is connection accepting.

 I hope to develop this framework to the point where I can host it as an oldschool server. I don't plan to attempt emulating RuneScape itself, but instead to emulate the original MoparScape servers and implement well written content. (Home being the eastern Varrock bank)

I would appreciate any critiques or corrections which can be left in the comments. You can act like a dick while correcting me if you feel so inclined I don't care, as long as there's logic behind what you say and it could better the project.

This project will be open source until after player updating has been finished. This will be a fair amount of time as I would prefer to have all my code in top shape before the link is removed.

Well, since you’re using NIO and the current implementation looks pretty standard (like rewritten RuneSource) not saying you did, but that is what it looks like, I would suggest looking into Naga, it is a simplified wrapper library for NIO and basic IO (also has support for the asynchronous counterparts!) Also, instead of jamming your NIO code inside your main class I’d suggest looking into some more design patterns to implement that code in a more readable and OO fashion. On another note you don’t seem to have any prewritten planning or design behind your logic it seems to me you are just writing as you go and optimizing as you see necessary – this is a bad way to approach a design, especially with a server. You should definitely take some sit downs and write some pseudo-code or a typed/written project hierarchy and see the flow of your designs before actually programming them.

It is looking much better!

But a few more suggestions if I may add them;

Iterator<?> ...

This is not mandatory, but I hate it when the inferred type arguments are not used properly (by properly I mean when unconstrained type arguments are left raw or when they infer <?>) You should instead use:

Iterator<SelectionKey> ...

again, not mandatory but just my way of doing things, it is easier on the eyes when reading code, when I see SelectionKey as the type argument I am like “Oh, an Iterator over selection keys” rather than “Oh, an Iterator over some class, oh, now I see an Iterator over SelectionKey’s”

on to this:

while (true)

I see these are very misleading and confusing, I would definitely look for an alternative, you do not need an infinite loop.
What I mean by this is to use some resources within your class that provide when to execute to loop and when to stop it.

For example in NIO I tend to do this:

[code=java]
while (channel != null && channel.open()) {

if (channel == null || !channel.open()) break;

... logic ...

}[/code]

This is very clear to whomever is reading your code that you will only process whatever it is that your processing whenever “this” flag is true, if it becomes false during the execution of the loop, it will break out of it.
