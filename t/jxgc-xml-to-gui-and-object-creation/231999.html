First I will mention little about what can be done:

You can create Objects through their Constructors, but only if the drawbacks mentioned later on does not apply. Also, if you have already created an Object, you can (if done correctly), re-use that Object as parameter-argument for another Object, by specifying the syntax “${ID}”, where “ID” is the ID for the referenced Object, to the Value-attribute of the new Object. You can also do the same with the Class-attribute, but this only copies the class-name from the referenced Object to the new Object, so you don’t have to re-write it (usually only if it is a long name etc.).

Also, another good thing is that you can call on methods on the particular Class (no static ones (that I know?)), in which you currently are in the XML-document.

If you fail on anything, an Error-message will be created, and you can create your own ErrorHandler Objects to handle Errors according to your own liking. The default ErrorHandler writes the Error-messages to a Container, which in turn will be written to a unique filename on the specified path determined by the Log-attribute in the main-Element, or “.” if not available… When done, it will close itself.

Now I will mention little about the drawbacks:

You cannot create an Object that have a Constructor which requires an Array of any type as parameter-argument (almost finished, but not quite yet).

You cannot create an Object that have a Constructor which requires another Class that can only be instantiated through a method as parameter-argument (Singleton comes to mind).

You cannot create an Object that have a Constructor which requires another Class that can only be accessed via a field in that (or any other) particular Class (constants comes to mind).

You cannot (yet) receive values from getter- (or accessor-) methods, to apply on other Objects. Neither can you call on static methods (that I know, never really tested :)).

Hope to hear some feedback, although it isn’t open-sourced… Maybe it will be someday?

You’re scripts are parsed and then run in the initialization part of the program. Activity is not enabled until after the script is run through. Look And Feel changes must be done before user interface objects are displayed (so, basically, in the initialization part of the entire program). Thus, to change the LAF, you need to do it inside the script. And also, it seems you have not changed the LAF in the Swing package before. It is done statically through the class, UIManager. It has nothing to do with JMenuItems and is a pain in the butt when run through an external plugin, but still is possible. As long as the LAF Jar file is included in the classpath and the same parameters apply for the method call, it should load. But any way you slice it, you need to modify the LAF through UIManager, thus needing access to call a static method during the initialization of the program, which is in your script.

system July 31, 2016, 1:44am #10

Taharok: The LookAndFeel is actually changed after initialization… All JComponents have a method called updateUI, with which you change the look of the JComponent, after you have changed UIManager.setLookAndFeel(…) to another LookAndFeel. I had to make a little “ugly-hack” in this package to actually show the LookAndFeel properly… Else it wouldn’t be shown.

Everytime you add a new Object, which in fact is an instance of JComponent, or JFrame with JRootPane and JLayeredPane, I also call their updateUI() respectively.

This GUI-creation tool is originally only made to set the LookAndFeel when the GUI is started… Changing the LookAndFeel can be manually done later in an ActionListener class or similar for different JComponents etc. This tool is not ment to handle everything for a user, but only the specific creation of the GUI.

lawl: The LookAndFeel I use, is “com.jtattoo.plaf.acryl.AcrylLookAndFeel”, which is a theme-LookAndFeel in the LookAndFeel-package called “JTattoo”.
