This method is contained in the Graphics class of the java.awt package. It draws a String to the screen (a line of text). I would like to point out quickly that this method is abstract, therefore the place where you use this method must me abstract, too. Also, the class may not be referred to for a method. You must instantiate a Graphics object and refer to that when calling a non-imported method. To call the drawString method, you would do the following:

Graphics.drawString(String str, int x, int y);

Or if you want you could import the class, or complete package, but that would require making the class you are putting it in abstract; not just the place where you are putting it in.

For the object reference, Graphics would be the name of a instantiated Graphics object. For the first parameter (String str), you would put the String you want to output. This can be either be a declared String, or just a line in quotations. For the second parameter (int x), you would put the X coordinate of where you want to output the String on the screen. For the last parameter (int y) you would put the Y coordinate of where you want to output the String on the screen. Coordinates start in the upper-left corner of the screen. X goes down, Y goes right. Therefore, Graphics.drawString("Thank you for reading.", 0, 0); Would output Thank you for reading at the top-left corner of the screen.

I didn’t feel I needed to mention that a Graphic object needed to be instantiated. I have modified my tutorial to mention it.

This is server-side, and it’s not my fault if for some reason on Runescape private servers it isn’t. But in this case, I know it’s server-side on Runescape private servers too. Sockets are already taken care of in your source, and this method should work server-side fine. The only reason everything isn’t server-side is because most people don’t have web-clients. If you create a web-client, with a private verification packet with your server, you can check to make sure they are using your web-client. You can also randomize it and even create your own formula if you really wanted to. The data parameter could be your secret code, and perhaps you could have many different packets come in an expected order and verify the client like that. I suggest making the data and pattern “fit-in” with the rest of your packets, so a simple packet sniffer could not manipulate it. With a web-client, you could put all of the server files on the website and you could play music, have custom sprites, everything you can do in Java. The point of this tutorial is to actually TEACH you REAL JAVA.
