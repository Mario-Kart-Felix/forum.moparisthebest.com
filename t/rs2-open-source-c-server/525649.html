https://code.google.com/p/c-plusplus-runescape-server/source/browse/#svn%2Ftrunk[/quote]Thanks, I appreciate it, but I’d like to write the entire server based on my own research and experience. This project is entirely for learning C++ after all.

So anyway, I wanted to let everyone know I haven’t quit this project (that would be pathetic :P). Development is very slow because of how little C++ I actually know. But, finally after several hours of pulling my hair out and dealing with annoying linking errors, I got logging in and adding players down. I had to fix quite a few things, like my buffer reading methods and some things for the node storage.

For those who are curious, here’s how things (should) work. Node -> Entity -> Character -> Player. NGICollection -> NodeStorage -> PlayerContainer. NGI stands for “no-gap indexable”, which is basically a fancy array I came up with (I’ve never heard of it, but I’m sure it’s been done before) that you can add and remove elements to without it resizing and worrying about gaps. It’s like a linked list but unsorted and you can index it. Nodes have what is called an NGI Index, which is its position in the NGICollection. The NodeStorage is basically the NGICollection but with a bitmap of all the nodes inside of it. It’s worth noting it’s set with a specified index (entity index) and not the ngi index. There is no bitmap for the ngi index because of the “no-gap” part of the collection. When a node is removed, the last node in the collection is moved to the empty spot and its ngi index is changed. To get the next available ngi index, one would just get the amount of nodes in the collection. The only thing left storage-wise is a list of player indexes and their corresponding ngi indexes, which should be pretty easy to implement.

So, why all these confusing storage methods? It’s simple (well in all actuality its a pain in the ass and hard to keep track of). The NGICollection, like I said, is like a list that never has any gaps or resizing, which makes it really fast and easy to iterate through when it comes time for updating or something (O(n)). The bitmaps, like I said, allow for fast lookups with player indexes. The reason these have to be different is because the player index is determined as soon as the client logs in and even though there is a packet that can change it, I think it would cause problems. I need to add the table of entity indexes and their corresponding ngi indexes because when packets are sent and use the entity index, I would have to iterate through every entity in the collection to see if the index matched the specified one.

I think this is coming along quite well. It’s quickly become what seems like a confusing nest, but it should be fast. Also I think I’m going to drop the multi-revision compatibility. It’s very annoying to implement and I don’t see many benefits. Besides, it can always be ported.
