the_bank July 30, 2016, 8:29pm #12

Solarwind, I hope this helps.

  1. I think it would help you if I make this post about Obfuscation rather then taking it away. Obfuscation, like already stated, is a method of making code very hard to read.

  2. Once again, this is for an obfuscator, not a deobfuscator. It varies. Usually all the variables are given useless names like a,b,c etc. Also, each data type will have it’s own “alphabet”. For example, there would be a variable named “a” for an int, string, double float etc. Also, most of the methods are given single letter names. That makes it hard very hard to decipher. For example:

String a;
Int a;
Double a;

public void a {
/*code shit goes here */
}

public string a{
return a;
}

A growing tool used in obfuscation is “flow obfuscation”. This is what Jagex uses. Basically, it makes the code very hard to read by having the JVM move to different parts of the code very often. Here is an example:

_L2:
        aStringArray76[92];
        log();
        return;
        aString5 = JOptionPane.showInputDialog(null, aStringArray76[80], aStringArray76[97], 3);
        aString6 = JOptionPane.showInputDialog(null, aStringArray76[89], aStringArray76[86], 3);
        aTimer8.timeLimit = 10000L;
        anInt13 = Integer.parseInt(aString5);
        anInt14 = Integer.parseInt(aString6);
        aTimer9.timeLimit = anInt13 + RandomGenerator.random(anInt13, anInt13 + 1) * 60 * 1000;
        aTimer9.reset();
        aTimer10.timeLimit = anInt14 + RandomGenerator.random(anInt14, anInt14 + 1) * 60 * 1000;
        aTimer10.reset();
        log((new StringBuilder()).append(aStringArray76[1]).append(aTimer9.timeLimit / 60L / 1000L).append(aStringArray76[6]).toString());
        log((new StringBuilder()).append(aStringArray76[0]).append(aTimer10.timeLimit / 60L / 1000L).append(aStringArray76[6]).toString());
        aString21 = JOptionPane.showInputDialog(null, aStringArray76[93], aStringArray76[88], 3);
        aString21.equalsIgnoreCase(aStringArray76[10]);
        if(flag) goto _L6; else goto _L5
_L5:
        JVM INSTR ifeq 400;
           goto _L7 _L8
_L7:
        aString69 = aStringArray76[95];
        theTabs.Options.setRun(true);
        if(!flag) goto _L9; else goto _L8

Please note that this code is semi-deobfuscated.

On top of this, all the String types are usually encrypted. The encryption method varies.

  1. A deobfuscator is responsible for removing all of these things. It will give the variables proper names and will decrypt the strings. When dealing with flow-obfuscation it will also put order back into the code. It does this by reconstructing the class file via bytecode.

  2. Refer to #1.

Hope this helps.

BAsh_CntLgn July 30, 2016, 8:30pm #17
frank_:

well i have no idea what you are talking about, if you have an identical method declararion in your classfile, the verifier will explode

Same arguments. Different return type. This sort of overloading is illegal in java but the sun jvm doesn’t do much checking at run time. The only way to tell the two apart is at the bytecode level when you call it. The signature has the different return type. Of course try passing that through a compiler and it won’t have a clue which one to choose.

So I made my own zkm style deob given that zkm has a zkm obbing applied to it. Same naming scheme and even the same order so you can compare zkm deobs with non-zkm deobs. [br][br][size=1]Posted on: 10 March 2008, 07:45:21 AM[/size][hr]A deobber is anything that makes jagex’s code easier to read. There is the zkm deobber which simply changes the naming and it is usually used before our own deobbers. The other deobbers are non-standard and usually perform a number of transformations on the method bytecode so the decompiler can decompile them correctly. One of these is the removal of ~ from if statements although this doesn’t help the decompiler as it usually trips up on flow rather than nested statements.

The deobbed client that I gave you has had a number of transformations applied to it that seek to remove unused/constant method arguments as well as a simplify number of simple bytecode patterns. While this can be helpful in understanding the client I suggest you focus on getting the client to run in a debugger at all. That means you need to take which ever deobber that can allow you to decompile the client correctly rather than producing the neatest code.
