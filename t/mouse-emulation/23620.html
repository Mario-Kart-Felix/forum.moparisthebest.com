/**

  • Spoofs mouse events to a specified Component.
    */

import java.awt.Component;
import java.awt.event.MouseEvent;
import java.util.Random;

public class MouseEmulator {

private Component target;
private int clickCount;

public MouseEmulator(Component target) {
	this.target = target;
	clickCount = 1;	
}

/* Internal Event construction */

//MouseEvent(Component source, int id, long when, int modifiers, int x,
// 			 int y, int clickCount, boolean popupTrigger, int button) 
private MouseEvent[] createLeftMouseClick(int x, int y) {		
	Random rand = new Random();
	
	MouseEvent pressed = new MouseEvent(target, MouseEvent.MOUSE_PRESSED, 
							System.currentTimeMillis(), MouseEvent.BUTTON1_MASK,
							x, y, clickCount, false, MouseEvent.BUTTON1);
	//lag var 47-219, usu 78-94; currently tends low		
	long releaseTime = System.currentTimeMillis() + rand.nextInt(100) + 40;
	MouseEvent released = new MouseEvent(target, MouseEvent.MOUSE_RELEASED, 
							releaseTime, MouseEvent.BUTTON1_MASK,
							x, y, clickCount, false, MouseEvent.BUTTON1);
							
	MouseEvent clicked = new MouseEvent(target, MouseEvent.MOUSE_CLICKED, 
							releaseTime, MouseEvent.BUTTON1_MASK,
							x, y, clickCount, false, MouseEvent.BUTTON1);
							
	MouseEvent[] clickEvents = new MouseEvent[]{pressed, released, clicked};
	return clickEvents;								
}

/* Public API */

public void setClickCount(int count) {
	clickCount = count;
}

public void clickLeft(int x, int y) {
	MouseEvent[] events = createLeftMouseClick(x, y);		
	target.dispatchEvent(events[0]);
	target.dispatchEvent(events[1]);
	target.dispatchEvent(events[2]);
}

}[/code]

I will now do my duty to give credit to psb for alerting me to the trap. I was busy doing ‘other’ things and so I completely neglected to look for this particular trap. He made me aware of it and I took measures to counter it. No, he and I are not working together on a system.

To Krich, your functionality looks fine. The only thing I would recommend is some form of a delay between events. As you all know, the listener methods set static fields. These fields are volatile. No, I don’t mean that they have the keyword volatile attached to them. What I mean is that they are dangerous to access in an asynchronous multithreaded system. Although single primitive fields are threadsafe by definition (@see JLS), you cannot rely on the same promise for multiple atomic fields. To compensate for this, they take the volatile fields and set their values to less volatile fields in a synchronized block. This is a common practice that allows them to coallesce events so they dont have to react to each and every single one. The state of the values is synchronized so that there is no corruption. The cpu utilization required to react to populating the entity array for every single atomic mouse movement would be insane. (Not that the current CPU utilization isn’t). So, the best way to handle this situation is to set the values directly the same way the game would, giving respect to the traps they have put in place. I do not create events. I would also recommend taking steps to alleviate some of the heap you will incur by creating so many mouse events.
