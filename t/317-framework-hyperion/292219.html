I think Graham released this in the server downloads section only to prevent anyone else from releasing it for him and getting rep for it.
Anyways, this server is something Graham and I have been working on for quite a while and as far as I know this is the most powerful 317 framework ever released open-source. This project has been codenamed Hyperion.

The whole engine runs on a task system which is thread pooled if the computer has more than one available core (it’s built so that multiple cores multiplies the power of this framework) and the entire updating system is run through these tasks. The update block is also cached so it runs pretty fast.

It has some (experimental and unfinished!) features such as an update-server (yeah, for 317) and a login-server (unfinished) for multiple worlds.

As far as I know, it’s 100% stable. As for networking, it uses apache MINA because it is so stable, professional, and scalable.
We have tested this server using a SYI-bot, and it supported 2000 players online simultaneously with little lag (this test was performed on a dual-core 2.0GHz linux box with 2GB of RAM, the SYI bot was run on another computer inside the local area network).

As far as content goes, for now it’s pretty bare. All content will be developed in JavaScript using the Mozilla Rhino script engine (see javax.script package) which allows development to be made very quickly without compromising performance too much. Once more content is made, Graham and I are considering entering this server into the server competition that Moparisthebest has set up.

[quote=“Graham”]There is a single ‘game logic’ scheduled executor service. This is where the majority of the work takes place i.e. packet processing, action processing (like woodcutting, mining). There is a special type of task that can be submitted to this service (the ConsecutiveTask) which blocks the logic service and submits the child tasks to a thread pool (set to the number of cores). Once they have all completed, the logic service no longer blocks and continues processing normally.

There is also a worker executor service, which is used for blocking operations e.g. loading and saving games. At the moment it is just a single threaded executor however this can be expanded. I’ll probably add some configuration so that users can fine tune the way the way these executor services work for performance (obviously depending on the number of cores you have, there will be different optimal configurations).

Some more things I’m going to look at for future versions is some sort of region system to avoid the sheer number of loops. I did a benchmark of splitting up a world into 8x8 sections and just using the current method, and doing this region system can speed it up five times on my system. Obviously I’m going to need to experiment with different sizes of sections as it can also be pretty hungry on RAM usage.

By a region system, I mean splitting up the world into chunks and having a list of entities inside each region. Then, whenever an entity is moved, if the region changes it can be removed from the old region and added to the new region. Obviously, this avoids so many loops + distance checks as you only need to loop through the regions directly around the player.

I was going to implement that into the server already, however, it ended up buggy so when I’m back from my holiday I’ll probably do it properly and see what the optimal configuration is (i.e. looking at region sizes, maintaining all regions or just ones with entities in, memory usage, CPU usage) and implement a decent version.[/quote]

    private static final Map<Integer, EquipmentType> typeMap =
            new HashMap<Integer, EquipmentType>();

    public static EquipmentType getType(Item item) {
        return typeMap.get(item.getId());
    }

    private enum EquipmentType {
        CAPE("Cape"),
        BOOTS("Boots"),
        GLOVES("Gloves"),
        SHIELD("Shield"),
        HAT("Hat"),
        AMULET("Amulet"),
        ARROWS("Arrows"),
        RING("Ring"),
        BODY("Body"),
        LEGS("Legs"),
        PLATEBODY("Plate body"),
        FULL_HELM("Full helm"),
        FULL_MASK("Full mask");

        private String description;

        EquipmentType(String description) {
            this.description = description;
        }

        public String getDescription() {
            return description;
        }
    }

    static {
        for (int cape : CAPES) {
            typeMap.put(cape, EquipmentType.CAPE);
        }
        for (int boots : BOOTS) {
            typeMap.put(boots, EquipmentType.BOOTS);
        }
        for (int gloves : GLOVES) {
            typeMap.put(gloves, EquipmentType.GLOVES);
        }
        for (int shield : SHIELDS) {
            typeMap.put(shield, EquipmentType.SHIELD);
        }
        for (int hat : HATS) {
            typeMap.put(hat, EquipmentType.HAT);
        }
        for (int amulet : AMULETS) {
            typeMap.put(amulet, EquipmentType.AMULET);
        }
        for (int arrowType : ARROWS) {
            typeMap.put(arrowType, EquipmentType.ARROWS);
        }        
        for (int ring : RINGS) {
            typeMap.put(ring, EquipmentType.RING);
        }
        for (int body : BODY) {
            typeMap.put(body, EquipmentType.BODY);
        }
        for (int legs : LEGS) {
            typeMap.put(legs, EquipmentType.LEGS);
        }
        for (int plateBody : PLATEBODY) {
            typeMap.put(plateBody, EquipmentType.PLATEBODY);
        }
        for (int fullHelm : FULL_HELM) {
            typeMap.put(fullHelm, EquipmentType.FULL_HELM);
        }
        for (int fullMask : FULL_MASK) {
            typeMap.put(fullMask, EquipmentType.FULL_MASK);
        }
    }

The benefits should be obvious with this approach: Instead of iterating every time you call the is() method, this simply checks the Map for the given item id, and returns the EquipmentType for that id.
The Map is static, so memory footprint should be quite small. That, and the reduced processing requirement should be worthwhile.

JS5 is multiple things:

[ul][li]Update server protocol[/li]
[li]Reference table format[/li]
[li]Client bundle format[/li][/ul]

The update server is, well, the update server. It deals with sending needed entries to the client.
Reference tables are the entries inside cache 255 except entry 255. Entry 255 is a checksum table of the reference tables. They store general reference information, like file name ‘hashes’ (frank_ proved that they are in fact not hashes, only behave like ones once they reach a certain size and overflow the integer), file checksums, subentry count, sizes, etc. for the files in the ‘index’ it describes (entry 0 describes index 0, entry 1 describes index 1, …).
Their client bundle format… what I am referring to is their .js5 files, like runescape.js5 or something similar. It’s format begins with a reference table, and then has x number of containers trailing behind it, where x = # of ref table entries. It then uses these entries inside a custom unpack200-implementation to rebuild the client classes. Sorry for the very non-detailed information on this, I HAD fully refactored it but I lost my refactor and I can’t reference it anymore.

JS5 likely stands for Jagex Server 5, or Jagex Service 5, or something similar.

Graham July 31, 2016, 9:47am #15

Lothy, you should note that not everything will run multi-threaded. In fact, parsing packets is not multi-threaded due to the issues that could arise from it (well they are decoded in multiple threads as MINA does that, but they are processed in the game logic thread). Updating is however, as that is safe. Due to the way tasks are executed, if we find opportunities for more multi-threading in the future it will be trivial (e.g. converting the use of a ConsecutiveTask to a ParallelTask).

In fact, since Blake hasn’t seemed to exactly explain it well, I’ll explain more about it here.

There is a single ‘game logic’ scheduled executor service. This is where the majority of the work takes place i.e. packet processing, action processing (like woodcutting, mining). There is a special type of task that can be submitted to this service (the ConsecutiveTask) which blocks the logic service and submits the child tasks to a thread pool (set to the number of cores). Once they have all completed, the logic service no longer blocks and continues processing normally.

There is also a worker executor service, which is used for blocking operations e.g. loading and saving games. At the moment it is just a single threaded executor however this can be expanded. I’ll probably add some configuration so that users can fine tune the way the way these executor services work for performance (obviously depending on the number of cores you have, there will be different optimal configurations).

Some more things I’m going to look at for future versions is some sort of region system to avoid the sheer number of loops. I did a benchmark of splitting up a world into 8x8 sections and just using the current method, and doing this region system can speed it up five times on my system. Obviously I’m going to need to experiment with different sizes of sections as it can also be pretty hungry on RAM usage.

By a region system, I mean splitting up the world into chunks and having a list of entities inside each region. Then, whenever an entity is moved, if the region changes it can be removed from the old region and added to the new region. Obviously, this avoids so many loops + distance checks as you only need to loop through the regions directly around the player.

I was going to implement that into the server already, however, it ended up buggy so when I’m back from my holiday I’ll probably do it properly and see what the optimal configuration is (i.e. looking at region sizes, maintaining all regions or just ones with entities in, memory usage, CPU usage) and implement a decent version.
