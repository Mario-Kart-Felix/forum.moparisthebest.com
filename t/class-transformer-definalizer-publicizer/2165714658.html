Anyway, this kind of thing is better done with a Visitor. Here’s a BCEL example I pulled out of my RS folder:

[code]//By Saev - http://www.kaitnieks.com/scr/showthread.php?p=7376

import java.io.;
import java.util.jar.;
import org.apache.bcel.classfile.*;
import org.apache.bcel.Constants;

public class Publicizer extends EmptyVisitor {

public static void main(String[] args) throws Exception {
    
    if(args == null || args.length == 0) {
        throw new RuntimeException("this program needs a jar file as an argument");
    }
    JarFile jar = new JarFile(args[0]);
    
    for(java.util.Enumeration<JarEntry> e = jar.entries(); e.hasMoreElements(); ) {
        JarEntry entry = e.nextElement();
        
        if(!entry.isDirectory()) {
            String entryName = entry.getName();
            InputStream instream = new BufferedInputStream(jar.getInputStream(entry));
            
            if(entryName.endsWith(".class")) {
                JavaClass clazz = new ClassParser(instream, entryName).parse();
                clazz.setAccessFlags(Constants.ACC_PUBLIC);
                new DescendingVisitor(clazz, new Publicizer()).visit();
                clazz.dump(clazz.getClassName() + "_.class");
            }
            instream.close();
        }
    }
}

public void visitMethod(Method m) {
    m.isPublic(true);
    m.isPrivate(false);
    m.isFinal(false);
}

public void visitField(Field f) {
    f.isPublic(true);
    f.isPrivate(false);
}

}[/code]

Further illustrating the point, here’s one I threw together with SERP for some reason:

/*
 * Publicizer.java
 * Created on November 16, 2005, 9:13 PM
 */

package serprenamer;

import serp.bytecode.*;
import serp.bytecode.visitor.BCVisitor;

public class Publicizer extends BCVisitor {
    
    public void enterBCClass(BCClass bc) {
        bc.makePublic();
        bc.setFinal(false);
    } 
    
    public void enterBCMember(BCMember bc) {
        bc.makePublic();
        bc.setFinal(false);
    }     
}

As I see it, this class is more of an Utility-class, than anything else. Utility-classes uses private constructors and static members (methods, variables etc.). If you take a look at java.lang.Math for instance, that class is an Utility-class, and it uses the exact pattern I just described.

If you look at it like this; there is no need for different instances of that class, because it have nothing that can change within itself, let alone affect itself compared to any other instance of it (I mean, all instances would be pretty much the “same” at all times of use).

Another thing is that it will be faster as static, than if not. Not that this is applicable in the most cases, but still.
