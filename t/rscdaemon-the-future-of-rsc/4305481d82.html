you do realize that ‘server side path generating’ is on current RSC sources, right?

Route finding is handled by the client using the following breadth-first search on the tile graph as described by the adjacency matrix:

    /*
      find (local) route from the start (x_i, y_i) to the goal rectangle [(x1_f, y1_f), (x2_f, y2_f)] with optional approximation
      route_x/y will store the list of pivot waypoints
     */
    public int route(int x_i, int y_i, int x1_f, int y1_f, int x2_f, int y2_f, int route_x[],
                     int route_y[], boolean approximate) {
        for (int k1 = 0; k1 < 96; k1++) {
            for (int l1 = 0; l1 < 96; l1++)
                route_direction[k1][l1] = 0;

        }

        /*
          here, route_x/y & len are used for a circular buffer
          head/tail are used for the deque in the search
         */
        int tail = 0;
        int head = 0;
        int x = x_i;
        int y = y_i;
        route_direction[x_i][y_i] = 99;
        route_x[tail] = x_i;
        route_y[tail++] = y_i;
        int len = route_x.length;
        boolean success = false;
        while (head != tail) {
            x = route_x[head];
            y = route_y[head];
            head = (head + 1) % len;
            if (x >= x1_f && x <= x2_f && y >= y1_f && y <= y2_f) {
                success = true;
                break;
            }
            if (approximate) {
                if (x > 0 && x - 1 >= x1_f && x - 1 <= x2_f && y >= y1_f && y <= y2_f && (adjacency[x - 1][y] & 8) == 0) {
                    success = true;
                    break;
                }
                if (x < 95 && x + 1 >= x1_f && x + 1 <= x2_f && y >= y1_f && y <= y2_f && (adjacency[x + 1][y] & 2) == 0) {
                    success = true;
                    break;
                }
                if (y > 0 && x >= x1_f && x <= x2_f && y - 1 >= y1_f && y - 1 <= y2_f && (adjacency[x][y - 1] & 4) == 0) {
                    success = true;
                    break;
                }
                if (y < 95 && x >= x1_f && x <= x2_f && y + 1 >= y1_f && y + 1 <= y2_f && (adjacency[x][y + 1] & 1) == 0) {
                    success = true;
                    break;
                }
            }
            if (x > 0 && route_direction[x - 1][y] == 0 && (adjacency[x - 1][y] & 0x78) == 0) {
                route_x[tail] = x - 1;
                route_y[tail] = y;
                tail = (tail + 1) % len;
                route_direction[x - 1][y] = 2;
            }
            if (x < 95 && route_direction[x + 1][y] == 0 && (adjacency[x + 1][y] & 0x72) == 0) {
                route_x[tail] = x + 1;
                route_y[tail] = y;
                tail = (tail + 1) % len;
                route_direction[x + 1][y] = 8;
            }
            if (y > 0 && route_direction[x][y - 1] == 0 && (adjacency[x][y - 1] & 0x74) == 0) {
                route_x[tail] = x;
                route_y[tail] = y - 1;
                tail = (tail + 1) % len;
                route_direction[x][y - 1] = 1;
            }
            if (y < 95 && route_direction[x][y + 1] == 0 && (adjacency[x][y + 1] & 0x71) == 0) {
                route_x[tail] = x;
                route_y[tail] = y + 1;
                tail = (tail + 1) % len;
                route_direction[x][y + 1] = 4;
            }
            if (x > 0 && y > 0 && (adjacency[x][y - 1] & 0x74) == 0 && (adjacency[x - 1][y] & 0x78) == 0 && (adjacency[x - 1][y - 1] & 0x7c) == 0 && route_direction[x - 1][y - 1] == 0) {
                route_x[tail] = x - 1;
                route_y[tail] = y - 1;
                tail = (tail + 1) % len;
                route_direction[x - 1][y - 1] = 3;
            }
            if (x < 95 && y > 0 && (adjacency[x][y - 1] & 0x74) == 0 && (adjacency[x + 1][y] & 0x72) == 0 && (adjacency[x + 1][y - 1] & 0x76) == 0 && route_direction[x + 1][y - 1] == 0) {
                route_x[tail] = x + 1;
                route_y[tail] = y - 1;
                tail = (tail + 1) % len;
                route_direction[x + 1][y - 1] = 9;
            }
            if (x > 0 && y < 95 && (adjacency[x][y + 1] & 0x71) == 0 && (adjacency[x - 1][y] & 0x78) == 0 && (adjacency[x - 1][y + 1] & 0x79) == 0 && route_direction[x - 1][y + 1] == 0) {
                route_x[tail] = x - 1;
                route_y[tail] = y + 1;
                tail = (tail + 1) % len;
                route_direction[x - 1][y + 1] = 6;
            }
            if (x < 95 && y < 95 && (adjacency[x][y + 1] & 0x71) == 0 && (adjacency[x + 1][y] & 0x72) == 0 && (adjacency[x + 1][y + 1] & 0x73) == 0 && route_direction[x + 1][y + 1] == 0) {
                route_x[tail] = x + 1;
                route_y[tail] = y + 1;
                tail = (tail + 1) % len;
                route_direction[x + 1][y + 1] = 12;
            }
        }
        if (!success)
            return -1;
        /*
          here, route_x/y is the resulting path described by a list of pivot waypoints
          head is the pointer into this list
         */
        head = 0;
        route_x[head] = x;
        route_y[head++] = y;
        int _dir;
        for (int dir = _dir = route_direction[x][y]; x != x_i || y != y_i; dir = route_direction[x][y]) {
            if (dir != _dir) {
                _dir = dir;
                route_x[head] = x;
                route_y[head++] = y;
            }
            if ((dir & 2) != 0)
                x++;
            else if ((dir & 8) != 0)
                x--;
            if ((dir & 1) != 0)
                y++;
            else if ((dir & 4) != 0)
                y--;
        }

        return head;
    }

There’s a reason why the path-generation is taken care of by the client. It would cause major load on the server-side, should it handle all players path-generation requests. It would be easy to crash the server if you sent new requests over and over to new random locations. So, the server should not generate a path. It should just validate the path generated by the client. You can fake it all you want client-side, but the server should just reject the request by stopping the player if it found out.

Indeed, this is the most effective way to do so – and the way that RSC was meant to do it, seeing as Jagex designed it that way. The only thing the server needs to do is validate each step per tick.

Kryptix August 3, 2016, 2:47am #477
veer:

Can you tell me one valid reason for redistributing that load to the server? You’re only making the server do far more work than it needs to, while only relieving individual clients of a minuscule burden. In the future try applying marginal cost-benefit analysis to decisions.

Client sided:

A user spam clicks to walk, the coords the client are aware of are sent to the server along with the coords it wishes to walk too. The server starts to move them but the client hasn’t been updated yet and sends the same information again. The server does the same action again creating a ‘walk back’

Server sided:

There’s no sync so this cannot happen. The server always knows the latest coords.

Pros:

No walkbacks
Less client-sided code

Cons:

More server usage
More server-sided code

That is literally the only reason. If there wasn’t that reason we wouldn’t be doing it.
