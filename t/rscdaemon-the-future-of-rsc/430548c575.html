Zilent August 2, 2016, 4:12pm #106

To clear up some confusion about what’s being changed / added / removed.

The client layout has been dramatically modified to accommodate for an event driven server.
The client has been largely replaced by native code (no script kiddies allowed).
The client’s GUI system now supports loading interfaces from XML / JSON / INI files.

The client is currently being cross compiled between GCC and MSVC.

In the client, the following classes have already been removed (replaced by a native (dynamic/shared) library):

mudclient
GameWindowMiddleMan
GameWindow
GameFrame
AppletLoader
PacketConstruction
StreamClass
DataOperations
there may be more.

In the future, the client may be fully “emulated” in native code, however I do not have the time to refactor the more cryptic classes to prepare for such a task.

The server has been written from scratch with the following main points:

There is a server written in C++ concurrently being developed along side a Java server.

  • Edit - If heap fragmentation or tenured cleanup times become an issue in the Java server, then there’s a back-up project.
    There is no tick - everything is event driven.
    Various networking libraries can simply be plugged into the network interface (MINA / Netty / Native ASIO have been tested)
    Various “Login Services” can simply be plugged in - be it another thread or another program.
    Various data loading services can be simply be plugged in - plain-text / SQL / binary files.

I can’t go into detail on the network protocol system as that information would dramatically help bot makers.

There’s a lot more to be said, but those areas, as well as the areas that I have touched upon above are subject to change as I see fit.
