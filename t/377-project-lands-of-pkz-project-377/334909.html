[left]Welcome to DeadLock™

This is my server that I have been progressivly developing. Its based around the 377 erra and uses a clean but slightly renamed 377 client with its original cache.
[center]The networking side of things is powered by a nio library I released a little while ago called RSNio
It uses JDBC with sqlite for storage
A custom engine I have been developing. [/center]

Now the idea here is to program a fully functioning server which is just as stable as current releases but raises above because of its unique content. The ‘tick’ off the server is the same as every other server it runs at 600ms intervals and has four tasks which are

[center]- update movement -

  • synchronize players -
  • reset updating flags -
  • process ground data-[/center]

Each stage is multithreaded but the stages happen in a sequential order. The server uses a whole lot of new idea’s which I havn’t seen before. This includes container flagging for inventories, a new interface system to manage interface actions with ease. The combat system is being developed from the ground up, a unique dialoge system for ease of chaining and flexability. I’ve been developing a ground system which is very unique and have not seen one as promising as the one implemented. This isn’t all though because the server is 100% ‘clip proof’ you simply cannot clip. This also applies for npc’s. Using this clipping system I have created a rather successful and universal following system which is entity based for both player and npc’s.

The amount of time I have spent making sure that the content is dupe free and unhackable, well you wouldn’t believe me, but I have spent hours opon hours off thinking about different idea’s to implement. For an example, the interface system I use, unless the server has an interface open it doesn’t actually know that the actions exist so therefor a player cannot modify the client to send the needed data for an example to transfer an item to the bank without it being open. It even validates that the ground objects exist when you interact with them.

I’m currently debating whether or not to release the source because I do agree with open source but I havn’t decided yet. Maybe only to those trusted on rune-server.

Anyway enough about the good stuff…

The game play isn’t limited. I am implementing everything I have time for and i’m certainly not limiting skills to certain erea’s. although i’m pushing it towards a combat server, the skilling will be just as good. I’ve even implemented cups of tea!

Update:
Work on combat and the RepetitivePacketFilter have been paused. During the mean time I have been developing a new core for the server. Before the core was made up of two threads one for timing and the other for processing. The timing thread would pass all its processing to the processing thread. That system had a few complications when making an event system because it was timed on one thread and then processed on the second. My new engine consists of two thread again. One for timing/updating and another for other logic processing such as packets. The timing thread sleeps roughly every 600ms and whilst its sleeping the logic processing thread kicks in. So no time is wasted between ticks.

I have been re-writing the current inventory/equipment system to a more efficient system which illiminates multple calls for sending equipment and inventory items to client per a method.
