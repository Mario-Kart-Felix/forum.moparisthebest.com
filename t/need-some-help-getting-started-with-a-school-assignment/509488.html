Arcik:

There are several reasons we have a jar file for example applications, organisation etc etc.

Your post was very misleading.

There are several components to a jar, a few including:

[ul][li]Smaller (disk space wise) than a group of classes (a jar is just a zip archive)[/li]
[li]You can optionally include a manifest into the archive, which allows for easy execution[/li]
[li]Ability to load a magnitude of classes into the JVM with ease. “java -jar foo.jar” ‘foo.jar’ may contain as many classes as you’d like.[/li]
[li]Finally, as Arcik stated, organization. Mainly when working with external libraries[/li][/ul]

Arcik:

Once you import a jar you would already have access to the methods etc inside it, but if you don’t know what those class’s called or what methods are within them i’d suggest decompiling it since there are only 4 class’s.

First of all, you don’t import a jar, you include them into your class path. Secondly, if you’re using a 3rd party jar and don’t know what it contains then you have issues, there are Javadocs for a reason.
