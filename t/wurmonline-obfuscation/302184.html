WurmOnline Obfuscation?

General Programming
Jython July 31, 2016, 10:59am #1

Okay, naturally, WurmOnline uses obfuscation. I’ve started on a quest to document all the different obfuscation techniques it uses:

  1. Renaming of methods, fields, classes, and packages into names that windows would deem the same files (e.g. aA.class aA.class aa.class and AA.class are all different classes but same files under windows) but the JVM would happily accept (due to having case-sensitive descriptors). There is no way to reverse these into the original names, but the cleanest thing would be to rename them all to names that will be easier to navigate through, usages of them be decompiled to non-ambiguous java source, and will end-up with class files that can be properly be manipulated under windows
  2. Local variable optimizing. It optimizes use of two local variable slots to use one; e.g.:

this = hp; if (this < 0.7f) { ... }
Obviously this cannot be recompiled. The closest to fixing this, I assume, would be to parse into a CFG. Then navigate through the code and see if two values or more of incompatible types were assigned to one index. You could find the start of where the later values were used, replace the indexes they are stored at, and replace all references to their indexes in their basic block’s successors with the new index.

I believe they use some light flow obfuscation on some methods (or maybe that’s just jad sucking), and an older post by Krichevosky appears to agree with this. If you have any information on this, and even how to attempt to fix it, please post on this thread. That goes for any other obfuscation I did not cover here.

Use ASM’s analyzer with a SourceIntepretor. That way you know which store corresponds to a load. The hard part would be separating out groups of them. I’m thinking take a store and add it to the group. Then for all of its corresponding loads add them to the group, then their corresponding stores and so on. Then when you have one group repeat until you don’t have any more ungrouped instructions. Then the easy part of just changing the indexes.

You will need to optimize the analyzer it a bit. Allowing frames to share SourceValues meant that the entire runescape client could be analyzed in 6 or so seconds with 50Mb or so of memory (I kept the result - you don’t need to). It still sucks though.
