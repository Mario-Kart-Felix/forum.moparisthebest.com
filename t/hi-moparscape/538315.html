So like I made a thing. It’s pretty cool and I’ve completed my goal for a rough demo before sitting down and designing a full-fledged game. Here’s a little backstory on it.

I researched the marching cubes algorithm for a long time, because I didn’t understand how it worked. After making a C#/XNA version of it, I realized how simple it was. I wanted to migrate it to the GPU, so since I had little knowledge of C++ I decided to use SlimDX to use DirectX 10/11. When it came time to implement MC, it occurred to me that C# wasn’t made for the DirectX API in a thin wrapper. So I switched to C++ and wrote down a series of major steps I would take, and if I succeeded I would sit down and design a true optimized and flexible engine. I did. Those steps were, taken from the word document:

  1. C++ Marching Cubes (direct conversion of C# version)
  2. C++ Marching Cubes on GPU
  3. C++ Marching Cubes Mesh Modifying
  4. C++ Marching Cubes Lighting

Step 1 turned out to be more than just a direct conversion because I had to completely write everything essentially from scratch. There were no basic shaders to use, no simple shader wrappers, no one-line texture loading/setting (well, you can sort of load textures with one line via D3DX11CreateShaderResourceViewFromFile but you’re still responsible for the sampler state and error handling), no automatic window creation, etc.

But enough of that. Here’s a screenshot with super basic terrain generation and modification. I dug into a mountain and build a sort of entrance around it. Notice how it’s dark inside the cave There is light occluding for sky and point light.

NOTE: I didn’t make the textures. The grass is Riemer’s and the rock texture is the first one that appeared on Google Images.

So why am I posting this here? Well I originally planned on keeping the development secret after finishing all 4 steps (I’ve been posting videos for each one), but I realized I really like talking about what I made. It’s great motivation and gives the feel of an accomplishment for a step. It’s also really important to build up a community if there is any interest in popularizing/commercializing a game. If I went away and months down the road showed off an alpha, it wouldn’t magically become widespread, so I must build it up over time.

In conclusion of the boring stuff: I made this entirely with C++ using nothing but the standard library, the Windows API, the DirectX SDK/DirectX 11, a conversion of some C# simplex noise code I used in my old voxel engine (I don’t remember where I found it), and some public marching cubes tables. No one with any sanity would create those themselves. I used Visual Studio 2013 because it has some helpful features, like C++ 11 (more extensively than 2012) and build-in HLSL compiling support.

Okay. So this engine is completely unoptimized. There are no “chunks”, level of detail support, CPU- and GPU-combined mesh generation, or anything like that. It’s very straight-forward. Not to say it performs terribly, but for example, all mesh is generated every frame. When you modify the terrain, the entire ‘noise map’ texture is sent for the whole map. If you have a high-performing graphics card that supports high bandwidth transfers, this won’t be an issue for you.

Requirements
-A graphics card supporting DirectX 11 and Shader Model 4. I’ve run this on a school computer with an integrated chip supporting DirectX 11 and the screen flickered on and off until the computer blue screened. This might happen to you, so be cautious.
-About 32MB memory for the program (high, I know)
-At least like 16MB RAM for your graphics card. This is a rounded-up estimate. You probably really only need like 8 to store the textures. I’m not sure how the geometry shader output will affect the memory.
-A graphics card/slot that supports high bandwidth operations is recommended
-Support for Direct Input 8
-You can likely get away with a poor CPU. The only “heavy” work being done on the CPU is the initial generation and the light calculations.
[size=10pt]-Visual C++ 2013 x86 redistributable - http://www.microsoft.com/en-us/download/details.aspx?id=40784 (The 64-bit build has no benefits over the 32-bit, so there’s no point in releasing a 64-bit build)[/size]
-If it doesn’t run for you, install this: http://www.microsoft.com/en-us/download/details.aspx?id=8109

Controls
-WASD to move
-Left click to remove mesh, right click to place
–If holding shift, it will remove/place whole blocks at a time. If not, it will do so gradually
–If holding control, it will just attempt to update the light. This was for testing purposes and has no use
-Space to move the point light
–You can hold space and it will follow you. It has a delay that will allow it to only update every 4 frames. It doesn’t impact performance much if it’s every frame, but I wanted to see how smooth it would look with a delay and I think it’s fine. It also has a light level of about 50%. Again, not that it’ll really impact performance if it was at 100%, but I wanted to differentiate it from the sky light. The engine has support for an infinite amount of point lights, but I was too lazy to implement multiple light placement, so this is all there is.

My specs as a reference (I get over 300 fps)
-AMD Radeon HD 7870 (2GB GDDR5 RAM)
-AMD FX-8350 @ 4.00GHz
-8GB DDR3 RAM

Download: http://www.mediafire.com/download/ort47r8vq2jatr0/Freeform%20Terrain

Notes
I’ve tested it and I think everything works fine. I’ve had pretty bad luck with the lighting though, so if anything seems like it’s wrong, let me know. As for artifacts regarding the lighting, I’m aware that the mesh might need to be about 2 blocks dense for the light to completely cut off. If the mesh is thin, it’ll look like the light is bleeding through but it doesn’t travel through the rest of the dark area. This is okay, since I haven’t tweaked that part much. It doesn’t actually impact the lighting either.

I’m also aware that resizing the window doesn’t resize the internal resolution. The code says it should, but the program suggests otherwise. What happens is the image just gets stretched and there is a weird offset with the camera position and target. I’d recommend just keeping it as the default resolution (720p).

Also the mouse doesn’t get centered or hidden. This is because it was faster to put the mouse at a random position that is likely to be inside the window rather than add the window’s position to half the size.

Finally, vsync is enabled and there’s no way to turn it off. I never added my own framerate limiter and this was the easiest way to do it. That said, there is no frame-skipping or any other method of benchmarking. I’ve just been using Fraps to measure my framerate because, even though it might be a little off, it’s accurate enough to determine if I’m getting 60fps or not.

Anyway, enjoy and let me know what you guys think. I do have plans to develop a full-fledged game out of this and have a lot already planned. Since you people are doing me a favor by testing it, provided I do get to a commercial release at some point, everyone here would get it for free.

~Lin
