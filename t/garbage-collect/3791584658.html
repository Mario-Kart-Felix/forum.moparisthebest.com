The initial capacity of a HashMap is 16 objects that is changed dynamically. Not defining a original size for the hashmap will cause it to resize quite a few times (time and memory consuming). Lets say you define a size thats half the size of the array. You have a hashmap that initialized 47 milliseconds (atleast on my PC) faster then a 2000x2000 array. So you have a 47 millisecond gain but spent 33% more memory.

Now comes in the part where you actually start adding and removing elements from the HashMap. Instead of actually knowing the direct location of the tile you’re looking for (x/y) you pass them to the HashMap that hashes them into a key and loads it from the internal array (assuming you don’t have a collision). If you have a collision, loading the object will take even longer depending on how collisions are handled (http://en.wikipedia.org/wiki/Hash_table#Collision_resolution).

The same issue starts with adding an item. The location is hashed and if a collision is detected it also needs to be handled.

Even though HashMaps are generally considered to be O(1) complexity you still spend quite a bit of time hashing and handling collisions which ultimately ruin your initial 47 second gain (adding 2000x2000 items to a hashmap was 4x slower then adding them to an array on my PC).

You do not win in speed, nor in memory.
