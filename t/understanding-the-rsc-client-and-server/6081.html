In this “series” of tutorials I hope to help you understand how the RSC client works and provide you with the tools you need to start developing your own server.
These tutorials will not be ‘noob friendly’ this meaning that you must have some knowledge of Java and there will be no copy&pasting.

I will be using Mudclient 203.
Netbeans would be a great help because of the built-in refactoring functions.

[size=20pt]Refactoring[/size]

Refactoring the RSC client is almost essential to the development of you’re own server. It will give you and understanding of how the entire client works together.

Refactoring is a disciplined technique for restructuring an existing body of code, altering its internal structure without changing its external behavior. Its heart is a series of small behavior preserving transformations. Each transformation (called a 'refactoring') does little, but a sequence of transformations can produce a significant restructuring. Since each refactoring is small, it's less likely to go wrong. The system is also kept fully working after each small refactoring, reducing the chances that a system can get seriously broken during the restructuring.
We will not be as much "refactoring" the code as simply renaming fields and method thus making the overall client much easier to read and understand. If you would like to know all about refactoring then check out www.refactoring.com

The most simple way to start is to look at the strings in the mudclient.
You should find something like this very quickly:

egc.cdn("Quest Points:@yel@" + egm, (l + c1 / 2) - 5, l1 - 13, 1, 0xffffff);

Looks kind of confusing at first huh?
But if you have any Java knowledge at all you should quickly notice that ‘egm’ is the amount of quest points this character has.
So you can now rename ‘egm’ to something like “questPts”.

It seems that ‘cdn’ is the method drawing this text on the screen.
Maybe we should look around for more uses of ‘egm’.

After a little bit of searching you’ll find that ‘cdn’ is used to draw text on: trade windows, bank windows, stat menus, etc.
What do all these have in common you ask?

They are all semi-transparent surfaces.
Now that we are fairly sure that ‘cdn’ draws text on semi-transparent surfaces we could rename it to “drawTextOnTransparency” or of the like.

Maybe we should have a look at ‘egc’ since “drawTextOnTransparency” was a member of it.

Search a bit for ‘egc’ and you will find more drawing of strings and even some unknown things like:

egc.cbc(byte0 + 5, byte1 + 222, 398, 0);

Eventually you will find the declaration of ‘egc’:

private k egc;

This should immeadiatly tell you that egc is just an instance of class ‘k’.

Maybe we should look into class ‘k’ to find out what it does.

Open class ‘k’ and look for the recently renamed “drawTextOnTransparency”.
What? It wasn’t in there?

Oh wait, look, class ‘k’ is just and extension of class ‘j’.
Open up class ‘j’ now and we’ll have a look at it.

Immediately notice that class ‘j’ has something to do with the game image from this code:

public class j implements ImageProducer, ImageObserver

Look around a bit more and find that maybe ‘j’ doesn’t just have a little something to do with the game image.
That it practically has everything to do with the game image.
If you’re wondering how I came to that conclusion search around more until you notice that class ‘j’ clips and plots sprites and many other things.

Now we are sure that ‘j’ does the game image so we could rename it to something like: “gameGraphicsFactory”

Well, wait, what does ‘k’ do then?
Head back to ‘k’ and you’ll see that, meh, it’s just a tiny bit of code. What can it do?

Look around and you will see that a tiny bit of code goes a long ways in this case.

Here is the only function in class ‘k’:

Hmm it seems to be calling a lot of methods from ‘dai’(which is just an instance of the mc)

Back to the mc and search for ‘fck’ because it’s one of the methods being called from class ‘k’.

You should find this code eventually:

final void fck(int arg0, int arg1, int arg2, int arg3, int arg4, int arg5, int arg6) { int l = eeh[arg4]; int i1 = dim[arg4]; if(l == 0) { int j1 = 255 + i1 * 5 * 256; egc.cal(arg0 + arg2 / 2, arg1 + arg3 / 2, 20 + i1 * 2, j1, 255 - i1 * 5); } if(l == 1) { int k1 = 0xff0000 + i1 * 5 * 256; egc.cal(arg0 + arg2 / 2, arg1 + arg3 / 2, 10 + i1, k1, 255 - i1 * 5); } }
Hmm it seems to just call some more methods from class “gameGraphicsFactory”.
You might be thinking “wtf is he talking about? I don’t see class “gameGraphicsFactory” anywhere.”
If so you might want to brush up on your Java skills.

‘egc’ is an instance of class ‘k’ which extends what is now “gameGraphicsFactory”.
Since ‘cal’ is not a method of class ‘k’ we can assume it is a method of “gameGraphicsFactory”.

So what class ‘k’ is really doing is just calling the methods that were declared in “gameGraphicsFactory”.
Class ‘k’ does a lot of the drawing with just this tiny bit of code.

We can safely assume that ‘k’ and now be renamed to something along the lines of “gameGraphics”.
We’ll call it that since “gameGraphicsFactory” does all the work but “gameGraphics” is almost like that boss. Telling “gameGraphicsFactory” (the workers) what to do.

Back to the code that you’ve most likely forgotten about. 

egc.cdn("Quest Points:@yel@" + egm, (l + c1 / 2) - 5, l1 - 13, 1, 0xffffff);

We know now that egc is really an instance of “gameGraphics” so we can probably rename it to something such as “gameImage”.

Whew!! All that work. Now Let’s see what we ended up with.

Little bits of renaming can go a very long way.

Keep on renaming using the methods that I’ve shown you:

[size=20pt]Client Server Communication and Data Management[/size]

Before we delve into how the client and server communicate with each other you should have a basic understanding of how input/output streams and sockets work. And because I do not wish to reinvent the wheel I will simply link you to a tutorial that I feel explains it well enough. Link: http://www.javaworld.com/javaworld/jw-12-1996/jw-12-sockets-p2.html
RS works on a TCP/IP connection in case you were wondering.

Client Side Data Management

Now from the tutorial you should of learned that input and output streams deal only in bytes.
Everything has to be converted to a byte before it is sent.

Quick vocabulary lesson:
Word - 16 bit integer
DWord - Normal 32 bit integer
QWord - 64 bit long

RS save a little bit of bandwidth by using 16 bit integers instead of the normal 32 bit when the number never reaches above 32,000.

Now onto how the client manages all these bytes that it recieves from the server.
Maybe by now you’ve noticed that in t.java there are some shifty operations going on.
Haha, get it? Shifty operations…

Yah well since that really isn’t funny I guess we will move on.
You’ll probably be wondering what all bit shifting is doing.
You should have a good understanding of how shifting and bitwise operators work.
So Google it until you have a firm grasp on it.

Well let’s take a look at one of the methods.

public static int fmi(byte arg0[], int arg1) { return ((arg0[arg1] & 0xff) << 8) + (arg0[arg1 + 1] & 0xff); }
Hmm this method seems shift an entire byte out of the arrray then combine it with the next byte, thus forming the 2 byte Word that is returned.
If you studied your bitwise operators well enough you should of immediately known that this returns a Word read from 2 bytes out of the supplied array of bytes.
2 Bytes = 16 Bits = Word

Keep looking throughout class ‘t’ and rename all the methods that you see fit.
If you didn’t catch it yet in t.java before each character’s name is sent to the server or client it is compressed/converted into a QWord.

This is a very important part as when you write functions for your server you need to know what type of data to send to the client.
If you send the wrong type then it will obviously not work.
Later on we will analyze how to know what kind of data to send when writing the server-side functions.

On to how the client and server communicate.

Client to Server Communication

The client and server communicate to each other using Frames.
Each Frame has it’s own unique ID that identifies what kind of function it is calling.
Most Frames have a payload of information that will be acted upon when the server or client recieves it.

Frame Types - http://rscd.eggsampler.com/wiki/?title=Packet_Format
That explains it much better than I could probably do so read it a few times.
If you’re wondering why does it send the last data byte or parts of the data.
I personally assume that the client/server does a check to make sure that it recieved all the data. I may be wrong though so please someone correct me if I am.

With the frames the only part we will usually be dealing with is the command and data. The length part is handled by the client or server but you may have to deal with in some of the more complex packets.

Hopefully you’ve noticed by now a lot of super.aag.* in the mudclient.
If you think they all seem to be related to actions than you are absolutely right.

If you haven’t figured this out yet then I suggest you go study the mudclient because it would be best if you learn it on you’re own. It will give you a much better understanding then if I just tell you.

You should of also noticed that whenever super.aag methods are called that it seems to fit to the packet structure that we discussed earlier.

How the super.aag methods work:
First the ID is sent.
Then if there is data that needs to be sent it is also.
The packet is constructed and send off.

This is relatively what you will see everytime except for a few special cases.

Each ID tells the server to execute a certain action.
If data is sent then usually the server will use that data in the action.

Let’s take a look at how the MC decides what packet to send and where it gets it’s data from.
When you click something like attack or trade first the client sets an action ID then later it sends off the packets that
the action ID said to send.

You should have probably noticed a lot of code like this by now:

efd[efm] = "Duel with"; egn[efm] = "@whi@" + dfm[i3].gme + s1; eig[efm] = dfm[i3].gmh; eih[efm] = dfm[i3].gmi; dnm[efm] = 2806; ege[efm] = dfm[i3].gmf; efm++;
Obviously it’s part the menu the client draws when you right click something.

In this case ‘2806’ would be the action ID.
We should probably search a bit for 2806 in the client and see what else we can find.

Hmm the only other occurence of ‘2806’ in the client is this:

if(i2 == 2806) { super.aag.hdn(103); super.aag.hcn(j1); super.aag.hdf(); }
Well we talked about what the aag methods do so you should know that this is the code that is sent to the server when you try to duel with someone.
Obviously 103 is the Duel packet’s ID.
You can make an accurate guess that j1 must the other player’s ID or name.
If you’ve refactored enough you won’t have to guess and will know that it is in fact the player’s serverside index.

You can keep doing this and quickly figure out what a lot of the packets functions are.

Recieving Data
This is a bit of a shorter one so if you need more explanation simply tell me “Hey I didn’t really get how (x) works can you explain it a little bit more” and hopefully I’ll be in a good mood and try to help out a bit.

Because Server -> Client communication works on the exact same concept as Client -> Server I’m not really going to do a tutorial on it.
You should be able to open up client.java and study it a bit and get as good as an understanding as at tutorial would give you.

Instead we’ll look at how the Client and the Server recieve their data and act on it.

Hopefully you’ve renamed most of t.java as it will be a big help in this part of the tutorial.

Open up the mudclient and search for the ‘adf’ method.
This is where the client recieves and acts on all the information it gets from the server.

I’ll start you out by letting you know that arg1 is the packet length and arg2 is the data.
If you can’t figure out what arg0 is then you probably should go back and reread the entire thing.

You should be looking at some well-renamed code by now and that will make it a lot easier to figure things out.
Otherwise there is a very small chance that you will get anything out of this part of the tut.

There is a lot of action on the data isn’t there?
Mostly from class ‘t’ this is because, as I said before, t.java is used to extract ints, longs, strings and what not out of byte arrays. This will be a big help here.

This is how you can figure out what kind of data to send along with you’re packet.

We’ll take a look at packetID 204.
This is probably one of the easiest packets.
Here’s the code:

            if(arg0 == 240)
            {
                ecm = t.fmh(arg2[1]) == 1;
                ede = t.fmh(arg2[2]) == 1;
                elc = t.fmh(arg2[3]) == 1;
                return;
            }[/code]
Not very obvious huh? 
Well hopefully you were smart enough to go about renaming it all and the function of it becomes very obvious.
I guess I'll give this one away as it's not very important so you can't go toting "i g0t teh rsc server!" if you try to C&P.

Here's what it should look like renamed:
 [code]          if (command == 152) {
                autoCameraAngle = dataConversion.getUnsignedByte(data[1]) == 1;
                mouseButtons = dataConversion.getUnsignedByte(data[2]) == 1;
                soundEffects = dataConversion.getUnsignedByte(data[3]) == 1;
                return;
            }[/code]
Now it should be extremely obvious how this works. 
But I guess I will go ahead and give an explanation anyways. :)

autoCameraAngle, mouseButtons, and soundEffects are all booleans.
What this code does is check to see if the specified byte (data[1], [2], or [3]) is set to 1. 
If it is then it will set the specified boolean to true otherwise it will be false.

The use of this code would be upon a player's login it would retrieve their settings from a database and then send it to the client so that the player does not have to redo his settings everytime he enters RS.

Other packets are more complex but if you renamed like I told you to you should have no trouble quickly finding out how to spawn item, fill shop windows, turn prayers on and off, etc.

On to how the server recieves data:
We'll be using frank_'s server source here. And although it's not the best (no offense frank_ :P) it's currently what's publicly available.

Open up client.java once again and take a look at the parseIncomingPackets method.
This is fully renamed so you shouldn't have a problem understanding it.

We'll take a look at how to create a simple dropping method. And yes you can copy and paste it but if you don't have an itemSpawn() and removeItem() method already it will be of no use to you. 

By using the methods we mentioned earlier find the id of the dropping command.
Or look at winterlove's tables and quickly find that it is 246. :)

So we need to add a case to handle the packet:
[code]case 146:

Obviously this will not work for you unless you have the right methods and arrays and variable and what not.

That was just an example showing how you could easily make a function to recieve and act on the packets you get from the client.
Very simple, if you’ve done RS2 server I think it acts in almost the same way so you should be relatively familiar with it.
