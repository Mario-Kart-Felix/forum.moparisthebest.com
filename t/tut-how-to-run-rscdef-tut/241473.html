Nice tutorial… And good work!

Also, note that there is one part in the res-folder called dist. This folder is used for distribution of your client, that is, the client you will upload for people to download and play on your server. Atleast that is the purpose of it. You have to change that in order to play on your server before you even distribute it to anyone else.

Here comes a brief explanation for my folder-names:

bin: Binary, which is where all the batch-files, MAKE-files and other files for running, compiling etc. will be located.
doc: Just documentation. For instance doc/jvadocs/ will contain the Javadocs.
ext: External, meaning external libraries needed to compile and run the application.
lib: This means library, and is where the compiled files (.class-files) will be stored.
res: Resources, and is the place to put all media (images, .xml-files etc.) for your application.
src: The normal source-folder, for your source-code.

Edit: I have seen one thing you forgot to do in your tutorial… You must also run the serializer.bat file, in turn to create the Area-files, to have it load all NPCs, ground-items, ground-objects and, when implemented, even the wall-objects, when you login.

system July 31, 2016, 2:55am #6

freaky5000: No, I don’t think you can actually edit these files without expecting it to be corrupted. But I may be wrong… Anyhow, the data is serialized, and must be deserialized using the same version of the class as it was when you serialized it. If you change different things in the class that will have it deserialized, it may be out-dated. The out-dating is not guaranteed, and can work, but usually you should think of it as if it will be out-dated if making changes in that class. And the holes you were asking for; I can say that the only holes I know is that it is harder to make it usable for a highscore-table (make an Applet and Servlet for this), aswell as having multiple servers (worlds) would make it harder… You would do best in remaking the system in that case. The third con is as stated above… You should not change the class being deserialized, else it will just reset in this server-source.
