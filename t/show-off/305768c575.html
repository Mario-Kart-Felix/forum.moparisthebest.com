Kryptix: You shouldn’t have to send a recorded video from the client. That would both be in-efficient (require alot of band-width) and easy to go around (noone should be able to disable it through changing the behavior of the client). Instead, there are two alternatives I can think of (one that is divided into two separate approaches):

  1. Just keep a fixed-size array of Strings (or possibly a List storing an Object containing the String-representation and a long for the time it was created) for every player. What the String should contain, is a formal representation of every action and event that a player do in-game. If you’re using the array-approach, you should never delete any of the String-representations of the actions and events, just add on-top of the others (override), for instance using the % operator (index++ % length). If on the other hand, you’re using the List approach, every once in a while, you should remove those that contains a time less than (current time - maximum store time) - if you want to keep the data to a minimum. Of course there are many other classes to use instead of a simple List, but List makes it easier to explain, since it is more commonly used (and people knows it better than many other similar classes).

  2. The other alternative is pretty much the same, just with the exception that you should store the data for all players globally (perhaps in a database), and in an ordered fashion (perhaps without ever removing any), and change one client to support reading the said data to produce a replay of any particular player in the game at any time. The saved data should include everything you need to create a replay - position, appearance, inventory etc. for all players affected (for instance the players that were logged in at the time).
