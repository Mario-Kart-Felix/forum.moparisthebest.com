Zorian and I have been working on a more secure system using both unique password salts and a much stronger SHA-512 encryption.

A unique salt is generated for every player and stored in the table containing all the players. It is then placed at the beginning of the password prior to being encrypted:

<?php $password_salt = generateSalt(); $password = hash("sha512", $password_salt . trim($_POST['password1'])); ?>

That salt seems a bit long… I believe it’s common practice to use 128 bits at most.

If you start making salts too long I’d suspect that you’ll run have more collisions, and then you’ll need to use another hashing algorithm to ‘ensure’ it’s not a collision.

In addition, why would you trim the password? What if my password has a space at the end?

Also, if you release the database all they have to do is bruteforce it, and seeing they have the salt it wouldn’t be adding any additional work… Granted, they couldn’t re-use the calculated hashes, but that’s beside the point.
