A few people have PM’d me about this so I thought I’d release it. There’s no error handling and giving it the wrong data will break it but providing the altars are located sensibly it will work fine (as it does on RSCEmulation). This won’t work at different map heights (underground and upstairs) but all the altars on RS2 are on the main ground level so it wasn’t programmed to be like that.

[code=java5]public static String locateAltar(int playerX, int playerY, int altarX, int altarY) {
String string = “”;

    if (Math.abs(playerY - altarY) > 0)
            string = (playerY < altarY ? "south" : "north");
   
    if (Math.abs(playerX - altarX) > 0)
            string += (string.equals("") ? "" : "-") + (playerX < altarX ? "west" : "east");
           
    return "The talisman pulls towards the " + string + ".";

You can’t compare a string value in java with ==, it doesn’t work like you think, so that needs changed. Also, you could simplify it a tad more with this (same operations, less code duplication):

                string += (string.equals("") ? "" : "-")+(playerX < altarX ? "west" : "east");

public static String locateAltar(int playerX, int playerY, int altarX, int altarY) {
        String string = "";

                string = (playerY < altarY ? "south" : "north");
       
        if (Math.abs(playerX - altarX) > 0)
                string += (string.equals("") ? "" : "-")+(playerX < altarX ? "west" : "east");

}

Would probably work at least with the Sun compiler right now, but not because it should work, but because of a compiler optimization which produces code that looks like this:

public static String locateAltar(int playerX, int playerY, int altarX, int altarY) {
        String compilerGeneratedStaticString = "";
        String string = compilerGeneratedStaticString;
       
        if (Math.abs(playerY - altarY) > 0)
                string = (playerY < altarY ? "south" : "north");
       
        if (Math.abs(playerX - altarX) > 0)
                string += (string == compilerGeneratedStaticString ? compilerGeneratedStaticString : "-")+(playerX < altarX ? "west" : "east");
               
        return "The talisman pulls towards the " + string + ".";
}

But you can’t always count on it being optimized like that, which is why you should use .equals.

Also, with code like this:

public static String locateAltar(int playerX, int playerY, int altarX, int altarY) {
        String string;
       
        if (Math.abs(playerY - altarY) > 0)
                string = (playerY < altarY ? "south" : "north");

                string += (string == "" ? "" : "-")+(playerX < altarX ? "west" : "east");

That doesn’t throw a nullpointerexception because you can compare ‘null == “”’ legally, but after you change it to .equals, you can’t call a null.equals without throwing an exception.

So a combo of dumb luck and compiler optimizations might have saved you.

The_Enforcer August 1, 2016, 11:53am #13

Using == on objects actually compares the memory addresses of these objects, so it should only be used to determine if two object references are equal.

I created an illustrative example to show this.

Example:

public static void main(String[] args)
{
	String str1 = "kryptix_needs_to_learn java";
	String str2 = "kryptix_needs_to_learn java";		
	System.out.println(str1 == str2);
	System.out.println(str1.equals(str2));
		
	str2 = new String("kryptix_needs_to_learn java");	
	System.out.println(str1 == str2);
	System.out.println(str1.equals(str2));
}

Output:

true
true
false
true

At first str1 and str2 point to the same object, that’s a static optimization, because the content is known at compile time. Then str2 is updated to contain the same data again, however this time it points to a different object and using == returns false.
