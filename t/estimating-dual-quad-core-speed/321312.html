moparisthebest July 31, 2016, 1:24pm #7

The number of instructions per second is what you should be interested in. I know back in the day, Intel Pentium 4’s @ 2Ghz and AMD Athlon XP 2000+ @ 1.2Ghz, even though the clock speed was lower on the AMD it performed more instructions per cycle and therefore ended up being the faster processor. Ghz isn’t everything.

In the case of multi-core CPUs, you can’t “estimate multi-core speed”, because it doesn’t work like that. It all depends how the program is written, if it is written in a very linear way, even if you have 80 cores it will only run on 1 core, and be the same speed as if you just had a 1 core CPU. Some programs can take advantage of many cores though, apache comes to mind since it spawns a new ‘instance’ with every client, theoretically if you had 80 cores and 80 clients they could each run on a separate core and be 80x faster than an equivalent 1 core CPU. Of course in practice this isn’t the case because the CPU isn’t the bottleneck there, rather the network, RAM, or hard disk would be.

Bottom line is unless you are running a server, or very CPU intensive programs like graphics rendering (not games) then a dual-core CPU will be more than sufficient, if you get a quad-core you probably won’t use half of it at all most of the time. And an 8-core would be a huge waste.
