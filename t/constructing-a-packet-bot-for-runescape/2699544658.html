BAsh July 31, 2016, 6:35am #29
At this point, I think a packet based proxy bot is well within reach. I've given it some thought, and I came up with a few benefits: 1.) Client footprint would be minimal (very little bytecode engineering... if any). 2.) Random event detection would be flawless. 3.) Easier to maintain in the long run. 4.) Can be designed to be absolutely undetectable, since we're operating at the packet level. That's all I can really think of off the top of my head, but there are probably a ton more benefits.
1.) You still have to load and run the client and that takes up resources. 2.) You can already write flawless anti-randoms with the right hooks. The reason why they aren't flawless in RSBot is because people are too lazy to write them properly. (Not sure if this is still relevant but we now have 'client script' running. The states of which weren't hooked before and needs to be for perfect anti-randoms). What you are proposing to do is exactly the same but on the byte level before the client has a chance to unpack it and store it in its internal structures. That requires you to understand the communication to the client while an easier way would be to simply rip it out of the client's memory. 3.) See 2. I would like to add that if jagex changes the protocol even slightly then it would break everything. If you simply read the client's variables some hooks will break but the majority will still work. 4.) It can't be designed to be undetectable. The only exception to this is if you build a packet bot as opposed to a packet proxy bot. Either option is infeasible. If you use a proxy bot there will be information that won't match up. To be honest while I've deobbed a client and mucked around with it a bit I've never actually understood how it runs but what little knowledge I do have should be sufficient. Assume you send a packet to the server saying you clicked on a specific object. Everything is fine. Then the server asks the client for some information. Lets say the number of times the mouse has been clicked or if the object is on screen or the number of actions taken place or a large number of other things. Sure you could modify the responses as well but then you start making a packet bot as opposed to a packet proxy bot and that is much harder.

Regardless of what form this packet based bot takes… it would require an incredible amount of effort and every revision that jagex released would require effort as well. Much simpler to follow rsbot and actually run the client. Let it do the donkey work which you know for a fact is undetectable in terms of internal state and packet parsing and then just rip the information out of its fields however you wish to go about that.
