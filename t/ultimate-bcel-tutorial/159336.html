[size=5]1 - Basics[/size]

First of all, lets start with something nice and simple. lets edit the title of the gui to “bob”. Right, much of the next bit can be found in the last tutorial in the series, this just goes into a bit more detail and I will say when I have finished with the parts repeated. First of all we will need to load up the dummy class and save this into a ClassGen object. A ClassGen object basically is a temporary class file which can be edited to your liking before being dumped as a new class. To create a new ClassGen object use the following:

try { cGen = new ClassGen(new ClassParser("DummyClass.class").parse()); } catch (ClassFormatException e) { e.printStackTrace(); } catch (IOException e) { e.printStackTrace(); }

Nice and easy :P Just so you know, in case you are hugely noobish, you will need to enter the path to the class you want to open where it says “DummyClass.class”.

Now the class is loaded, we can actually do something to it. First of all, the most commonly used method of finding code to edit would be to search through all the methods in the code and see if the [b]unique[/b] thing you are looking for is there. I really do mean unique when I say that as you really don't want to edit certain stuff accidentally. We can get all the methods in our DummyClass by using cGen.getMethods() which will return a Method[] object. (Info on the Method class can be found at http://jakarta.apache.org/bcel/apidocs/org/apache/bcel/classfile/Method.html). We can now check through the all the methods to see if there are any similarities to ours. For simplicities sake, we can check to see if the name of the Method is buildGUI. This is an example of how you could look through the code to see if we have found the buildGUI method:

Method build = null; for(Method m : cGen.getMethods()) { if(m.getName().equals("buildGUI")) { build = m; System.out.println(“Found buildGUI”); break; } }

Right, we now have a way to find the method we want (how sexy :P) now we need to search for the code which sets the title of the GUI. We now need to open up your Bytecode editor and search for where the title of the GUI (which is “Dummeh!”) is. If you read the BCEL manual like I told you earlier, you would most likely know that the Opcode ldc is used to get a constant from the Constant Pool. Look at the image below:


These are the first few lines of the Bytecode of the buildGUI method from DummyClass. You should now be able to see where the ldc loads the constant “Dummeh!”. It is also loaded from the 20th index ([i]not sure if the correct phrasing used[/i]) in the Constant Pool. As this is the first LDC in the method, we can search for it easily (Runescape Bytecode editing would most likely be more difficult). Before we search we will need to get all the Bytecode instructions from the buildGUI method. To do this, we need to add the following lines to our code:

The ConstantPoolGen object basically creates a temporary ConstantPool, same with MethodGen yet, obviously, with a Method. The instructionList object will basically return all the Bytecode instructions found in the current method, simple really :D

So, we now want to edit the LDC so that it is something else. The first step of this is to make a new LDC object (BCEL contains objects for all the Opcodes that Bytecode has to offer). The constructor of an LDC objects wants an integer as its parameter and this is the same as the location in the ConstantPool that the String can be found. Basically, the easiest way to add any string to the Constant Pool would be to use:

cPoolGen.addString(String str);
This will return the index at which the new addition to the Constant Pool is stored. Now we have a way of adding a string to the Constant Pool, lets save this to an LDC object:

LDC ldcIns = new LDC(cPoolGen.addString("spoon"));
Its pretty easy now to replace the old LDC instruction with the new one with the setInstruction(Instruction ins); method of the original Instruction object. Lets replace our old instruction with the new one:

titleLDC.setInstruction(ldcIns);
We have now successfully edited our first bit of bytecode, now to add it to the InstructionList and do a bit of cleaning up (you will need this at the end of all your BCEL stuff):

iList.setPositions(); methGen.setInstructionList(iList); methGen.setMaxStack(); methGen.setMaxLocals(); methGen.removeLineNumbers();

Now all we need to do is replace the old method with the new method and then dump it in place of the old Class. To replace the method we will need to call the setMethod method for our ClassGen object and the first parameter is the old method and the second is the new one:

cGen.replaceMethod(build, methGen.getMethod());
To dump our new class just use the following:

try { cGen.getJavaClass().dump("DummyClass.class"); } catch(IOException ex) { ex.printStackTrace(); }

Lovely :P Lets bung that all together and it should give us this:

[size=5]2 – Basic Searching[/size]

There are other ways to search through the Bytecode which are more sophisticated, better suited for certain situations and are much more accurate. Seeing as there are a large amount of Strings loaded from the ConstantPool in our Dummy class, it seems logical to see if its possible to find a string loaded from the middle of our buildGUI class. There are many ways of doing this, the first, being the easiest, would be to search through the class for an LDC instruction and if one is found, check it if it equals what we are searching for. Such as, if we want to change the ActionCommand of our button from “Moose” to, lets say, “Spamdiddleyamerous”, all we would need to do would be to find the instance of an LDC instruction and if it equals “Moose”, change it to what we want.

[size=3]2.1 – Searching Specifying String[/size]

So, our searching from before was the following:

for(int f = 0; f < iHandles.length; f++) { if(iHandles[f].getInstruction() instanceof LDC) { titleLDC = iHandles[f]; System.out.println("found LDC"); break; } }

We want to add a way to check the instruction to see if it equals “Moose” and the best way to do this would be the following:

((LDC)iHandles[f].getInstruction()).getValue(cPoolGen).equals("Moose"))
Obviously this looks disgusting, yet it is easy to understand to be honest. Basically, the first part is getting the instruction and casting it to an LDC object, we then get the value, and have to specify the ConstantPoolGen object as one of our parameters, then check if it equals what we are searching for.

for(int f = 0; f < iHandles.length; f++) { if((iHandles[f].getInstruction() instanceof LDC) && ((LDC)iHandles[f].getInstruction()).getValue(cPoolGen).equals("Moose"))) { LDC tempLDC = (LDC)iHandles[f].getInstruction(); if(tempLDC.getValue(cPoolGen).equals("Moose")) { titleLDC = iHandles[f]; System.out.println("found LDC"); break; } } }

[size=3]2.2 – Searching through the Constant Pool[/size]

There is yet another way to search for a certain piece of data (as long as it is a constant) and edit it, this is by searching through the ConstantPool. Here we are not limited to searching for strings, we can search for any constant in the ConstantPool. There is a huge amount that could be done here, yet it can be relatively difficult to do. Anyway, lets see how we get on (seeing as I have never tried editing the ConstantPool before writing this tutorial and I thought of it part way through. This could turn abysmally wrong :P).

Right, after a bit of research I found out it is actually easier to edit the Constant Pool although in the BCEL api, it says in the description about some of the methods I will be using, “Use with care!” :S Well, looks like this isn't a good idea after all but I'll show you how to do it anyway. First, you need to know that all Strings are stored in the Constant Pool as two values, a Utf8 value and a String value. Don't ask me why, they just are. We also need to know that when editing the one value, we need to edit both the Utf8 and the String for it to work. One last thing that we need to know is the fact that you cannot create a new String in the Constant Pool by specifying a String in its constructor in BCEL, we first need to create the Utf8 Constant object, then use the index of this to create our String. If you still understand, continue, if not, don't worry too much, this isn't too important. 

Anyway, we need to first create our ConstantPoolGen object, like before, then we need to search through for a Utf8 Constant with the value “Moose”. We then use the index of this to edit both the Utf8 constant then the String Constant:

[code]ConstantPoolGen cPoolGen = cGen.getConstantPool();

int utf8StringPointer = cPoolGen.lookupUtf8(“Moose”);
if(utf8StringPointer != -1)
{
cPoolGen.setConstant(utf8StringPointer, new ConstantUtf8(“Spoon”));
cPoolGen.setConstant(cPoolGen.lookupString(“Moose”), new ConstantString(utf8StringPointer));

[size=3]3.1 – Integer Constants[/size]

Right, lets create a new hugely basic dummy class in which all it does is print a constant number. Heres what I came up with quickly:

public class IntegerSearchDummy { public static void main(String[] args) { System.out.println(23); } }

As I said, unbelievably basic and trivial yet it works for demonstration purposes. So, how would we go about editing what is printed to 77? Now, its nice and easy really! Lets look at the bytecode, that is always good for giving us a clue as to what to do...


Right, as you can see, the place in which our integer is called has the Opcode “BIPUSH”. This shows that integers can be pushed onto the Operand Stack using the BIPUSH Opcode. Nice eh? Now, remember our basic searching function from before?

for(int f = 0; f < iHandles.length; f++) { if(iHandles[f].getInstruction() instanceof LDC) { titleLDC = iHandles[f]; System.out.println("found LDC"); break; } }

Thinking about this logically, what would you change to this so that it checks if the instruction is an instance of BIPUSH? We'd need to change LDC to BIPUSH of course!! Easy eh? So, now we know how to find a BIPUSH, lets edit the original BIPUSH to push the number 77 onto the Operand Stack. Because integers are not stored in the Constant Pool, we need to change the instruction setting the LDC instruction quite a large amount. Lets have a look at what it was before:

LDC ldcIns = new LDC(cPoolGen.addString("spoon"));
We are using the BIPUSH Opcode so lets change LDC to BIPUSH. As integers aren't stored in the Constant Pool, we know that cPoolGen will not be needed! This then leaves us with a predicament as to what goes in the parenthesis. Looking at the BCEL API it tells us that a new BIPUSH object takes a Byte as its parameter, it must be obvious to you now what the final result will be:

BIPUSH bipushIns = new BIPUSH((byte) 77);
How sexy! The source as always can be found in the attachment. It just saves me space by putting it there...

[size=3]4.1 – Basic setting of fields[/size]

Right now we're getting onto the good bit, but to begin with, lets just get an understanding of how to create a new field. I know we are unlikely to need this in the creation of our a bot but it's still good to know.

4.1.1 – Creation of new Fields

So, the actual creation of a field should be very easy. You first need to know that even though a Field is not a constant, the name and type of the Field is stored in the constant pool. This means that when we create a new Field using BCEL, we will need to have something in it which has a reference to our constant pool. So, we have our Modifying method and we have our cPoolGen object initialised, we now need to invoke the addField method of the cGen. Now, the parameter of addField is a field object but that just looks way to complicated, lets use a FieldGen object and invoke getField() to return its field. This is how I've created my new field in a dummy class:

[code]ConstantPoolGen cPoolGen = cGen.getConstantPool();

cGen.addField(new FieldGen(Constants.ACC_PUBLIC, Type.INT, “rabbi”, cPoolGen).getField());[/code]

Now, there's a few things that I need to go over with this. The first parameter asks for the access rights of the field being created. These could be “public static final”. Now, to transpose this into our method we would have add the following into our first parameter:

Constants.ACC_PUBLIC | Constants.ACC_STATIC | Constants.ACC_FINAL
Basically, all you do is bung all the different access rights to our new Field together with a “|” separating them all. Now, the second parameter is the type of field we wish to create. So, if our field was of type boolean, the second parameter would be Type.BOOLEAN. Obviously, you can look here http://jakarta.apache.org/bcel/apidocs/org/apache/bcel/generic/Type.html for all the different primitive types that can be chosen. The third parameter should be extremely obvious but if you didn't know, it is the name of the field and the fourth, if you don't understand it, what the hell have you been doing whilst reading this tutorial!!! :P

4.1.2 – Setting our first field

Now we should be able to create a new field and you should have a bit of an idea of how Fields and the Constant Pool work. Now, to tell you a bit more about it. Every time a Field is referenced in our code, a reference is put into the constant pool. So, anyway, lets work out how to first search for our field.

To start off, lets get all the fields currently in our class using the following:

cgen.getFields();
This will return all the Fields in our class as Field[]. Now, lets search through all the fields to check if their names are equal to the field we want to set (this obviously wont work in runescape bot making but it gives you an idea for the example. I will go into more detail very soon :P). So, what have we got so far:

Field rabbiField = null; for(Field f : cGen.getFields()) { if(f.getName().equals("rabbiMaster")) { rabbiField = f; System.out.println("rabbiMaster found!"); break; } }

Now we have a way to find the field named rabbiMaster (sorry for the random names) we now need to search for the place to add the code to set the rabbiMaster field to whatever we want. You should know already about searching for the method you want to add the field reference to and you can probably work out a way to find out where but for simplicities sake, we will just whack it at the end of our <init> method (which is the constructor for our class), but before the return. So, seeing as we are going to have to put it before the return, we will need to search for the return instruction at the end of our code... Nice and simple:

for(int f = 0; f < iHandles.length; f++) { if(iHandles[f].getInstruction() instanceof RETURN) { returnInstruction = iHandles[f]; System.out.println("found RETURN"); break; } }

Now comes the difficult bit. We need to add a new line of code to set our field rabbiMaster (which, in fact is an integer) to a certain number, lets say, 92. Now, I find the easiest way to find out how to add stuff to code is to actually add it to the source then see what it looks like in bytecode. Lets add a line to our source setting rabbiMaster to 9 to see how it is converted to bytecode:


This shows we need to add an ALOAD instruction, a BIPUSH instruction (although that was kind of obvious) and a PUTFIELD instruction. Now, the ALOAD opcode basically loads up a local variable at the index specified but seeing as rabbiMaster is not a local variable, it is set to 0 as its index. BIPUSH you should already know. Now, PUTFIELD is interesting. This will put the last instruction into the field specified by the PUTFIELD opcode. To add these to our method using BCEL is actually very easy fortunately. For ALOAD, we just have the following code:

iList.insert(returnInstruction, new ALOAD(0));
returnInstruction is the instruction that the ALOAD will be placed before and it is equal to the RETURN instruction at the end of our method. Now, the BIPUSH insertion is just as easy:

iList.insert(returnInstruction, new BIPUSH((byte)92));
Now, the last insertion is quite difficult. I touched briefly about how references to any Field is in the Constant Pool, now, this is where we use it:

iList.insert(returnInstruction, new PUTFIELD(cPoolGen.addFieldref(cGen.getClassName().toString(), rabbiField.getName(), rabbiField.getSignature())));
Right, we insert a new PUTFIELD instruction after the return, as you could guess, and the argument for the PUTFIELD constructor is the index at which the reference to our Field is in our class path, so we add a new reference to our field. This method asks for the Class Name of the Field, the Field name and the Signature. Now, you can either write it all out by hand or get each from the classGen and Field objects respectively.

Now you should be able to edit any primitive data type field in any class, how great do you feel now! You still have a long way to go but you are getting close to becoming a master of BCEL!

So, lets say we want to search to find the field chickenStick. There are two field integers in this class so it could be difficult trying to search for our field. Obviously there are easy ways to search for our field, yet in the runescape client, it would be no way near as easy so to emulate this, I am going to tutorialise to you how you could go about doing this.

First of all, we need to think of ways in which we could find the chickenStick field. In the runescape client, to find the correct field, I would get all the fields in the class that I know our field is in, search through all the instructions of that class to see if we can find the field being used in a certain way. This is, basically, like the Advanced Constant search I did earlier, look back or look in the attachment to see it. If our field is used in this way, we'd then know this was the field we were looking for. Sexy eh? This could even be extended to look through other classes to see if the field is used there in a certain way.

So, lets look in the bytecode for our dummy class to see a unique place in which chickenStick is bunged on the Operand Stack (I will use quotes for bytecode from now on as you should be familiar with it all by now):
getfield DummyClassWithLotsOfFields/chickenStick I invokevirtual java/io/PrintStream/println(I)V
We'd first check to see if we could find a GETFIELD instruction and check if it equals the name of the field we're currently searching through the code for. Then we'd check if the next line is an invokevirtual instruction. Obviously, there is just going to be one getfield instruction in this code but in the runescape client there would be hundreds!

Lets see how this would now be implemented into our code so that it find it:

[code]ConstantPoolGen cPoolGen = cGen.getConstantPool();

[size=5]5 – Creation of Methods[/size]

So far we should know how to search and edit constants and how to set fields. It doesn't sound like much but believe me, it's loads! You are well on your way to creating a worthy bot, yet there is just one hurdle left. This section is a bumper packed section which will tell you how to create a hook in the runescape client to return any field you want. How fittage is that! Now, lets start off by creating a new method that returns, lets say, an int. So, first off, we will need to create an InstructionList object which will hold all the instructions and a MethodGen object to generate our method:

InstructionList methodIList = new InstructionList(); MethodGen getMethod = new MethodGen(Constants.ACC_PUBLIC, returnField.getType(), Type.NO_ARGS, new String[]{}, methodName, cGen.getClassName(), methodIList, cPoolGen);

So, the InstructionList object seems easy enough yet the MethodGen object is very daunting, especially for someone new to BCEL. I even think of it as being extremely scary, yet fortunately, we have the BCEL api to help. Anyway, seeing as this is quite a complicated object constructor, I will take the constructor summary from the bcel api:
public MethodGen(int access_flags, Type return_type, Type[] arg_types, String[] arg_names, String method_name, String class_name, InstructionList il, ConstantPoolGen cp) Declare method. If the method is non-static the constructor automatically declares a local variable `$this' in slot 0. The actual code is contained in the `il' parameter, which may further manipulated by the user. But he must take care not to remove any instruction (handles) that are still referenced from this object. For example one may not add a local variable and later remove the instructions it refers to without causing havoc. It is safe however if you remove that local variable, too. Parameters: access_flags - access qualifiers return_type - method type arg_types - argument types arg_names - argument names (if this is null, default names will be provided for them) method_name - name of method class_name - class name containing this method (may be null, if you don't care) il - instruction list associated with this method, may be null only for abstract or native methods cp - constant pool
Ok, to start off with, I talked a wee bit about access_flags before, I can't remember where but I am quite sure I did. These can be expressed as “Constants.PUBLIC | Constants.STATIC” for a public static method. Now, the return type can be easily guessed, basically it will be Type.[i]objectType[/i] or, you can create a Type of an object by using Type.getType(class c1) where c1 is the objects class. Now, arg_types, are the types for the parameters in the method, and arg_names are the names of the parameters. Obviously, if there are no parameters, there will be no args so an empty array can be used. Now, the name of the method and the class name are very obvious so I'm not going into them. il is obvious and cp is obvious. Now, it really wasn't too hard was it! Lovely :P

Now what do we need to do? Lets add some bytecode to our method now as all we have at the moment is an empty shell of a method, yet, as we are using some relatively complicated instructions, we will need to use something called an InstructionFactory object to help us build some instructions. We will also append our instructions to the end of our InstructionList to add the bytecode, to aid simplicity. You will need to know how fields are accessed first before doing any coding. First of all an ALOAD_0 command is called, then the field is accessed, after this we will want to create our return statement. Now, examine the following piece of code to see how it's done:

methodIList.append(new ALOAD(0)); methodIList.append(iFactory.createFieldAccess(cGen.getClassName(), returnField.getName(), returnField.getType(), Constants.GETSTATIC)); methodIList.append(InstructionFactory.createReturn(returnField.getType()));

This should be relatively self explanatory now, the first append instruction is obvious. The only part of the second that may not be understandable would be the Constants.GETSTATIC. This basically asks for the opcode of the instruction. Now, the field of the Constants class can either be GETSTATIC or GETFIELD depending whether the field is static or not, respectively. Now the last append instruction should be self explanatory again.

Now, the final parts are just the same old end sorta stuff:

getMethod.setMaxStack(); getMethod.setMaxLocals(); cGen.addMethod(getMethod.getMethod());
