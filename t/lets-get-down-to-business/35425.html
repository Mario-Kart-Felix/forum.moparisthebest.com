Ok well it appears that the whole rs cheating scene is quiet at the moment so why don’t we get things moving again. These past few days I’ve been working on a new project of mine called rumacs. I hope to eventually develop it into a fully featured rs2 bot.

Rumacs is a native agent library built with the jvmti tool. JVMTI is basically an event driven api that lets you communicate directly with the jvm with events (and JNI). My first attempt to utilize this tool to my advantage in my quest to make a functional bot for rs failed. I quickly found out that a lot of events that the jvm was firing were lagging the client. I was quite upset because the model I was using would of allowed for a great functional bot. I was faced with the dilemma of re-constructing my bot to still allow this superb potential functionality and not cause the client to lag at an unplayable rate.

I was hoping to find a native BCI library that would allow me to make up for some routines that I had cut out due to the lag problem. I eventually found java_crw_demo. It wasn’t what I had hoped (in terms of functionality), but it did the job. This library only gives me 4 options, 2 of interest at the moment in development:

  1. Inject hook at the beginning of client methods.
  2. Inject hook at the end of client methods.
  3. Notify on object allocation events.
  4. Notify on object de-allocation events.

All we need is one home made java class for this bot and that’s all.

public class Rumacs
{
	private static native void hook_entry_native(Object thr, int cnum, int mnum);
	public static void hook_entry(int cnum, int mnum)
	{
		hook_entry_native(Thread.currentThread(), cnum, mnum);
	}
	
	private static native void hook_exit_native(Object thr, int cnum, int mnum);
	public static void hook_exit(int cnum, int mnum)
	{
		hook_exit_native(Thread.currentThread(), cnum, mnum);
	}
}

Let me now get into the technical details. During the jvm initialization, the methods in which classes we are interested in are modified by java_crw_demo. It basically injects hook_entry and hook_exit into, respectively, the beginning and the end of the rs methods of that class. My bot has a hook_entry_native and a hook_exit_native function which are called whenever a rs method is entered and exited. In my native function I use the current thread to get stack frame info to find out exactly what rs method was called (since it’s impossible for java_crw_demo to modify only a few selected methods of a class, it must modify all of the methods of one class). Processing goes from there.

Anyone who is still following me at this point should realize that this as potential of a native, functional bot. I am releasing a small demo of rumacs, open source, so you can see for yourself how it works. I included a script so you can run it, all it does at the moment is output whenever you click your mouse.

I’d like to thank pplsuqbawlz for his loader, I’m currently using it for my testing on rumacs. Although the beauty of rumacs is it can be run right off the official runescape client on their website, this will be default behavior once I can get a decent beta out of it.

One last comment before I leave you, if you want to modify rumacs.c and compile it go into src/ and run the following command:

bcc32 -tWD -n../bin rumacs.c

I can’t be assed to find the equivalent gcc command because… well… honestly, I hate gcc. Oh and also don’t forget to add jvmti.h, jni.h and other includes to your bcc config file.
