A simple misunderstanding. I am using the term hook in a more abstract way.

I agree with you that a jvmti approach is much cleaner and easier to implement. Also, because it is c-based it pretty much draws a line in the sand and just about everyone in the community will be left scratching their heads. However, I think you misunderstand the complexity involved with finding sense out of every event that you are going to encounter. As a case and point, how do you plan to figure out what classes you intend to hook for callbacks for class load or for field access/mutation events? You have to have a means of figuring out which fields or method calls are of importance to you. Otherwise you will be inundated with thousands of events (assuming you hook all of them) and no method of sifting the wheat from the chaff. One of the first issues I ran into when first trying what you’re doing is finding context. Sure it is fantastic to be notified of every single modification/access of every single field… but where do I draw the correlation to what change means what. This is the part that makes things a bit hairy.

Sure it’s incredibly easy to obtain a hook to get client data using jvmti events. The problem is once you have that hook in your hands you’ll be left with the ‘wtf face’ look thinking ‘ok… now what?’. Even if you hook on the main game loop (I’m assuming you mean the real main game loop code location and not some abstract ‘main game loop’), you are still left wondering 'ok now which of the 150 of these fields on this client is the field I’m looking for.

There is a solution to this problem. I just wanted to get that out there so that the kiddies have something to think about before they step out onto the field and play with the big boys.

byte July 30, 2016, 2:29am #27

Ah I see where you’re coming from this time.

In our case we encounter 2 types of hooks:

  1. Hook injected by java_crw_demo at the main game loop (yes, I meant the real main game loop).
  2. Hook created by JVMTI when a field is modified.

For #2, the events are only fired on fields of interest. What happens is when the rs class files are loaded I call a function which set watches on only several fields. This means that my field modification callback function is only called when those fields of interest are modified and not all fields of all rs classes. The callback function gives us tons of arguments to determine exactly which field of interest was modified then it updates its value in the agent.

You have to have a means of figuring out which fields or method calls are of importance to you.

http://java.sun.com/j2se/1.5.0/docs/guide/jvmti/jvmti.html#SetFieldModificationWatch. That’s the function I use to set fields of importance.

Using SetFieldModificationWatch means I need an updater to get wanted field names + signatures (should stay constant anyways) every update so I can get their new jfieldID. An updater shouldn’t be too hard, it could probably even be done in an awk script.
