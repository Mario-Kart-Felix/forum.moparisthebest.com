If you’re referring to depth-cueing, here’s how I did it.

private static final int ALPHA = ((COLOR >> 24) & 0xFF);
private static final int BLUE = ((COLOR >> 0) & 0xFF);
private static final int GREEN = ((COLOR >> 8) & 0xFF);
private static final int RED = ((COLOR >> 16) & 0xFF);

...

int[] depthBuffer = zBuffer.getDepthBuffer();

double distance1 = 150.0D;

for(int i = 0; i < zBuffer.size(); i++) {
    if(pixelData[i] != 0) {
        double distance2 = depthBuffer[i];
        double factor = Math.min(distance1 / distance2, 1.0D);
        
        int color1 = pixelData[i];
        int red1 = ((color1 >> 16) & 0xFF);
        int green1 = ((color1 >> 8) & 0xFF);
        int blue1 = ((color1 >> 0) & 0xFF);
        int alpha1 = ((color1 >> 24) & 0xFF);
        int red2 = (int)(red1 + (RED - red1) * factor);
        int green2 = (int)(green1 + (GREEN - green1) * factor);
        int blue2 = (int)(blue1 + (BLUE - blue1) * factor);
        int alpha2 = (int)(alpha1 + (ALPHA - alpha1) * factor);
        int color2 = ((alpha2 & 0xFF) << 24) | ((red2 & 0xFF) << 16) | ((green2 & 0xFF) << 8) | ((blue2 & 0xFF) << 0);
        
        pixelData[i] = color2;
    }
}

Protosstribe August 4, 2016, 1:36pm #4

No, then you have to use view frustum culling. All this does, is blend the colors of objects with some blend color (usually the background color), based on their distances to the camera. The further away, the closer their color will be to the blend color.

Edit: You actually could use the ZBuffer to do what you want. The ZBuffer contains the depth value for each pixel. So if you iterate all values in the ZBuffer and find a depth that is further away than you accept, just set that pixel to black.

Protosstribe August 4, 2016, 1:36pm #8

Of course it would be more computationally expensive than using view frustum culling, as you would have to do the rendering anyway. If you’re after speed, then don’t do that. But if you’re just after the visual effect, then it would be possible.

View frustum culling is essentially clipping the polygons to the viewing frustum (pyramid-like shape). My implementation does not have a far-clipping plane (if I remember correctly). But you need that too, in order to discard more unwanted polygons.

However, you could also check the polygons distance to the camera before deciding whether you should include them in the rasterization phase, or not. I did that at some point at least. Not sure if it’s in the code I released.
