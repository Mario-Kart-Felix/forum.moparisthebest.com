Krichevskoy July 29, 2016, 8:35pm #14
frank_:

I have been lead to believe that you need to use saevion’s (or something similar) deobber to finish the job. The source is readable to a certain extent, though obviously you cannot recompile it. Is saev’s deobber “broken” because patterns it searches for are no longer there, or for another reason? If the latter, please elaborate a little bit

Saevion’s program doesn’t work for three reasons that I’ve found:

  1. It can longer find the control variable upon with the ZKM-created opaque predicates are based. This is a result of a (possibly unintentional) interaction between ZKM and whatever program Jagex is using to scramble variable locations. Saevion’s “zkmstripper” transformer depends on its ControlFinder to locate opaque predicates. The particular ControlFinder that saevion created (“CPControlFinder”) just checks the final (constantpool.getSize() - 1) entry in the main class (i.e., client) constant pool. This works fine against Zelix alone because that’s usually where it puts the variable. However, Jagex’s additional scrambling changes that location and CPControlFinder fails.

  2. It doesn’t address all the obfuscation. Even if “zkmstripper” works perfectly, the classes are still hopelessly scrambled. In addition, I’ve heard that it breaks audio support in the client (I usually provide my own music, so I have no idea if it’s true or not).

  3. The code produced does not compile without renaming. I haven’t got a clue what causes this, but if you try to compile the code that JAD gives you without running the deobfuscated client through Zelix (or whatever you rename with), you get compiler-stopping numbers of errors about dereferencing a variable. I don’t see where it’s being dereferenced, so the only thing I can imagine is a name conflict of some sort (which I also haven’t found).
