Yes it would break the “independence” of Java. What Runtime.getRuntime().exec() does can be thought of as opening the shell (the CMD prompt on windows, or Terminal on Linux/Mac) and executing the String.

For example, doing

Runtime.getRuntime().exec("start restart.bat");

would work for a windows machine, but since Linux doesn’t have a “start” command, it would result in an exception being thrown (hopefully). If it was altered so that the OS would know what to do with the command (let’s say, Linux had the command “start”), it still wouldn’t know what to do with the bat file, because linux doesn’t natively support batch files. I mean, you could probably download a program that parses the batch code, and executes the contents correctly (ie. changing “copy” to “cp”). But that also results in a problem since most people here use absolute paths. So the following code wouldn’t execute cross-platform

"C:\Program Files\Java\jre6\bin\java.exe" SomeClass

first of all because linux doesn’t use the letter naming for drives.

If however, you made a link to a file called “java.exe” in the specified directory, it wouldn’t run the program because it doesn’t know how to execute exe files. (unless it was built with Mono, in which case, you’d still have to do “mono java.exe”)

In general, it’s not considered a good idea to do things like this.

You could check the operating system of the host computer running the application using System.getProperty(“os.name”)

Runtime.getRuntime().exec("java some.package.stuff.Main");

The above would work on any operating system that has Java (except Windows 7 x64).
EDIT: No it should work on ANY operating system.
