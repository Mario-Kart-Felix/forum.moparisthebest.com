Purpose

To create simple and straightforward framework that people who aren’t geniuses can understand. I hate seeing shit like AsynchronousAbstractSelectorEngineDecoder when it literally could have been named something more practical that would define its purpose better. The only thing worse than that is seeing almost useless over-complicated code that could be written so much simpler. Then you guys wonder why no one wants to use your frameworks? What are you trying to achieve? With rs2scape I can assure you once again that everything is simple and straight to the point. People with a decent amount of java and protocol knowledge should be able to understand this relatively well.

Design

In the simplest terms this is pretty much a multithreaded asynchronous nio framework.

The GameEngine is the “control center” if you will which contains methods to start/shutdown the server and gives access to the IoEventSelector, TaskDispatcher, and the World. It also runs the game logic thread which polls queued i/o events, fires tasks, and then updates entities every tick.

The IoEventSelector (or in other words the reactor) runs on its own dedicated thread and basically queues i/o events for clients (since you cannot handle any game logic on the networking thread or you’ll run into weird state errors and race conditions) that need them (accept, read, write).

The TaskDispatcher runs on the game logic thread and fires previously submitted tasks when the delay is up.

The World handles all in-game entities.

I don’t really have any plans (yet) on how I’m going to do entity updating and all of that ish, when I get an idea of how I’m doing it I’ll update this topic.
