I started to work on my own open source game engine a while ago. Just to learn game engine architecture and design. My ultimate goal would be to make it into a general purpose game engine, but I have focused on FPS games in 3D so far. And I’m sure I will continue that path.

The features of this game engine includes, but are not limited to, the following:

[ul][li]A Java2D based scan-line rasterizer, for 3D rendering.[/li]
[li]A camera that can be controlled in the 3D environment.[/li]
[li]Keyboard- and mouse input.[/li]
[li]A depth-cue rendering option.[/li]
[li]A wire-frame rendering option.[/li]
[li]Groups of polygons can be rotated and moved independently of others.[/li]
[li]A particle system, that currently only supports one type of particle, namely blood splatter.[/li]
[li]Point light sources.[/li]
[li]Shaded textures, created using the point light sources.[/li]
[li]A progress update screen, for loading resources and configuring a game.[/li]
[li]Map loading, currently only via XML-files.[/li]
[li]Toggling of mouse re-centering, useful for FPS games.[/li]
[li]Toggling of paused mode, in which various aspects of the game is not updated, but you can roam around.[/li]
[li]Toggling of a cursor, that is, you can show it or hide it.[/li]
[li]Toggling of FPS rendering.[/li]
[li]Toggling of texture rendering.[/li]
[li]A simple sound manager, with support for guitars, tones and byte arrays of sound data.[/li]
[li]Configurable map boundaries for X and Z, instead of real collision detection, as of yet.[/li][/ul]

A few days ago I GitHub:ed it, so it can be used in competitions, such as Ludum Dare. I did use it in the latest Ludum Dare 29. My game entry is called Surface Invaders and can be found in the links section below.

If anyone has any feedback or suggestions for improvement, features or critisism, please let me know.

Protosstribe August 4, 2016, 1:05pm #8

I usually follow strict rules when I program. For instance, I usually throw subclasses of RuntimeException for all invalid parameter arguments, like null or numbers that are out of range for the current task at hand. I also usually create many short methods that do one thing. That makes it easier to understand what a larger method does. If it is named well. Not to mention you can reuse the code in other methods in the same class.

However, I need as fast code as possible in this project. Having a lot of checks for invalid parameter arguments slows things down. Writing many small methods takes more time and I just want to make progress. And of course, it can also take longer to execute more methods, if they’re not inlined. But HotSpot will usually take well care of that, so that is not of great concern. I have actually experienced the opposite, that having many small methods can improve performance significantly. That has happened a few times in this game engine.

Other than that, I have tried to separate a lot of the different parts of the game engine. But it can always be improved more. I try to keep away from package level dependency cycles (both direct and indirect ones).
