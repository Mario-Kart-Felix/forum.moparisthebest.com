If you hardcode the public key into the client (as is done at the moment) then it‚Äôs not an issue unless people don‚Äôt use the official client. There‚Äôs absolutely no reason for a secretive distribution of a public key.
As suggested in one of my previous posts though, do not hard code the private key into the server source code. Use the method I suggested previously and then the security of the private key boils down to the trust placed in the server administrator.

If people aren‚Äôt responsible enough to use the official client then that goes beyond the scope of our obligation to protect them. You can‚Äôt protect somebody from themself.

EDIT: Here‚Äôs the query to create a view over the players table that will not make the password available. You can select from it just as you would the regular player table, but the results returned will not contain the password field.

CREATE VIEW vw_player_nohashes AS

FROM pk_players;

And here is the stored procedure to perform the user credential check. The hash argument is VARBINARY instead of VARCHAR to enforce case-sensitive comparisons, otherwise strcmp will act in a case-insensitive manner.
Due to the use of the COALESCE function, this function will return 0 instead of NULL when a record does not exist for the provided username, so you don‚Äôt have to worry about checking for NULLs. If it returns 1 then the credentials are valid, otherwise it returns 0 to indicate an incorrect password or non-existent username.

DELIMITER $$

CREATE FUNCTION validate_user_credentials(username VARCHAR(255), hash VARBINARY(255))
RETURNS BOOL
BEGIN
  RETURN COALESCE(
             strcmp(hash, (SELECT pass FROM pk_players WHERE user = username LIMIT 1) ) = 0, 
             0
         );
END $$

Invoke this stored procedure like so:

SELECT validate_user_credentials('username', 'password');

All you need to do now is set up a couple of roles in the database and give them the correct permissions. A game_server role should basically have select on every table, minus pk_players, and the view I‚Äôve just provided a query for.
It should also have execute permission on the stored procedure provided.
