I recommend you pop open a deob and break out the old eye sockets.

Theres many things that java, like c++, cannot change, and that’s the pointers to the base libraries/packages. From this you can then work your way up without ever running the code once. Another great way to get started is to look for static unchanging strings in obfuscated code. Static strings give insight into the engine, such as the “T” exception strings, t of course stands for type, and it’s accomponied by a number which of course is the type number… Type 1, I can’t recall off the top of my head, but type 2 is the unit* update method, and type 3 is the js5 cycle method.

The last and final means of reversing that I utilize is looking at the architecture of the cache files. One thing is certain with computers, they don’t magically whip up stuff, it comes from somewhere; and you can assume the cache is the base for all of that data the game needs to operate.

As far as patterns go, there’s no necessity for patterns, nothing is static with obfuscation… However, the structure has to be there in order to produce the same results, that’s where the ingenious instruction hook that I believe pplsuqbawlz came up with comes into play: It allows you to roll over changing code and find things that don’t change, or merely change in location. Although, the public ones are bad, I’ve seen pplsuqbawlz latest, which uses a static repository to automaticly ignore obfuscater data such as fields, if statements, dummy arguments, and as he says “pointless inane math statements”.

  • I call it unit(forgive me, I worked with d2 before this), it’s mobile in franks_ schema, and character in psb’s.

  1. Get a deob. (But not necessary.) Deompile and look at source code. Also look at the bytecode with either bytecode viewer (JBE) or have JAD annotate bytecode inline as comments.
  2. Make a simple BCEL program that inserts a println for every String, int or whatever you want to examine. I’d go about it this way: have it print the name of the variable and print whatever it contains.
  3. Notice things with the strings. If you see anything funny, go trace that variable, example: where it gets initialized, where it gets changed, how it gets changed and so on.
  4. Once you find stuff, refactor it in the source code in your favourite IDE.
  5. Learn the data structures and class structures.
  6. Only option right now to edit the client is by working with BCEL. There are no working deobbers out there that deob nicely and let you recompile code well. BCEL is great for this stuff anyway.
