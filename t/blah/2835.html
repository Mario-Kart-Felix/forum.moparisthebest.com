BobDole81 July 29, 2016, 8:10pm #4

You have added the “random” filename to a static variable in Builder.java. A static variable’s value is a part of the class, so whenever you call Builder.run(), it uses the same filename

Static variables and methods are useful, but not all the time. What I would do is create a Builder object before your loop in NodeGUI, and add another method in Builder called randomize or something:

import java.util.Random;
import java.io.*;

public class Builder {
	private String randomFile;
	private String randomFile_JAVA;
	private String nodeContents;
	private File nodeFile = new File(randomFile);
	private File nodeFile_JAVA;
	private File nodeFile_CLASS;

	public void run() throws IOException {
		try {
			PrintWriter out = new PrintWriter(new BufferedWriter(new FileWriter(nodeFile_JAVA)));
			out.print(nodeContents);
			out.close();
			if(f.exists()) {
				System.out.println("File " + f + " already exists, exiting script.");
				System.exit(0);
            }
		} catch(IOException e) {}
	}

    public void randomizeFile() {
        randomFile = new String(Utility.randomstring(15, 15));
        randomFile_JAVA = new String(Utility.randomstring(15, 15) + ".java");
        nodeContents = new String("public class " + randomFile + "{public " + randomFile + "(){}}");
        nodeFile_JAVA = new File(randomFile + ".java");
        nodeFile_CLASS = new File(randomFile + ".class");
    }

    public Builder() {
        randomizeFile();
    }
}

Also, I got rid of the parameters since they are now local to the object. So in the loop in NodeGUI, have this instead:

    try {
        myBuilder.randomize();
        myBuilder.run();
    }

I didn’t test it or anything, but that should work; if not, you got the idea of what I was saying

Also, it’s a nicely done application, but I have a couple of suggestions. Again, stay away from static methods and variables. They’re use well like you have in the Utility class, but they’re not always the best option. Also, in your run method in Builder, I would recommend returning a boolean value to tell whether it ran properly or not, instead of just forcing an applicaiton shutdown in the middle of the loop. Just better control flow

BobDole81 July 29, 2016, 8:10pm #6

Right, because java starts the application that way…NodeGUI.main(args). In most small applications like this, main should be one of the only static methods used. I actually didn’t notice that that class used all static methods as well, but I’ll help you get away from that, which will start to take you to true object oriented programming. Also, even though I converted the Builder class to more of an object form, it won’t create any problems when you compile it. Why? Because even in static classes/methods, you can create objects.

Forgive me if you know all of this, I’m just going to try to explain this as simply as possible, since I know when I was learning java I had the same problem. In your application, you have 3 classes; classes are typically the blueprints for objects, which allows for reusability and flexibility of code.

So at the moment, your NodeGUI class is composed entirely of static classes. If you want to simply reform it to get rid of static methods, first remove the static call in each method and instance variable declaration and then dump all of the code in the main method into a constructor method for NodeGUI…so it would be:

    public NodeGUI() {
		try {
        	JButton createnode = new JButton("Create Node");
        	createnode.addActionListener(new ActionListener() {
        	    public void actionPerformed(ActionEvent evt) {
					init();
        	    }
        	});
        	JLabel info = new JLabel("Node creator running - by White Castle");
        	nodesCreated = new JLabel("Created " + created + " nodes.");
        	JPanel controlPanel = new JPanel(new BorderLayout());
        	controlPanel.add(createnode, BorderLayout.EAST);
        	jf = new JFrame("Node Creator");
        	jf.setPreferredSize(new Dimension(300, 100));
        	jf.setLayout(new BorderLayout());
        	jf.add(info);
        	jf.add(nodesCreated, BorderLayout.NORTH);
        	jf.add(controlPanel, BorderLayout.SOUTH);
        	jf.pack();
        	jf.setVisible(true);
		}
		catch(Exception e) {
		}
	}

Then, once the main method is empty, add this line inside it: NodeGUI myNodeGui = new NodeGUI();
Simply put, the NodeGui class can now be used as an object. This object is now local to the main method, so when it ends (when the application ends) the object is destroyed, just like the PrintWriter object in Builder.run.

Once you create an object of the class in the main method, you can make non-static calls to it. For example, by using that code, you could call myNodeGui.init() (but don’t do that, as it’s already called), which would now be a non-static method, just like you would call methods of an i/o object.

I can always explain a bit more specific topics for you as well, but yeah, stuff like this I find makes much more sense with practice/trial and error.
