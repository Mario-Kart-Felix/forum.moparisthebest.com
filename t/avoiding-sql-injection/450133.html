sppt August 3, 2016, 3:00pm #2

The best way to prevent SQL injection is to use prepared statements, do not rely on mysql_real_escape_string and such, since these methods can be quite faulty.

And regarding the question of passing links through POST or GET, it doesn’t really matter, since both can be read by a cunning individual, so you should just make sure there’s nothing private in there

sw0o0sh August 3, 2016, 5:17pm #8

Every input field should have anticipated data, if you don’t defensively program your validation script to only accept the ANTICIPATED data than you leave yourself vulnerable for things such as XSS and Inject.

Example, if you have an e-mail field, you’d simply use a regular expression to validate that it meets the e-mail syntax. A valid e-mail syntax would not be able to inject a database.

/^([*+!.&#$\A6\'\\%\/0-9a-z^_`{}=?~:-]+)@(([0-9a-z-]+\.)+[0-9a-z]{2,4})$/i

If an input box should only contain letters upper lower, numbers, and spaces, regular expressions could again save the day.

/^[a-zA-Z0-9\s]+?$/i

Bottom line is I recommend diving deep into regular expressions if you want to avoid using tacky and inconsistent functions like is_numeric() for validation. Also the mysqli_real_escape_string() is the current recommendation, not mysql_. You can simplify the need of writing out these long functions by utilizing PHP5’s OOP model or simply create a variable function.

$escape = 'mysqli_real_escape_string';
echo $escape($link_id, 'input');
