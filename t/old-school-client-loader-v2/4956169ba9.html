Ruby:

This is literally no different to loading it through your browser.

I don’t believe you know Java or the Applet related API. What makes you think you know that?

Yes, it is different.

They could see that the necessary resources are not being loaded by analysing HTTP access logs.

They could see that the User Agent (or other HTTP request properties) are not similar to most popular web browsers.

They could see that when the client requests for a document to be shown it just crashes because of the RuntimeException. I believe it sends a showDocument request when an error occurs.

They could see that it does not declare the applet to be inactive when it is expected to be inactive (imthenull: Please read the documentation for isActive. It should never always return true.)

An applet is active [b]just before[/b] its start method is called.

Therefore the code should be something like this:

applet.init(); stub.setActive(true); applet.start();

I’ll paste something i found on the internet to explain it a little better than i could,

Using the default package may create namespace collisions. Imagine you're creating a library which contains a [i]MyClass[/i] class. Someone uses your library in his project and also has a MyClass class in his default package. What should the compiler do? Package in Java is actually a namespace which fully identifies your project. So it's important to not use the default package in the real world projects.
