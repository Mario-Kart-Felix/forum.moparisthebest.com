regecks1 July 30, 2016, 7:59am #4

Ignore him, for he is a douchebag.

There are a few things. First and foremost, we have reflection checks.
Search for “java.lang.reflect” among the client classes, and you will come up with some methods that invoke methods, get information about classes/methods/fields, and can send this information to the server. So what ends up happening is, if the client is edited/refactored &c, the server might know about it, depending on how Jagex implements these checks.

We have all heard about how editing the client automatically flags your account
... untrue. There are dozens of edits you can do that go beneath the radar on Jagex's part. They _cannot_ CRC their own classes, apart from checking their .dat cache files, and those checks are done in the loader, not the true client.

There are of course things like method counters and execution timers, but these are only relevant if you severely edit the code.

The extent of the edits, and whether Jagex have sufficient API or resources to check them decide whether you will be found out or not.

Krichevskoy July 30, 2016, 8:01am #9

We know several hard facts about detection, but a lot of what people talk about is just speculation.

Facts:

  • Jagex has the ability, via reflection, to check method/field names and access flags, to invoke methods and send their return values to the server, and to get and set the values of fields. However, they don’t necessarily do all or any of these at a given time.

  • Jagex has placed, in many methods, variables which are referred to as method counters.
    The code looks something like this:

private static int errechnenScheisse() {
  counterAB34++;
  ....
}

In theory, they could use the above mentioned reflection methods to grab the value of a counter and compare it. I haven’t thought about it in a long time, to be perfectly honest - someone who’s more up to date with the client should start logging reflection actions so we know what they actually do.

  • When the client is run as an application (or any time that the game’s own frame isn’t null), the client sends a unique number to the server (like an error code).

  • When a certain byte in the login packet equals one (which it always does now), the client maintains and transmits a log of your last 40 (or something near) mouse clicks in a hash that also contains position delta and timing information. This is not the only information about your game state that the server harvests. In case of an extensive investigation, a Jagex employee could perfectly recreate what you see while playing the game (so humanlike mouse behaviour is important).

Most people’s deobfuscation processes or sloppy edits interfere with the client in some way that is detectable to the above methods. However, there are some things that people only think might contribute to being banned.

Conjectures:

  • Sharing a UID, IP address/hostname, or client instance with a ‘dirty’ account might cause your ‘clean’ accounts to come under scrutiny. The UID-purge in particular is an item of faith among SCAR users.

  • It has been suggested that Jagex has implemented unique identifiers on all items, so that the flow of duped and macroed items/gold can be easily traced and the characters involved with it banned. The common solution to this fear seems to be transfers through multiple characters. Personally, I would consider REAL laundering via market sites.

  • Some people think that Jagex checks whether or not you’ve loaded the game via the RuneScape website by comparing your game IP to recent web logs (or something similar). This is entirely possible- but, as with other guesses about what Jagex does serverside, there’s no proof one way or the other.
