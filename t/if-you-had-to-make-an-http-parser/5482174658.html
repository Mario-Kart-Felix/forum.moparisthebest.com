sinisoul: A scanner is a lexer (a lexical analyzer). I’m not using an AST in that language, as you noticed, but that’s just how it turned out. I could easily do that. That could be done in a graph as well, but it’s probably easier in a list.

Disclaimers:

Have to verify HTTP header size, have to verify content length, have to implement chunked transfer encoding, yes Im aware the body isnt always encoded as US ASCII, just to assure I didn’t fudge up the callback.

Designed to have a low memory footprint, be as fast as possible. Multiline header values come in chunky (multiple callbacks), have yet to make it so that it ignores the LWS and just keeps it to a single WS character.

package com.nyphrex.rs2.http;

import com.google.common.base.Charsets;
import org.junit.Test;
import org.junit.runner.RunWith;
import org.junit.runners.JUnit4;

import java.util.Arrays;

/**
 * Created by sini on 8/26/14.
 */
@RunWith(JUnit4.class)
public final class HttpParserTest {

    @Test
    public void def() {
        HttpParser parser = new HttpParser();
        parser.register(new HttpHandler() {
            @Override
            public void parsedRequestMethod(String method) {
                System.out.println("Method: " + method);
            }

            @Override
            public void parsedUrl(String url) {
                System.out.println("URL: " + url);
            }

            @Override
            public void parsedVersion(int major, int minor) {
                System.out.println("Version: " + major + "." + minor);
            }

            @Override
            public void parsedHeaderField(String field) {
                System.out.println("Header field: " + field);
            }

            @Override
            public void parsedHeaderValue(String value) {
                System.out.println("Header value: " + value);
            }

            @Override
            public void parsedBody(byte[] bytes, int off, int len) {
                System.out.println("Body: Off-" + off + ", Len-" + len + ", Val-" + new String(bytes, off, len, Charsets.US_ASCII));
            }
        });
        System.out.println(parser.parse("GET /happy.html HTTP/1.1\r\nContent-Length: 4\r\n\n0123".getBytes(Charsets.US_ASCII)));
    }
}

Pwnd:

Ah I see. I have seen it used in the official Java library source code. It had a few methods here and there, so I guess I’m more strict about it.

The benefits in my opinion, is that you can easier reuse parts of your code, as each method should preferably do one indivisible operation. What an indivisible operation means in the current context, is pretty much up to the person writing the code. It could theoretically be a set of instructions on the bytecode-level. But I’m not that strict. In my case it could for instance be that I separate the precondition assertions, like checking for valid arguments and throwing Exceptions, from updating some field. Then, if I need to notify Observers of a state change, that is also in a separate method.

Another benefit I have noticed is that a lot of smaller (and less complex) methods can be easier for the JIT-compiler to optimize. Method-inlining will usually not be performed on large and complex methods.
