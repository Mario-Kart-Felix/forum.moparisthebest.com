Client updates also have new frame opcodes; so, while 60 may have been used for the prayer activation frame in the revision you’re dealing with, in the future it may be randomized to any other legal opcode (i think the maximum width is 16-bits).

Jython August 1, 2016, 3:02pm #18

There isn’t a clean way to do this with reflection. Besides that, you have no guarantee reflection will work for these members at all – it relies on assumptions that only the JLS, and not the JVM specification, makes.
With obfuscated code, you can have methods of the same name with identical argument types, while you can’t do this in source code. This is because the compiler faces ambiguity, while at runtime, all invokes point to method reference constants, which internally rely on name-and-type constants – these embody an entiresignature/descriptor, which incorporates return type.

So, while:

class a { void a(int i); boolean a(int i); }

… is illegal in Java, it’s perfectly legal to the JVM:

{ class: "a", { name: "a", descriptor: "(I)V" } } { class: "a", { name: "a", descriptor: "(I)Z" } }

This also applies to fields:

class z { long z; Object z; }

{ class: "z", { name: "z", descriptor: "J" } } { class: "z", { name: "z", descriptor: "Ljava/lang/Object;" } }

Reflection however via:
Field getDeclaredField(String name)
Method getDeclaredMethod(String name, Class<?>… parameterTypes)

… only take half of the descriptor, which is assumed to be unique by the JLS. Jagex’s obfuscator, however, could easily exploit the fact the JVM doesn’t require these restraints, and render your approach useless.

Typically, people modify the client code to add their own interfaces to the client’s code. They might inject methods like “SendID” and “SendVar” which delegate to their obfuscated counterparts, handling argument reordering and dummy parameters. This way you have a consistent layer for your main bot code to interface with the code that isn’t nearly as low-level and difficult to update. This may prove to be unnecessarily low-level in itself, and you may find yourself preferring something akin to Aryan in that it utilizes /action-forcing/.

Internally, the client keeps a queue of game /actions/ to process every game tick. This queue is represented by what in STS is the “OptionId” and “OptionX/Y”, “dfm”, “dfn”, “dga” arrays. Instead of manually constructing the packet yourself in a concurrent manner (which is unsafe, btw – you may be accessing the output buffer at the same time the client is), you instead write to the slightly higher-level action queue and let the client handle the nitty-gritty data encoding. The way Aryan did this was by hijacking the client’s action processing routine, which in STS is named “RunCommand”, to push the desired actions into the queue (which is executing in the client thread and therefore safe). You may find it easier to work with the action queue than the network logic itself.

Following the Aryan mass-ban, however, most ditched this method of botting in favor of something based on synthesizing fake UI mouse events for the client, which is an even higher-level facade. This method attempts the most at being human-like however lacks the ease of implementation that action-forcing exploited. Instead it focuses all its attention on coming up with realistic mouse paths which is a very difficult problem in its own right.
