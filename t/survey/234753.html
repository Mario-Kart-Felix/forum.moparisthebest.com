Taharok, you probably won’t pay any attention to this post as you think I am an idiot. But regardless, I will still post it.

The thing which a lot of people get stuck on is the structure. They know the logic behind it, but not so much how to put it into clean and tidy code.

Structure wise, you have a few simple classes to make for a few categories. You could create a single “sound” class, which handles the playing of all sounds, perhaps based on a String argument as the name of the sound. If you wanted to go above and beyond you could treat it like your own player, which will play your own format. But that is unneeded.

In my opinion, the two hardest parts you will find is the Game AI and the “Scene Graph System”. You have not revealed exactly what sort of game this is, however any new age game requires a complex graph system in order to hold nice renders for your graphics.

With the Game AI, we already know a few things. For example, it will require more than one class. Otherwise it would be very messy. There are many things you need to take into account. You need to allow it to “think” logically. Say you were making a strategy game, you need it to “think” about what to build first or what tanks to make when etc.

But more than that, it needs to have multiple different patterns, else it would act the same way every time. It needs to be able to react to what’s going on around it, so lets say you send your tanks in to attack the enemy base, the enemy (computer AI) will send it’s tanks in to defend it. This is where I think ActionListeners could come into play. If you made a class that handles certain aspects via an ActionListener (ex. When you run out of tanks, the ActionListener kicks in telling you to build more).

Yes, the User-Interface system is going to be entirely made from scratch for the library. I don’t want to go into detail on the benefits of doing this, because it will give away some of the parts of the library I don’t want people to know yet. In this engine, there are only two ways to display something on the screen. These include the UI system, and the Scene Graph system. TeleNubby July 31, 2016, 2:09am #15

An abstract user interface system would mean the user would have to play a fairly large role in the whole scene. When it comes down to a commercial project, especially a game engine, I’m sure the user would like to have that part of it packaged and ready to be used rather than spending time filling in the gaps you left open. Keep it the way the majority of your game engine is already set up, and stick with that concept. I’m sure you can incorporate a basic component system (class) that can be used if the user wishes to create and manage their own components. All in all, keep your current setup (implementation-based), create a component base/structure for others to build upon (so the entire abstract concept may be used just for this part), and create a system that will manage all prepackaged components and ones that could be built by users.

Make sure all components on-screen are being monitored and are repainted when they need to be. Sun set up their UI system poorly, in my opinion, and there a few mishaps that can occur that leave your component only half-drawn or half-showing. I can’t recall any specifics occurences at the moment, but I’m sure there have been a few in the past that I’ve run into.

Plan out your LAF sytem/engine. There are many ways to create custom LAFs for the engine. Two examples include an abstract core, like the way Sun has it set up, and a scriptable system in programs such as Trillian (multi-protocol IM client). I’d prefer something other than an abstract engine, because Sun’s method irritates me. Trillian’s method is just plain scripting. Sure, it’s a lot more resource dependent and costly when speaking on a technical level, but it’s also more user-friendly and would be a great start to a public community for the engine (the makers of Trillian mastered this concept over at their online forum).

Now let’s talk about component interaction, or more specifically so, the dragging of a component. Be sure to allow the ability to drag components, objects, or whatever you like to call them; as this is a big thing especially for games. Sun already supports the dragging of the mouse and sends the coordinates of the new position the mouse has been dragged to anytime dragging occurs to the appropriate listener. You should look into that, as I really do not feel like going into detail on this one.

Enable support for external, but commonly used, file formats/specifications. This includes things such as FLV playback and modifications (so an API built into your API to be used by your API - this goes for all supported formats), WMA, MP3, WMV, AVI, MPEG, BMP, PNG, etc.; but only when you have extra time available in the end. Possibly even networking protocols like FTP, HTTP, PHP, etc? This isn’t needed at all, but could be a big help and a handy extension to a commercial game engine (if you plan on going really big with this one). Ah, I know! Create a plugin system for all this, set up a community, and let your users/members help with this! If you want, you can even make the plugin “language” “universal” (by that I mean your own script-system or anything similar - but only to stay away from Java and allow others to build on your engine as well).

If I can think of anything else I’ll come back. I’m fried right now and need to get back to homework.

Excellent ideas. I like the one especially about monitoring active components within the UI system. The UI and Scene Graph systems both pool directly into the rendering system, so as the rendering system enters its next phase of rendering, the UI and Scene systems do the same. This means I wouldn’t actually need to monitor the components.

About the abstraction, I was envisioning a library closer to AWT’s in that way. There are master component classes which all subclasses must follow. Then each of the subclasses provide color palettes, look and feel information, (if it is a child of an abstract container, then information on held components), and other stuff like borders. Then, of course, information about the corresponding subclass (like a button would store information about its click status, etc).

I don’t know how effective the formats you are suggesting will be, but it led me to some other ideas. The idea of using actual video in an UI system seems very appealing. As for specific video, sound, and image formats that the engine can load, well, that depends on the implementations. And, I don’t know what you meant exactly by the protocols you listed (since they seem more fit to be in the networking section), but that also led me to think of the possibility of having some simple HTML parsing and rendering in specific text boxes as the Swing package as.

The LAF system will be similar to Swing’s as well. Though, your suggestion of scripting also led me to another idea. This engine will eventually, in the long wrong, have various scripting supports (ranging from pre-processed scripts to bytecode based scripts). Scripting works very well with the cache system already built into the IO engine. As for allowing a LAF to be programmed either (or both) in a scripting and normal implementation manner, well then I would need to change a lot now before then, or it will be almost possible to convert. Thanks for the scripting idea.

And finally, for the dragging of components idea. I like this idea, and can’t wait to start experimenting with it. I am going to make it a long-term goal, as it is not going to be easy to do and will take a bit of work to finish.
