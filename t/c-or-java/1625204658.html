On Topic: C++/C are great due to the functionality of the preprocessor… You can make thousands of lines of code, with only a few macro’s.

#define receive(buffer) \ Packet packet = proxy->makePacket(sizeof(buffer)); \ packet.setData(buffer); \ proxy->receive(packet); \ delete packet; \ delete buffer[];

Now instead of having to copy/paste 5 lines of code for the sum of 73 different packets, I can just use receive(varname);

The only real downsides to C/C++ is the lack of stability, portability, and transparent memory management. We can leak at the seems in C/C++ while Java will manage it and prevent such things from ever occurring. For example, scar (coded in delphi, which has the same functionality as C/C++ only it’s written in Pascal syntax, instead of C style coding) scripts can define images, but some never make the call to clear them when they either crash or recall the define method… Thus we just stack memory up until scar has a meltdown and crashes on us.

Java on the other hand is the epitome of OOP. It ONLY lets us manipulate objects as a whole… About the only pointer’s you will ever have in Java are Array’s, however, this is arguable, but as a test you can pass an array to a method, set its value, and return nothing… Then iterate the array to console and the change is PRESENT.

Java is one of the fastest virtual machines out there, able to outperform most others so well that even microprocessors of weak ability can execute Java… (Cell phones, calculators… etc)

2.1 What Java API’s are supported? How complete is the support?
Matt Welsh writes:

Just look in the ‘libjava’ directory of libgcj and see what classes are there. Most GUI stuff isn’t there yet, that’s true, but many of the other classes are easy to add if they don’t yet exist.

I think it’s important to stress that there is a big difference between Java and the many libraries which Java supports. Unfortunately, Sun’s promise of “write once, run everywhere” assumes much more than a JVM: you also need the full set of JDK libraries. Considering that new Java APIs come out every week, it’s going to be impossible to track everything.

To make things worse, you can’t simply run Sun’s JDK classes on any old JVM – they assume that a bunch of native methods are also defined. Since this native method requirement isn’t defined by the JDK specs, you’re effectively constrained to using Sun’s JVMs if you want to use Sun’s JDK libraries. Oh yes – you could also reimplement all of those native methods yourself, and make sure they behave exactly as Sun’s do. Note that they’re undocumented!

2.4 What is the state of AWT support?
Work is in progress to implement AWT and Java2D. We intend to support both GTK and xlib peers written using CNI. Some components are already working atop the xlib peers.
2.5 How about support for Swing?
Once AWT support is working then Swing support can be considered. There is at least one free-software partial implementations of Swing that may be usable.
