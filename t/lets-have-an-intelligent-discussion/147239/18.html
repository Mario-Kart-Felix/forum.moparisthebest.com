I’m sure it would be possible to have a bot where EVERY tile is node and there are edges between the closest 8 (diagonally as well). These edges would only exist if a wall was not in the way and the nodes would only exist if they were not in an object. You can then have “action” edges for opening doors and the need to run as it is likely that you are under attack.

You can then extend this concept by having actions for ladders allowing you to map dungeons and so on.

On top of this you can randomise the path, making it a few squares longer and avoid detect ability. Or pass a integer argument specifying how random it needs to be allowing time critical applications such as pking to move faster or slower ones such as essmining.

Ruler already has an algorithm to do this which he has optimised the hell out of so I wouldn’t say it would be too CPU intensive. The advantages is that you can walk anywhere, you don’t need to give multiple paths and things like pking becomes much faster because instead of your character finding a route through masses of trees you can do it, avoiding the walking back that is so annoying.

? Isn’t that what the game does?

They use a multidimensional array to perform bitwise checks to validate if parts of the tile is traversal open or traversal closed (as well as projectile open / projectile closed). As bash pointed out, doors would have to be treated differently as the game itself treats doors as any other traversal/projectile blocker.

I use a ray scanning approach that, in my opinion, looks quite human. I use data that is shared among all instances that contains global location information. I found this to be very memory friendly. Doors and other path traversable entities are marked and at runtime their state is updated. The system only loads the data as the bot needs it and the data is unloaded as soon as a path is generated and the runtime logic will handle any portals, doors, gates, etc as they come. The tedious portion is mapping out connectors. From what I could see, there were no hard-coded locations that indicate where a path object will teleport/connect to.

As far as I’m concerned, there is no right answer to this. One implementation, in my opinion, is just as good as any other; so long as it handles movement completely. I see it much like updating code; reliability > speed. The only stipulation I put on this is that it works every time from any location to any location. It took me quite a while to be able to successfully traverse from literally any location in the game to literally any other location in the game; handling doors, ladders, portals, etc as they arise. wL will say otherwise ( ), but I happened to think the feat was nontrivial. Getting from one location to another from anywhere on the same level or story is relatively simple. (Note I said relatively). Getting from one level to another level elsewhere and even validating if the location even exists is tricky if the location does not happen to be loaded in the game client.

After successful path finding comes the actual act of traversing it successfully without looking completely like a bot. Any idiot can create a node-to-node map-clicking movement function. It’s a bit trickier to reliably move from point to point along a path without giving away the fact that you are a bot. I know that if I can tell a bot from a human just by watching them move from place to place, anyone can. Getting your bot to move fluidly like a human would is not as cut-and-dry as one would think. One issue people find they have to get over is the movement of the map while the bot is moving. Training the mouse over the exact location is cumbersome, can be cpu intensive, and will increase the odds of detection (abnormal mouse jittering over an exact location).

Good discussion though, much more interesting to talk about this stuff over how fast one’s updater was able to find getMaxHps.
