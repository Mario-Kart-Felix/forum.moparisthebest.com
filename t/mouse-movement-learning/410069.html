Mouse movement learning?

General Programming
mod_taharok August 1, 2016, 2:04pm #1

I came up with a theory yesterday regarding the idea of using prerecorded mouse paths to generate new mouse paths. I record a series of data points representing the movement of a mouse, and then linearly interpolate between the points to fill in the holes. I then compare this data set to a curve (right now, just a line) and determine a deviation value of how much different the dataset is from the curve. The curve has an orientation of using either independent x values, or independent y values (determined based on whether y or x deviates more in the dataset, respectively). I calculate the “progress” along the curve by taking the distance from the start point to the current point, divided by the distance between the endpoints. This creates a percentage of how far along the curve I am. I then map this percentage to the deviation value for each point in the curve.

When I want to generate a new dataset, I can create the same curve between any two datapoints, and then use my progress along that curve to gather a new deviation value from the deviation table. If the percentage cannot be exactly matched, the deviation values for the next percentage lower than the inputted percentage and the next percentage higher than the inputted percentage are interpolated. The new deviation value becomes an estimate in a scaled (or rather, irregular) curve. This means I can create any sort of mouse movement I want, and then produce any sort of point a to point b dataset, no matter how its size or orientation compares to the original mouse movement.

The program I made implements this algorithm along with an idea of randomness. I store a new table for each input mouse movement, then have a means of switching between tables in the runtime of the generation algorithm. Basically, it’s currently setup to randomly pick one of any of the created tables, then generate the line with a 50% chance in the course of the algorithm to randomly switch to any other table, thus combining two entirely separate mouse movement schemas into one.

Another idea to develop this algorithm is I theorized the behavior of generating new datasets would differ greatly if one used a curve other than a line, and interpolated nonlinearly. For example, generating a bezier curve and calculating differentiation based on that, then creating a new bezier curve to grab the deviation values would create a system of far more curved behavior, obviously. One could also interpolate values from the table using cosine or a cubic approach. There is a lot of room for customization in how the algorithm actually determines the table and determines new datasets. I propose these customizations for what some people have called “profiles” of certain behavior. A linear approach is a much more direct means of finding the point, whereas a bezier curve approach would represent people taking their time to move their mouse from point a to point b. One could even create a noisy linear approach for people that are really slow at moving their mouse, and tend to have a ridiculous about of zigzagging in their pathway.

But anyway, here is a picture that hopefully illustrates my algorithm better:

Here you have a red curve with a blue dataset of points, assumed to have been made by a mouse. This particular line progresses faster along the y-axis, thereby making the y values of the line the independent variable, and the x values dependent. This implies x differs far more nonlinearly in the dataset than y does, which obviously is true. So, we progress along the line and compare the x values, acquiring deviations from the line to the dataset (these values are seen on the right). I also calculate the progress along the line, which can be thought of as a percentage complete (these values are seen on the left). This allows me to approximate any dataset based on another line, by simply comparing the progress values to the deviation values in a table.

Moving onto the actual program I wrote. I did it rather poorly, so don’t immediately crucify me. My program does employ my theory in the most crudest and simplest of forms. It does a poor job, overall, but it gets the job done. There are much better solutions than the ones I have, and I hope they can be used by someone else in place of mine. I am simply proposing a theory and hoping it’s unique. My program sometimes messes up on creating the curves (especially when you record really wild ones), and the result curves are usually really noisy (easily fixable with a smooth filter). [li]F2 - switches program mode to generating, where a click starts a path and another click ends it. [li]F3 - switches program mode to randomized generating where, as explained above, it will choose a random dataset to source the new path on, and then have a 50% chance of switching to another one midpath.[/li][/ul]
