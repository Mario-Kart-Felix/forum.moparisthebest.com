#include <string> // for std::string

if you’re dealing with header files, show that you give a fuck, and an include guard:

app.hpp

#ifndef _APP_HPP
#define _APP_HPP

// a bit extra
#pragma once

#include <string>

extern std::string hello();

#endif // _APP_HPP

app.cpp

#include "app.hpp"

using std::string;

string hello()
{
    return "hello";
}

if you are a bad-ass:

std::cout << "hello" << std::endl; // print "hello" and flush the output buffer

other than that there was never a problem in your code, really. great job. sorry for the rant to a beginner about something really small. it does make a difference.
[/quote]how using the whole standard namespace hints at poor organisation i do not know, it’s just preference. so many people use the whole namespace anyway that it’s never going to cause you any problems in using it since almost everyone knows better than to introduce a class named exactly like one in the std namespace and scare the user when he’s used both namespaces and there’s a name resolution conflict. also, i think from anyone who isn’t a Java programmer’s perspective, a wall of used classes at the top is ugly.

btw he shouldn’t need a header for a single file. nothing is ever going to include it (i mean, who the fuck has a main.h anyway) so it’s completely excessive.

Some C++ programmers are purists and always write std::. When using MSVC++ it’s also nicer to look at Intellisense when it only lists members of the standard namespace, not everything that’s in the global one. Also, you should never declare anything with ‘using’ in a header file - not very nice towards those who include it. This means that in header files you’ll have to write std:: everywhere. You could then do ‘using namespace std;’ in the cpp files, but are you really going to make your function definitions different from the prototypes to write just a little less?[/quote]personally i’m a namespace purist (std:: everywhere plZ!1), but i think it’s also common sense to not put a ‘using’ in the header file. as for making the prototypes and the actual implementation different, i’m fairly indifferent to it for as long as it’s clear.
