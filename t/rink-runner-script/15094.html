Arkanor July 29, 2016, 10:17pm #2

I came back briefly to look at something, and I happened upon this thread.

If you want to create SCAR scripts, debugging them yourself prior to posting them for public viewing is a good idea. It’s critical that you understand what your script is doing and how it is doing it - seems to me as if you are simply looking at tutorials and the SCAR manual, taking code snippets, and plugging them in.

This script doesn’t work, and won’t, because you don’t call half your procedures in it, and the procedure ‘hitnotes’ doesn’t actually do anything besides wait for a total of 50 ms.

In the future, try to plan out your scripts to mimic a set of movements/responses. Play the game yourself manually, write down/remember what you have to do, and then build your script to mimic those actions. If you click on certain colors, use the FindColor and ClickMouse procedures. If you move the mouse, use the MoveMouse procedure, etc, etc. Build in debugs such as using the WriteLn command, so you know what the script is doing and so that you can tell when it effs up. Most people can’t code perfectly and never need to revise, so expect to have to fix bugs and rework portions of scripts sometimes.
