mod_taharok August 4, 2016, 7:23am #26

That’s still a rather arbitrary means of measuring efficiency. You can’t determine what underlying operations are happening for each ‘operation’ or even if they are happening in the same order that you are specifying. In computer science, it’s much easier to stress test an algorithm and measure various timings of the algorithm, then compare similar tests for ports of the algorithm in different languages. To determine if a particular algorithm is efficient, we can use Big-O and other notations to specify an approximation to the time or memory complexity of an algorithm. You can roughly model the time complexity using polynomial and algebraic expressions (this is without using Big-O approximation), but it’s still arbitrary since the time to execute the instructions is entirely dependent on factors well beyond the scope of this challenge.

The_Enforcer August 4, 2016, 7:23am #29

It would be better to evaluate the resulting machine code. The optimizer is free to do anything as long the behavior of the code stays intact - eliminating, reordering, transforming operations. The resulting machine code might be totally different from what you expect.

And then there’s cache aware programming. As you start exploring this area you will become very careful with writing your code, because it’s scary when at one point you find out that switching the order of two increment operations speeds up your algorithm by 2.5x.

These days it’s best to evaluate algorithms in real-world situations by performing benchmarks, not by looking at the source code. But of course - looking at the source code gives you some general idea how efficient the algorithm is - but this can’t be used to determine the actual efficiency.
