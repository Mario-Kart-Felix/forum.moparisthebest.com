Before I start, let me say I don’t know jack shit about Java. All I did was downloaded an IDE, fired it up, and set a web page to Google. It was supposed to resemble my C# game engine.

Some things you should know:
-Java conventions are nowhere near perfect. I leaned towards C# on some (Although there’s some things that break convention rules overall).
-Key press doesn’t work worth shit. It’s why I abandoned this project. Like I said, I don’t know anything about Java, so I can’t imagine why stuff happened like it did (Although I have an idea. From what I googled, Java GUIs have their own thread, which means I would have had to time things right, which I didn’t want to do)
-Some things you might find messy. This is because I encountered a series of problems and got too lazy to clean up the experimented code. This includes unneeded imports.
-Some of you are gonna get all pissy cause there are some really small classes, but I had later plans for them (Like a scripting engine processor in the EventProcessor class that never got added) and don’t feel like explaining them here.

So! This is a small game engine I made in about a day. It has regular and animated objects and some input stuff. You’ll find the MapObject class is empty. It was going to be the object for maps that included views and its own objects. It’s pretty fast. At an uncontrolled speed, I got over 100 FPS with 101 animated objects on the screen. [quote=“cows1471, post:9, topic:342239”]sigh yes, i know what it is the soul, but why do you need to use it?[/quote]In Java, all class functions are by default virtual. In other words, by default it allows a subclass to override it. Because of this, the compiler doesn’t force the override annotation upon you, though it is better practice. The annotation makes the compiler perform validation checks on whether that function is actually overriding a superclass’ function or not. This is useful in case you intended to override a function but mistakenly misspelled its name definition, or something similar.

I for one, have the bad habit of never using these annotations since I don’t expect myself to make a mistake while naming the function, but most Java developers will tell you to err on the side of caution and always implement it, as this also adds for code legibility.
