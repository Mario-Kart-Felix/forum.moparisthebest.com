http://www.splitmedialabs.com/media-toolkits/vh-java-media-toolkit

That looks promising. I would not recommend trying to build something yourself. Java SE does not have strong features for something like realtime screen capture.

For starters, the RobotPeer’s getPixels method for natively grabbing the active screen buffer is extremely slow. For a 1280x800 screen, I measured it taking around 50-150ms (granted, I’m trying to remember these numbers from years ago, so I could be a bit off). That comes down to 6-20 frames per second. Clear animation takes about 24 frames per second, or more.

Secondly, you have network latency. As Davidi2 implied, serializing a bufferedimage and feeding it over a stream is highly inefficient, especially at a moderately high frame rate. Even if you pulled the raw bitmap from the image and fed it over a compressed gzip stream, it’s still very inefficient. Now you would have to look into image compression techniques (lossy or unlossy), and of course, the processing requirements of doing that for each captured buffered for each given frame.

All in all, it’s not easy. Unless you plan on writing the native code yourself, I highly recommend using an existing application or, at least, library (if you really insist on using Java for something like this :P)
