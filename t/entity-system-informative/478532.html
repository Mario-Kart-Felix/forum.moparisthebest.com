[anchor=clickme4]IV. Inheritance/Polymorphism explained[/anchor]
Inheritance includes includes Interfaces as well as abstract classes, but we won’t be talking about Interfaces in this particular tutorial.

Just like how every class extends Object, we can make our own subclasses that inherit from parent classes! you should now be beginning to understand what we can do. You should also already know that when a a subclass extends a parent (super) class, all the fields and methods are inherited from the superclass. You can see this example in most sources in your Player -> Client classes! You can declare a method in Player and use it in Client. How? Because Client extends Player! You can also only do

Client player = b[/b] Server.playerHandler.players[index];

because Client is a subclass of Player!

I suck at explaining things and abstraction is pretty complicated. I recommend reading one page which will take no longer than five minutes here.

[anchor=clickme5]V. What is an Entity?[/anchor]
An entity is basically a single representation of a Player or NPC. Meaning an Entity can be an npc, or a player. The benefits of having an Entity system are that we don’t have to write the same code twice, plus having an Entity based combat system is KICK ASS. why? Because combat is exactly the same for players and npcs.

We’re going to look at a shit diagram I made in paint, and that’s exactly how our Entity system is going to look. It’s simple, and trust me It’s not hard to make… at all.

[anchor=clickme6]VI. Making our Entity system[/anchor]
Now we start with the superclass Entity

[code=java]public abstract class Entity {

 // common values that both Player and Npc share here...

 public int absX, absY, heightLevel;

 public int index;

 public Attribute entityAttributes = new Attribute();

 // etc.

}[/code]

Then you make your Player and Npc classes extend Entity:

[code=java]public class Player extends Entity {

 // we don't need to include those values here, Entity already holds them :)

}[/code]

[code=java]public class Npc extends Entity {

 // we don't need to include those values here, Entity already holds them :)

}[/code]

And we’re done! That’s it. The system is successfully implemented!

We can also do:

[code=java]public void doAction(Entity entity) {

}

// doAction(player instance here)
// or
// doAction(npc instance here)[/code]

Meaning one method, that works for both player and npcs, or any other type that extends Entity. It follows the same concept from step one

A field that can be a player or npc…

A list of players and/or npcs…

Endless possibilities! See what you can do with proper programming and design techniques? This makes things easier, not harder.

The index of an entity is the index within the global entities “list” so to speak in which is was registered, you typically do not use index 0,

take this code for example:

register(new Player(...));
register(new Mob(...));
register(new Player(...));

the first registered players index is 1, the first mob registered is also 1, the second player registered is 2, there are separate lists for each entity type.

The ID on the other hand would only specify to something that has a definition, for example, players technically have a definition but they do not have an “ID” so to speak, their definitions include username, password, player privilege, etc and so forth, mobs have an ID to denote their specified definition, each ID has its own unique definition with the same attributes as another, so, for example, I have a mob with the ID of “3456”

when I register that mob

register(new Mob(MobDefinition.getDefinition(3456), ...);

it’s ID is 3456, while its index is 1
