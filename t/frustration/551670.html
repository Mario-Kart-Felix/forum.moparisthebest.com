Lin August 4, 2016, 3:09pm #9

This is completely normal, but it’s best not to scrap everything and rewrite (unless it’s a test project to learn). To give you an idea, I’ve been working on Codename::World for like over a year now and I still tweak and rewrite code I wrote at the very beginning - even the very first classes I wrote I’ll change to make to better. You can always go back and make changes later, even to internal system workings. And you probably will have to as well, due to design changes, additional requirements, optimizations…

Lately I’ve actually been using a whiteboard and a sort of different approach to programming I never used to do. Instead of just writing code the first time to make it as good as possible, I write/draw out how something will work on a whiteboard and then roughly program it in to try and get it working - because it might not, and I would’ve tried optimizing it for no reason. This makes problem solving easier too because you aren’t taking shortcuts or worrying about if one of your optimizations is the cause of the problem. And of course, premature optimization is the root of all evil anyways.

This could also a good case for test driven development. If you write all of the cases you can think of near the beginning, then when all of your tests pass can be a good indicator that you’ve met the acceptance criteria (being the tests you’ve written). Then, when you want to make any changes, you can rely on your previous test to help validate that it works.

I find that this is really common, especially for people learning it on there own. Something that may help is automated testing. It’s not something that is really discussed in communities like this, but it is a means to an end of frustration. Testing helps preserve the longevity of your code, prevent bugs, help explain what the code is capable of, and in some cases also proves correctness (or, in the least, functionality). The first part is the most important of all of them. I find that disliking code is often due to missing oversights later in the project, leading to either major refactoring or scraping parts of the project. Although this cannot be permanently prevented (hence why project management in software is so much different than other fields), testing is a very large help in trying to minimize it.

If you’re working on a team, I also strongly suggest code reviews. They help provide feedback for every line of code you write. It’s hard to write bad code if it’s well-tested, documented, formatted nicely, shown to work, and looked at by two different software engineers. It’s still possible, but much less likely. If you’re not working on a team, feel free to ask a fellow programmer to do a code review for you sometime. It’s extremely helpful.

If you want to know more about either topics, I’ll be happy to share.
