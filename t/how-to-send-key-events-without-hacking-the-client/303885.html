just dispatch the event from the Canvas , example:

public static void pressBackSpace(int sleep) { char c = KeyEvent.VK_BACK_SPACE; int code = (int)c; getCanvas().dispatchEvent(new KeyEvent(getCanvas(), KeyEvent.KEY_PRESSED, System.currentTimeMillis(), 0, code, c, KeyEvent.KEY_LOCATION_STANDARD)); getCanvas().dispatchEvent(new KeyEvent(getCanvas(), KeyEvent.KEY_RELEASED, System.currentTimeMillis() + sleep, 0, code, c, KeyEvent.KEY_LOCATION_STANDARD)); }
^some part of the Keyboard event dispatcher from Armsbot (my bot)

Edit: note: if you don’t know how to get the canvas, you just cast the inner component at 1,1 of the applet:

public Canvas getCanvas(){
return (Canvas)getApplet().getComponent(0).getComponentAt(1,1);
}

Potentially the client could surround its code with timers. I haven’t checked but since I haven’t seen a thread warning us about it I find it unlikely.

It would be very effective though. By moving all of the code that calls system functions you can remove most sources of delay and so benchmark graphics code against core game logic code. A bot could defeat this with a little work by benchmarking sections of code every now and again (to cope with changing system load) then replace the graphics code with waits of the appropriate (random) duration. Of course any direct modifications to the client are inherently risky.
