Graham July 31, 2016, 4:45pm #24

[quote=“the elve age, post:5, topic:341636”]Lol I have been saying this for ever, in every topic talking about event managers.

Swings are easier to use, and they are professionally made (assuming your talking about Graham Event Manager vs Swing Timers)[/quote]

The Swing timer runs stuff in the EDT (Event Dispatch Thread) which is for doing GUI-related tasks (e.g. creating/using forms/buttons/etc) because Swing isn’t thread-safe. Shouldn’t be used for doing your game logic.

There are other options in the Java API which do the same thing:

  • java.util.concurrent.ScheduledExecutorService
  • java.util.Timer

I made the ‘EventManager’ because I didn’t know that those classes exist. Now I tend to use a single-threaded scheduled executor service and run all game logic stuff in that. (In fact, it’s probably better to use one of those than the EventManager - old code always haunts you).

The reason I originally made it was to show that sticking stuff in the process() method is just stupid, for a few reasons:

  • Stuff that doesn’t need to be there is put there.
  • Not very accurate timing (process can actually be called more than once per cycle - it’s actually called once per packet parsed during a cycle, hence why mass clicking speeds things up).
  • You can have multiple instances of each event with the event manager.
  • With the event manager only events that are executing or scheduled to be executed are dealt with. With process() timers you are continually checking a load of crap.
