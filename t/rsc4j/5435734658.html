Rodgerwilco:

Your software should not rely on a 3rd party and that’s the point.

This is categorically false. There isn’t a real non-trivial program in the world that doesn’t rely on some third party library. Know of course that standard libraries are third party libraries in the sense that they’re external to your own program code, so just using something like a string technically involves utilisation of a library.

Rodgerwilco:

[font=Verdana]What happens when the Netty API gets a major update like it did recently, the switch is quite a pain in the ass and you basically have to rewrite everything. A developer who is relying or utilizing an API doesn’t know what they want, they don’t know if it’s going to be outdated or updated. [/font]

[font=verdana, arial, helvetica, sans-serif]There are other ways to mitigate the impact of a library change that don’t involve adding program layers. One such approach is encapsulating the use of the library to only part of your own code. In the case of RSCD, as far as I can recall, that is performed confining networking code to the net package.[/font]
[font=verdana, arial, helvetica, sans-serif]So that painful change you’ve spoken of isn’t actually that bad, or at least it shouldn’t be. Perhaps the real problem is that haphazard development has taken place, and this has led to tight coupling between RSCD code and mina/netty code in more areas than originally planned.[/font][/font]
