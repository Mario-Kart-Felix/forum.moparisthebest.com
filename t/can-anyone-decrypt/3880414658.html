zyle1992 August 1, 2016, 8:29am #29

My whole thing is that no one should ever need to know the password, and you shouldn’t “encrypt” it either. If you hash the password, with a private key/salt and store that, then all you have to do is take the password, hash it, compare it to the stored result. If they are the same, continue doing whatever, if it’s not, stop.

For example, let’s say I had a password of apple123. The Whirlpool hash of that is

d3e649364bd23dfffcd7cea29b5e902f247c43badd25b473f89b62dce5f26428b20027da6ee053ff40f1c37053cea5181c95e0cea39fb7e51746afdfa3e52c3e

Now, there is NO possible way for ANYONE to decrypt this password. If they try for example, Apple123, the hash is

973f0069eb29bb8db148808476f5e5e9b2a2c723d4874f9881b99717e76e73c2b422cca36d594b4f30e9156672699e800e5b0c611780f06d1684939368ccc8b6

As you can see, changing the case of one letter drastically changes the hash. So, if the programmer wanted to make sure that the security of it’s clients is maintained, they would simply store the first hash I showed, and then all login/validation attempts would hash the password that was entered, and if it didn’t match the stored value, it would respond with an invalid login/validation.

So assuming that the person who wrote the program that generated that hash had any sense of security, they would basically do what I said. Another thing they could do is hash it an undisclosed amount of times. So for example, they could hash “apple123”, convert the result to Base64, repeat for n number of times, and store the final value. As long as the parameters(salt, number of iterations, etc) doesn’t change, the result will always remain constant.
