k00dr July 30, 2016, 12:49am #9

In short, private methods and fields can only be accessed in the class itself, they aren’t even visible to other classes, not even subclasses, while protected methods and fields of e.g. class FOO can be accessed by another class in the same package, or by a subclass of FOO.

Demonstration:

public class Foo {

private String message;

public Foo(String message) {
this.message = message;
}

public final String getMessage() {
return getMessage(this);
}

public final void copyMessage(Foo other) {
message = other.message;
}

private final void invisible() {
System.out.println(“I am only visible to myself.”);
}

private final static String getMessage(Foo instance) {
return instance.message;
}

}

final class Bar extends Foo {

public Bar(String message) {
super(message);
}

protected void invisible() {
System.out.println(“I do not override Foo.invisible(), because I do not know Foo.invisible() exists.”);
System.out.println(“That is why I am allowed to have the same method signature and name.”);
System.out.println(“However, as I don’t belong to a package, I am only visible to my subclasses.”);
}

}

A quick note for you guys though, a private member can be accessed by the class as a whole, meaning that any instance of the class can access the private members of another instance of the same class.

So:

public class Number implements Comparable<Number> {
    private int number;

    public int compareTo(Number n) {
        return (number == n.number) ? 0 : ((number > n.number) ? 1 : -1);
    }
}

That’s perfectly legal. I just mentioned that because I realized that quite a while after I started java programming.

Also, @k00dr, even though class Foo is not technically in a package (it’s technically in the default nameless package), protected members can still be accessed by any local class also not in a package. They can’t be from packaged classes because…they can’t access the default nameless package
