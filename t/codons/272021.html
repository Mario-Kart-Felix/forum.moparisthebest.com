This is a small paper that I’m writing on some “updater theory” that I have been working on. There is no code created to support this yet, however I will be working on a simple implementation when I have time. I have been talking to Ace about practical applications for such code.

As we all know, one of the largest issues that we face when developing patterns for the fields we find in our updaters, is the constant change of structure in the client, making finding certain things harder than they should be.

The basic idea is to have a set of classes that extend BCEL classes (ClassGen, MethodGen, etc). These classes divide class/method/etc information into codons. Basically, a codon is an instruction bundle that references something, such as an “if block”. In a better explanation, codons are sets of instructions that represent source code level structures. Naturally, you would still be able to work at an Instruction level, but, searching and differentiating between methods while taking into account it’s source code in addition to it’s bytecode, is invaluable in my opinion.

Right off the bat, you can know that, for example, method A has X amount of for-loops in a specific order, followed by Y amount of if-blocks. Let’s say you’re looking for an array, you go to the third if-block, because you know that those sorts of things MUST remain in a standard, specific order, algorithms etc.

Lets take a minute to talk about the structure of what I’m talking about. First, I’d have a generic Codon class, which would extend InstructionList, and be fairly abstract. Then, I’d have classes such as IfCodon, LoopCodon, ExceptionCodon etc. Also, ideally, id set it up to essentially combine all method invocations, stack movements into codons, for example:

sipush 1234 istore_0Would be translated into a LocalVariableCodon, that held information stating that you’re operating on a local variable.

All the instructions within a codon are accessible, they’re just simpler for traversing the client. And, theoretically, more static from update to update.

That’s about all I have for now, the biggest issue involved is parsing the instructions and generating the codons, then back, it would be a fairly long task. However, having said that, it should be in reach.

As I said, I will work on a prototype when I have time, but I would love to hear your views on the matter.

BAsh July 31, 2016, 6:47am #2

While I don’t call it that I use the ‘codon’ idea on the instruction level.

I can break the flow into a bunch of blocks but currently I keep those blocks in order as a list and dump them to an instruction list sequentially. My issue is creating a tree or “control flow graph” from those blocks. If code was linear it would be quite simple but then there are loops and I’m not sure how to parse them. Its harder than you think due to if statements that tend to get folded. Then there are || and && and else blocks that can jump for no apparent reason. If you find a way around it let me know.

BTW: The best way to go around parsing and choosing the right architecture is to be as lazy as possible. If you don’t know how to parse it into the right codon or you haven’t built the right codon for the job then make a generic one that can accept anything and build your dumping method so that it can dump raw instructions. The only issue is that you might not be able to build the next codon since it relies on the first. In that case you can get a list of unknown codons but its better than dealing with the complexities of a architecture that can handle everything. You need to remember how you are going to be manipulating the tree and if you make it too complex it will make it harder or impossible to catch all the cases.

BAsh July 31, 2016, 6:48am #20

I currently use a sort of source code “line level” codon. Basically whenever the stack goes back to zero (or I can’t figure out how to parse something) a new codon is created.

StaticClass.Class77_method733(II)I 0: P[] N[2] C[] F[ 1] [ISTORE [ISHR [IMUL [IAND [ILOAD, Const[127]], ILOAD], Const[7]]], IF_ICMPGE [ILOAD, Const[2]]] 1: P[0] N[4] C[] [ISTORE [Const[2]], #GOTO] 2: P[0] N[4] C[] F[ 3] [IF_ICMPLE [ILOAD, Const[126]]] 3: P[2] N[] C[] F[ 4] [ISTORE [Const[126]]] 4: P[1,2,3] N[] C[] [IRETURN [IADD [IAND [ILOAD, Const[65408]], ILOAD]]]
P=Previous, N=Next, C=Catch, F=Fall Through, #=Unknown

It is quite effective at stripping out the xor crap and swapping stuff around to get a normalized instruction list.

What I have issues with is the If/else swapping. They invert some of the If statements and then swap around the If and Else blocks.
