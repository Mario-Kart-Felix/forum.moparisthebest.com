Best way to handle/save the large number of player variables?

Games
Runescape
davidi2 August 4, 2016, 2:09pm #1

I’d like to start a discussion on the different ways to save and handle the large number of player variables that come with the territory when you start to develop content for a server. I’m not talking about the variables every server has, like the username, PID, items, equipment, blah blah. I mean the variables that are little nuances that can really add up. Things that make you question whether or not it’s worth it to even save the variable and hinder your content because of that.

Things like:
Item attributes (player specific, like durability, timers for plant pots, etc)
Huge quantities of booleans for various things like tracking open world changes for instancing, counters for things like the coal trucks, etc
Quest tracking (if you do this properly it shouldn’t take more than a few variables per quest, but with a hundred+ quests it still adds up)

Starting off the conversation, there are a few ways to handle this server sided without worrying about how you save them. The first way which is the most common practice is simply creating a large number of variables in a player/client class to keep track of everything. I find it ugly but I guess it is quite simple and easy to use and if done properly is technically the most reliable.

The second way I know of is to create a hashmap of <String, Object> where you simply think of a magic string and use it throughout all of your code when you want to find a variable.

For example, [tt]player.setVariable(“seersCoalTruck”, 15);[/tt]

The idea works, but using magic strings is hard to maintain and bad practice.

A third way very similar to that is to use something like this:

enum PlayerVariable { SEERS_COAL_TRUCK, WATERFALL_QUEST, }
and use a map of <PlayerVariable, Object>
(this is virtually identically to creating a class of static string constants by the way, which accomplishes the same thing and could be easier to understand for new people)

A fourth way would be to create clusters of those previous types thus letting you be more specific with your implementation. This is what a lot of people would do automatically, for example they would have a cluster for item information which lets you easily find it when incrementing timers and such. But you would still need a more ambiguous one for the random variables you might run into.

If you have any other ideas feel free to bring those up too.

Now for saving, let’s cover a few different ways people do that as well.

The first way is simple flat file saving which is probably the easiest to discuss. You go through, add another writeLine (or xml marker, or json whatever etc) for each variable, and you’re done.

The way I really want to discuss is SQL saving, because no matter which way I choose to handle what I talked about above I’m not sure how I’d save it all. I don’t want to create new columns for everything, but I don’t know if just having a table of string | value is good practice.
