So, I’ve been out of programming for a while (doubt anyone who knows me is still here).

Anyhow, I’m taking C++ in my CS-106 course, and it’s got me interested (I’ve done/still do Java also). I’ve been using some emulators, mainly for psx and gba, and started being curious as to how they’re made. It seems relatively complex, so I started looking into something simple to emulate, like chip8 and atari. Here’s what I’ve found out so far:

-ROMs (game files) consist of two-byte hex instructions, in the form 0xABCD, where A is an opcode specific to a certain action, B is a memory address, and CD is a replacement value. Addresses until 0x2000 are reserved for various reasons.
-Emulators work by pretending to be the CPU, GPU, Sound Chip, and Memory of the target system. Code is written to translate opcodes and functions are written for each, then when run, a two-byte instructions are read from the ROM one at a time and executed.
-The graphic and audio output are then produced (normally using DirectX or OpenGL) and refreshed, “emulating” the system’s screen.
-To translate opcodes, you have to get a hold of the documentation for the system.

It seems like a hard push to the boundaries of my programming knowledge at the moment, but it seems like a really interesting project. Having a working emulator you can call your own and play some classic roms on seems interesting. If this goes well and I’m able to create something efficient, I may move up to NES, SNES, or Gameboy…but I’ll take it one step at a time.

So far this is what I see needs to be coded:
Main Class (program startpoint…GUI goes here)
ROM Reader
CPU
Graphic Display
Sound

Tell me your thoughts and correct me where I’m wrong.

ownagesbot July 31, 2016, 12:41pm #3

You need to pick the device you want to emulate before you start work, you need to emulate each piece of hardware.

Not so long ago I started work on a sega master system emulator, it went ok but it never got finished because there were so many Z80 instructions I just didn’t have time.

You need to research the console you’re going to emulate, and then emulate it. When the SMS booted, in read the ROM into ‘frames’, these were paged into the RAM. (So you firstly had to emulate this). Then there was loads of shit in between (I can’t remember it was a while ago), but eventually the CPU booted, you passed the interrupts/io controls/ram instance to the CPU, had the CPU running on a loop, faked different cycle times etc. and all was good. You then need to consider the VDP and audio devices/chips. In reality, the actual process is very simple, especially if you’re emulating something far inferior to the machine you’re running on. Theres alot of nice open source projects out there, its definately worth taking a look at them.
