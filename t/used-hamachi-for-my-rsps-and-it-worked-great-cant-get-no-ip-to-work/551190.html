sk8rdude461 August 4, 2016, 3:04pm #3

uTorrent works for portforwarding by making an exception in the windows firewall, and it typically uses uPnP. Some routers will have uPnP disabled, so if using uTorrent, you need to make sure that it’s not.

Another issue with uTorrent is, it likes to bind to the port you specify… So running it on port 43594 will bind it to 43594, therefore making it impossible for your RSPS to… The solution to that is to set uTorrent to a port that’s not 43594, run the rsps, and after it’s done and loaded to set the port back to 43594.

However, I’d rather not use uTorrent if I didn’t have to. If you portforwarded in the router already, then I’d close out of uTorrent completely. No-IP is only needed if your IP address (Not your local one, your broadcast IP) is dynamic. Most ISPs these days make them static, so you don’t “need” No-IP, but people still use it. Also make sure your router allows anonymous connections (I’ve heard this being the issue on some routers).

To check that your port forwarded, make sure your server is fully running, and then head over to http://canyouseeme.org and enter 43594, then run the test.

If canyouseeme comes back that its working, then other people can connect to you(assuming your server is running), only you cannot.
Reason is NAT redirection which does not allow intermingling between network nodes through an external network source, without your permission.

EX: Client:192.168.1.10 Server: 192.168.1.15 Router: 192.168.1.1 Client --> connect: 192.168.1.15 --> success! Client --> connect: your_no_ip --> failed! [192.168.1.10 -->router(192.168.1.1)-->isp-->no_ip_redirection_service-->isp-->router(192.168.1.1)-->ERR] The router saying no because the start and end node are both in your local network. Your ideal way to connect is through the direct ip (192.168.1.15).

How is this a security threat?
Data integrity is vulnerable, prone to spying/alterations.
[client -> router -> OUTSIDE_YOUR_NETWORK -> router -> Server]
Gives a chance for anyone to read/tamper with the data.

Some router’s give you the ability to disable this feature, which is a mistake IMO.

If you think it is not a big deal, consider the fact that it is your server. Try and think of a reason someone would want the username/password of the owner.
