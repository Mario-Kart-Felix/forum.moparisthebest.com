And lastly, the field “myPlayer”, which is the player you have logged into Runescape as.

Also, how do current bytecode bots work? I’m guessing that they modify the client at run-time to implement hooks (which are really just interfaces defining what methods a class must have in order to function), and when you need information about the surrounding area, such as if and where an NPC is standing you use the hooks that will in turn call methods in the runescape client itself.
Bingo! It’s really just author’s preference whether the client is modified on runtime or modified beforehand and the modified classes substituted for the originals.
Why do you need to use an interface, why can’t you just modify the class to contain a method to return the data of an obfuscated field? Is that just good Object Oriented Programming style, or is there something that I’m missing?
In method injection, implementing interfaces in the classes is almost vital. With an interface, it’s just ((Interface)object).method(). Without injecting interfaces, one way (praps there are more? I don’t feel like thinking on it) would be object.getClass().getMethod(“methodName”,new Class[]{}).invoke(object). See how much easier the implementing is?
What about when they come to a “loading please wait” does the bot have to retrieve information from the surrounding area again? If it does, how does it detect the "loading please wait."
The “loading please wait” stage is detected using the “login index”, which is a horribly name for what is really a symbol of the game state. The “login index” is an integer used to represent the client’s current state, e.g. loading, at the welcome screen, logging in, logged in. Applet clientApplet = new client();

clientApplet.start();

Client client = (Client)clientApplet;
