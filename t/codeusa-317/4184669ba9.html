October 22 -

My focus in the past two days has been -as I stated before my post was ruined because of the splash screen incident- design. The entire idea revolving around the cache editing GUI is not to hard code everything in, but rather develop two libraries that are side by side and minimize the amount of compiled code. Although I do not expect to do everything in Java, the only things I have found useful in Java are the concurrency, networking and swing packages. Although all do their job well, the entire point of what I want to eventually get to is a plugin based system for just about everything where each plugin is treated independent. Although the availability to bridge variables in plugins will be available, I will be suggesting that it shouldn’t be. These features where they are not stand alone, might just be done in Java. I have yet to further map it out.

The final product for building the frame was completed today, although I will need to invest some time into investigating how I will do action listeners.

--[[ buildframe.lua
    	Written by Hadyn Fitzgerald
]]

--Bind the toolkit 
dofile("./bin/scripts/bind/bindtoolkit.lua")

--Create the frame table
local Frame = {}

--Create the constructor function
function Frame.new()

	--Create the local table
	local self = {}
	
	--Private Variables:
	local f = luajava.newInstance("java.awt.Frame")
	local m = dofile("./bin/scripts/app/frame/menu/buildmenubar.lua")
	
	--Attach the menu to the frame
	m:attachTo(f)
	f:setSize(500, 500)
	
	--Get the menu bar for this frame
	function self:getMenuBar() 
		return m
	end
	
	--Set the icon of this frame
	function self:setIcon(icon)
		f:setIconImage(icon)
	end
	
	--Sets this frame visible or not visible
	function self:setVisible(bool)
		f:setVisible(bool)
	end
	
	return self
end

--Create a new frame
frame = Frame.new()

--Load the frame icon
local icon = toolkit:getDefaultToolkit():getImage("./media/icon.png") or nil

--Set the icon for the frame if needed
if icon ~= nil then
	frame:setIcon(icon)
end

frame:getMenuBar():addMenu("File"):addMenu("Sub"):addMenu("Sub"):addMenuItem("Finally")
frame:getMenuBar():addMenu("File"):addMenu("Sub"):addMenu("Su"):addMenuItem("Finally")
frame:getMenuBar():addMenu("Edit"):addMenuItem("Moparscape")

--Set the frame to visible
frame:setVisible(true)

return frame

As you can see from this, I encapsulated a frame and then added other supporting methods to help add or remove features. This is because I need to control the amount of features that a plugin developer has available to them, to help constrict and streamline the design process for creating plugins. In cases like this however, I may add some support to make sure a script such as this does not get called by outside sources unless they have specific permissions to do so. Creating a new frame such as this (minus the adding menu, setting the size, and making it visible) will be solely exclusive to this script just as the main backbone for the application. This is not a script to make new Frames as you could easily do that with the luajava.newInstance(“classname”, …params). This is to help support plugins and make their job a bit easier to add new menus and what not.

frame:getMenuBar():addMenu("File"):addMenu("Sub"):addMenu("Sub"):addMenuItem("Finally")
frame:getMenuBar():addMenu("File"):addMenu("Sub"):addMenu("Su"):addMenuItem("Finally")
frame:getMenuBar():addMenu("Edit"):addMenuItem("Moparscape")

One of the neat features that I am sad to report that I cannot show you, is the idea behind the tree system for menus. Since the idea of having two menus for a screen sounded fairly stupid, I just created a table for each ‘build’ script that had a table with the purpose of storing and checking what items were on a menu or menu bar.

I know development is slow guys, but from here on out as soon as the plugin API is finished thing will start picking up majorly. Thanks for reading this if you did, and have a great Saturday.
