As some people here may or may not know, I’ve been working on a game for over a month now, and I recently decided to improve some code to make it run a lot faster. However, the optimizations needed to be made due to C#'s faults. I’ve found a performance gain of at least 500%, going from a couple seconds for one chunk of generation to a small fraction of a second. Here’s all I did:

-Changed the world block array from a multi-dimensional ([, ,]) array to a jagged array ([][][])
-Reduces method calls by copying the method’s code and putting it where it’s needed

These kinds of things shouldn’t have to be changed to speed up a program. Does Java have these issues? I’m almost considering learning C++ because of how much faster it is in this category of code, from what I read.

0xCAFEBABE August 4, 2016, 6:26am #3

Java HotSpot (that uses JIT etc.) performs method inlining and other optimizations, whenever possible. Microsofts equivalent runtime also contains a JIT compiler, that does the same. But there seems to be quite a few limitations to it. Methods greater than 32 bytes of IL won’t be inlined. Neither will virtual methods, methods with complex control flow (control flow with constructs such as switch and while) or methods that contains exception-handling.

This just in, not understanding how the IL compiler works produces slower code. See: simple stack instructions [][] vs further method overheads [,]. Calling a method always has an overhead regardless of the language, especially if you’re using an interpreted language to write a software renderer? (are you not using gl/dx ?)

I’m not sure what .net runtime you’re targeting but there are caveats to jit inlining in c# which you might not be adhering to and are improved in 3.5, even more so in 4.5 and newer mono. Alternatively you may be subroutin’ing way too much useless stuff you don’t need to.
