BobDole81 July 29, 2016, 8:39pm #4

The Appendable interface is more for appending data to streams, not gui text elements. And unless he was developing his own type of text area independant of the standard libraries, I don’t think he’d need to worry about such an interface.

Anyway, you seem to come up with some really interesting problems White Castle
Basically, it all has to do with how the applet and application versions are run. Applications are run by calling the main method, initializing all static variables first. Therefore, you’re already using an initialzied version of the ChatClient as your main object (you call client.init() in main(), which initializes everything.

However, the applet’s main object is initialized by the browser/appletviewer, and so when init is called, it’s already running as a completely different object than the static object initialized in the first place. Therefore, when you’re calling the static ChatClient object from the ChatDisplay class, you’re calling an entirely different object altogether – fun how these things work, huh?

You can solve this problem very easily (using your current code), by just sticking this code at the top of the init() method:

client = this;

To be honest though, the code design isn’t very logically written and I wouldn’t stick to that solution. Static methods and fields really aren’t always the best option, and can sometimes make things difficult (as in this situation). I would recommend passing the ChatClient object to the ChatDisplay as a parameter instead to simplify things. Also, in swing elements (such a JFrame and JPanel), the default layout manager used is BorderLayout, so you don’t need to restate that. I’m not trying to pick on it at all, but I’m just trying to give you some tips to help improve your coding style

@White Castle: You can use a customized JDialog or even a new JFrame as a popup (but I would stick to the JDialog). Check out this page on creating custom dialogs: http://java.sun.com/docs/books/tutorial/uiswing/components/dialog.html

What are the advantages of JDialog over JFrame? I’ve never used JDialog…is it just basic?[/quote]
Simply put, yes. They are extremely similar classes, but were written for different purposes. JFrame is technically meant for a standalone interface while JDialog is meant for use as a child window. It has fewer features, and therefore is easier to set up, but its not that different from setting up a JFrame for the same purpose. JDialogs are also used by the JOptionPane class to display certain types of dialogs.

It really doesn’t matter too much which you pick, but I would stick to JDialog since it was meant for such a use.
