When people think of NIO they think of faster faster faster, this is not always the case.
One of the big reasons they released the java.nio package is because it basically made it so that sockets were not limited anymore by thread count. stream-based I/O could possibly turn out to be faster in some cases because it doesn’t have to poll, then go off to another thread for reading. NIO’s purpose was to allow programmers to create more scalable servers, etc. Not to make the fastest server in the world.

well, if you still think nio(channel) is always faster than old i/o(stream) you can run some tests yourself and see the results.
