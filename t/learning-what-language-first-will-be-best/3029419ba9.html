lothy July 31, 2016, 11:19am #42

There’s not a chance in the world that I’d recommend a ‘game oriented’ programming book to a beginner; more often than not it will overlook many important aspects of the language, or simply skim over them - and that’s if it even includes a fundamentals section which, if the book is any good, it won’t.
Game development books that try to be everything (jack of all trades) often lack substance in the areas necessary to make the book worthwhile.

For the record, there are actually books geared at new programmers that are written by Sun employees and officially supported by Sun. They bear the Sun logo in holographic form and are, more often than not, worth the effort to acquire (or the money to acquire legitimately).
In my opinion though, the Sun website offers a wealth of knowledge and education for free - do try using that (the tutorial, which has already been linked to in an earlier post) before forking out money if you can.

EDIT: Left out my addendum to the information about Java that was provided by zstars.
Yeah, it’s a simple language in that it lacks features that are, quite frankly, misused more often than they’re used correctly.
Yeah, it’s an interpreted language, and yes there’s a good chance that some java software isn’t quite on par with a comparable native implementation written in C or C++ (however modern hotspot compiler technology has closed the gap, and the performance differences in the majority of applications are inconsequential – only in the most performance critical programs would you need to resort to a native language for fear of Java ‘not cutting it’. Ahead Of Time compilers are also available now, but most of the good ones are commercial).

Now for the good points about java that make it worth your while.
The API is incredibly rich, even in the standard edition. It’s so vast, in fact, that you should be able to keep boilerplate (mundane) code to a minimum in your applications.
Java software can be incredibly secure if written correctly; memory leaks are only ever an issue if you are working with objects that manage their own memory (ie, collections classes such as Stack and Queue implementations). All you have to do is declare variables within the smallest scope possible to ensure that memory management does a proper job; you don’t have to worry about deleting or free()ing pointers, you don’t have to worry about accidentally trying to re-delete or re-free() a pointer that hasn’t had NULL or zero assigned to it.
Bounds checking is built in to the JVM, so you never have to worry about accessing non-existent indexes of an array (a block of memory), or writing beyond the end of that block of memory (overflows).

In short, I don’t think there’s anything inherently easier about Java than, say, C++ (or any given OOP language).
You are ultimately dealing with a paradigm; the language, once you learn the grammar and feature set, becomes a non-issue.

On to projects.
People are probably going to beat me over the head for this, but I always suggest an IRC bot, because you get to deal with so many aspects of the language (and programming/OOP design in general).

Firstly, IRC bots deal with string manipulation. You parse strings using regex, you create strings. The experience that you get with regards to regular expression usage is quite valuable.

Secondly, you deal with socket programming and the client-server model. This isn’t language-specific at all, but it’s something that’s nice to play with and certainly worth getting your head around.

Thirdly, most bots require things such as permissions and users; this means that you have to implement data types for those users and permissions. Will you take a highly granular approach to permissions (attributing permissions without basing them on a user class; for example, Joe Blogs may be able to kick and ban with the bot while someone else may only be able to run games), or will you take a far simpler approach (class-based permissions, where a user with class x can do something, and a user with class y can do all things that class x can do plus all of the things class y can do).
When dealing with structures such as users you then start to deal with the collections API and generic programming. This is another aspect of the language that is well worth experiencing and using.

Another reason to do an IRC bot is that you learn to implement something designed by someone else, based on existing documentation. This is an important skill to have if you ever program professionally as most projects will revolve around you receiving an interface to implement, which will then be expected to just work when combined with whatever a second group is developing.

I’m sure there are more, but I think those reasons alone justify it as a project for the intermediate programmer.
