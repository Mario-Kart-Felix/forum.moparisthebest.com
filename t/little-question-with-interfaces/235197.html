system July 31, 2016, 2:08am #10

An interface can be seen as a specification fow a minimum requirement of methods in a certain class. When you implement an interface in a class, that class must contain all the methods that is contained in the implemented interface. You can of course still add new methods to this class aswell, as a normal class, but to add all methods contained in the interface is a minimum requirement.

Using interfaces would be a good idea when you know what methods you will be using in your project/package, but not the exact implementation. This means, people that uses your interface in a class, can implement these methods anyhow they like, but they need to be there.

Another good thing with interfaces is that you can have them as a container for constants, that is ‚Äúpublic static final‚Äù. So, instead of importing a constant by class or static import, or extending a class that have constants, you now can implement (using ‚Äúimplements‚Äù) it instead.

The last but not least thing about interfaces I will mention, is that you can change the implementing classes without compiling the classes that uses these interface. That is, you only have to compile the implementing classes, unless you change more.

But you are recreating a new Player everytime you are creating this new PacketHandler. You should instead create the Player class when someone starts to play, because that is this users Player-model.
