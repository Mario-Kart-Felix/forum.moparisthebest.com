The textures on rocks, grass, dirt, wood, concrete, plastic, carpet, drywall, glass, marble, paper, steel, asphalt, rubber… sure, you have textures like this that would be impossible to generate from noise:

But this itself is not a texture. It’s a bunch of objects that all have a noisy texture. Regardless, if you read in the original post under “planned technical features”, you’ll note I listed World “models” (aka spots in the terrain that require sub-sampling) - aka small stones, bricks, finer details, etc. I can always have some sort of hardware tessellation and do other GPU tricks to make up for what randomly generated textures can’t do. Also I plan on adding bump map and specular light map generating to elements once I eventually get around to adding deferred rendering, which not only will allow hardware tessellation like I said, but also give the illusion that parts of the elements are sticking out when being lit from different angles. Due to current time constraints and serious additional memory requirements, this is on the back-burner until I figure out a way to optimize vertices, such as instancing coupled with using an index buffer.

Don’t worry, your concerns are as great as mine. I would LOVE to just be able to use pre-made textures, non-blocking point lights, shadow mapping, level of detail rendering, texture blending, and all of these features every modern game has that makes them look great. But by doing so, you’re increasing your limits - hard. Pre-made textures is obvious and won’t work for this kind of game. Non-blocking point lights not only means liquid illumination would be difficult, but making inside structures dark with expensive ambient occlusion and then having point lights either have a limited amount or bleed through walls not only increases the GPU load, but makes the game feel less genuine and free. Shadow mapping is something I’d like to implement but don’t know how it would look when objects already cast natural shadows, and shadows would disappear when a mountain goes out of view and such. Level of detail rendering isn’t an option because the bottleneck isn’t the GPU - it’s the memory required for having to store all of the world data at full resolution because of the light calculations. Implementing level of detail mesh generation just means more CPU work for no reason (the game is memory-bound). Texture blending is something I looked into for days straight, and unfortunately, it’s not going to happen. It’s not possible without SERIOUS limits (like a max of 5-9 terrain textures) and expenses both on the GPU and CPU, and you all know my stance on limits.

So there you go. Several things we’d all love to see, but will likely not happen due to the gameplay features they eliminate and the extra resources necessary to make them happen.
