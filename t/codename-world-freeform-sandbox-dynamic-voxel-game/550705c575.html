EDIT: Here’s a product of the 2D and GUI systems I’ve been working on that will be used for testing (not a part of the final game). The 2D system is my own version of the XNA SpriteBatch class, but it’s extended to allow primitives like rectangles, triangles, and segmented circles. That said, the GUI is built entirely using shapes. I was considering Direct2D but it draws way too many polygons for shapes, so I decided to do it myself. The GUI system is very similar to WinForms (.NET), with a recursive parent-child event-driven system. Mouse input is handled automatically. It’s very powerful, fast, and simple. I’m happy with it.

http://gfycat.com/ImpossibleTestyBoubou

For some details, it operates similar to how iOS and Android scrolling works. It’s snappy, quick, and smooth. Visually, it’s pretty responsive, with various parts lighting up depending on where the mouse is. If unpinned, it shows and hides itself depending on the mouse position, and stays when pinned (done by clicking the left bar). You can either scroll to select an element or click one, which it scrolls to. I like it a lot for just a test feature.

As tyb97 said, work on the game. If your game idea doesn’t work, everything you’re doing now is for nothing.[/quote]Interesting. If you want to see poor optimization go and look at the first few videos. Just because you see voxel engines that have massive view distances doesn’t mean they’re optimized better. They aren’t doing recursive light calculations, which is the number one most expensive thing. If you take that out, everything would be instant and I could implement some sort of level of detail. Of course, there might still be a delay when generating the world but that’s just from generating noise, which also has nothing to do with my engine. Light calculations are extremely optimized, as is mesh generation. The biggest bottleneck is memory read/write speeds and there is only so much cache optimizing you can do. Trust me, I’ve done all the profiling and consistently improved the code and tweaked the design to be better. It always comes down to having to sample the world data. Also worth noting is that a chunk only has its mesh generated and appears visible once every chunk surrounding it is loaded and has its light calculated. So while it may look like the world is loading very slowly, it’s in a sense loading 9x faster sometimes than you see.

There could be ways around the memory bottleneck, but they would require using a substantial amount of additional memory. By using memory pools, cache hits are more likely and there’s very little overhead for allocating chunks. This is all stuff I’ve considered, but I don’t want to spend days reworking the core when it ups one of the other requirements and might not even work.
