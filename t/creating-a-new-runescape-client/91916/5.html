The chance of success is very low, if it is for a grade you might want to choose something easier and safer.
In anyway, to get you an idea, to make a gfx-less but functional client, you would basically need to:

-Reverse Engineer most of the Runescape packet protocol and structure.
-Reverse Engineer the RSA encryption implementation they use for a few but required packets.
-Reverse Engineer the ISAAC random number generator they use for op-code encryption.
-Reverse Engineer the Huffman compression algorithm implementation they use for string compression. (Which I wanna do some day…).

*These algorithms are published, and their implementation should be similar or identical though.

-Program the client to send and respond to packets as needed.

-Many more things which can’t be foreseen, possibly the client will need updates weekly, as the RS protocol is said to change weekly.

In conclusion, though it would be fun to try, and if you got at least part of it right and released it, me and others would be happy, might be unwise to do it as a project for a grade

Krichevskoy July 30, 2016, 8:59am #10

It shouldn’t be that hard to emulate the RuneScape client well enough to access the game. You can rip the protocol classes straight out of the client along with the necessary large primes. The problem is that they do randomly change things. In the past they’ve added random packets and for years they’ve changed the header IDs with each version. I remember some discussions from a while back that they might even be changing payload structure or endianness now. So it’s the interpretation that gets you.

No one does it because there’s no reason to. It would take a horrible amount of work to make a clientless system that undetectably handles every possible game state in the correct manner. And since it’s impossible to play the game normally with a clientless system, the only purpose for having one would be botting (and yet, I would love to see someone write an RSMUD client just to prove me wrong :P). Since avoiding detection would be such a massive (not to mention recurrent) task, people avoid packet bots now. Near-instant detectability is generally a Bad Thing in a bot.

Don’t misunderstand - I can definitely envision a pattern recognition tool to handle the packet updates for you. I just wouldn’t want to write it myself, nor would I want to map the game logic attached to each packet.
I certainly wouldn’t try it for a time-limited project with a grade attached.

zstars July 30, 2016, 9:08am #12

Your teacher won’t be able to start seeing progress easily. Getting started is probably the hardest part. (Well, and maintaining it too). Most bots which do connect to Runescape are modified clients, which have been deobbed and decompiled. In fact there has only been one Runescape packet bot, and that was Autorune. Sadly, its source never went public. In any case, to get started, you will probably have to first break the encryption.

Runescape encrypts the login packet using RSA. I’m not 100% on any of this, as I have not checked myself, but I think that a CRC or other hash of the runescape cache is also sent encrypted, so that the server can check that the cache is unmodified.
Probably, at this step the seed for the ISAAC random number generator is also sent.

Most packets are supposed to not be encrypted. Their op-codes, however, are encrypted using a synchronized ISAAC random number generator. That way, if the server and client share the same seed, the numbers it will generate will be the same. If the bot or a potential proxy-bot isn’t running a generator using the same seed, however, it will be unable to know what the op-code of each packet is, and especially, it will be unable to insert a packet. After breaking the RSA encryption, you will have to deal with this. You can do that by finding how exactly this system works, so that you can get get or write your own ISAAC generator, to emulate the real client’s.

If you manage to do all that, one of the hardest parts will be done. The rest is basically just a lot of work, sniffing packets and reversing the client till you know the protocol and have documented enough different packets so that you can emulate a real client.

Note that, although it’s not really an encryption, strings are also compressed. I’ve been trying to find the algorithm to decompress them myself.

Anyway, if you finally decide to try this I hope you keep us informed, as I am really interested in this myself
If you need some help I will try to answer, my experience with Runescape is very limited but I have worked on packets on games such as Diablo II or Starwars Galaxies.
