lothy July 30, 2016, 11:37pm #2

Looks like you didn’t add the weight field that I suggested in your other topic.

Anyway, with ItemDefinition, you can make it immutable. There’s no reason why any of that information will ever need to change (once the Item.xml file is properly configured), so you can safely remove all of the setter methods from there, along with making all of the fields final.
Other than that, I still recommend adding a weight field - if something reduces the weight of a player then obviously the weight field’s value should be less than 0 (ie, -4 for boots of lightness). Then when they equip the item, it’ll turn out to be playerWeight += itemdef.itemWeight(), which could be something like 20 + -4, which is 16. It’d work fine.

I’ll add a bit more soon. Come onto IRC sometime, it makes discussing this kind of stuff easier.

EDIT: Is this based off the last modified version that I uploaded?

MORE EDITING: I don’t agree with your approach to the ItemDefinitionManager class. What happens if you pass an id of 10000 as an argument to getById(int) ? That’s 10k iterations.
You’d be better off having an array (even if it does need to be quite large). That way it’d be something like this:

public ItemDefinition getById(int id) {
    ItemDefinition def = itemDefinitions[id];
    if (def == null) {
        //Do something here to handle the problem. Either throw an Exception of some sort, or simply inform us that an error has occured via the console.
    }
    return def;
}
