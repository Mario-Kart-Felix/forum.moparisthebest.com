Introduction

Over the past few days I’ve been working on this little project which is probably one of the most powerful frameworks that is in existence for RuneScape privatee servers. I’ve thinked about the design of the core parts of the server and came up with a few unique solutions to help improve performance.

It’s around the same level/same style as rs2d, but without some of the bugs (can’t guarantee it’s bug free yet!!!).

General Overview

The source uses Apache MINA. William has done some testing (thanks!) and it appears that the packet spamming bug is no longer present (due to a few things I’ve changed). At the moment it uses version 2.0, milestone 6. When a more stable version comes out (such as the release candidate or proper version), I’d encourage you all to upgrade however I have noticed no significant problems with using it so far.

There is one unit test (in the test/ folder). Might add more as the source gets more complicated.

Everything with the exception of the ISAAC cipher code has been written from scratch.

Benchmark

I’ve done some testing and it seems the server can handle 2000 players without crashing. However, this is not real players so not accurate. Also, bear in mind the server in fact has very few features and this will have an impact on performance (as in a real server if you had 2000 players they would be performing actions).

For those interested in how the benchmark was setup, the SYI attacker was run on the same PC as the server. The PC is dual core, 2.2ghz. With the way in which the server is setup it would perform better on more cores, as not just the input/output is done in a multi-threaded manner. The throttle filter was disabled. With it enabled, they will connect slowly. Also, there is no limit per IP yet.

While this number of players are online, the server is still useable (logins are slower than normal - as expected - but there isn’t much lag).

Now, here are some details of the exact optimizations that have been put in place:

Caching update blocks

Say you had 100 players in a region, and then a new one logs in. In a traditional server, each 100 players will reconstruct the new player’s appearance update block when they login. This server will cache update blocks for the duration of the cycle, so it only happens once, at the expense of slightly more memory usage.

This should give some improvements when there are a lot of players in an area, especially if the updates are large (like appearance blocks).

Pooling player updating

Player updating is a read only task. This means you can execute it in multiple threads safely. The server creates a thread pool with the same number of cores as the current computer it is running on, and dispatches update tasks to this pool.

To prevent threading issues, the main thread will block until all the updates are complete. Then the main thread may run as usual, processing packets and performing reads and writes safely with little worry of thread-safety.

This can give vast improvements when you have a PC with multiple cores and/or processors. If you look at the screenshot above, which was taken on a dual core PC, the server is using around 80% of the CPU which means it is taking advantage of both cores (otherwise it would be at or below 50%).

There are classes allowing you to do this as well (namely the ParallelTask class), however, there are few remaining cases in which this is possible to do safely.

When the server is idle, it really is idle

A final performance optimization, that has more impact on other processes running on your server, is the fact that unlike rs2d (or rs2d iirc?) it does not ‘tick’ every 30ms or so.

At the core of the server, is one main thread which has a BlockingQueue of pending tasks. It continually takes events off the queue and processes them. When there are no tasks to process, this blocks, allowing other processes to do their stuff.

Different thread for loading and saving players

Players are loaded and saved in a different thread meaning if the disk is busy, the main thread will not be interrupted.

Event management and all the other standard ‘good stability’ stuff

It has an event manager (look into World.getWorld().submit(Event event) that should be more efficient than the others people may have seen in my previous servers. It operates directly on a ScheduledExecutorService (the same executor service that tasks get submitted to) meaning it is thread safe and pretty accurate. process() doesn’t even exist (the equivalent is TickTask but please use an event instead!).
