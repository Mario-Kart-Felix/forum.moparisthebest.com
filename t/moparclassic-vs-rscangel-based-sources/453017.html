CodeForFame August 3, 2016, 10:39pm #10

[quote=“waj, post:8, topic:453017”]How are the commands handled now?

Just curious, will check later when on the computer if no-one sheds any light on how there handled.[/quote]Previously there was a huge mess of if statements. Then, to make it even messier, in the single method, you had to make sure you put it in the right place, to ensure the user had proper permissions.

Now, permissions are in a configuration file, and commands are matched to the proper function.

Let’s look a command to see what else is different…

Before:

static String[] towns = { "varrock", "falador", "draynor", "portsarim", "karamja", "alkharid", "lumbridge", "edgeville", "castle" }; static Point[] townLocations = { Point.location(122, 509), Point.location(304, 542), Point.location(214, 632), Point.location(269, 643), Point.location(370, 685), Point.location(89, 693), Point.location(120, 648), Point.location(217, 449), Point.location(270, 352) };

if (cmd.equalsIgnoreCase("town")) { try { String town = args[0]; if (town != null) { for (int i = 0; i < towns.length; i++) if (town.equalsIgnoreCase(towns[i])) { player.teleport(townLocations[i].getX(), townLocations[i].getY(), false); return; } } } catch (Exception e) { e.printStackTrace(); } }
And that needed to be placed between the check for the player being a pmod and being a mod.

Now:

val towns = List[(String, Point)]( ("varrock", Point.location(122, 509)), ("falador", Point.location(304, 542)), ("draynor", Point.location(214, 632)), ("portsarim", Point.location(269, 643)), ("karamja", Point.location(370, 685)), ("alkharid", Point.location(89, 693)), ("lumbridge", Point.location(120, 648)), ("edgeville", Point.location(217, 449)), ("castle", Point.location(270, 352)) )

<!-- Teleports the player to the specified town --> <permission name="town">1</permission>

def town(p : Player, args : Array[String]) { val town = args(0); CH.towns.find(e => e._1 equalsIgnoreCase town) match { case Some(x) => p.teleport(x._2.getX, x._2.getY, false) case None => } }

Now, let’s look at what has been improved or is different…

Lines of code wise, the new version is smaller (if you properly format the Java code), even if you include the XML that is required to set the permissions level.
Where you put the method doesn’t matter anymore.
We don’t have to explicitly loop through all the towns; we simply say, find me a town that matches.
We don’t have to worry about ordering of towns and their locations, along with making sure there is a matching pair.
We don’t have to worry about catching exceptions, but we can if we want to.
We don’t have to have guards making sure things aren’t null.
We can easily change who has access to the command, and we don’t even need to re-compile or change any code.
Our mapping of towns to locations is now immutable.
We don’t have to make sure we jump out of the method (calling return at the end) to make sure other commands won’t be executed and additional conditions won’t be checked.

Because of these changes, the Scala version is much more readable, even if you don’t know how to program it isn’t hard to figure out what’s going on. The same really can’t be said for the Java version.
