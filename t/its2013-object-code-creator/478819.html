CodeForFame August 4, 2016, 2:16am #18

Making an editor when things are stored using MySQL is going to be much more work than using JSON. Your editor will need to be able to read/write to a MySQL database, and have access to it. You won’t be able to edit anything unless you can connect to it. That would be a huge pain in the ass.

Using JSON, you can use the existing classes for the objects (they’re pretty annoying to make yourself) and edit them whenever or wherever. Online or offline.

All the gameserver needs to know is the information contained in the XML/JSON files, and then when it needs to save/load it should query the loginserver. It should know nothing about MySQL being used in the loginserver, so saying it’s already being used is completely irrelevant.

Also, the data we’re talking about is essentially unchanging. It loads it once and doesn’t change it. It’s simply loading, not saving.

“Needlessly separating them into inferior storage mediums instead of keeping everything on one system doesn’t make sense.” They’re already separated… Making the gameserver depend on MySQL, which it shouldn’t even know about, doesn’t make sense. There is absolutely no benefit of using MySQL over JSON.

Is MySQL faster? Nope.
Does it lower overhead? Nope.
Is it easier to use with a web front-end? Nope.
Is it easier to write an editor for and would it be better? Hell no.
Is it easier to switch to? Hell no.

So really, why use MySQL over JSON?

“By my workings out a long time ago, It took 350ms to load itemAffectedTypes 2000 times using xstream And a total of 1.5 seconds to load it 2000 times using JDBC library.”
That is extremely surprising to me… Was the MySQL server on the same machine or half-way across the world? Did you remember to divide the time by 2000? Did you remember to start each from scratch or clear their cache, because they both do caching? Was thee MySQL shared or under heavy load? Can you post your benchmark code?

You can still make an editor export SQL statements instead of directly connecting to the MySQL server if you wanted to. Even if the server doesn’t change it, who said you can’t use MySQL for data that isn’t changed through one medium? It could still be changed often through another.
