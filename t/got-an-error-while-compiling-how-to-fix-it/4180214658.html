Lower-casing it because it most commands that are ever used (especially in linux) are lowercase.

Splitting it because it breaks the command into a group of usable parameters. For example take this command

Now, the traditional way of handling something like this is to substring everything. There is a much nicer way to do it however.

The “::” I believe is omitted when the client sends the command packet, so we don’t have to worry about it. That leaves us with the string

The split(String) method takes a string and splits it into an array of different strings (as the name suggests). The string argument is the delimiter that we want to split. What we have now is 3 distinct Strings

item 995 100

Now since it is stored in a String[] we now know what each piece of the command represents, due to the commands contract, or syntax

The first part (arg[0]) will always be the command name. All the data after that can be manipulated as we wish.

@dzeko: The first error is due to me not knowing your particular server’s API. You will of course need to replace getName() with whatever field represents the player’s name. (often playerName or username).

Or (for some OOP points) you could create the getName() method and have it return either playerName or username.

The second error is, again, due to me not knowing your server’s API. As I said in my previous post, your method (getPlayerID) returns an index in the players[] for the particular player you are trying to reference, whereas my method returns the actual Player instance. You could simplify your code to

final Client recipient = (Client) PlayerHandler.getPlayerByName(name);

And it would almost work. The reason I say almost is because you will have to rename the getPlayerID() method in PlayerHandler to getPlayerByName because that more correctly describes what the method is doing.
