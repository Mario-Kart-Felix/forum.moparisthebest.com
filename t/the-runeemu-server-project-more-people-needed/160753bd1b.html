Indeed, Java is big endian. It is however not C++ or Windows what is little endian, but the PC x86 architecture itself. Reversing the byte order is however a trivial task. Also, it would still need to be done in Java because Runescape uses several different non standard byte orders and encodings, possibly for obfuscation purposes.

C++ streams are not intended for that use, so that might be the reason for your troubles. We are using some custom ByteBuffer classes, which are in a way similar to Java byte streams. If anyone wants to have a look at them, they can be found in the SVN repository, in the Shared project.

zstars July 31, 2016, 6:17am #655

Not exactly. There isn’t much automagic at this respect. Data is serialized over the network, and in the end it’s all just a bunch of signals. If a 4 bytes integer is serialized, sent over the network, and received by a different computer as the byte stream DE AD CA FE, without additional information you actually have no way of knowing the integer’s real value. It all depends on how the integer was serialized. Could be DEADCAFE, FECAADDE, or something completely different. Runescape especifically uses several different byte orders and encodings, probably to make packet sniffing / forging harder, which means that often it would be neither of the above.
