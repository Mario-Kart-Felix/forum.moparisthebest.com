zstars July 30, 2016, 10:23pm #444

Graham is right, our “client-side” feature set is nothing especial yet

However, note that the project was actually started on November 25, as the date of my first post shows, which is less than 6 months ago, not 1 year.

Also, we have a framework which is considerably stronger and more efficient than that of an average server:

-We have an asynchronous networking system. (As opposed to the less efficient, standard networking).

-We have a great events system. (Taken from / Inspired in “Game Coding Complete, 2nd edition”, intended to resolve, among many others, these problems that the average server solves with a large switch).

-We have a proper task scheduler, somewhat similar to the events system. (As opposed to the thousand-inefficient-timers approach to handle timed events).

-We use the right ammount of threads. (At the moment, we use three threads, enough to benefit from multi-core processors, but not enough to delay it all with excessive thread-switching, which is the case of these servers that create one thread per client).

And more.

Also, we have some interesting server-side features already in, such as support for an scripting system. As of now, many of the classes are already usable from python scripts, and python scripts are (or can be) already called on certain events.

Yes, the event system is, in general, queued The system also provides a trigger() method for instant execution, but it can be somewhat less flexible so we don’t usually use it. Basically, there is a single EventManager class (as of now) and several EventListeners and Event classes. The system also supports repetition of events and priorities, plus an optional timeout for the ExecuteQueuedEvents method. If you are interested in the exact workings of the system, you can download very similar source from http://www.mcshaffry.com/GameCode

Sorry, we are not teachers. Best way to learn C++ is from a book. Learning a programming language, requires effort and real willingness, and this is especially true in the case of C++, which is large and somewhat complex. We can recommend good books though, both for newbie and more advanced programmers, and answer questions about RuneEmu itself, or advanced C++ features, or C++ design, or basically any other interesting topic.

On any case, anyone who is seriously interested in joining, should contact us at #RuneEmu. Knowing C++ is not strictly required, but, for practical purposes, it is, as of now (until we have more features, the python scripts we need are limited, and are somewhat specialized, so they require knowledge about the structure of the RuneEmu core anyway), somewhat required that you are willing to learn it.
