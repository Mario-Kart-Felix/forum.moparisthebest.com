Ashi:

Mina also manages most of the pipeline stuff for you.

I don’t know where you have got this misconception that the way the pipeline works in Netty is different to how things work in MINA… IoFilterChain in MINA is (very nearly) completely analogous to ChannelPipeline in Netty. The only difference is that MINA enforces that you have a single IoHandler at the end of the pipeline (with the rest of it being composed of IoFilters), whereas Netty does not have a distinction between IoHandler and IoFilter - their functionality is rolled into a single ChannelHandler interface.

Ashi:

Opinions not valid unless you can actually back up that the pipeline is worth something. I’m pretty sure Mina simplifies what Netty over complicated.

The pipeline allows you to separate decoding packets from the logic that handles them. You can also have several different decoders in your pipeline for different types of packet, or decode a single packets in a sequence of several stages. It leads to cleaner code, and you can re-use code by mixing and matching existing decoders/encoders.

For example, if you wanted to read textual data from the client line by line, you can simply add the built-in DelimiterBasedFrameDecoder to the pipeline first, configured to split the incoming data by a newline character (\n), and then add the built-in StringDecoder, which will decode the array of bytes into a String (with an encoding such as UTF-8). Finally, your custom handler at the end of the pipeline would end up dealing with String objects, thus separating the handling logic from the decoding/encoding logic.

Of course, MINA has the exact same support for this kind of design in its IoFilterChain class, as I discussed above.

Ashi:

I don’t think Netty has session attributes either.

Netty 3 supports attaching a single object to a ChannelHandlerContext.

Netty 4 and above have got support for multiple attributes per channel again (in a rather nice type-safe manner) - see http://netty.io/4.0/api/io/netty/util/AttributeMap.html (which Channel, ChannelHandlerContext, etc. implement).

It’s generally very rare to need to use them anyway - most people use Netty in a manner such that you need separate instances of your codecs for each channel anyway (in fact for some classes e.g. FrameDecoder (now known as ByteToMessageDecoder in Netty 4+) this is enforced), in which case you can just use fields in your codec class to store state.

Ashi:

Mina also has a much better design

The author of both Netty and MINA (Trustin Lee) seems to imply Netty addresses some design issues in MINA (and thus has a better design) in this StackOverflow answer:
“Some of those features were integrated into the core too tightly to be removed even if they are not needed by a user. In Netty, I tried to address such design issues while retaining the known strengths of MINA.”
(http://stackoverflow.com/a/2000128/15755)

From having used both, I definitely prefer Netty.
