lothy August 1, 2016, 11:52am #2

That practice is known as encapsulation, or data hiding.

With that approach, note that nothing outside of that class can modify the value. On the other hand, anything would be able to change the value of a public variable.

Even if you are going to have an equivalent ‘set’ method though for a variable, it’s still good practice to encapsulate with the set/get methods for access, rather than just public variables.

moparisthebest August 1, 2016, 11:52am #5

Ok, your class is (using pseudo-code here):

Bob {
public HashMap myField;
}

You set it wherever you like inside your class, all is well and good, now I create a class:

Tom {
Bob bob = getABobInstanceFromSomewhere();
bob.myField = null; // or whatever the hell I want
}

Your code might rely on myField never being null, which you ensure via a constructor or methods, now your code randomly crashes. Or maybe you decide to change myField to a HashTable, or an ArrayList or something that suites you better, now my class won’t compile because you changed the type. If you had getters I could still get the value of myField, but I couldn’t change it, and if you had setters, I could try to set the value of your field, but you could do checks to make sure I never set it to null, or if you decided to change the type of myField internally, then you could leave the setters and getters the same and my class would never have to know you changed, and would still work as expected.

Making variables declared final public is alright though, since the compiler assures that those values can never be changed for you. Otherwise, you should make them private or protected.

public int X

public int Y
{

            set

                {
                    y = value;
                }
            }
}

private int y;

This basically means that even with properties you probably end up writing almost the same amount of code as with setters/getters. And to illustrate why you shouldn’t let others access a field of a class directly I created an example from the C++ world - you probably can’t get away with something like this in Java, but there are similar errors.

#include <iostream>

class TheClass
{
public:
	TheClass(void) : x(0) { };
	int x;
};

int main(void)
{
	TheClass tc;
	
	if (tc.x = 1)
	{
		std::cout << "Oops, wanted to check if tc.x == 1." << std::endl;
	}

	std::cin.sync();
	std::cin.get();
	return 0;
}

In this case tc.x gets the value 1. C++ is backward compatible with C where every nonzero value is true, so this is allowed. Mistakes of this kind can be avoided by using encapsulation.

Protosstribe August 1, 2016, 11:54am #14

Actually, “protected” means it is public to child classes and classes within the same package.

There’s another big difference, though. (This is the case, only if the API is published and used by a lot of programmers (perhaps even around the world). No client wants their programs broken because of backwards-compatibility issues.) All “protected” members are part of the published API and may not be removed (the same as with “public”). However, “private” and “package-private” may be changed any way shape or form, as long as the contract of the API is followed.
