Welcome, to the one and only development topic for Lex! What is Lex, you ask? Well, Lex is the very first 355 server, and soon-to-be the first far-developed C# server you’ve seen. Currently it is being worked on by Jayden and myself and our main goal is to change servers altogether.

You see, a problem with the servers today is people edit code using Notepad. In-existent studies have shown that those who use an IDE have a hypothetical improvement. Sure, one can download Netbeans or Eclipse, but they are even more confusing than setting environment variables. Lord knows we’ve had enough of that…

More commercial/professional and irrelevant text to pull you in…

Okay, onto serious stuff. Lex is pretty much done as far as ground basics go, which include all forms of updating and walking. What we want to do is get Lex to a point where someone can actually customize their server, which means it will have combat, items, etc… working fine. Then, thanks to you Blakeman#### (Sorry, I’m too lazy to look up your name), we’re probably going to go even further and make an old, old server. You know the kind I’m talking about: Varrock Bank as a starting point, the training room being the safe area with the shops, and the wilderness everywhere else.

In terms of stability, it’s very stable. In terms of speed on the other hand… well, it’s fairly decent. I wouldn’t call it great because I’m not good at networking, but it’s a lot better than the servers out there today (~350MS player update cycle in a region of 500 players with packet processing still right-on… on my computer. Pentium 4 processor @ 2.4GHz and 2GB RAM. It was on localhost though, so these results cannot be too accurate). Here’s how the system works:

A connection is received -> The connection is queued until the previous connection, if there is any, has been fully processed. It then is handled to be logged in -> If successful, the server searches for an open thread to put the player on (More on threading below) -> On each cycle (not updating cycle), packets and other actions for the players in that thread are processed. ->> The update thread interrupts whatever is going on and updates the players. Everything does remain in sync so there isn’t anything to worry about there.

The threading system is rather simple. There are objects I call PlayerThreads, which are threads that hold and process a certain amount of players. This is faster than one thread for all players and way more efficient than one thread for all players. Currently we’re still playing around with how many players get put on each thread, but the above statistics were tested with 20 players per thread. If you’re decent at networking and have experience, I’d like a recommended number.

Current Server Features
-Full player updating
-Player movement (Thanks to Hyperion’s code. :))
-Full NPC updating
-Basic items
-Client is somewhat refactored (It is a 355, remember)

Also, since the last post I discovered why packet processing was getting to be so difficult with many, many players in one region. The way I threaded Lex does actually cause problems, but I avoided them by disabling packet processing when the updating is going on. This means that if the update cycle reaches 600ms packets won’t ever be processed. I’m not sure how I’m going to avoid this, because processing packets while the updating cycle is going causes problems (Like mismatched updating packet sizes, thus crashing the client).

What I think I’ll do is make it so incoming data is always accepted and parsed right away, but if the updating is going on the data is queued. If the average update cycle is >= 600ms, I’ll make it so the queued data gets handled by the updating thread. This way data will always get handled.

In Lex news… Because of packets possibly building up and taking several seconds to process due to a long updating cycle, I’ve made some adjustments.

The first one is the server will decrease the time between player processing cycles depending on the updating time. This is because the player processing cycle will only take place if the update cycle is not taking place to keep everything in sync.

Second, because this can still hold up packets, I’ve implemented a feature where if either “EnsurePacketSafety” is on (Which means this feature will always happen regardless of the update cycle time. It’s optional) or the update cycle is taking a long time, the earliest packet caught in the middle of the update procedure will be processed by the update thread. This means no matter what, atleast one packet will be processed during the update cycle, thus breaking up large buildups and further lag.
