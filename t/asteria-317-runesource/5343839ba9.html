More fixes and updates to the task factory

I wanted to do a little cutscene just for fun before releasing this and started thinking about how to design the code, as the last cutscene I did was very messy. So I did a little thinking and spit out some code, and the ScheduledChainFactory was born! Chain factories are chain of runnable tasks that can be executed in an orderly fashion. Tasks can be appended to the internal queue to later be ran in the order that they were appended. Chain factories are a one-shot operation meaning that once ran, they cannot be re-ran. These can prove to be useful when programming operations that require an extensive amount of tasks executed in succession (eg; cutscenes, random events, holiday events, etc.).

As you know, this task system is not cycle based, but runs on multiple threads and workers are dispatched asynchronously by ScheduledExecutorService. This proved to be a problem for me however, because when using ScheduledExecutorService you cannot dynamically change the delay which proved to be hassle when sketching out my ideas for how I wanted combat to be designed (and can cause problems for lots of other content). I solved this problem by making another implementation of the ScheduledWorker, and the result was: the DynamicScheduledWorker. This is basically your average cycle based task, that uses the worker from the parent (ScheduledWorker) class, to process it.

[code=java] @Override
public void task() {
this.cyclesPassed++;

    if (this.isRunning() && this.getCyclesPassed() == this.getCycles()) {
        this.cycle();
        this.setCyclesPassed(0);
    }
}

[/code]

this allows for dynamic (runtime) changes to the delay, rather than having to stop the task and start a new one with a different delay
