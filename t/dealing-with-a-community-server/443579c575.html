Alright, since David mentioned the source I created earlier Ill help paint a light as to what features it has, what issues are currently problematic, and what updates it needs.

Features it has

When I was searching for a source, I was originally planning on using Apollo but I concluded that it was a bit too bloated with unneeded features and not exactly in my eye for the style of what I wanted to use. So I played around with ScapeEmulator, but realized it needed a ton of work.

So, first things first I added in the plugin system which was inspired by Apollo. It uses JRuby/JSON, and its fairly straight forward. The plugin loader loops through the plugin folder and in each child folder looks for a ‚Äòplugin.json‚Äô file. This file holds all the needed information about the plugin like which scripts to load and what dependencies the plugin has.

I added in dependencies because I have plugins like ‚Äúdoors‚Äù which then will be used by city plugins to setup all the doors in a city without cluttering a single file. Dependencies are loaded before the plugin script (duh).

Heres an example:

{
	"scripts" : [ "doors.rb" ],
	"dependencies" : []
}

Plugins use common bindings for player inputs like item on item, clicking an item option, clicking a button. These bindings are called through the bootstrape class through the runeemu module.

Here is the current bootstrap:

require 'java'

java_import 'net.scapeemulator.game.button.ButtonHandler'
java_import 'net.scapeemulator.game.model.ExtendedOption'
java_import 'net.scapeemulator.game.item.ItemHandler'
java_import 'net.scapeemulator.game.command.CommandHandler'
java_import 'net.scapeemulator.game.item.ItemOnItemHandlerAdapter'
java_import 'net.scapeemulator.game.item.ItemOnObjectHandler'
java_import 'net.scapeemulator.game.object.ObjectHandler'
java_import 'net.scapeemulator.game.player.PlayerHandler'
java_import 'net.scapeemulator.game.model.World'

module RuneEmulator

	# The constants for each permission type
	ADMINISTRATOR = 2
	MODERATOR     = 1
	PLAYER        = 0

	# The list of NPCs to populate
	NPC_LIST = World::getWorld().getNpcs()

	class Bootstrap
		class << self
			def bind_cmd(name, permission=ADMINISTRATOR, &block)
				$ctx.add_command_handler(ProcCommandHandler.new(name, permission, block))
			end

			def bind_button(*args, &block)

				# By default bind button one
				option = ExtendedOption::ONE

				# If there are more arguments than the parent/child ids set the option
				if args.length == 3
					option = args[2]
				end

				$ctx.add_button_handler(ProcButtonHandler.new(option, args[0], args[1], block))
			end

			def bind_item(option, &block)
				$ctx.add_item_handler(ProcItemHandler.new(option, block))
			end

			def bind_object(option, &block)
				$ctx.add_object_handler(ProcObjectHandler.new(option, block))
			end

			def bind_item_on_object(item, object, &block)
				$ctx.add_item_on_object_handler(ProcItemOnObjectHandler.new(item, object, block))
			end

			def bind_item_on_item(item_one, item_two, &block)
				handler = ProcItemOnItemHandlerOne.new(item_one, item_two, block)

				# If the block has more than 4 arguments assume it is using handler two
				if block.arity > 4
					handler = ProcItemOnItemHandlerTwo.new(item_one, item_two, block)
				end

				$ctx.add_item_on_item_handler(handler)
			end
		end

		class ProcButtonHandler < ButtonHandler
			def initialize(option, parent, child, proc)
				super(option, parent, child)
				@proc = proc
			end

			def handle(player, param)
				@proc.call player, child, param
			end
		end
               
                # Other stuff...

		class ProcPlayerHandler < PlayerHandler
			def initialize(option, proc) 
				super(option)
				@proc = proc
			end

			def handle(player, selected_player)
				@proc.call player, option
			end
		end
	end
end

Second thing I added was item lists. I did a bit of thinking on this one, and finally concluded that there needs to be a local item list for the player and one for the world so that the solution to how the item list would be done wouldn‚Äôt be hacky. The item lists are synchronized with the player via a listener.

Third thing I did was the object list, and also map/landscape loading. The original source had a map/landscape loaded and I refitted it so that I could attach listeners for when the data was parsed. I wanted to make sure that you couldn‚Äôt cheat by clicking an object that doesn‚Äôt exist so the source keeps all the objects recorded on the map in memory for simplicity. It doesn‚Äôt actually consume too much memory.

Fourth thing I did was the pathfinding and traversal map. I looked at the traversal map refactor by super_ and for the most part followed the same layout. The traversal map has methods on if certain directions are traversalable from a tile, which also includes projectiles. The map is populated by data from a map loader listener and object list listener. All the data is loaded on start and consume a good chunk of memory, but since it wasn‚Äôt a huge concern at the moment to fix it I haven‚Äôt made it more memory efficient yet. I used the A* pathfinding class created by graham.

Fifth thing I did was researching how interfaces/interface scripts exactly work. I knew how they worked in 317, but wasn‚Äôt exactly sure for anything ‚Äòmodern‚Äô. I consider modern post 400, where they started using their new game engine. Turns out that there are root interfaces, two to be exact which have components that are replaced and then the way they are rendered is by drawing each root component and rendering the component that replaces that. Kinda like a peg board with a bunch of holes. There were many misconceptions, which was‚Ä¶interesting to say the least. I had to completely rewrite the interface set class that was provided to suite my needs and now there is a listener on each interface component for when the interface is closed or an input is clicked, and they are all enumerated.

Sixth thing I did was adding in some minor content, just some level up messages and making the skill icons flash. I had to add in the state set class so that I can update/manipulate/access configuration values. It actually saves me the ability of making redundant classes/fields for something like a boolean for if the attack skill icon is flashing.

From there it was mostly research and tiny bug fixes. Graham added a client script dumper which was useful for figuring out the correct way in doing some content pieces, like bank tabs. When the scripts in the client make dynamic components (aka child components) you actually have to tell the client which options to display. I don‚Äôt know exactly why though the solution I added feels hacky. The access set class is used to store which permissions are active for a component (such as if you can click a certain option or swap items).

Problematic Issues

The server is a memory hog because I cut corners with the clipping data, it currently uses about 500-600MB of memory. I will fix this in the future, as I said before it wasnt a big priority to fix that issue as I was the only one developing the server.

Needed Updates

There are two major systems off the top of my head that I need to add; dialogues and cutscenes. Im currently independently getting all the data for dialogues currently, and haven‚Äôt started on cutscenes. I haven‚Äôt exactly finished my dialogue class model yet, but I have an idea of how its going to work.

Some core updates that need to happen:

  • Memory fixes

Some content system updates that need to happen:

  • Ability to spawn global projectiles, still graphics
  • Dialogue system/builder
  • Cutscene system/builder
