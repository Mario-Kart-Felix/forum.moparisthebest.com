_Vengeance August 4, 2016, 2:35am #68

hi lothy (;

Needs less talk more do, guys. Now, true to fashion, I’m going to talk about what we should do!

I don’t think there should necessarily be a community server, rather a communal core and API that others can work with and submit content for, in the form of plugins, and in some cases, bug patches.

Release a stable package containing a core that users don’t need to touch, and allow people to submit their own plugins for quests/minigames/etc if you want to monetize it. You can leave the core open source, and design a simple system for downloading and installing the plugins. Since people won’t actually need to touch the core, it won’t matter whether it’s open source or not - this isn’t where you’ll be making most of your money anyway. Make sure the plugin installer includes a system for verification of content, and compatibility with other plugins. Of course, these plugins can either be free or paid, with a small % going to us, and the rest to the plugin devs.

By including a basic plugin package with some standard quests/activities etc, you can quickly have a lot of people running basic servers. When they realise how easy everything is, they’ll get excited and start wanting more content. The way they’ll get this is by either coding it themselves or downloading it from others. When this happens, if they release it, we get money from their downloads, and if they download someone else’s release, we get money from that download as well.

Currently there are about 200 servers on the mscp status page, and I presume there would be a bunch more on other site’s pages. So let’s say after the release of an easy to install and run server, we can have about 150-250 people running our product at a time. Each server could easily be running an average of as many as 50 plugins. Say they’re priced at around $1-$2, that’s anywhere from $7500-$25000 in total. To save us paying out contributors too much, we can just credit them with cheaper/free downloads, and give them about 50% of the profit. Of course, larger content packages would cost more, and mean more money to us. We could also allow for a similar system to the server status page and have certain packages promoted at a cost.

Aside from raking in money, this kind of system has some other benefits, too.

For a start, we end up running a consistently stable server, with a standardised protocol. This has all kinds of obvious benefits, but it also means that real development will be much faster and more practical. Tutorials can be standardised, meaning anyone can learn to code for the server much faster, without all of the confusion of having many different names across different servers for the same methods. Development will obvious be much more organised. Everyone will have fast access to all of the content that has been released, meaning there will be constant forward movement, rather than stagnation and backwards development.

I’m eating the most delicious nachos right now

Such a system also means minimal development on our part. Aside from the core, we’ll only have to release a small amount of content to demonstrate the capabilities of the system and get the community started. Then we can develop the rest of the content we want and release our own servers and plugins for profit!
