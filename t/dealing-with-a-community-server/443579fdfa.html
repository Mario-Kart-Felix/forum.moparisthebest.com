Graham August 4, 2016, 10:39am #147

btw JRuby will allow you to use Ruby naming conventions to call methods which are named with Java naming conventions. E.g.:

if @player.getInventory().freeSlots() < 1
  @player.sendMessage("Your inventory is too full to hold any more logs.")
  stop()
  return
end

can actually be written as:

if @player.inventory.free_slots < 1
  @player.send_message "Your inventory is too full to hold any more logs."
  stop
  return
end

and an example of setting a variable:

@object.setId(@original_id)

this can be written as:

@object.id = @original_id

Also in TreeType/Hatchet you are using attr_accessor, I suspect you intend for these classes to be immutable, in which case you can use attr_reader instead. Similarly in Tree, I haven’t checked but I suspect tree_type and object are probably read-only, so you could use attr_reader there too.

You’ve also got brackets in lots of places where Ruby lets you drop them and it would be convention to leave them off e.g.

def create_tree_types()

could be

def create_tree_types

You can also leave them off on a lot of your method calls too, but that’s probably more debatable than dropping them from zero-arg function definitions.

Convention for blocks is to use do/end for multiple lines, and braces for a single line, so e.g.:

HATCHETS.each { |id, hatchet|
  next if(!best_hatchet.nil? && HATCHETS[id].speed < best_hatchet.speed)
  (best_hatchet = hatchet) if (@player.getInventory.slotOf(id) > -1 && get_wc_lvl() >= hatchet.level)
}

would generally be written like so (also removing brackets you don’t need, using Ruby-style naming, using contains instead of slotOf and using hatchet instead of HATCHETS[id]: )

HATCHETS.each do |id, hatchet|
  next if !best_hatchet.nil? && hatchet.speed < best_hatchet.speed
  best_hatchet = hatchet if @player.inventory.contains(id) && get_wc_level >= hatchet.level
end

Lines 247-270 are pretty bad, is there a way you can do this avoiding you having to repeat code for each tree type?
