The agent class will be loaded by the system class loader (see ClassLoader.getSystemClassLoader). This is the class loader which typically loads the class containing the application main method. It appears you don’t understand the main_file_cache.

First you must understand the main difference between the signed and unsigned clients; i.e. the signed client uses a “loader” which runs trusted, and has various abilities to load native libraries, open sockets, read/write from disk, etc.
The signlink class provides a service to execute actions like the above. If you look you’ll see the signlink in the client archive and the one in the loader archive differ a fair bit. The method providesignlink() is called from the loader to inject it’s privileged signlink to replace the client’s dummy, non-privileged default one.

So what does the unsigned client do, which is run without the loader stub? It runs limited.
For example, it can only use the Java “safe-mode” software graphics engine, while signed code can run using native hw3d hardware engines, like jaggl for OpenGL and jagdx for DirectX, as well as their native sw3d software graphics engine.
Another draw-back as you mentioned is the inability to do disk I/O, meaning it can not utilize the main_file_cache which acts as a local repository for cached resources. When running unsigned the client must submit on-demand requests to the update server far more frequently, which can easily slow the whole process down.

Frell:

Couldn’t Jagex create a launcher that is not Java and use it to check client files before launch? And someone make the server HAVE to verify that you in fact ran and launched the game from the launcher?

But what exactly can the server do to fool-proof verify that all the checks passed? Anything that relies on the client can be falsified and spoofed; that is, the server doesn’t trust the client, the client trusts the server. This is a pretty simple and universal rule for virtually /anything/ modeled in the canonical client-server fashion.
