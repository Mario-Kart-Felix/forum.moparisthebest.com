Wintercanon August 1, 2016, 2:58pm #27

The thread per client approach is not recommended. It’s a better design for when you are serving a single request to a client such as a HTTP server. For a consistent connection I’d advise against it. Sure the DataInputStream and DataOutputStream is useful, but does it have support for the little endian byte order or Runescape’s 2 custom byte orders? Also you will have to implement Runescape’s custom data types: special A, special C and special S.

To others, the IO streams are not blocking, it’s the ServerSocket.accept() method which is blocking. There is a simple workaround for this though.

Lastly instead of using “serverSessionKey = ((long) (java.lang.Math.random() * 99999999D) << 32) + (long) (java.lang.Math.random() * 99999999D);” you could use “serverSessionKey = new SecureRandom().nextLong();”.

Hope my advice helps.
